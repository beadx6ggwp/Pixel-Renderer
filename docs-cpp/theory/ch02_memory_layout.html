打開你的 application.h，你會看到兩行很不一樣的宣告：

ScreenManager screen; // 直接宣告
RenderDevice* device = nullptr; // 指標，之後用 new

然後在 application.cpp 的建構函數裡：

if (running) device = new RenderDevice(&screen);

而在解構函數裡：

Application::~Application() {
    delete device; // 手動釋放
    screen.Close(); // screen 不需要 delete
}

問題來了：為什麼 screen 不需要 delete，但 device 需要？ 它們不都是 Application 的成員嗎？差別在哪裡？

答案藏在一個根本問題裡：這兩個物件住在記憶體的不同地方。 而住在哪裡，決定了它的壽命由誰管理。

本章範圍 — 理解程式執行時的四個記憶體區域（Stack、Heap、Static、Code）， 以及它們如何決定變數的壽命。當你搞清楚「東西住在哪裡」，很多 C++ 的奇怪規則就會瞬間合理。

1 從 Ch01 到執行期

在 Ch01 裡，你看到了 .cpp 如何一步步變成 .exe：預處理、編譯、連結。 但 .exe 只是一個靜靜躺在硬碟上的檔案。當你雙擊執行它，作業系統會做一件關鍵的事： 把這個檔案載入記憶體，並為它分配一塊獨立的地址空間。

你可以把這塊地址空間想像成一棟四層樓的建築。每一層有不同的用途和規則：

這四層樓就是本章的主題。讓我們從最上面的 Stack 開始，一路往下走。

2 Stack：自動管理的快速記憶體

Stack（堆疊）是你最常接觸的記憶體區域，卻可能從來沒意識到它的存在。 每當你在函數裡宣告一個區域變數，它就住在 Stack 上。

LIFO：後進先出

Stack 的運作方式像一疊盤子：最後放上去的，最先被拿走（Last In, First Out）。 每次你呼叫一個函數，系統會在 Stack 頂端推上一個 stack frame（堆疊框架）， 裡面放著這個函數的所有區域變數、參數、和返回地址。 函數結束時，整個 frame 被彈出（pop），裡面的變數全部自動消失。

想像你的渲染器在跑一幀畫面。OnRender() 呼叫了 DrawTriangle()：

void Application::OnRender() {
    Vertex v1 = {100, 50, 0, 0xFF0000};
    Vertex v2 = {200, 250, 0, 0x00FF00};
    Vertex v3 = {50, 200, 0, 0x0000FF};
    device->DrawTriangle(v1, v2, v3, 0xFFFFFF);
}
void RenderDevice::DrawTriangle(const Vertex& v1, const Vertex& v2,
                                const Vertex& v3, uint32_t color) {
    int min_x = (int)(std::min)({(int)v1.x, (int)v2.x, (int)v3.x});
    int max_x = (int)(std::max)({(int)v1.x, (int)v2.x, (int)v3.x});
    int min_y = ...;
    int max_y = ...;
    // ... rasterization loop
}

當這段程式碼在跑的時候，Stack 長這樣：

當 DrawTriangle() 結束 return 時，它的整個 frame（包括 min_x、max_x 那些區域變數） 被 pop 掉。記憶體並沒有真的被「清除」，只是 stack pointer 往回移了。 這也是為什麼 Stack 這麼快：分配和釋放只是移動一個指標，不需要搜尋可用空間。

Stack 的限制：空間有限

Stack 很快，但它有一個致命的限制：大小固定。 在 Windows 上預設通常是 1 MB，某些配置下可以到 8 MB，但就是這麼多。

這意味著你不能在 Stack 上放太大的東西。想像你的渲染器是 800x600 的視窗：

void bad_idea() {
    uint32_t buffer[800 * 600]; // 800*600*4 = 1,920,000 bytes = ~1.83 MB
    // BOOM! Stack overflow -- 超過預設 1 MB
}

這就是為什麼 frame buffer 絕對不能放在 Stack 上。 你的 ScreenManager 用 CreateDIBSection 把 frame buffer 分配在別的地方（我們馬上會講）。

Stack 的三個特性：
1. 自動管理 — 變數在函數結束時自動消失，不需要你手動釋放
2. 速度極快 — 分配只是移動 stack pointer，一個 CPU 指令
3. 空間有限 — 通常 1-8 MB，放不下大型資料結構

3 Heap：手動管理的大片土地

Stack 的限制把我們推向了 Heap（堆積）。Heap 是一大片自由的記憶體空間， 理論上可以用到系統允許的上限（通常是幾 GB）。 但自由是有代價的：你必須自己分配，也必須自己釋放。

new 和 delete

回到你的渲染器。在 application.cpp 裡：

device = new RenderDevice(&screen); // 在 Heap 上建立 RenderDevice 物件

這行程式碼做了兩件事：

  1. 在 Heap 上找到一塊足夠放 RenderDevice 物件的空間，分配它
  2. 在那塊空間上呼叫 RenderDevice 的建構函數，傳入 &screen

new 回傳的是一個指標（地址），告訴你「你的物件住在 Heap 的這個位置」。 這個指標本身（device 這個變數）存在 Application 物件裡，但它指向的 RenderDevice 實體在 Heap 上。

然後在解構函數裡：

delete device; // 呼叫 RenderDevice 解構函數，然後歸還 Heap 空間

如果你忘了 delete，那塊 Heap 記憶體就永遠不會被回收 — 這就是所謂的 memory leak（記憶體洩漏）。 你的程式每跑一次那段程式碼就少一塊可用記憶體，跑久了就會吃光系統資源。

Frame Buffer：Heap 的實際應用

你的 screen_manager.cpp 裡有這段：

void* ptr = NULL;
dib_bitmap = CreateDIBSection(mem_dc, &bi, DIB_RGB_COLORS, &ptr, NULL, 0);
frame_buffer = (unsigned char*)ptr;

CreateDIBSection 是 Windows API，它在內部分配了一塊記憶體來放像素資料。 這塊記憶體的行為跟 Heap 非常類似 — 它是由系統在背後分配的大塊空間， 需要通過 DeleteObject(dib_bitmap) 來釋放（你的 Close() 有做這件事）。

為什麼 frame buffer 要用這種方式？因為 800x600 的 32-bit 畫面需要 800 × 600 × 4 = 1,920,000 bytes，將近 2 MB。 Stack 放不下，只能放在 Heap（或類似的區域）。

注意 Heap 和 Stack 的根本差異：Stack 是整齊的一疊框架，而 Heap 像一塊空地， 你想在哪裡蓋房子就在哪裡蓋。這帶來自由，也帶來麻煩 — 碎片化、分配速度慢（需要搜尋可用空間）、容易忘記歸還。

Stack
  • 自動分配 & 釋放
  • 速度極快（移動指標）
  • 大小固定（~1 MB）
  • LIFO，不能跳著用
Heap
  • 手動 new / delete
  • 速度較慢（搜尋空間）
  • 幾乎無限大
  • 自由分配，容易碎片化

4 Static/Global 與 Code 區

剩下兩個區域你比較少直接打交道，但知道它們存在很重要。

Static / Global 區

全域變數和 static 區域變數住在這裡。它們的壽命是整個程式的執行期 — 程式啟動時分配，程式結束時才釋放。

// 全域變數 -- 住在 Static 區
int g_frame_count = 0;

void countFrames() {
    static int call_count = 0; // static 區域變數，也住在 Static 區
    call_count++; // 函數結束後 call_count 不會消失
    g_frame_count++;
}

注意 static 區域變數和一般區域變數的差異：一般區域變數住在 Stack，函數結束就消失； static 區域變數住在 Static 區，函數結束後值還在，下次呼叫時繼續用。

字串字面值（string literals）也住在這裡：

const wchar_t* title = L"Pixel Renderer"; // "Pixel Renderer" 這串字住在 Static 區

Code (Text) 區

你寫的所有函數 — OnRender()、DrawTriangle()、SetPixel() — 在 Ch01 的編譯過程中被翻譯成機器指令。這些指令就存放在 Code 區。

Code 區通常是唯讀的。CPU 從這裡讀取指令來執行， 但你的程式不能（也不該）修改自己的機器碼。 這也是為什麼函數名稱在 C++ 裡可以被取地址 — 函數本身就住在記憶體裡的一個位置。

區域內容壽命管理者
Stack區域變數、函數參數函數執行期間自動（編譯器）
Heapnew/malloc 分配的物件直到 delete/free你（程式設計師）
Static全域變數、static 變數整個程式執行期自動（系統）
Code機器指令整個程式執行期唯讀（系統）

5 你的渲染器裡的記憶體地圖

理論說完了，讓我們把你的渲染器每個重要物件都標上它住在哪裡。 回到 application.h：

class Application {
protected:
    ScreenManager screen; // 嵌入在 Application 物件裡
    RenderDevice* device = nullptr; // 指標嵌入在 Application 裡，指向 Heap
    bool running = false; // 嵌入在 Application 物件裡
};

關鍵問題：Application 物件本身住在哪裡？

如果你在 main() 裡這樣寫：

int main() {
    MyApp app(800, 600, L"Pixel Renderer"); // app 住在 Stack
    app.Run();
    return 0;
}

那 app 整個物件就住在 Stack 的 main() frame 裡。 而 screen 是 app 的一部分，所以 screen 也住在 Stack。 device 這個指標變數也住在 Stack（因為它嵌在 app 裡），但它指向的 RenderDevice 物件 住在 Heap。

這就是為什麼 screen 不需要 delete 而 device 需要：

  • screen 嵌在 Application 物件裡，Application 消滅時 screen 自動消滅（解構函數被自動呼叫）
  • device 指向 Heap 上的物件，Heap 不會自動清理，你必須手動 delete

讓我們逐一確認每個元件住在哪裡：

物件住在哪裡為什麼
app (MyApp) Stack (main frame) 在 main() 裡直接宣告
screen (ScreenManager) Stack (嵌在 app 裡) 是 Application 的值成員，不是指標
device (指標本身) Stack (嵌在 app 裡) 指標變數是 Application 的成員
*device (RenderDevice 實體) Heap 用 new 分配
frame_buffer 資料 Heap (系統分配) CreateDIBSection 在內部分配
DrawTriangle 的 min_x, w1 等 Stack (函數 frame) 區域變數，函數結束即消失
L"Pixel Renderer" Static 字串字面值在編譯時決定
DrawTriangle() 的機器碼 Code 編譯產生的指令，唯讀

現在回頭看開頭的問題：screen 不需要 delete 因為它嵌在 Application 裡， 跟著 Application 一起自動消滅。device 需要 delete 因為它指向 Heap 上獨立分配的物件， Heap 不會自動清理。

常見陷阱

1. Dangling pointer（懸空指標） — 回傳區域變數的指標。函數結束後那塊 Stack 記憶體已被回收， 指標指向的是垃圾。

2. Stack overflow — 在 Stack 上放太大的陣列（例如 int buf[1000000]）， 或者無限遞迴，都會把 Stack 撐爆。

3. Memory leak — new 了卻忘記 delete。 渲染器如果每幀都 leak 一點，跑幾分鐘就會吃光記憶體。
絕對不能做的事 — 回傳區域變數的地址

int* bad_function() {
    int local_value = 42; // 住在 Stack
    return &local_value; // 回傳 Stack 上的地址
} // local_value 在這裡消失，地址變成垃圾

int* p = bad_function(); // p 現在指向已經被回收的記憶體
printf("%d", *p); // Undefined Behavior -- 可能印出 42，可能 crash，可能任何事

編譯器通常會給你警告（warning: address of local variable returned）， 永遠不要忽略這個警告。如果你需要回傳一個物件，用值回傳（會複製）或用 new 分配在 Heap 上。

三件事帶走

1. Stack 自動管理但空間有限 — 區域變數住在 Stack，函數結束就自動消失。 不需要手動釋放，但不能放太大的東西（frame buffer 就放不下）。

2. Heap 需要手動 new / delete — 大型物件和壽命需要超越函數範圍的物件放在 Heap。 用 new 分配，用 delete 釋放，忘了就是 memory leak。

3. 變數住哪裡決定它的生命週期 — ScreenManager screen 嵌在 Application 裡， 跟著 Application 一起生滅。RenderDevice* device 指向 Heap，必須手動管理。 理解住址就理解壽命。
