PART 1 · CHAPTER 01
編譯模型
從 .cpp 到 .exe 的旅程
compilation linker header

打開你的渲染器專案資料夾，你會看到這些檔案：main.cpp、application.h、application.cpp、screen_manager.h、screen_manager.cpp、render_device.h、render_device.cpp。為什麼不把所有程式碼寫在一個檔案裡就好？為什麼要把 .h 和 .cpp 分開？當你在 VS Code 按下 Build，那短短幾秒鐘裡到底發生了什麼事？

更重要的是，你一定遇過這個令人崩潰的錯誤訊息：undefined reference to `RenderDevice::DrawTriangle(...)`。編譯明明通過了，為什麼最後一步又炸了？要理解這些問題，你需要知道 C++ 的編譯模型 — 從原始碼到可執行檔，中間經歷了哪些階段，每個階段各自負責什麼。

本章範圍 — 建立 C++ 編譯流水線的心智模型。你會學到預處理、編譯、連結三個階段各自做什麼，以及常見錯誤出現在哪個階段。我們會用你的渲染器專案作為貫穿全章的範例。

1 一步到位的幻覺

如果你寫過 Python 或 JavaScript，你習慣的世界是這樣的：寫好一個 .py 檔，在終端打 python main.py，程式就跑了。沒有中間步驟，沒有等待，所見即所得。

帶著這個經驗來到 C++，你很自然會這樣想：「寫好 .cpp，按一個按鈕，直接變成 .exe，結束。」好像整個過程是一個黑箱，一步到位。

但真相是：C++ 的建構過程至少有三個截然不同的階段，每個階段由不同的程式負責，做完全不同的事。我喜歡用一個比喻：

比喻：出版一本書
  • 預處理 = 寫信 — 把散落的草稿拼成完整手稿（純文字處理）
  • 編譯 = 翻譯 — 把中文手稿翻譯成英文，每章獨立翻譯
  • 連結 = 裝訂 — 把翻好的各章裝訂成一本書，解決跨章引用
C++ 實際流程
  • 預處理器：展開 #include、#define，輸出 .i 檔
  • 編譯器：每個 .cpp 獨立翻譯成 .o 目的檔
  • 連結器：把所有 .o 拼成最終的 .exe

接下來我們逐一拆解每個階段，用你的渲染器程式碼當作例子。

2 預處理器：文字替換大師

預處理器是整個流程的第一步，它做的事情出奇地簡單：純文字替換。它不懂 C++ 語法，不知道什麼是類別或函式，它只是一台文字搜尋與取代的機器。

#include = 複製貼上

當預處理器看到 #include "application.h"，它做的事情就是：打開 application.h 這個檔案，把裡面的內容原封不動地貼到 #include 那一行的位置。沒有什麼魔法，就是複製貼上。

看看你的 main.cpp 開頭：

#include "application.h"
#include <stdio.h>

const int WINDOW_WIDTH = 800;
const int WINDOW_HEIGHT = 600;
// ...

預處理器處理 #include "application.h" 時，會打開 application.h，發現裡面又有 #include "screen_manager.h" 和 #include "render_device.h"，於是繼續遞迴展開。最後，原本只有 45 行的 main.cpp 可能膨脹成數千行（因為 screen_manager.h 引入了 <windows.h>，那個檔案本身就有上萬行）。

Header Guards：防止重複貼上

這裡有個問題。你的 application.h 引入了 screen_manager.h，而 render_device.h 也引入了 screen_manager.h。如果預處理器老老實實地兩次把 ScreenManager 類別的定義貼進來，編譯器就會抱怨：「class ScreenManager 被定義了兩次！」

解法就是你在每個 .h 開頭看到的那三行：

#ifndef SCREEN_MANAGER_H // 如果還沒定義過這個符號
#define SCREEN_MANAGER_H // 定義它（佔位用，值不重要）

// ... 整個 header 的內容 ...

#endif // 結束條件編譯

第一次遇到 #include "screen_manager.h"，SCREEN_MANAGER_H 還沒被定義，所以內容正常展開，同時定義了這個符號。第二次遇到時，符號已經存在，#ifndef 判斷為假，整塊內容被跳過。這就是所謂的 header guard（頭檔保護）。

另一種更簡潔的寫法是 #pragma once，效果一樣，但不是 C++ 標準的一部分（雖然所有主流編譯器都支援）：

#pragma once // 告訴預處理器：這個檔案只要引入一次就好
重點 — 預處理器不懂 C++。它做的全部事情就是文字替換：展開 #include、替換 #define 巨集、處理 #ifdef 條件。輸出的 .i 檔仍然是純 C++ 原始碼，只是所有 # 開頭的指令都被解決了。

3 編譯器：翻譯成機器語言

預處理器把文字整理好之後，輪到編譯器上場。編譯器的工作是：把 C++ 原始碼翻譯成機器語言（目的碼，object code），產出 .o（或 Windows 上的 .obj）檔案。

這裡有一個關鍵概念你必須記住：每個 .cpp 是獨立的編譯單元（translation unit）。編譯器一次只看一個 .cpp（準確說是它展開後的 .i），完全不知道其他 .cpp 裡寫了什麼。

宣告 vs 定義

這就是為什麼 .h 和 .cpp 要分開。Header 裡放的是宣告（declaration） — 告訴編譯器「這個東西存在，長這樣」。而 .cpp 裡放的是定義（definition） — 告訴編譯器「這個東西的實際內容在這裡」。

render_device.h (宣告)
class RenderDevice {
public:
  void Clear(uint32_t color);
  void SetPixel(int x, int y, uint32_t color);
  void DrawLine(...);
  void DrawTriangle(...);
};
// 只說「這些函式存在」，沒有實作
render_device.cpp (定義)
#include "render_device.h"

void RenderDevice::Clear(uint32_t c) {
  // 實際清除 frame buffer
  memset(frame_buffer, ...);
}
void RenderDevice::DrawTriangle(...) {
  // 實際畫三角形的程式碼
}

當編譯器在處理 main.cpp 時，它看到 device->DrawTriangle(...) 這行呼叫。它翻開（展開後的）header，確認 DrawTriangle 的簽名正確，然後就安心了：「好，我相信這個函式在某個地方有實作，我先記下這裡需要呼叫它，等會兒連結器會幫我找到。」

編譯器在 main.o 裡留下一個「待填」的記號：「這裡要呼叫 RenderDevice::DrawTriangle，地址待定。」

為什麼要獨立編譯？ — 想像你的渲染器專案越來越大，有幾十個 .cpp。如果你只改了 render_device.cpp，編譯器只需要重新編譯這一個檔案，其他的 .o 可以直接重用。這就是增量編譯的好處，也是 make 工具存在的意義。

4 連結器：拼圖的最後一步

現在我們有了四個 .o 檔案。每個檔案裡都有一些已翻譯好的機器碼，以及一些「待填」的空位。連結器的工作就是：把所有碎片拼在一起，填上所有空位。

具體來說，連結器會建立一張符號表（symbol table），列出每個 .o 檔提供了哪些符號（函式名、全域變數名），以及需要哪些符號。然後它進行配對：

.o 檔案提供的符號需要的符號
main.o main(), MyApp 類別 Application::Application(), Application::Run()
application.o Application::Application(), Application::Run() ScreenManager::Init(), RenderDevice::RenderDevice()
screen_manager.o ScreenManager::Init(), ScreenManager::GetFrameBuffer(), ... Windows API (由系統庫提供)
render_device.o RenderDevice::Clear(), RenderDevice::DrawTriangle(), ... ScreenManager::GetFrameBuffer(), memset()

如果每個「需要」都能在某個 .o 或系統庫中找到對應的「提供」，連結成功，輸出 PixelRenderer.exe。

undefined reference 的真相

現在你終於能理解那個恐怖的錯誤了。假設你在 render_device.h 宣告了一個新函式：

// render_device.h
class RenderDevice {
public:
  void DrawCircle(int cx, int cy, int r, uint32_t color); // 新增！
  // ...
};

然後你在 main.cpp 裡開心地呼叫它：

device->DrawCircle(400, 300, 50, 0xFF0000);

但你忘了在 render_device.cpp 裡寫實作。會發生什麼事？

  • 預處理：完全沒問題。#include 正常展開。
  • 編譯：也沒問題！編譯器看到 header 裡有宣告，簽名吻合，它很滿意，在 main.o 裡留下一個「待填」記號：「這裡要呼叫 RenderDevice::DrawCircle」。
  • 連結：爆炸。連結器翻遍所有 .o 檔，找不到 DrawCircle 的定義。
undefined reference to `RenderDevice::DrawCircle(int, int, int, unsigned int)'
collect2: error: ld returned 1 exit status
記住這個規則 — 如果錯誤訊息出現在「編譯」階段（syntax error, type mismatch），那是你某一個檔案的語法或型別問題。如果錯誤出現在「連結」階段（undefined reference, multiple definition），那是跨檔案的問題 — 宣告和定義沒有對上。

5 你的渲染器的編譯流程

把所有東西串起來，你的渲染器專案從原始碼到可執行檔的完整旅程長這樣：

你的 makefile 正是在控制這個流程。當你執行 make 時，它會：

  1. 對每個 .cpp 執行 g++ -c（-c 表示「只編譯，不連結」），產出對應的 .o
  2. 把所有 .o 加上系統庫（-lgdi32 -luser32）一起餵給連結器，產出 PixelRenderer.exe

如果你只改了 render_device.cpp，make 會聰明地只重新編譯這一個檔案，然後重新連結。這比每次都重新編譯所有檔案快得多。

常見陷阱
  • 循環引入（Circular Includes） — A.h 引入 B.h，B.h 又引入 A.h。預處理器會無限遞迴（實際上 header guard 會阻止它，但結果是其中一方的宣告不完整）。解法：使用前向宣告（forward declaration），例如 class ScreenManager;，只在 .cpp 裡才 #include 完整 header。
  • 忘記 Header Guard — 沒有 #ifndef / #pragma once，同一個類別被定義兩次，編譯器報 redefinition of class。
  • 在 Header 裡定義函式（ODR 違規） — 如果你在 .h 裡寫了非 inline 的函式定義，每個引入這個 header 的 .cpp 都會產生一份該函式的機器碼。連結時連結器看到多份相同的定義，報 multiple definition of ...。這就是 One Definition Rule (ODR) 被違反了。解法：在 header 裡只放宣告，定義放到 .cpp；或者加上 inline 關鍵字。
本章學到的三件事
  1. 預處理是純文字替換 — #include 就是複製貼上，#define 就是搜尋取代。預處理器完全不懂 C++ 語法。
  2. 每個 .cpp 是獨立的編譯單元 — 編譯器一次只處理一個檔案，它靠 header 裡的宣告來「相信」其他檔案會提供實作。這就是 .h / .cpp 分離的原因。
  3. 連結器把 .o 碎片拼成最終程式 — 它負責解決所有跨檔案的符號引用。undefined reference 是連結階段的錯誤，意味著某個宣告找不到對應的定義。
