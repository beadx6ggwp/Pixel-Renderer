1 你現在在哪？

你目前寫的 render_device.cpp 是一個軟體光柵器（software rasterizer）： 完全跑在 CPU 上，你親手控制每一個像素怎麼被畫出來。 Unreal Engine 和 Unity 做的事情在概念上完全相同——只是它們把這些工作交給了 GPU， 並且在你手工完成的每個步驟上，疊加了大量效能最佳化和視覺技術。

2 光柵化的兩條路線：掃描線 vs 包圍盒

你的 DrawTriangle 用的是包圍盒法（Bounding Box + Barycentric Test）： 找出三角形的外接矩形，對矩形內每個像素計算重心座標，判斷是否在三角形內。 這是學術上最簡潔的方法，也是現代 GPU 硬體光柵器的原型。

另一條更古老的路線是掃描線法（Scanline Rasterization）： 把三角形分成梯形，逐行掃描，只計算每行的左右端點。 早期軟體渲染器（Quake、Doom 時代）幾乎都用這個方法， 你的參考資料 mini3d.c 也是這個架構。

mini3d.c 的掃描線架構

mini3d.c 用一個叫 trapezoid_t 的結構來表示梯形（上下兩條水平邊 + 左右兩條斜邊）， 任意三角形最多可以拆成 2 個梯形（以中間頂點的 Y 座標為切割線）：

mini3d.c — 三角形拆梯形（核心概念）
// 三角形按 Y 排序後，分三種情況： // 1. 平頂三角形 → 1 個梯形（直接就是梯形） // 2. 平底三角形 → 1 個梯形 // 3. 一般三角形 → 2 個梯形（以中間頂點 Y 切割） int trapezoid_init_triangle(trapezoid_t *trap, vertex_t *p1, p2, p3) { // 按 Y 座標排序 p1 < p2 < p3 ... if (p1->pos.y == p2->pos.y) { // 平頂 → 1 梯形 trap[0] = { top: p1.y, bottom: p3.y, left:(p1→p3), right:(p2→p3) }; return 1; } if (p2->pos.y == p3->pos.y) { // 平底 → 1 梯形 trap[0] = { top: p1.y, bottom: p2.y, left:(p1→p2), right:(p1→p3) }; return 1; } // 一般 → 2 梯形，以 p2.y 為切割線 trap[0] = { top: p1.y, bottom: p2.y, ... }; trap[1] = { top: p2.y, bottom: p3.y, ... }; return 2; }

兩種方法的比較

方法包圍盒 + 重心座標（你目前用的）掃描線梯形法（mini3d.c）
原理 逐像素測試是否在三角形內 直接算出每行的左右邊界，只畫邊界內的像素
實作複雜度 低，邏輯直接 中，需要梯形拆分 + 邊緣插值
效能（CPU） 包圍盒過大時浪費（測試很多不在三角形內的像素） 精確，沒有無效測試
GPU 並行化 天生適合（每個像素獨立，可並行） 掃描線有相依性，不適合 GPU
現代用途 GPU 硬體光柵器的核心概念 CPU 軟體渲染器、早期遊戲引擎
參考論文 Mileff et al. — Accelerated Half-Space Triangle Rasterization The Setup for Triangle Rasterization
▌ 論文：Accelerated Half-Space Triangle Rasterization（Mileff, Nehez, Dudra）
你的資料夾裡有這篇 paper。它研究的就是包圍盒法的硬體加速—— 把「點是否在三角形內」的判斷用 edge function（有號面積的符號）表示， 讓三個邊函數可以增量計算（從上一個像素加一個差值），避免每次重新算叉積。 這是現代 GPU 光柵器的真實做法。

3 透視正確插值：rhw trick

你目前的 DrawTriangle 在三角形內插值顏色用的是線性插值（屏幕空間）。 這對顏色在許多情況下看起來還好，但對 UV 座標（紋理）卻是錯誤的—— 透視投影之後，屏幕空間的線性插值對應的是 3D 空間中的非線性分布。

問題的根源

透視除法之後，頂點的 x/y 被除以 w（深度）， 但 UV 座標沒有被除——它們還在原始空間。 如果你直接在屏幕空間線性插值 UV，相當於假設 UV 隨屏幕像素均勻分布， 但實際上越近的部分格子越大、越遠的越小（透視變形）。

解決方法：rhw trick（1/w 插值）

mini3d.c 在 vertex_rhw_init() 裡實作了這個：

mini3d.c — 透視正確插值（rhw trick）
void vertex_rhw_init(vertex_t *v) { float rhw = 1.0f / v->pos.w; // rhw = 1/w（Reciprocal Homogeneous W） v->rhw = rhw; v->tc.u *= rhw; // UV 先乘以 1/w 再插值 v->tc.v *= rhw; v->color.r *= rhw; // 顏色同理 v->color.g *= rhw; v->color.b *= rhw; } // 掃描線中，對每個像素取回原始值： float w = 1.0f / scanline->v.rhw; // w = 1 / rhw，取回真正的深度 float u = scanline->v.tc.u * w; // UV 還原：(UV/w_orig) / (1/w_orig) = UV_orig float v = scanline->v.tc.v * w;

數學上的原理是：1/w 在屏幕空間是線性的（可以正確插值）， 而 UV/w 也是線性的——所以把 UV 除以 w 後插值，再乘回 w，就得到透視正確的 UV。 這就是 GLSL 裡為什麼 varying 可以正確插值的原因： 驅動和硬體在底層做了完全相同的事情。

▌ 論文：Perspective Texture Mapping（Paul S. Heckbert, 1989）
你資料夾裡的 Perspective Texture.pdf 和 lowk_persp_interp_techrep.pdf 就是研究這個問題的。 前者推導了透視正確插值的完整數學；後者研究在低成本硬體上如何用近似方法（每隔幾個像素才算一次精確值，中間線性補間）減少計算量—— Quake 就用了這種近似，那個略微的紋理扭曲感是刻意的效能取捨。

4 深度緩衝的設計細節

Z-buffer vs W-buffer

最直覺的深度緩衝是存Z 座標（NDC 的 z，範圍 [0, 1]）。 但 Z 在透視投影下有個問題：大部分精度集中在近處，遠處的精度急劇下降， 導致Z-fighting（遠處兩個面互相閃爍）。

W-buffer 存的是原始的視圖空間深度 1/w（也就是 mini3d.c 的 rhw）—— 它在視圖空間是線性的，精度分布比 Z 均勻，但硬體支援度差，幾乎已被廢棄。

Reverse-Z（現代標準做法）

現代引擎（Unreal Engine 4.2+ 起）預設使用Reverse-Z： 把深度值的範圍從 [0→near, 1→far] 改成 [1→near, 0→far]。 這樣做利用了浮點數的特性——靠近 0 的區域有最多的精度—— 讓浮點 depth buffer 的有效精度分布接近 W-buffer 的效果。 現在 32-bit float depth + Reverse-Z 已是 AAA 引擎的標配。

▌ 論文：Optimal Depth Buffer for Low-Cost Graphics Hardware
你資料夾裡有這篇，分析了各種 depth buffer 精度方案的取捨（Z-buffer、W-buffer、log-depth、reverse-Z）。 如果你之後遇到遠處物件閃爍的問題，回來讀這篇會很有幫助。

Hierarchical Z-Buffer（HZB）

把 depth buffer 建立 mip chain（HZB），讓每一層都存比下一層 2×2 像素的最小深度。 這樣可以快速做遮蔽剔除（Occlusion Culling）： 對一個物件的包圍盒，在 HZB 的適當 mip level 查詢，如果最近深度比包圍盒的遠端還近， 整個物件就可以跳過，不需要送進 GPU 光柵化。 Unreal 的 Early Z PrePass + HZB 是整個可見性剔除系統的基礎。

5 Unreal Engine 5 深入

5-a 執行緒架構與 Render Dependency Graph

Unreal 把 CPU 端的工作切成三條流水線化的執行緒：

Game Thread
跑遊戲邏輯、物理、動畫。用 ENQUEUE_RENDER_CMD 把「這幀的場景描述」推送給 Render Thread。比 Render Thread 早一幀。
Render Thread
執行各 pass 邏輯（剔除、排序、生成 draw call）。不直接呼叫 GPU API，產生平台無關的 RHI 命令。
RHI Thread
把 RHI 命令翻譯成真正的 API 呼叫（Vulkan vkCmdDraw、D3D12 DrawIndexedInstanced）。D3D11 沒有這一層。

Render Dependency Graph（RDG）

UE5 所有的 pass 都用 RDG 宣告。RDG 的運作分兩階段：

  1. 宣告階段：Render Thread 宣告所有 pass（每個 pass 說明「我讀什麼 resource、我寫什麼 resource」）
  2. 執行階段：RDG 編譯 → 剪掉 unreferenced pass → 計算 resource 生命週期（讓 transient resource 共享記憶體）→ 插入 GPU barrier → 執行所有 lambda

這等同於 Vulkan/D3D12 的 render pass 同步模型，但在高層次自動處理了大量 barrier 管理的細節。 你在未來實作 shader 系統時，也可以把「宣告 pass 資源依賴、批次執行」作為架構設計的靈感。

5-b Material 系統與 Shader 編譯

Unreal 的 shader 有三個層次，分別在不同粒度上編譯：

類型每個什麼編譯一次用途
FGlobalShaderType整個引擎全螢幕 pass、工具用途（Bloom、SSAO 等）
FMaterialShaderType每個 Material（相同 Material 的所有物件共用）Base Pass、光照 pass
FMeshMaterialShaderTypeMaterial × Vertex Factory 組合不同類型 mesh（靜態/蒙皮/Nanite）用相同材質時各自的排列

Material 節點圖（Node Graph）在編輯器裡編譯成 HLSL，再用 DXC 編譯成各平台格式： DXIL（D3D12）、SPIR-V（Vulkan）、GLSL（OpenGL/ES）、MSL（Metal）。 編譯結果快取在 DDC（Derived Data Cache），下次啟動不需要重新編譯。 一個複雜的 Material 可能有幾百個 permutation（靜態開關不同、品質設定不同）。

5-c Vertex Factory

Vertex Factory 是 Unreal 抽象頂點輸入格式的機制。同一個 Material Shader 需要能處理不同種類的 Mesh：

  • FLocalVertexFactory — 靜態 Mesh（Static Mesh）
  • FGPUSkinVertexFactory — 蒙皮動畫 Mesh（Skeletal Mesh）
  • FNaniteVertexFactory — Nanite 的 Visibility Buffer path
  • FInstancedStaticMeshVertexFactory — 實例化的靜態 Mesh

這就是為什麼一個 Material 在引擎內部會編譯出多份 shader： Material × Vertex Factory 的每個組合都需要一份。 你的 IShader 設計（Ch23）雖然簡化很多，但 payload_t 裡的各種矩陣和屬性陣列， 和 Unreal 的 Vertex Factory 解決的是同一個問題：如何讓 shader 跟輸入格式解耦。

5-d 延遲渲染 Pass 順序

① Early Z PrePass
只寫 depth，不計算光照。後面所有 pass 用 EQUAL depth test，完全消除 overdraw。
② HZB 生成
Depth buffer 降採樣建 mip chain，用於 GPU 遮蔽剔除（Nanite 的 instance culling 也依賴這個）。
③ Base Pass（GBuffer 填充）
畫所有不透明物件，把材質屬性（Normal、BaseColor、Metallic、Roughness）寫入 GBuffer，完全不做光照。靜態預計算光（Lightmap）在這步一起合併。
④ Shadow Depth Pass
從光源視角渲染 shadow depth map（或 VSM 的 page 更新）。
⑤ Deferred Lighting
讀取 GBuffer，對每個可見像素計算所有光源貢獻。光源先用 Tile/Cluster 剔除，再批次計算。
⑥ SSAO / Lumen GI
環境遮蔽、全域光照（Lumen 的間接光）。
⑦ 透明物件（Forward）
透明物件無法寫 GBuffer，改用 Forward 渲染，按距離排序後疊加。
⑧ 後製 Stack
Bloom → Tonemapping → TSR（時域超解析度）→ 色彩分級 LUT。

5-e Nanite 虛擬化幾何

Nanite 解決的問題：

  • 傳統 LOD 系統需要美術手動製作多個精度版本的模型
  • 高多邊形模型（雕刻細節）不能直接放入遊戲場景
  • 小三角形的 GPU 效率極差（setup 成本 > 實際填充成本）

離線預處理

Mesh 在 import 時被切成約 128 個三角形的 cluster（一個 GPU wavefront 的大小）。 Cluster 之間建立一個 LOD DAG（有向無環圖）：葉節點是最高精度，根節點是最低精度， 每層的 cluster 覆蓋相同的表面積但用更少的三角形。 相鄰 LOD 層的 cluster 邊界被「鎖定」（錯誤容差限制），讓不同精度 cluster 混用時沒有接縫。

執行時管線

  1. Instance Culling：GPU 用上一幀的 HZB 做視錐 + 遮蔽剔除
  2. Cluster LOD 選擇：persistent compute shader 走 BVH/DAG，根據每個 cluster 的屏幕像素大小選擇精度（目標：每像素 ~1 三角形）
  3. 雙路光柵化：大三角形走硬體光柵器，小三角形（sub-pixel）走 compute shader 實作的軟體光柵器（就是你現在做的事，只是在 GPU 上）
  4. Visibility Buffer 輸出：不輸出顏色，輸出 ClusterID + TriangleID
  5. 材質評估：根據 Visibility Buffer，每個材質執行一次全螢幕 pass，從 Visibility Buffer 重建頂點屬性，寫入標準 GBuffer
▌ 為什麼要軟體光柵化小三角形？
當一個三角形小於 1–2 個像素時，GPU 硬體的 rasterizer setup（interpolator 計算、ROP 同步） 的固定成本遠大於實際計算成本。一個 compute shader 可以對大量這樣的三角形並行處理， 完全跳過硬體 setup，效率更高。這和你的 DrawTriangle 概念完全相同，只是規模不同。

5-f Lumen 動態全域光照

Lumen 解決傳統 GI 的困境：預烘焙靜態光照無法應對動態光源，即時 Ray Tracing 又太貴。

Surface Cache（表面快取）

每個 Mesh 有一組「card」——從不同角度覆蓋 mesh 表面的低解析度 atlas（類似 lightmap）。 Card 存儲 radiance、irradiance、albedo、normal。 當光源改變時，受影響的 card 重新計算（不是整個場景重算）。

Ray Tracing 後端（二選一）

  • Software RT（預設）：每個 Mesh 有 SDF（Signed Distance Field，一個存距離的 3D 體積紋理）。光線用球形步進（Sphere Marching）在 SDF 裡快速找到命中點。全局 SDF 用於長距離光線，mesh SDF 用於近距離精確計算。不需要 RTX 硬體。
  • Hardware RT：使用 GPU 的 BVH 加速結構（RTX）。更精確但更貴，遠景用 SDF 近景才用 HW RT 是常見配置。

Final Gather（最終收集）

不對每個像素追蹤光線。而是維護一組 screen-space 的探針（Probe）， 每個探針每幀追蹤幾條光線並累積到 Surface Cache。最後每個像素從最近的探針插值取得間接光。 結合時域累積（exploiting temporal coherence），把每幀的實際光線數壓到可接受範圍。

5-g Virtual Shadow Maps（VSM）

傳統 CSM（Cascaded Shadow Maps）的問題：解析度固定，近處細節不足。 VSM 的解法：維護一張虛擬的 16K×16K shadow map，用頁表（Page Table）按需分配物理記憶體。

  • 只分配「實際被光照、且可見的」像素對應的 page（128×128 像素一個 page）
  • 靜態幾何的 page：永久 cache，不重新渲染
  • 動態物件的 page：每幀重新渲染（但只有動態物件所在的 page）
  • 光源方向改變 → 所有 page 失效（虛擬地址空間重新映射）

這讓靜態場景的陰影幾乎免費，動態物件的陰影也只支付最小必要成本。 Nanite 可以直接光柵化到 VSM page，解析度達到 sub-pixel 精度——傳統 CSM 做不到這一點。

▌ 跟你的 Ch31 Shadow Mapping 的關係
你的 Ch31 實作的是最基礎的兩次渲染法（Basic Shadow Map）。 理解了它的原理（深度比較），才能理解 PCF 為什麼可以軟化邊緣、 CSM 為什麼要分段、VSM 為什麼要用虛擬頁表——每一步都是解決上一步的問題。

5-h Visibility Buffer

Visibility Buffer 是 Nanite 的輸出格式，也是現代 GPU-driven rendering 的核心概念。 傳統的 G-Buffer 在 base pass 直接寫入材質屬性（albedo、normal 等）——每個像素的材質資料都被寫入。 Visibility Buffer 的做法不同：base pass 只輸出 ClusterID + TriangleID（兩個 32-bit 整數）， 延遲到 material pass 再根據這些 ID 重建完整的材質資料。

好處：

  • 大幅減少 base pass 的頻寬（只有 8 bytes/pixel vs GBuffer 的 32–64 bytes/pixel）
  • 材質評估可以精確到每個像素只算一次（不因 overdraw 浪費）
  • 適合 GPU-driven culling，base pass 可以完全在 GPU 上決定畫什麼

6-b SRP 架構詳解

SRP（Scriptable Render Pipeline）是 Unity 開放渲染管線給開發者用 C# 自訂的框架。核心類別：

  • RenderPipelineAsset：存在 Project 裡的設定資產，CreatePipeline() 工廠方法產生 pipeline 實例
  • RenderPipeline：你 override Render(context, cameras)，這是每幀的入口
  • ScriptableRenderContext：延遲命令錄製器——所有呼叫都是「記錄」，不是執行。只有 ctx.Submit() 才真正送 GPU
  • CommandBuffer：渲染命令列表（清屏、設定 RT、blit、draw 等）
Unity URP — Render() 核心流程（示意）C#
protected override void Render(ScriptableRenderContext ctx, Camera[] cameras) { foreach (var camera in cameras) { // 1. 視錐剔除 → 取得可見物件清單 camera.TryGetCullingParameters(out var cullingParams); CullingResults culled = ctx.Cull(ref cullingParams); // 2. 設定相機矩陣（View / Projection）到 GPU constant buffer ctx.SetupCameraProperties(camera); // 3. 畫不透明物件（LightMode = "UniversalForward"） var drawSettings = new DrawingSettings(shaderTagId, sortingSettings); var filterSettings = new FilteringSettings(RenderQueueRange.opaque); ctx.DrawRenderers(culled, ref drawSettings, ref filterSettings); // 4. 畫天空盒、透明物件... ctx.DrawSkybox(camera); // 5. 提交所有命令給 GPU（在此之前都只是「記錄」） ctx.Submit(); } }

注意 ctx.Submit() 之前的所有呼叫都不會執行——這和你的 SetPixel() 同步直接寫入完全不同。 GPU 是非同步的：「記錄指令，批量提交」是現代 GPU API（Vulkan、D3D12、Metal）的核心設計。

ScriptableRendererFeature

URP 的插件點。你可以繼承 ScriptableRendererFeature，在指定的 pass 之後注入自訂的 render pass （例如：在所有不透明物件之後加一個 outline 效果、在後製之前加自訂的 SSAO）。 這相當於 Unreal 的 FSceneViewExtension。

6-c ShaderLab 與 HLSL

Unity 的 shader 格式是 ShaderLab，一個包裝 HLSL 的容器格式：

Unity ShaderLab 結構HLSL
Shader "MyShader" { Properties { _Color ("Color", Color) = (1, 1, 1, 1) _MainTex ("Texture", 2D) = "white" {} } SubShader { Tags { "RenderPipeline" = "UniversalRenderPipeline" } Pass { Name "ForwardLit" Tags { "LightMode" = "UniversalForward" } // 告訴 URP 這是哪個 pass HLSLPROGRAM #pragma vertex vert #pragma fragment frag #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl" // SRP Batcher 要求：所有 per-material 屬性放進這個 CBUFFER CBUFFER_START(UnityPerMaterial) float4 _Color; float4 _MainTex_ST; CBUFFER_END // ... vertex / fragment shader ... ENDHLSL } } }

LightMode tag 告訴 URP 在哪個 pass 呼叫這個 shader。 Unity 用 DXC 編譯 HLSL，再透過 shader compiler toolchain 轉成 GLSL/MSL/SPIR-V。

6-d Batching 策略

策略原理限制效益
Static Batching Build 時把靜態 mesh 合并成一個大 vertex/index buffer 只適用靜態物件；記憶體增加（頂點資料複製） Draw call 最少，效果最好
GPU Instancing 一次 DrawIndexedInstanced 畫 N 個相同 mesh，per-instance 資料存 buffer 必須相同 mesh + 相同 material；與 SRP Batcher 不相容 大量相同物件（樹、石頭）效果極佳
SRP Batcher 不減少 draw call；把 per-material uniform 常駐 GPU，每幀只 upload per-object transform Shader 必須把 material 屬性放在 CBUFFER UnityPerMaterial 消除 state change 開銷，CPU render loop 快很多
Dynamic Batching CPU 端複製小 mesh 頂點到共用 buffer，一次 draw 頂點數限制 (<300)；已幾乎過時 現代幾乎棄用
▌ SRP Batcher 的關鍵觀念
SRP Batcher 不減少 draw call，而是消除 draw call 之間的 state change。 傳統渲染每次 draw 前都要上傳 material 資料（顏色、貼圖偏移等），這個 CPU→GPU 上傳很昂貴。 SRP Batcher 把 material 資料永久存在 GPU 的 CBUFFER 裡，相同材質的物件只需要更新 per-object transform—— 大量 draw call 但材質不換的場景，CPU 時間可以從幾毫秒降到幾乎為零。

6-e Volume 系統

兩個引擎都有 Volume 系統：把後製效果（DoF 焦距、曝光、霧氣濃度等）用一個空間碰撞體來控制， 相機進入 Volume 範圍時，效果參數按距離/優先度混合。

這讓設計師可以在場景不同區域設定不同的大氣、光照風格，完全不需要程式碼介入。 Unreal 叫 Post Process Volume，Unity 叫 Volume Override。

7 GBuffer 設計原則

最低需求：PBR Deferred 所需的資料

  • 法向量：至少 2 個 channel（XY 存儲，Z 用 sqrt(1-x²-y²) 重建），需要精度夠高（16-bit per channel 或 octahedral 編碼）
  • BaseColor（Albedo）：sRGB，8-bit × 3 channel
  • Roughness：8-bit
  • Metallic：8-bit
  • Depth：從 depth buffer 重建 world-space position，不需要額外存

常用 format 選擇

資料Format理由
AlbedoRGB8 sRGB硬體 sRGB decode/encode 免費，比 RGB16F 省頻寬
NormalRG16F 或 RG16_SNORM精度需求高；RG16F 在大角度下精度更均勻
Roughness/Metallic打包進 Albedo 的 A 或 Normal 的 BA節省 render target 數量
Emissive / 光照累積B10G11R11 或 R11G11B10FHDR，無 alpha；頻寬比 RGBA16F 低 33%
DepthD32F + Reverse-Z精度最大化；32-bit float + reverse 分布

ShadingModelID（Unreal 的做法）

Unreal 在 GBufferB.A 存一個整數 ID，標記每個像素用哪種 BRDF 模型（Default Lit、Subsurface、Clear Coat、Hair、Eye 等）。 Deferred Lighting pass 讀這個 ID，用 switch 分派到不同的 BRDF 函數。 這讓一個場景可以混用多種材質類型，而不需要多個 GBuffer layout 或多個 lighting pass。

Forward+ 的核心機制

在 Forward pass 之前執行一個 compute shader，把 frustum 切成 3D 格子（tiles × depth slices = clusters）， 把每個光源的影響範圍分配到相交的 cluster。 Forward pass 中每個像素只查自己所在 cluster 的光源清單，大幅減少不必要的光源計算， 同時保留 Forward 對透明物件和 MSAA 的天然相容性。

9 陰影貼圖的演進

Basic Shadow Map
從光源視角渲染深度圖，渲染時比較每個像素的深度。問題：精度不足（perspective aliasing）、shadow acne（自遮擋）。
PCF 軟陰影
對 shadow map 取多個樣本，平均結果。核心大小決定陰影柔化程度。GPU 硬體支援 2×2 PCF（bilinear compare）。
PCSS 接觸軟化
先搜尋 blocker 的平均深度，再根據 blocker-receiver 距離決定 PCF kernel 大小。接觸處陰影硬，遠離處柔——物理正確。
CSM（Cascaded）
把 camera frustum 切成 N 段（cascade），每段一張 shadow map，解析度分配更合理（近處高、遠處低）。Unreal 預設 4 cascade。
VSM（Virtual）
虛擬頁表，按需分配。靜態場景永久 cache，動態物件只更新所在 page。Nanite 可直接光柵化到 VSM，解析度達 sub-pixel。
▌ 為什麼 shadow acne（自遮擋）會發生？
Shadow map 的解析度有限——一個 texel 代表某個面積。 當我們從光源視角取樣這個 texel 的深度，和從相機視角看到的表面深度比較時， 浮點誤差和紋理解析度的不足導致表面「遮蔽了自己」，出現條紋狀的錯誤陰影。 解法：加一個 depth bias（讓陰影圖的深度稍微往光源方向偏移），代價是 peter panning（陰影和物件分離）。 選一個適合場景的 bias 值需要反覆調整。

10 抗鋸齒的演進

SSAA
超採樣：以 N倍解析度渲染，再縮小。效果最好，成本最高（O(N²)）。
MSAA
多重採樣：每個像素多個 depth/coverage 採樣點，fragment shader 只跑一次。只抗幾何邊緣鋸齒，不抗 specular 高光鋸齒。與 Deferred 不相容。
FXAA / SMAA
後製抗鋸齒：對最終圖像做邊緣偵測、模糊。便宜但會模糊細節。SMAA 品質優於 FXAA。
TAA
時域抗鋸齒：每幀 jitter 相機，累積多幀結果（指數移動平均）。現代引擎標配；問題：鬼影（ghosting）需要用 velocity buffer 修正。
DLSS / FSR / XeSS
AI/機器學習超解析度：以低解析度渲染，用神經網路上採樣。DLSS（NVIDIA）最成熟，FSR（AMD）開放標準，XeSS（Intel）介於之間。
UE5 TSR
Temporal Super Resolution：UE5 自研，與 DLSS 類似但不依賴 NVIDIA 硬體。品質接近 DLSS，跨平台支援。

現代引擎幾乎都用 TAA 作為基礎，再加上 AI upscaling（DLSS/FSR）或引擎自研方案（TSR）。 你的 Ch32 學的 Wu's Line 和 MSAA 是理解這些方法「為什麼必要」的基礎—— 鋸齒的本質是採樣頻率不足（Nyquist 定理），所有抗鋸齒方法都是在不同層次解決這個問題。

11 開源軟體渲染器導讀

你的資料夾裡有幾個很有價值的開源軟體渲染器，每個架構思路不同：

mini3d.c — 單檔案 C 軟體渲染器
skywind3000 · C · Win32 GDI
架構：掃描線梯形法 + 透視正確插值（rhw）+ Z-buffer + 紋理採樣 + 矩陣系統。 全部在一個 .c 檔，適合一次性通讀理解整個管線的實作細節。

重點學習：vertex_rhw_init()（透視正確插值）、trapezoid_init_triangle()（掃描線三角形拆分）、 device_draw_scanline()（含 Z-test + 紋理採樣的掃描線）。

與你的 render_device.cpp 的差異：mini3d 用掃描線法，你用包圍盒法； mini3d 已有透視正確插值和 Z-buffer，這是你接下來要實作的 Ch11 和 Ch24。
SRender — C++ IShader 架構渲染器
C++ · CMake · TGA + OBJ
架構：有 IShader 抽象介面（vertex_shader / fragment_shader）、Phong Shader、PBR Shader（含 IBL：irradiance map + prefilter map + BRDF LUT）、Skybox Shader。

重點學習：shader.h 的 payload_t 結構（uniforms 的組織方式）、 iblmap_t（IBL 資源的載入與使用）。 這和你的 IShader 設計（Ch23）在概念上完全相同，可以對照參考。

還有 homogeneous clipping（齊次空間裁剪，Ch30 的內容）的 in/out buffer 設計。
▌ 建議閱讀順序
  1. 先看 mini3d.c 的矩陣系統（LookAt、Perspective）——對應你的 Ch13-18
  2. 再看 mini3d.c 的 vertex_rhw_init + device_draw_scanline——對應 Ch24 透視正確插值
  3. 最後看 SRender/shader/shader.h——對應 Ch23 IShader 設計
