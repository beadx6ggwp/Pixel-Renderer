# 重構紀錄

目的是希望穩住根基 以此劃分兩條branch做開發:
- main: 持續完成render pipeline
- feature/ui: 學習ui系統

架構:
- 全域協議層 (types.h)
    - 角色: 專案的「通用語言」。
    - 內容: 定義 FramebufferConfig、Vertex、Vec3f 等純資料結構。
    - 意義: 核心層與渲染層不需要互相知道對方的實體類別，只需要知道這些資料結構。這解決了 C++ 中最麻煩的循環依賴 (Circular Dependency) 問題。

- 基礎設施層 (Core / Infrastructure):
    - `ScreenManager`: 直接與 Win32 API 對接。負責視窗建立、硬體畫布 (GDI) 獲取與輸入事件監聽。它不處理任何渲染邏輯。
    - `RenderDevice` (RHI): 渲染硬體接口。它封裝了對記憶體緩衝區 (Framebuffer) 的直接操作。重構後它變得非常純粹，只負責 SetPixel 和 Clear。

- 渲染邏輯層 (Render / Graphics Logic)
    - Rasterizer: 渲染器的「大腦」。它負責執行光柵化演算法（如 Bresenham 或重心坐標三角形填充）。它透過指標依賴 RenderDevice，將算好的顏色寫入畫布。
    - Application: 引擎的控制中樞。負責管理計時器、驅動每一幀的 Update 與 Render 循環。

# 🛠️ 像素渲染器 (Pixel-Renderer) 重構與架構演進紀錄

## 一、 遇到的問題 (The Problems)
在重構之前，專案面臨了典型的「胖類別 (Fat Class)」與「高耦合 (High Coupling)」問題，這會嚴重阻礙後續 3D 功能的開發: 

1.  **職責不清與邏輯混雜**: 
    原本的 `RenderDevice` 不僅負責管理記憶體緩衝區（Framebuffer），還包含了 `DrawLine` 與 `DrawTriangle` 等幾何繪製演算法。這讓「硬體操作」與「數學計算」綁死在一起。
2.  **循環依賴與標頭檔地獄**: 
    各個模組之間互相 include（例如 `RenderDevice` 需要知道視窗大小，可能依賴了 `ScreenManager`；其他模組又依賴 `RenderDevice`），導致編譯時間增加，且架構難以拆分。
3.  **缺乏全局通訊協議**: 
    基礎的資料結構（如 `Vertex`, `FramebufferConfig`）散落在不同的標頭檔中，模組之間難以乾淨地傳遞資料。

---

## 二、 怎麼解決 (The Solutions)
我們採取了「由下而上，資料驅動」的重構策略，核心目標是**解耦 (Decoupling)**: 

1.  **建立全局協議層 (`types.h`)**: 
    將 `FramebufferConfig`、`Vertex`、`Vec3f` 等純資料結構抽出，放在專案根目錄的 `types.h`。這成為了所有模組共用的「通用語言」，徹底打破了模組間的循環依賴。
2.  **淨化硬體抽象層 (`core/render_device`)**: 
    將所有與「幾何、數學、演算法」相關的程式碼移出 `RenderDevice`。現在它只專注於最底層的內存搬運: `SetPixel`、`Clear`、初始化與獲取寬高。它變成了一個純粹的「畫布」。
3.  **獨立渲染邏輯層 (`render/rasterizer`)**: 
    建立專屬的 `Rasterizer` 類別，負責承接 `DrawLine` 與 `DrawTriangle`。它透過**依賴注入 (Dependency Injection)** 的方式（在建構時傳入 `RenderDevice*` 指針）來作畫。同時使用了**前置宣告 (Forward Declaration)** 來進一步降低標頭檔耦合。
4.  **目錄結構實體化**: 
    將檔案嚴格分類為 `core/` (基礎設施/作業系統相關) 與 `render/` (圖形學/數學相關)，並由根目錄的 `main.cpp` (Application) 負責組裝它們。

---

## 三、 整體思路與架構設計 (Architectural Vision)
這次重構的核心思路，完美對應了現代商業遊戲引擎（如 Unreal / Unity）的底層設計哲學: 

* **RHI (Render Hardware Interface) 概念**: 你的 `core/` 目錄就是 RHI。它把 Windows 繁瑣的 Win32 API、訊息迴圈 (Message Loop) 和 GDI 繪圖隱藏起來。如果未來想移植到 Linux，只需替換 `core/`，上層邏輯完全無感。
* **Renderer Module 概念**: 你的 `render/` 目錄就是渲染核心。這裡不談作業系統，只談純粹的 C++ 與數學（重心坐標插值、Bresenham 畫線等）。

這種分層設計，讓開發者的心智模型非常清晰: **「在 `core` 處理系統工程，在 `render` 處理圖形數學。」**
