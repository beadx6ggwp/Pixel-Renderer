Dependency Management / Third-party Integration
接入 SDL、OpenGL、D3D、ImGui 時，真正要管理的是 header、link、runtime 與 package boundary
dependency third-party linker cmake

I-19 解決的是 compile-time configuration：哪些 source 被編譯、哪些 macro 被定義。這一章接著處理更實際的問題：當你想在 macOS 用 SDL、Windows 也能編譯、未來又可能接 OpenGL / D3D / ImGui 時，第三方 library 到底是怎麼進到 C++ 專案裡的？

直覺上好像是「include 一個 header 就能用」。但 header 只讓 compiler 看懂 declaration；真正能不能產生 executable，還要經過 linker、runtime loader、package discovery、platform SDK。這也是很多跨平台 C++ 專案最常卡住的地方。

本章範圍 — 不做特定 package manager 速查，而是建立 dependency 的底層模型：include path、compile definition、link library、runtime artifact、package discovery/version。SDL、OpenGL、D3D、Metal、ImGui 的接入方式不同，但都應該被隔離在 platform/backend target，不應污染 renderer core。

1 naive model：include header 就代表 library 接好了

你可能會先寫：

#include <SDL.h>

int main() {
  SDL_Init(SDL_INIT_VIDEO);
}

如果 compiler 說找不到 SDL.h，你會補 include path。然後下一個錯誤可能變成 undefined reference / unresolved external symbol。這代表 compiler 已經看懂 declaration，但 linker 找不到 implementation。 

compile-time problem:
  fatal error: SDL.h: No such file or directory
  -> compiler cannot find header declarations

link-time problem:
  undefined reference to SDL_Init
  unresolved external symbol SDL_Init
  -> linker cannot find library implementation

runtime problem:
  SDL2.dll not found
  image not found: libSDL2.dylib
  error while loading shared libraries: libSDL2-2.0.so
  -> executable was linked, but OS loader cannot find runtime artifact
失敗點 — header 不是 library 本體。C++ dependency 至少同時跨越 compiler、linker、runtime loader 三個階段；package manager 只是幫你描述或取得這些東西。

SVG：第三方 library 不是單一東西，而是一組跨階段 resource。CMake target 的價值，是把這些 usage requirements 綁在正確 target 上。

3 SDL 可以怎麼接

SDL 是 Pixel-Renderer 第一個適合接的 cross-platform platform layer，因為它能處理 window、event、texture upload、present。接入方式大概有幾種：

方式優點代價適合時機
system install快速、符合 OS package 習慣不同機器路徑和版本可能不同個人開發、macOS Homebrew、Ubuntu apt
vendored / submodule版本固定，可離線repo 變重，需要自己維護 build options需要完全可重現 dependency
FetchContentCMake 內取得 sourceconfigure 時可能需要網路，版本升級要管小型跨平台專案
vcpkg / Conan能管理多平台 binary/source package多一層工具鏈設定依賴開始變多，想穩定管理版本
pkg-configLinux 生態常見Windows/macOS 一致性較差Ubuntu / Unix-like package discovery

第一版不要急著追求「最完整 dependency manager」。真正重要的是把 SDL 限制在 sdl_display_backend target 或 executable adapter，不讓 renderer_core 的 public header 需要 include SDL.h。 

renderer_core
  owns Framebuffer / Rasterizer / RenderDevice
  must not include SDL.h

display_sdl
  includes SDL.h
  owns SDL_Window / SDL_Renderer / SDL_Texture
  presents const Framebuffer&

app
  chooses backend
  links renderer_core + display_sdl

4 OpenGL / D3D / Metal 和 SDL 不同在哪

SDL 是 platform layer，也可以給你 SDL_Renderer 這種 2D render API；OpenGL、D3D、Metal 則是 graphics API。它們接進專案時的 dependency 型態也不同。

APIdependency 型態Pixel-Renderer 近期態度
SDLthird-party library；header + library + runtime artifact可用來做 DisplayBackend，把 CPU framebuffer present 到 window。
OpenGL平台提供 driver/API；通常還需要 loader，例如 glad/glew如果只是 software renderer，不應讓 renderer core 依賴它。
D3D11/12Windows SDK / system API；link 到 Windows libraries未來 hardware backend 可研究，現在不應混進 software backend。
MetalApple framework；通常走 Objective-C++ / framework linkmacOS 原生 backend 的未來選項，不是第一版跨平台 present 的必要條件。

這裡的結論不是「永遠不要碰 OpenGL/D3D」。而是 dependency direction 要對：software renderer core 的存在價值，是你完全掌握 CPU framebuffer 和 rasterization。OpenGL/D3D/Metal 若出現，應該是在 hardware backend 或 display/present adapter，而不是把 DrawTriangle 偷換成 GPU API。

5 ImGui 的接入方式又不同

ImGui 常見接入方式是把 source files 直接編進專案，再加上一個 backend implementation。例如官方 backend 會有 imgui_impl_sdl2.cpp、imgui_impl_opengl3.cpp 之類的檔案。

但如果 Pixel-Renderer 想保持 software renderer 作為主要實驗場，ImGui 不應直接變成「使用 OpenGL backend 畫 UI」。比較符合主線的設計是： 

ImGui / self-made immediate UI
  |
  +-- produce UI draw data
        vertices
        indices
        texture id / font atlas
        clip rect
        draw command
  |
  v
UI software adapter
  |
  v
RenderDevice / Rasterizer -> CPU Framebuffer

也就是說，ImGui 的 dependency 可以存在於 UI adapter target；但 RenderDevice 不應變成「知道 ImGui 是什麼」的 class。它應該只知道比較一般化的 draw command、texture view、clip rect、blend state。

6 CMake target 如何表達 dependency

CMake 裡最關鍵的不是 include_directories() 或 link_directories()，而是 target 的 usage requirements。

add_library(renderer_core
  src/render_device.cpp
  src/rasterizer.cpp
)

add_library(display_sdl
  src/platform/sdl_display_backend.cpp
)

target_link_libraries(display_sdl
  PRIVATE renderer_core SDL2::SDL2
)

target_compile_definitions(display_sdl
  PRIVATE PIXEL_RENDERER_USE_SDL_BACKEND=1
)

這裡的 PRIVATE 很重要：SDL 是 display_sdl 的實作細節，不應變成 renderer_core 的 public requirement。若你的 core public header 沒有出現 SDL_Window、SDL_Texture，那就不該讓 core 的使用者也被迫 include/link SDL。

target boundary 檢查 — 如果移除 display_sdl target 後，renderer_core 仍然可以被編譯和測試，代表 dependency boundary 大致健康。若 core 立刻找不到 SDL.h，代表 SDL 已經漏進 core。

7 Runtime dependency：build 成功不代表執行成功

Dynamic library 還有最後一關：OS loader。不同平台的錯誤型態不同：

平台常見 runtime artifact常見處理
WindowsSDL2.dll放在 executable 同資料夾，或加入 PATH。
macOSlibSDL2.dylib、SDL2.framework處理 framework path、rpath、Homebrew prefix、bundle 佈署。
Ubuntu / LinuxlibSDL2-2.0.so系統 package、ldconfig、rpath、LD_LIBRARY_PATH。

這就是為什麼「在我的 VSCode 裡能 build」不等於 dependency 管理完成。真正的檢查至少要包含：fresh build directory、clean machine 或 CI、debug run、runtime library discovery。

8 Pixel-Renderer 第一版 dependency policy

以目前專案方向，第一版可以保守一點：

  1. renderer_core 不依賴 SDL / Win32 / OpenGL / D3D / ImGui。
  2. display_win32 可以 include windows.h，但不把 Win32 type 放進 core public API。
  3. display_sdl 可以 include SDL.h，擁有 SDL_Window / SDL_Renderer / SDL_Texture。
  4. 第一版 SDL backend 建議用 std::vector<uint32_t> owned framebuffer + SDL_UpdateTexture，保持 framebuffer lifetime 穩定。
  5. 若未來接 ImGui，先把它當 UI frontend / draw data producer，不讓 renderer core 直接 depend on ImGui。
  6. OpenGL/D3D/Metal 先不接進 software renderer core；未來若做 hardware backend，再新增獨立 target。
correct direction:
  Rasterizer
    -> RenderDevice::SetPixel
    -> CPU Framebuffer
    -> DisplayBackend::Present
    -> Win32 BitBlt / SDL UpdateTexture / future platform present

wrong direction for this project:
  DrawLine     -> SDL_RenderDrawLine
  DrawTriangle -> SDL_RenderGeometry
  SetPixel     -> SDL_RenderDrawPoint
  RendererCore -> OpenGL/D3D/Metal by accident

9 verification：怎麼知道 dependency 沒漏進 core

可以用幾個很直接的檢查： 

1. rg "SDL_|SDL\\.h|windows\\.h|HWND|HDC" src/core include/core
   core headers should not mention platform API

2. inspect compile_commands.json
   renderer_core source should not have SDL include path unless truly needed

3. build renderer_core tests without display_sdl target
   core tests should still compile

4. create a clean build on macOS / Windows / Ubuntu
   runtime dependency failures should be display/app deployment issues, not core compile issues

這些檢查比「我感覺有分離」更可靠。Dependency boundary 是可以用 source dependency、target graph、compile command、link line 觀測出來的。

本章學到的三件事：
  1. 第三方 library 不是單一 header；它跨越 include path、compile definition、link library、runtime artifact、package discovery/version。
  2. SDL、OpenGL、D3D、Metal、ImGui 的接入方式不同，但都應被放在正確 target 與 adapter layer，不能污染 renderer core。
  3. Pixel-Renderer 第一版應保持 Rasterizer -> CPU framebuffer -> DisplayBackend::Present 的方向；SDL 只負責 window/event/texture upload/present，不取代 software rasterization。

到這裡，工具鏈、preprocessor、dependency boundary 已經足夠支撐下一段主線：開始把目前 Win32-owned framebuffer 拆出來，讓 renderer core 真正擁有自己的 CPU framebuffer。
