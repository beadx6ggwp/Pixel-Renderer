IMPLEMENTATION TRACK · I-07
CMake Target-based Build
CMake 的核心不是比較高級的 Makefile，而是描述 target graph
CMake target third-party

I-06 把 makefile 拆成 target、prerequisites、recipe、timestamp。那為什麼還需要 CMake？如果答案只是「跨平台大家都用」，那還是不夠。從第一性原理看，CMake 解的問題是：當專案開始有多個 module、多個 platform backend、多個 third-party library 時，compile/link 設定應該跟著 target 走，而不是全部丟到全域變數裡。

Pixel-Renderer 接下來會走向 renderer_core、win32_backend、sdl_backend、可能還有 imgui 或 opengl helper。這時候「哪個 source 參與哪個 build」、「哪個 include path 只屬於哪個 backend」、「哪個 library 只在 Windows link」都必須成為 build graph 的一部分。

本章範圍 — 從 makefile 的 global variable model 過渡到 CMake target model。你會理解 configure/generate/build 三階段、generator、out-of-source build、add_library、add_executable、target_include_directories、target_compile_definitions、target_link_libraries、PUBLIC/PRIVATE/INTERFACE、platform-specific source、以及 SDL/OpenGL/D3D/ImGui 這類 library 在工程專案中如何接入。

1 naive model：CMake 是跨平台 Makefile

很多人第一次看 CMake，會把它想成： 

CMakeLists.txt
  -> generate Makefile
  -> run make

這只描述了某一種 generator。更完整的模型是： 

CMakeLists.txt
  |
  | configure: read options, compilers, packages, platform
  v
CMake build tree
  |
  | generate: emit backend-specific build files
  v
Ninja files / Unix Makefiles / Visual Studio solution / Xcode project
  |
  | build: actual compiler and linker commands
  v
object files + libraries + executable

CMake 不是 compiler，也不是 linker，也不是 debugger。它產生 build graph，實際編譯仍然是 g++、clang++、cl.exe 這些 toolchain 在做。

第一個修正 — CMake 不取代理解 make/compiler/linker。它把 project model 描述成 target graph，再交給 Ninja/Make/Visual Studio/Xcode 執行。

2 out-of-source build：source tree 和 build tree 分開

CMake 常見工作流是：

cmake -S . -B build -G Ninja
cmake --build build
參數 意思 為什麼重要
-S . source directory。 讀 repo root 的 CMakeLists.txt。
-B build build directory。 把 generated files、object files、cache 放到 build/。
-G Ninja generator。 選擇要產生 Ninja build files；也可用 Makefiles、Visual Studio、Xcode。

source tree 是你的原始碼；build tree 是某次 configured build 的產物。Debug build、Release build、Windows build、macOS build 都可以有不同 build tree。 

Pixel-Renderer/
  CMakeLists.txt
  src/
  docs/
  build-debug/      generated files + objects
  build-release/    another generated build tree

3 target 是 build graph 的節點

Makefile 裡常見的是全域變數： 

SOURCES = all source files
INCLUDES = all include paths
LIBS = all libraries

$(TARGET): $(OBJECTS)
  $(CXX) ... $(LIBS)

target-based CMake 會改成：

add_library(pixel_core
  src/core/framebuffer.cpp
  src/render/rasterizer.cpp
)
target_include_directories(pixel_core PUBLIC src)

add_executable(pixel_renderer src/main.cpp)
target_link_libraries(pixel_renderer PRIVATE pixel_core)

這裡 pixel_core 和 pixel_renderer 都是 target。每個 target 擁有自己的 source、include dirs、compile definitions、compile options、link libraries。依賴關係用 target_link_libraries 表達。 

pixel_renderer executable
  |
  +-- links pixel_core library
        |
        +-- owns core/render sources
        +-- publishes include path src
target 的 aha moment — target 不只是最後的 exe。Library target 也可以是 build graph 的節點，用來封裝一組 source 與使用它所需的 compile/link requirements。

4 PRIVATE / PUBLIC / INTERFACE：dependency 會不會傳出去

CMake 最容易卡住的是 PRIVATE、PUBLIC、INTERFACE。從使用關係看會比較清楚：

Keyword target 自己需要？ 使用 target 的人也需要？ 例子
PRIVATE 需要。 不需要。 win32 backend 內部 link gdi32。
PUBLIC 需要。 也需要。 pixel_core 的 public headers 在 src/。
INTERFACE 自己不編譯，只有使用者需要。 需要。 header-only library 或 compile options bundle。

例如：

target_include_directories(pixel_core
  PUBLIC src
)

target_link_libraries(pixel_platform_win32
  PRIVATE gdi32 user32
)

src 可能需要傳給使用 pixel_core 的 target，因為 public headers 可能用 #include "core/..."。但 gdi32/user32 是 Win32 backend 的 implementation detail，不應污染 renderer core。

5 不要把全域 include/link 設定搬進 CMake

這種 CMake 很常見，但長期不好：

include_directories(src third_party/SDL/include)
link_libraries(gdi32 user32 SDL2)
add_executable(pixel_renderer ${SOURCES})

問題是它讓每個 source 都看得到 SDL headers，每個 target 都可能 link Win32 libraries。這會模糊架構： 

bad:
  renderer_core can accidentally include SDL.h
  macOS build sees gdi32/user32 link settings
  platform boundary exists only in your head

target-based 寫法會把需求放回正確 owner：

add_library(pixel_core
  src/core/framebuffer.cpp
  src/render/rasterizer.cpp
)
target_include_directories(pixel_core PUBLIC src)

add_library(pixel_platform_win32
  src/platform/win32/win32_display_backend.cpp
)
target_link_libraries(pixel_platform_win32 PRIVATE gdi32 user32)

add_executable(pixel_renderer src/main.cpp)
target_link_libraries(pixel_renderer PRIVATE pixel_core pixel_platform_win32)
不要用 CMake 掩蓋架構問題 — 如果 renderer core 不能依賴 Win32/SDL/macOS，那 build target 就不應把那些 include/link requirements 傳給 core。

6 platform-specific source：用 target 選 backend

現在 repo 主要是 Win32/GDI path。未來加 SDL/macOS 時，不應讓所有 platform source 同時被編。比較清楚的模型是： 

pixel_core
  no Win32
  no SDL
  no macOS

pixel_platform_win32
  Win32DisplayBackend.cpp
  link gdi32 user32

pixel_platform_sdl
  SDLDisplayBackend.cpp
  link SDL2

pixel_renderer
  links pixel_core + exactly one platform backend

第一版 CMake 可以先只 reproduces current build：

add_library(pixel_platform_win32 src/platform/win32/win32_display_backend.cpp)
target_link_libraries(pixel_platform_win32 PRIVATE gdi32 user32)

add_executable(pixel_renderer src/main.cpp)
target_link_libraries(pixel_renderer PRIVATE pixel_core pixel_platform_win32)

之後才加 option：

option(PIXEL_USE_SDL "Build SDL display backend" OFF)

if (PIXEL_USE_SDL)
  target_link_libraries(pixel_renderer PRIVATE pixel_platform_sdl)
else()
  target_link_libraries(pixel_renderer PRIVATE pixel_platform_win32)
endif()

這裡的重點不是語法，而是讓 build graph 表達「這次 build 選哪個 backend」。

7 third-party library 怎麼接：header、binary、runtime 三層

你前面問過 SDL、OpenGL、D3D、ImGui 這類 library 怎麼接進工程。不要把它們想成「下載後 include 就好」。通常要拆三層：

層 解決什麼 CMake 對應 例子
header / compile compiler 找得到 declarations。 target_include_directories SDL.h, imgui.h
library / link linker 找得到 symbols。 target_link_libraries SDL2.lib, libSDL2.a, opengl32
runtime 程式啟動時找得到 dynamic library。 copy DLL、rpath、bundle setting SDL2.dll, .dylib, framework

OpenGL/D3D 這類 platform graphics API 還會牽涉 SDK 或 system framework： 

Windows OpenGL:
  headers from Windows SDK / MinGW
  link opengl32
  load extensions separately if needed

Windows D3D:
  headers/libs from Windows SDK
  link d3d11/d3d12/dxgi/d3dcompiler or dxcompiler depending path

macOS OpenGL/Metal:
  framework linkage, platform-specific source

SDL:
  headers + SDL library + runtime DLL/dylib/framework

ImGui:
  usually source files compiled into your target
  backend files chosen per platform/renderer
工程思考 — third-party library 的接入方式取決於它是 header-only、static library、dynamic library、system library、framework，還是 source-level dependency。CMake target 的工作是把這些需求綁到真正使用它的 target。

8 find_package、FetchContent、submodule、vendored source 的取捨

接第三方 library 常見有幾條路：

方式 優點 代價 適合
find_package 使用系統或 package manager 安裝的 library。 環境差異較大，初學時容易找不到 package。 SDL、OpenGL、Vulkan SDK、平台 SDK。
FetchContent CMake configure 時抓 dependency。 需要網路；版本和 cache 要管理。 小型 library、教學用 dependency。
git submodule 版本固定，repo 內可重現。 submodule workflow 需要紀律。 ImGui、microui 這種 source dependency。
vendored source 最直接，沒有外部安裝。 更新與授權管理要自己負責。 單檔/少量 source library。

Pixel-Renderer 第一版不需要急著引入 dependency manager。可以先把 target boundary 設好，之後再決定 SDL/ImGui 用哪種方式接。

9 一個適合 Pixel-Renderer 的 CMake skeleton

這不是要求現在立刻改 repo，而是用來建立方向：

cmake_minimum_required(VERSION 3.20)
project(PixelRenderer LANGUAGES CXX)

add_library(pixel_core)
target_sources(pixel_core PRIVATE
  src/core/framebuffer.cpp
  src/core/render_device.cpp
  src/render/rasterizer.cpp
)
target_include_directories(pixel_core PUBLIC src)
target_compile_features(pixel_core PUBLIC cxx_std_17)

add_library(pixel_platform_win32)
target_sources(pixel_platform_win32 PRIVATE
  src/platform/win32/win32_display_backend.cpp
)
target_link_libraries(pixel_platform_win32 PRIVATE gdi32 user32)

add_executable(pixel_renderer src/main.cpp)
target_link_libraries(pixel_renderer PRIVATE pixel_core pixel_platform_win32)

這個 skeleton 暗示一個架構： 

pixel_core
  owns renderer-independent C++ sources
  publishes public include path
  does not know Win32/SDL

pixel_platform_win32
  owns Win32 present path
  privately links gdi32/user32

pixel_renderer
  executable composition root
  chooses core + backend

這和未來的 software renderer 前後端分離是同一件事的 build-system 版本。Code boundary 沒切乾淨，CMake 也救不了；但 CMake target 可以幫你檢查 boundary 是否真的存在。

10 VSCode / CMake Tools 的位置

VSCode CMake Tools 不應該被理解成另一套 build system。它只是替你操作 CMake configure/build/debug： 

VSCode CMake Tools
  |
  +-- choose kit: compiler/toolchain
  +-- configure: cmake -S . -B build
  +-- build: cmake --build build
  +-- debug: launch selected executable with debugger

這和 I-05 的 tasks/launch 是同一層問題：editor 是 orchestration，真正的 build graph 還是 CMake 產生，真正的 compile/link 還是 toolchain 做。

11 verification：怎麼確認 CMake target 沒污染

引入 CMake 後要檢查的不是「能不能 build」而已。更重要的是 target boundary 是否乾淨。

看 compile commands

cmake -S . -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
cmake --build build

compile_commands.json 會列出每個 source 真正使用的 compile command。你可以檢查 core source 是否不該看到 SDL/Win32 include path。

看 target link libraries

Win32 libraries 應該只出現在 Win32 backend 或最終 executable link line，不該成為 core target 的 public requirement。

測平台 source 是否被選錯

Windows backend build:
  should compile win32_display_backend.cpp
  should not compile sdl_display_backend.cpp unless option selected

SDL backend build:
  should compile sdl_display_backend.cpp
  should not require gdi32/user32 on macOS
常見陷阱
  • 把 include_directories()、link_libraries() 當預設做法，導致全域污染。
  • 用 file(GLOB ...) 全掃 source，讓平台 source 邊界消失。
  • 把 third-party library 只加到 VSCode IntelliSense，忘記 build/link/runtime 三層。
  • 把 CMake 當成不用懂 compiler/linker 的捷徑。
  • 還沒切 code boundary 就期待 target boundary 自動成立。
本章學到的三件事：
  1. CMake 的核心是描述 target graph，然後產生 Ninja/Make/Visual Studio/Xcode 等實際 build files；它不是 compiler/linker 本身。
  2. target-based CMake 把 source、include dirs、compile definitions、link libraries 綁在真正需要它們的 target 上，避免 renderer core 被 Win32/SDL/ImGui 污染。
  3. 第三方 library 要拆成 header、link、runtime 三層思考；OpenGL/D3D/SDL/ImGui 的接入方式不同，但都應該透過 target boundary 表達。
