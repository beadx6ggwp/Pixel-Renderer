IMPLEMENTATION TRACK · I-11
SDLDisplayBackend
用 SDL 做 window/event/present，不把 SDL 變成 renderer core
SDL UpdateTexture LockTexture present

I-10 把 Win32 path 說清楚了：DIBSection 是 present staging buffer，真正的 renderer target 是 project-owned Framebuffer。SDL backend 也應該用同一個思路。

容易混亂的地方是：SDL 既可以幫你開 window，也可以幫你畫 line/rect/texture，甚至底層可能走 GPU API。這會讓人誤以為「用了 SDL 就是在用 SDL renderer 畫圖」。對 Pixel-Renderer 來說，正確切法不是這樣。

本章範圍 — 建立 SDLDisplayBackend 的工程模型：SDL_Window、SDL_Renderer、SDL_Texture 是 backend-owned resources；Framebuffer 仍由 project owns；第一版用 SDL_UpdateTexture 在 Present 階段 upload/copy pixels，再用 SDL_RenderCopy 或 SDL_RenderTexture 畫到 renderer backbuffer，最後 SDL_RenderPresent。

1 這章延續的疑問地圖

前面討論 SDL 時，真正卡住的不是 API 名稱，而是「SDL 到底站在哪一層」。這章會把那串疑問整理成同一張地圖：

當時的疑問 這章的回答 落到 Pixel-Renderer
SDL 本質是封裝 framebuffer 嗎？ 不是。SDL 是 window/event/input/audio/render/present 的 cross-platform platform layer。 我們只拿它做 DisplayBackend，不拿它當 renderer core。
SDL_Renderer 背後是不是走 GPU pipeline？ 可能。SDL_Renderer 是 2D rendering abstraction，底層可走 D3D/Metal/OpenGL/software。 即使底層用 GPU，也只是拿來 present final texture，不是替你 rasterize triangle。
為什麼要把 framebuffer 畫成 texture 再交給 SDL？ 因為 window system 通常不接受你直接把任意 CPU pointer 當螢幕；你要走平台 present path。 Framebuffer -> SDL_Texture -> renderer backbuffer -> compositor。
UpdateTexture 和 LockTexture 差在哪？ 前者是你 owns CPU framebuffer，SDL copy/upload；後者是 SDL 暫時借你 writable update memory。 第一版用 UpdateTexture，因為 lifetime 最乾淨。
SDL_RenderPresent 後面發生什麼？ SDL submit renderer backbuffer，交給 OS compositor，再進 display scanout。 這是 display boundary，不是 software rasterizer boundary。

2 naive model：把 DrawLine 換成 SDL_RenderDrawLine

如果目標只是做一個普通 2D app，SDL 的 2D renderer API 很方便： 

SDL_RenderDrawPoint
SDL_RenderDrawLine
SDL_RenderFillRect
SDL_RenderCopy / SDL_RenderTexture
SDL_RenderPresent

但 Pixel-Renderer 的學習核心是你自己實作 rasterizer：Bresenham、edge function、barycentric、depth test、shader interface。把 DrawLine 改成 SDL_RenderDrawLine，等於把你原本要理解的 pipeline 交給 SDL。

不要替換錯層 — SDL backend 不應提供 SetPixel、DrawLine、DrawTriangle 給 renderer core。它只應該實作 Present(const Framebuffer&) 和 event/window lifecycle。

3 SDL 在這裡的真正角色

SDL 對這個專案的角色是 cross-platform platform layer： 

SDLDisplayBackend owns:
  window lifecycle        SDL_Window
  event pump              SDL_Event / SDL_PollEvent
  present context         SDL_Renderer
  upload/present image    SDL_Texture

Pixel-Renderer core owns:
  Framebuffer             CPU pixels
  RenderDevice            SetPixel / Clear
  Rasterizer              line / triangle / shader pipeline
  UI renderer             future software UI draw commands

這個分工讓 Windows 和 macOS 都可以走同一份 renderer core。Windows 可以選 Win32 backend，也可以選 SDL backend；macOS 短期可以先選 SDL backend，之後再評估 native Cocoa/Metal path。

4 SDL2 / SDL3：不是同一組 API 名稱，但層次相同

SDL2 和 SDL3 對你現在最重要的差異，不是「哪個比較高級」，而是 API 名稱、event enum、include/link 方式、部分 function signature 有整理。抽象層次沒有變：你仍然是開 window、建 renderer、建 texture、poll event、update texture、render texture、present。

概念 SDL2 常見名稱 SDL3 常見名稱 本章關心的語意
Include #include <SDL.h> 或 platform package path #include <SDL3/SDL.h> 只出現在 SDL platform target。
Renderer copy texture SDL_RenderCopy SDL_RenderTexture 把 SDL_Texture 畫到 renderer backbuffer。
Quit event SDL_QUIT SDL_EVENT_QUIT 轉成 project-level ShouldClose()。
Texture update SDL_UpdateTexture SDL_UpdateTexture 從 project framebuffer copy/upload pixels。
版本選擇的工程判準 — 教學文件可以同時講 SDL2/3 的語意，但實作時要選一版固定下來，否則 sample code 會被 API 名稱差異污染。若目標是穩定跨平台、資料多、教學資源多，SDL2 較保守；若你想跟新 API 走，SDL3 也可以，但 build/package 細節要另外處理。

5 SDLDisplayBackend owns 什麼

和 Win32 backend 一樣，SDLDisplayBackend 是 resource-owning type。差別是它 owns SDL handles，不 owns renderer pixels。

class SDLDisplayBackend : public DisplayBackend {
public:
  SDLDisplayBackend() = default;
  ~SDLDisplayBackend() override { Shutdown(); }

  bool Init(int width, int height, const char* title) override;
  void Shutdown() override;
  void PollEvents() override;
  bool ShouldClose() const override;
  void Present(const Framebuffer& fb) override;

private:
  SDL_Window* window_ = nullptr;
  SDL_Renderer* renderer_ = nullptr;
  SDL_Texture* texture_ = nullptr;
  int width_ = 0;
  int height_ = 0;
  bool should_close_ = false;
};

真正 implementation 時可以用 custom deleter 包成 unique_ptr，或先用 raw SDL pointer + 明確 Shutdown。重點不是語法，而是 ownership：誰 SDL_Create*，誰就必須 SDL_Destroy*。

6 Init：SDL lifecycle

最小 lifecycle 是： 

SDLDisplayBackend::Init(w, h)
  SDL_Init(SDL_INIT_VIDEO)
  SDL_CreateWindow(...)
  SDL_CreateRenderer(...)
  SDL_CreateTexture(
      renderer,
      pixel_format,
      streaming_or_static_access,
      width,
      height)

這裡每個 resource 的語意不同：

Resource 意義 不是什麼
SDL_Window native OS window 的 SDL handle。 不是 framebuffer。
SDL_Renderer SDL 2D rendering context，負責 texture copy/present。 不是你的 software rasterizer。
SDL_Texture SDL renderer 可使用的 image resource。 不保證是永久可寫的 linear CPU memory。

7 第一版 Present：SDL_UpdateTexture

第一版建議走 SDL_UpdateTexture。理由很直接：你的 CPU framebuffer lifetime 穩定，RenderDevice 永遠寫 project-owned memory，SDL 只在 Present 階段接手。 

CPU side                                      SDL / driver / display side
--------                                      --------------------------

Framebuffer
  vector<uint32_t> pixels
  owned by project
  stable pointer
       |
       | RenderDevice::SetPixel
       v
final CPU pixels
       |
       | SDL_UpdateTexture(texture, pixels, pitch)
       | copy / convert / upload
       v
                                            SDL_Texture
                                                 |
                                                 | SDL_RenderCopy / SDL_RenderTexture
                                                 v
                                            renderer backbuffer
                                                 |
                                                 | SDL_RenderPresent
                                                 v
                                            OS compositor / display
void SDLDisplayBackend::Present(const Framebuffer& fb) {
  if (fb.Width() != width_ || fb.Height() != height_) {
    // first version: reject or recreate texture
    return;
  }

  const void* pixels = fb.Bytes();
  int pitch = fb.Pitch();

  SDL_UpdateTexture(texture_, nullptr, pixels, pitch);
  SDL_RenderClear(renderer_);
  SDL_RenderCopy(renderer_, texture_, nullptr, nullptr);
  SDL_RenderPresent(renderer_);
}

如果使用 SDL3，最後的 copy call 可能改成 SDL_RenderTexture。但 abstraction 不變：Present 只讀 Framebuffer，不讓 renderer core 依賴 SDL_Texture。

8 SDL_Renderer 背後到底做了什麼

SDL_Renderer 不是你的 renderer core。它比較像 SDL 提供的一個 2D render context。你對它下的命令，例如 clear、copy texture、present，SDL 會轉到某個 backend： 

SDL_Renderer API
  SDL_RenderClear
  SDL_RenderCopy / SDL_RenderTexture
  SDL_RenderPresent
        |
        v
SDL renderer backend
  Windows: maybe Direct3D
  macOS:   maybe Metal
  Linux:   maybe OpenGL / software / platform-dependent
        |
        v
backend render target / window surface / compositor path

所以你前面問「SDL 自己有自己的繪圖 func，底層可能走顯卡 pipeline，是這樣嗎？」答案是：對，可能。但那是 SDL 的 2D renderer pipeline，不是你的 software rasterizer pipeline。 

Your pipeline:
  vertices
    -> your CPU rasterizer
    -> your CPU framebuffer

SDL renderer pipeline used here:
  uploaded texture
    -> textured quad/copy
    -> renderer backbuffer
    -> present
兩條 pipeline 不要混在一起 — 你的 pipeline 決定每個 pixel 是什麼顏色。SDL renderer pipeline 只負責把這張已經算好的 image 搬到 window 的 present path。

9 為什麼 CPU 不能假設永遠直接寫 GPU texture

你前面問「等於我只是畫一張貼圖給 SDL 讓他再傳給 GPU，這不怪嗎？」從 software renderer 角度看，這其實是合理的。因為你的 framebuffer 是 CPU 計算結果，而現代 window/display path 通常期待的是某種 OS/driver 管理的 surface 或 texture。 

CPU-friendly memory
  linear layout
  address = base + y * pitch + x * 4
  stable pointer
  good for RenderDevice::SetPixel

GPU/driver texture
  may be in VRAM or driver-private memory
  may use tiled/swizzled layout
  may be used by GPU command queue
  may require synchronization / staging upload

所以 graphics API 常見的模型是 staging/update： 

CPU framebuffer
  stable, linear, debuggable
        |
        | update/upload/copy
        v
GPU/driver texture
  optimized for sampling/copy/present
        |
        v
renderer backbuffer / compositor

SDL_UpdateTexture 和 SDL_LockTexture 就是 SDL 把這段 update lifecycle 包起來的兩種入口。它們不是多此一舉，而是在承認 CPU memory、driver resource、GPU texture、OS compositor 不是同一塊東西。

10 為什麼不是先用 SDL_LockTexture

SDL_LockTexture 很容易讓人覺得「我終於直接拿到 framebuffer pointer 了」。這句話只對一半。你拿到的是 SDL 在 lock/unlock 期間提供的 writable texture update memory，不是可以永久保存的 render target。 

SDL_LockTexture(texture)
       |
       v
temporary writable pointer
  valid only until SDL_UnlockTexture
  pitch chosen by SDL/driver
  old contents may not be preserved
       |
       | CPU writes during lock
       v
SDL_UnlockTexture(texture)
       |
       | SDL/driver applies, converts, uploads
       v
SDL_Texture usable by renderer

如果你的 RenderDevice 要直接寫 locked pointer，就會需要新的 frame lifecycle： 

BeginFrame()
  SDL_LockTexture()
  create FramebufferView from returned pixels/pitch
  bind RenderDevice to this temporary view

Render()
  Rasterizer -> RenderDevice::SetPixel

EndFrame()
  unbind RenderDevice view
  SDL_UnlockTexture()
  SDL_RenderCopy / SDL_RenderPresent

這不是不能做，但它會讓 RenderDevice 的 target pointer 每幀變動，而且有效期被 SDL lock/unlock 包住。對第一版 backend abstraction 來說，這會增加太多 lifecycle coupling。

LockTexture 的正確定位 — 它是後續效能實驗選項，不是第一版架構分離的基礎。先讓 Framebuffer ownership 穩定，再討論是否把 present path 改成 lock/unlock streaming。

11 UpdateTexture vs LockTexture

問題 SDL_UpdateTexture SDL_LockTexture
CPU framebuffer owner Project owns stable Framebuffer。 SDL owns temporary writable memory during lock。
RenderDevice target Stable view over project memory。 Temporary view, must be rebound per frame。
Debug / screenshot Easy: framebuffer always exists。 Harder: valid pixels may exist only during frame window。
Copy cost Likely one copy/upload per frame。 May reduce one application-side copy depending on backend。
Best first use Yes, for clean boundary。 Later, after lifecycle abstraction is explicit。

12 SDL_RenderPresent 後面的資料流

SDL_RenderPresent 不是「把 triangle rasterization 交給 GPU」。到這一步，你的 CPU rasterizer 已經把 final bitmap 算完了。SDL renderer 做的是 texture copy / draw 到 backbuffer，再把 backbuffer present。 

CPU rasterizer already done
       |
       v
SDL_Texture contains uploaded frame
       |
       | SDL_RenderCopy / SDL_RenderTexture
       v
SDL renderer backbuffer
       |
       | SDL_RenderPresent
       | flush/submit backend commands
       | maybe wait for vsync
       v
OS compositor
       |
       | compose desktop windows
       v
display scanout

底層如果是 D3D/Metal/OpenGL，SDL 可能把 texture copy 變成 GPU draw/copy command；底層如果是 software backend，SDL 可能自己管理 CPU buffer 再交給 window system。對 Pixel-Renderer 來說，這些都在 display backend 之後。

OS compositor 這層也很重要。現代桌面通常不是每個 app 直接掃到螢幕，而是每個 window 產生自己的 surface，然後 compositor 把所有 window 合成成最後桌面畫面。 

Your app window surface
Browser window surface
Terminal window surface
Mouse cursor / overlays
        |
        v
OS compositor
  Windows: DWM
  macOS:   WindowServer / Quartz Compositor
  Linux:   compositor under Wayland/X11 environment
        |
        v
final desktop image
        |
        v
display engine scanout
        |
        v
monitor

所以 SDL_RenderPresent 比較精準的意思是：把 SDL renderer 的 current backbuffer submit 到平台 present path。它可能等待 vsync，也可能只是把 frame 交給 OS/driver queue；不應理解成「函式 return 的瞬間螢幕物理像素已經變了」。

和 Win32 BitBlt 對照 — BitBlt 是 GDI 的 blit request；SDL_RenderPresent 是 SDL renderer 的 present boundary。兩者後面都還可能經過 driver、compositor、display scanout，不是你程式中的最後一行就等於 monitor 已更新。

13 event/input：SDL_PollEvent

SDL backend 也會替換 Win32 的 PeekMessage / WindowProc path：

void SDLDisplayBackend::PollEvents() {
  SDL_Event e;
  while (SDL_PollEvent(&e)) {
    if (e.type == SDL_QUIT) {
      should_close_ = true;
    }
    // translate SDL key/mouse events into project InputState
  }
}

這裡的設計原則和 Win32 一樣：不要讓上層依賴 VK_ESCAPE，也不要讓上層直接依賴 SDL scancode。backend 應該把 native key 表示轉成 project-level KeyCode 或 InputState。

14 Shutdown 與 RAII

SDL resource release order 通常反過來： 

Shutdown()
  if texture_  SDL_DestroyTexture(texture_)
  if renderer_ SDL_DestroyRenderer(renderer_)
  if window_   SDL_DestroyWindow(window_)
  SDL_Quit()   if this backend owns SDL global video lifecycle

如果專案未來可能有 audio、gamepad、ImGui SDL backend、或多個 SDL-using subsystem，就要小心 SDL_Init / SDL_Quit 的 global ownership。第一版可以讓 SDLDisplayBackend owns video lifecycle；之後再抽成 SdlContext 也可以。

和 T-06 的連接 — SDL_Window、SDL_Renderer、SDL_Texture 都是 C API resource。你可以先用 explicit Shutdown，但最終應讓它們進入 RAII wrapper 或 custom-deleter unique_ptr，避免 early return leak。

15 build boundary：SDL 是 platform target dependency

SDL 不應該滲進 core target。比較乾淨的 target graph 是： 

pixel_core
  Framebuffer
  RenderDevice
  Rasterizer
  DisplayBackend interface
  no SDL.h

pixel_platform_sdl
  SDLDisplayBackend
  includes SDL.h
  links SDL2/SDL3

pixel_app
  links pixel_core + selected platform backend

工程上這表示 sdl_display_backend.cpp 可以 include SDL header，但 render_device.h、rasterizer.h、framebuffer.h 不可以。這點比使用哪個 SDL function 更重要。

16 和 Win32 backend 對照

概念 Win32 backend SDL backend
Window HWND SDL_Window*
Event pump PeekMessage / WindowProc SDL_PollEvent
Present staging DIBSection memory SDL_Texture update/upload path
Present call BitBlt SDL_RenderPresent
Renderer core Rasterizer -> RenderDevice -> Framebuffer，完全相同。

17 實作 commit sequence

Step Change 驗證
1 新增 SDLDisplayBackend skeleton，只負責 init/shutdown。 能開 window 並正常關閉。
2 建立 SDL_Texture，格式對齊 Framebuffer 的 uint32_t packing。 test pattern 顏色不紅藍顛倒，alpha 不造成黑屏。
3 Present 用 SDL_UpdateTexture upload framebuffer。 同一個 frame 在 Win32 與 SDL 顯示一致。
4 接 SDL_PollEvent 到 project InputState。 ESC、mouse position、close button 行為一致。
5 build system 增加 SDL platform target。 Windows/macOS 都可選 SDL backend；core target 不 include SDL。

18 Summary

本章學到的三件事：
  1. SDL 在 Pixel-Renderer 裡應該先當 platform/present backend，不應接管 SetPixel、line、triangle 或 shader pipeline。
  2. 第一版 SDLDisplayBackend::Present 建議用 SDL_UpdateTexture，因為 project-owned Framebuffer lifetime 穩定、debug/test 容易。
  3. SDL_LockTexture、SDL_Renderer、OS compositor 都在 present lifecycle 裡；它們不改變你的核心邊界：Rasterizer -> RenderDevice -> Framebuffer -> DisplayBackend::Present。

下一章可以暫時離開 backend 實作，回到 C++ theory：如果你要讓 math type、framebuffer view、shader input、command buffer 都能用泛型表達，真正問題不是「template 語法」，而是 type requirement 與 compile-time contract。
