IMPLEMENTATION TRACK · I-12
Cross-platform Development Path
以跨平台為主軸整理 Windows / macOS / Ubuntu 的 toolchain、debugger、SDL、backend selection
cross-platform Ubuntu macOS CMake

前面已經把 Win32 backend、SDL backend、owned framebuffer 的邊界講清楚。現在回到更大的工程問題：如果 Pixel-Renderer 要能在 Windows、macOS、Ubuntu/Linux 都繼續開發，哪些東西必須是共用 core，哪些東西必須變成 platform backend？

直覺答案可能是「Windows 用 g++、Mac 用 clang++、Ubuntu 用 g++」。這只講到 compiler driver，還沒處理 source boundary、link library、debugger、window system、runtime dependency、VSCode model。真正的跨平台不是換 compiler 指令，而是把平台差異隔離到清楚的 build target 和 backend implementation。

本章範圍 — 建立跨平台開發路徑：分清 Windows / macOS / Ubuntu 的 compiler、standard library、binary format、debugger、window system、SDL dependency、VSCode、CMake preset、backend selection 各自負責什麼。短期目標是讓 pixel_core + SDLDisplayBackend 能在 macOS 與 Ubuntu 上開發，同時保留 Windows Win32 backend 作為 baseline。

1 naive model：每個平台換一條 compiler command

如果你現在把專案搬到 macOS 或 Ubuntu，最直覺可能是： 

Windows:
  g++.exe src/*.cpp -lgdi32 -luser32

macOS:
  clang++ src/*.cpp

Ubuntu:
  g++ src/*.cpp

這會失敗，因為目前 source 和 build 還混著 Win32-only 假設： 

compile phase:
  windows.h not found
  HWND / HDC / HBITMAP unknown
  VK_ESCAPE unknown

link phase:
  gdi32 / user32 do not exist
  Win32 symbols unavailable

debug phase:
  launch.json expects .exe
  debugger path is Windows/MinGW-oriented
  macOS normally uses LLDB
  Ubuntu commonly uses GDB or LLDB

這些錯誤不是某個 compiler 不會 C++。它們在說：platform-specific source 還沒有被隔離。

不要修錯層 — 在 macOS 或 Ubuntu 上看到 windows.h not found，不是去找一份假的 windows.h，也不是把 Win32 code 改成一堆 #ifdef 塞在 core 裡。正解是讓非 Windows build 不編 Win32 backend。

2 跨平台開發其實有七層

一個平台能不能跑，不是單一設定。至少有七層要分開：

層 Windows macOS Ubuntu / Linux 該怎麼處理
Compiler driver MinGW g++.exe Apple clang++ GCC g++ 或 Clang 交給 CMake preset 或 platform task 選。
Standard library libstdc++ 或 MSVC STL libc++ 通常 libstdc++，也可用 libc++ 避免依賴非標準 extension。
Object/executable format PE/COFF, .exe Mach-O ELF 不要把 output path 寫死成 Windows 型態。
Platform API Win32/GDI SDL/Cocoa/Metal/Quartz SDL/X11/Wayland/OpenGL/Vulkan 短期非 Windows 都先走 SDL backend。
Debugger GDB / Visual Studio debugger LLDB GDB 或 LLDB VSCode launch 要用對 debugger。
Editor model c_cpp_properties.json 手寫 最好讀 compile_commands.json 同樣讀 compile_commands.json 讓 VSCode 跟 build graph，不要複製設定。
Window/compositor DWM WindowServer / Quartz Compositor Wayland compositor 或 X11 window manager/compositor 由 SDL backend 隱藏初期差異。

3 短期路線：macOS / Ubuntu 都先跑 SDL backend

短期最穩的路線不是同時寫 MacOSDisplayBackend 和 LinuxDisplayBackend，而是讓 macOS / Ubuntu 都走已經定義好的 SDL present path： 

macOS short-term target:

pixel_core
  Framebuffer
  RenderDevice
  Rasterizer
  math / shader / UI data
  no windows.h
  no SDL.h

pixel_platform_sdl
  SDLDisplayBackend
  SDL_Window / SDL_Renderer / SDL_Texture
  SDL_UpdateTexture present path

pixel_app
  links pixel_core + pixel_platform_sdl
Ubuntu short-term target:

pixel_core
  same C++ renderer core
  no windows.h
  no SDL.h

pixel_platform_sdl
  SDLDisplayBackend
  SDL uses X11/Wayland backend internally

pixel_app
  links pixel_core + pixel_platform_sdl

這條路徑的意義是：你不是為了某個平台改掉 renderer；你只是換掉 display backend。Win32 backend 仍然可以留在 Windows 作為 native/reference backend。

短期判準 — 如果 macOS 與 Ubuntu build 都能編過 RenderDevice、Rasterizer、Framebuffer，並透過 SDLDisplayBackend::Present 顯示同一張 test pattern，就代表抽象方向正確。

4 Toolchain matrix：Clang / GCC / MSVC 不是同一種東西

跨平台時要避免一句「都是 C++ compiler」就混在一起。它們會產生不同 object format，搭配不同 standard library、debugger、system SDK。 

Windows MinGW:
  g++.exe
  PE/COFF executable
  links gdi32/user32
  debugger: GDB

macOS:
  clang++
  Mach-O executable
  links frameworks / dylibs / SDL
  debugger: LLDB

Ubuntu:
  g++ or clang++
  ELF executable
  links shared objects / SDL
  debugger: GDB or LLDB

所以 build system 不應該假設：

  • binary 一定叫 .exe。
  • debugger 一定是 gdb。
  • system library 一定是 -lgdi32 -luser32。
  • 所有 source 都能在所有平台編譯。

5 CMake preset：把平台選擇顯性化

I-07 已經講過 target-based CMake。到了跨平台階段，重點不是先追求最漂亮的 CMake，而是要讓平台選擇變成 build graph 的一部分。 

CMake options:
  PIXEL_BACKEND=Win32
  PIXEL_BACKEND=SDL
  PIXEL_BACKEND=MacOSNative   future

Windows preset:
  compiler: MinGW GCC or MSVC
  backend: Win32 or SDL

macOS preset:
  compiler: Apple Clang
  backend: SDL

Ubuntu preset:
  compiler: GCC or Clang
  backend: SDL

概念上會變成：

add_library(pixel_core ...)

if(PIXEL_BACKEND STREQUAL "Win32")
  add_library(pixel_platform_win32 ...)
  target_link_libraries(pixel_platform_win32 PRIVATE gdi32 user32)
elseif(PIXEL_BACKEND STREQUAL "SDL")
  add_library(pixel_platform_sdl ...)
  target_link_libraries(pixel_platform_sdl PRIVATE SDL2::SDL2)
endif()

這段不是要你現在照貼，而是要理解：gdi32/user32 只屬於 Win32 backend，SDL 只屬於 SDL backend，core 不 link 任何平台 library。

6 SDL dependency：include、link、runtime 是三件事

在 macOS 或 Ubuntu 接 SDL 時，常見錯誤是以為「include path 加進 VSCode」就完成了。實際上 SDL dependency 至少有三層：

層 問題 CMake/工程上要表達
Compile Compiler 找得到 SDL headers。 include directories / imported target usage requirements。
Link Linker 找得到 SDL library。 target_link_libraries(pixel_platform_sdl PRIVATE SDL...)。
Runtime 執行時 loader 找得到 dylib/framework。 runtime path、bundle、package manager 或安裝位置。

SDL 可以來自 package manager、framework、vendored source、或系統套件。macOS 可能走 Homebrew 或 framework；Ubuntu 可能走 distro package、pkg-config、CMake config package 或 vendored source。這些選項不應該散落在手寫 compile command；它們應該被 build target 表達，讓 VSCode、compiler、linker、runtime 有同一份真實來源。

7 Linux / Ubuntu 特別要注意什麼

Ubuntu 的好處是 compiler / debugger / package manager 比較接近傳統 Unix 開發環境；但 display layer 比 macOS/Windows 更容易分岔：X11、Wayland、不同 compositor、不同 GPU driver、不同 distro package version。 

Ubuntu display reality:
  SDL_Window
       |
       +-- X11 backend
       |     X server / window manager / compositor
       |
       +-- Wayland backend
             Wayland compositor owns display protocol

Pixel-Renderer should not care:
  Rasterizer -> Framebuffer -> SDLDisplayBackend::Present

這正是 SDL backend 的價值：你不需要第一版就學 Xlib、Wayland protocol、GLX/EGL、desktop compositor 差異。這些都先被壓在 SDL platform layer 底下。

Ubuntu 不是沒有平台問題 — Linux 不是單一平台。不同 distro、display server、driver、package version 都可能造成差異。第一版使用 SDL backend，是為了把這些差異隔離在 pixel_platform_sdl，不要污染 renderer core。

8 VSCode：不要讓 editor 設定假裝是 build system

目前 Windows-oriented VSCode 設定大致是： 

tasks.json
  invokes make / g++.exe path

launch.json
  launches .exe
  assumes GDB

c_cpp_properties.json
  tells IntelliSense include paths
  does not actually build the program

跨平台後比較乾淨的模型是： 

CMake configure
  chooses compiler
  chooses backend
  finds SDL
  writes compile_commands.json
        |
        v
VSCode
  reads real include paths / defines
  launches GDB/LLDB against generated binary
  does not duplicate build graph by hand
IntelliSense 不是真相來源 — 如果 VSCode 紅線消失，但 terminal build 仍然失敗，表示 editor model 和 build model 不一致。長期應讓 editor 讀 build system 產出的 compile database。

9 Debugger：GDB / LLDB 是同一類問題

macOS 上預設 debugger 是 LLDB。Ubuntu 常見是 GDB，也可以用 LLDB。它們的核心概念類似：都控制一個 process，靠 debug symbols 對應 source line。 

Debug build:
  compiler -g ...
        |
        v
platform binary with debug info
        |
        v
GDB / LLDB
  breakpoint
  run
  step
  frame variable
  backtrace

你要保留的 debug 能力不是某個 GDB 指令，而是這些問題：

  • RenderDevice::SetPixel 有沒有寫到正確 address？
  • Framebuffer::Pitch 和 SDL texture update pitch 是否一致？
  • SDLDisplayBackend::Present 每幀是否被呼叫？
  • 關閉視窗時 event 是否轉成 ShouldClose()？

也就是說，debugger 在 GDB / LLDB 之間切換，不會改變你的 debugging reasoning；它只改變 tool command 和 launch configuration。

10 Native backend 都不是第一優先

未來如果要做 native platform backend，各平台都會引入新知識： 

Future MacOSDisplayBackend:
  Cocoa window / NSView
  maybe Metal texture upload
  maybe CAMetalLayer present
  Objective-C++ bridge
  framework links

Future LinuxDisplayBackend:
  X11 or Wayland window
  maybe EGL/OpenGL/Vulkan present path
  compositor-specific behavior
  package/driver differences

Future WindowsDisplayBackend:
  Win32/GDI baseline
  maybe D3D present path later

Still same boundary:
  Present(const Framebuffer&)
  no SetPixel replacement
  no rasterizer replacement

但短期不是要同時學 Cocoa、Wayland、X11、D3D、Metal。SDL backend 已經可以同時解 window/event/present，並且反過來驗證 DisplayBackend interface 真的有用。

11 backend selection：runtime 與 build-time 都要有邊界

你可以從兩層選 backend：

選擇層 做什麼 例子
Build-time 決定哪些 platform source 被編譯、link 哪些 library。 macOS/Ubuntu preset 編 SDL backend，不編 Win32 backend。
Runtime 在已編進 binary 的 backend 裡建立哪個 object。 CreateDisplayBackend(BackendKind::SDL)。

對現在的專案，先做 build-time selection 會比較簡單。等 build graph 穩定後，再讓 app 支援 runtime selection 也不遲。

std::unique_ptr<DisplayBackend> CreateBackend(BackendKind kind) {
  switch (kind) {
    case BackendKind::SDL:
      return std::make_unique<SDLDisplayBackend>();
    case BackendKind::Win32:
      // only available when Win32 backend target is built
  }
}

12 最小可行 commit sequence

Step Change 驗證
1 把 core source 和 Win32 source 分開。 macOS / Ubuntu build 不會 include windows.h。
2 新增 project-owned Framebuffer。 RenderDevice 不再從 ScreenManager 拿 DIB pointer。
3 接 DisplayBackend interface。 Application 只看 interface。
4 實作 SDLDisplayBackend。 macOS / Ubuntu 能開 window 並 present test pattern。
5 建立 CMake preset / platform-aware build。 Windows Win32、Windows SDL、macOS SDL、Ubuntu SDL 至少概念上可選。
6 VSCode 改讀 CMake/compile database。 IntelliSense 與真實 build graph 一致。

13 常見錯誤清單

  • 把所有 platform source 都 wildcard 掃進同一個 executable。
  • 在 core header 裡 include windows.h 或 SDL header。
  • 把 SDL texture pointer 傳給 RenderDevice 當長期 framebuffer。
  • 為了 Mac/Linux 能編，直接刪掉 Win32 backend，失去 Windows baseline。
  • 只改 VSCode includePath，卻沒有處理 link/runtime。
  • 用 native macOS/Linux backend 當第一步，導致同時學 Cocoa/Metal/X11/Wayland/CMake/SDL/renderer refactor。

14 Summary

本章學到的三件事：
  1. 跨平台開發問題不是換 compiler command，而是把 compiler、linker、debugger、window system、runtime dependency、backend source 邊界分清楚。
  2. 短期最穩路線是 pixel_core + SDLDisplayBackend：macOS / Ubuntu 用 SDL present，Windows 保留 Win32 backend baseline。
  3. CMake preset、SDL dependency、VSCode IntelliSense、GDB/LLDB launch 應該跟同一個 build graph 對齊，不要各自手寫一套世界觀。

下一章會把前面幾章收束成完整工程架構：Rendering Frontend、Software Rendering Backend、Owned Framebuffer、DisplayBackend 到底各自負責什麼，以及 UI / ImGui / shader pipeline 未來應該接在哪一層。
