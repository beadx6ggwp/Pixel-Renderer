Build Workflow：Compiler 到 CMake
先理解一次編譯與一次 debug 怎麼發生，再談 make 與 CMake
compiler VSCode make CMake

前一章 I-02 把 toolchain 拆成 editor、build driver、compiler、linker、library、debugger。這章不要直接跳 CMake。先從最小的 C++ 編譯指令開始，看 VSCode 的 tasks.json 和 launch.json 到底只是包了哪些命令，再看 make 解決什麼問題，最後才談 CMake 為什麼會出現。

如果這個順序反過來，一開始就學 CMake，你會背很多指令，但不知道它們在包裝哪一層。

本章範圍 — 建立工程化 build workflow：g++ driver、preprocess/compile/assemble/link、常用 flags、VSCode task、VSCode launch、GDB/LLDB、makefile dependency workflow、最後過渡到 CMake target graph。這章仍不重講圖學，也不直接改 repo build system。

1 naive model：按下 Run 就是編譯

很多人在 VSCode 裡按下 Run 或 Ctrl+Shift+B，會把這整件事想成： 

VSCode runs my C++ program

這個模型太粗。VSCode 本身不是 compiler，也不是 linker，也不是 debugger。它比較像一個控制台： 

VSCode UI
  |
  +-- tasks.json  -> run build command
  |
  +-- launch.json -> start debugger
  |
  +-- c_cpp_properties.json -> feed IntelliSense

Compiler / linker / debugger are external programs.

所以遇到問題時，第一步不是亂改 VSCode 設定，而是先判斷：錯的是 compile、link、build driver、debugger，還是 IntelliSense。

2 g++ 不是單純 compiler，而是 driver

先把 IDE 拿掉。一個最小 C++ 程式可以這樣編： 

g++ main.cpp -o app.exe

這看起來像「g++ 編譯 main.cpp」。更精準地說，g++ 是 driver：它會依照輸入檔、副檔名和 flags，幫你呼叫 preprocessor、compiler frontend、assembler、linker。 

main.cpp
  |
  | g++ drives the pipeline
  v
preprocessor
  |
  v
compiler frontend
  |
  v
assembler
  |
  v
linker
  |
  v
app.exe

所以 g++ main.cpp -o app.exe 是一個方便的合併指令，不代表 C++ 真的只有一個階段。

3 把 g++ pipeline 拆開看

你可以用不同 flags 停在不同階段：

指令 停在哪裡 產物 你會看到什麼
g++ -E main.cpp -o main.ii preprocess 後 .ii #include 被展開，macro 被替換。
g++ -S main.cpp -o main.s compile 後 .s 產生 assembly。
g++ -c main.cpp -o main.o assemble 後 .o 產生 object file，但還沒有 executable。
g++ main.o -o app.exe link 後 .exe 解析 symbols，接 runtime/libraries。

這跟 T-01 的 translation unit/linkage 模型接起來： 

main.cpp + included headers
  |
  | preprocess
  v
main.ii
  |
  | compile
  v
main.s
  |
  | assemble
  v
main.o
  |
  | link with other .o and libraries
  v
app.exe

如果專案有多個 .cpp，可以分成兩步：

g++ -I. -Isrc -c src/main.cpp -o build/main.o
g++ -I. -Isrc -c src/core/application.cpp -o build/application.o
g++ build/main.o build/application.o -o build/PixelRenderer.exe -lgdi32 -luser32

第一組指令是 compile，產生 object file。最後一行是 link，解析 symbols，接上 libraries，產生 executable。

4 常用 flags 分屬不同階段

很多 flags 看起來都塞在同一行，但它們影響的是不同階段。

Flag 影響階段 意思 目前專案關聯
-I., -Isrc compile / preprocess 增加 header 搜尋路徑。 讓 #include "core/application.h" 找得到。
-Lpath link 增加 library 搜尋路徑。 接第三方 library 時常用。
-lgdi32, -luser32 link 要求 linker 連進某個 library。 目前 Win32/GDI present path 需要。
-g compile / link metadata 產生 debug symbols。 GDB 才能對回 source line。
-O0, -O2 compile optimization 控制最佳化程度。 Debug 通常低最佳化，Release 才開最佳化。
-std=c++17 compile language mode 選 C++ 語言標準。 應該明確寫入 build，而不是只寫在 IntelliSense。
-Wall, -Wextra compile diagnostics 開更多警告。 工程化後應逐步打開。

這也解釋為什麼「include path」不能修 link error。-I 只幫 compiler 找 declarations；linker 找的是 object files 和 libraries 裡的 symbol definitions。

診斷順序

如果 compiler 說找不到 header 或 type，通常是 compile 階段問題。如果 linker 說 undefined reference 或 unresolved external symbol，通常是 object file 或 library 沒有被 link 進來。

5 VSCode task：把 build 指令包起來

目前 .vscode/tasks.json 的核心是：

"command": "make",
"args": ["debug"],
"options": {
  "cwd": "${workspaceFolder}",
  "shell": {
    "executable": "C:\\Program Files\\Git\\bin\\bash.exe"
  }
}

它的意思不是「VSCode 會編譯 C++」。它只是說： 

When user builds:
  cd ${workspaceFolder}
  use Git Bash
  run: make debug

所以 Ctrl+Shift+B 只是啟動 build command。真正知道哪些 .cpp 要編、輸出到哪裡、link 哪些 library 的，是 makefile。

6 VSCode launch：把 debugger 啟動方式包起來

launch.json 則描述 debug 如何啟動：

"program": "${workspaceFolder}/build/debug/PixelRenderer_debug.exe",
"MIMode": "gdb",
"preLaunchTask": "make debug Pixel-Renderer"

這代表按 debug 時，VSCode 會先跑 build task，再用 GDB 啟動產出的 exe。 

F5 / Debug
  |
  v
preLaunchTask
  |
  v
make debug
  |
  v
build/debug/PixelRenderer_debug.exe
  |
  v
GDB launches program and controls breakpoints

Debugger 不是 compiler。GDB/LLDB 需要的是已經編好的 binary，以及能對應 source line 的 debug symbols。這就是為什麼 debug build 會加 -g。

7 GDB 到底控制了什麼

從第一性原理看，debugger 做的事情不是「讀 C++」。它控制的是一個正在執行的 process，並利用 binary 裡的 debug information，把 machine address 對回 source file、line number、function、local variable。 

source code
  |
  | compile with -g
  v
executable + debug symbols
  |
  | launched by GDB
  v
process in memory
  |
  | debugger controls execution
  v
breakpoint / step / inspect stack / inspect variables

所以 GDB 能做幾類基本操作：

操作 GDB 指令 VSCode 對應 底層意義
設 breakpoint break main, break file.cpp:42 點行號紅點 在某個 address 停住 process。
開始執行 run Start Debugging 啟動 program，直到結束或遇到 breakpoint。
繼續執行 continue Continue 從目前停住的位置繼續跑。
單步進入 step Step Into 如果下一行有 function call，進入 function。
單步越過 next Step Over 執行下一行，但不進入 function call。
看 call stack backtrace Call Stack panel 看目前停在哪一串 function call 裡。
看變數 print x Variables / Watch 讀目前 stack frame 或 memory 裡的值。

對 Pixel-Renderer 來說，這些操作會直接用在：

  • 在 RenderDevice::SetPixel 設 breakpoint，看 x、y、color、frame_buffer。
  • 在 Rasterizer::DrawTriangle 裡 step through barycentric 計算，確認 inside test。
  • 在 ScreenManager::UpdateScreen 停住，確認 framebuffer 已寫完才 BitBlt。
  • 在 crash 時看 backtrace，判斷是 out-of-bounds write、null pointer，還是 platform API 失敗。
-g 和 -O0 的差別

-g 是加入 debug symbols，讓 debugger 能把 binary 對回 source。-O0 是降低最佳化，讓執行順序、變數存在位置比較接近 source code。Release build 開 -O2 後，變數可能被最佳化掉、inline 掉、重排，debugger 看到的狀態就不一定像原始碼那麼直覺。
這一節只是入口

GDB/LLDB 本身值得獨立一章。下一章 I-04 會完整拆 process、breakpoint、stack frame、memory inspect、watchpoint，以及如何用它們追 Pixel-Renderer 的 SetPixel、DrawTriangle、UpdateScreen。

8 c_cpp_properties：IntelliSense，不是真正 build

c_cpp_properties.json 目前有：

"includePath": [
  "${workspaceFolder}",
  "${workspaceFolder}/src"
],
"compilerPath": "g++",
"cppStandard": "c++17"

它主要餵給 VSCode C/C++ extension，讓 editor 知道 include path、compiler mode、標準版本。它不決定真正 build。

檔案 控制誰 常見誤解
tasks.json build command 不是 compiler，只是呼叫外部命令。
launch.json debugger 啟動方式 debug 設定錯不代表編譯錯。
c_cpp_properties.json IntelliSense 紅線消失不代表真實 build 成功。

9 make 解決什麼問題

手動打 g++ 可以理解流程，但專案一大就不適合手打。make 解決的是：

  • 把 build command 寫成規則。
  • 把 debug/release 輸出分開。
  • 根據檔案時間戳決定哪些 object 需要重編。
  • 把 link libraries 集中管理。

目前 makefile 的模型是： 

SRCDIRS = . tool src src/core src/render
SOURCES = all *.cpp in those dirs
OBJECTS = build/debug/*.o
LIBS = -lgdi32 -luser32

%.cpp -> %.o
all .o -> PixelRenderer_debug.exe

這對單一 Windows/MinGW 工作流很合理。它的問題不是「make 不好」，而是目前規則沒有描述平台邊界：所有 source 都被掃進同一個 executable，所有 include path 和 libraries 都全域套用。

10 make 工作流的自然極限

當你加入 macOS、SDL、Win32 backend、可能還有 MSVC，makefile 會開始變成一堆條件： 

if Windows:
  use g++.exe or cl.exe?
  compile win32_display_backend.cpp
  link gdi32/user32
  debugger = gdb or msvc debugger

if macOS:
  use clang++
  compile sdl_display_backend.cpp or cocoa backend
  link SDL2 or framework
  debugger = lldb

這時真正需要的不是更多 shell trick，而是一個能描述 target、dependency、platform、compiler、configuration 的 build graph。

11 CMake 出現的位置

現在才輪到 CMake。CMake 不是第一步，也不是魔法。它的角色是產生各平台實際 build system： 

CMakeLists.txt
  |
  | configure
  v
build graph for selected generator
  |
  +-- Ninja / Makefiles
  +-- Visual Studio solution
  +-- Xcode project

then compiler/linker still do the real work

所以 CMake 不取代 compiler/linker/debugger；它只是把「這個專案由哪些 target 組成」描述清楚，然後幫你產生對應平台的 build files。

12 Target-based CMake 解決的是 dependency 邊界

如果直接把 makefile 翻譯成：

include_directories(src external/SDL/include)
link_libraries(gdi32 user32 SDL2)
file(GLOB SOURCES src/**/*.cpp)

你只是把全域污染搬到 CMake。比較好的方向是 target-based：

add_library(pixel_core
  src/core/render_device.cpp
  src/render/rasterizer.cpp
)
target_include_directories(pixel_core PUBLIC src)

add_library(pixel_platform_win32 src/platform/win32/win32_display_backend.cpp)
target_link_libraries(pixel_platform_win32 PRIVATE gdi32 user32)

add_executable(pixel_renderer src/main.cpp)
target_link_libraries(pixel_renderer PRIVATE pixel_core pixel_platform_win32)

這裡的重點是：Win32 libraries 只屬於 Win32 backend，SDL headers 只屬於 SDL backend，core target 不應該知道任何平台 API。

13 循序漸進的重構順序

所以 Pixel-Renderer 的 build 工程化可以分成四步： 

Step 1: understand current workflow
  g++ driver -> object files -> linker -> executable
  VSCode task -> make debug -> g++.exe -> gdi32/user32 -> exe -> GDB

Step 2: make current workflow explicit
  document compile/link/debug commands
  keep makefile as baseline

Step 3: introduce CMake to reproduce current build
  one executable
  current sources
  Win32 libs on Windows

Step 4: use CMake target graph after architecture split
  pixel_core
  pixel_platform_win32
  pixel_platform_sdl
  pixel_renderer

也就是說，CMake 不是替代理解的捷徑。你先理解 compiler、linker、VSCode、GDB、make，CMake 才會變成有意義的抽象。

不要跳過底層 workflow

如果你不知道 tasks.json 呼叫什麼、不知道 launch.json 何時啟動 GDB、不知道 makefile 哪一行在 link gdi32/user32，那直接學 CMake 只會把問題包得更厚。
常見陷阱
  • 把 VSCode 當 compiler，結果分不清 editor error、compile error、link error、debugger error。
  • 把 c_cpp_properties.json 當 build 設定，修了紅線但真實 build 還是失敗。
  • 還沒理解 makefile 的 compile/link 流程，就直接導入 CMake。
  • 用 CMake 全域 include/link 設定重現 makefile 的全域污染。
本章三個 takeaways
  1. g++ 是 driver：它可以一路驅動 preprocess、compile、assemble、link，也能用 -E、-S、-c 停在中間階段。
  2. GDB/LLDB 控制的是執行中的 process；-g 提供 source 對應，breakpoint、step、backtrace、print 都是在操作 process 狀態。
  3. makefile 把 compile/link 規則自動化；CMake 應該在你理解 compiler、linker、debugger、make 之後出現，用來描述 target dependency graph。
