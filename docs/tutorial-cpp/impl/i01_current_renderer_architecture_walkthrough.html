Implementation Track 不應該一開始就跳到 CMake、SDL 或 macOS。你要先知道現在這份 Pixel-Renderer 到底長什麼樣子：誰建立 window？誰 owns framebuffer？SetPixel 寫到哪裡？BitBlt 發生在 frame 的哪一步？如果這些沒有先釐清，後面講「抽象分離」很容易只是在替 class 換名字。

這章的功能是建立一張目前架構地圖。它不是 refactor 教學，也不會把 graphics math 重新推導一次；它只回答一件事：以目前 code 來看，資料、ownership、platform dependency 是怎麼流動的。

本章範圍 — 讀 main.cpp、Application、ScreenManager、RenderDevice、Rasterizer 的關係。重點是 frame lifecycle、CPU framebuffer、Win32 present path、header dependency，以及下一步為什麼要拆成 owned Framebuffer + DisplayBackend。

1 naive model：Rasterizer 直接畫到螢幕

剛開始看 software renderer，很容易有這個直覺： 

DrawLine()
DrawTriangle()
SetPixel()
  |
  v
screen

這個模型太短。它把「算出哪些 pixel 要亮」和「把一塊 CPU memory 顯示到 OS window」混成同一件事。以目前專案來看，真正的資料流比較像： 

User demo code
  |
  | calls Rasterizer::DrawLine / DrawTriangle
  v
Rasterizer
  |
  | calls RenderDevice::SetPixel(x, y, color)
  v
RenderDevice
  |
  | writes CPU bytes through frame_buffer pointer
  v
ScreenManager-owned DIBSection memory
  |
  | ScreenManager::UpdateScreen()
  | BitBlt(window_dc, ..., mem_dc, ...)
  v
Win32 window / OS compositor / display
第一個判斷點 — Rasterizer 是 geometry-to-pixel layer，不是 display layer。ScreenManager::UpdateScreen() 才是目前的 present path。

2 現在的 object graph

目前 Application 是 orchestrator。它 owns ScreenManager，用 raw owning pointer 建立 RenderDevice 和 Rasterizer：

SVG：目前 object graph。藍色實線是 ownership，黃色虛線是 borrowing relation。
Object 目前責任 語意問題
Application 控制 frame loop，owns ScreenManager、RenderDevice、Rasterizer。 device / rasterizer 是 raw owning pointer；可讀性不如 std::unique_ptr 或 value member。
ScreenManager Win32 window、DIBSection framebuffer、input、message pump、present。 責任過重；platform resource 和 renderer target ownership 混在一起。
RenderDevice Clear、SetPixel，用 FramebufferConfig 借來的 pointer 寫 pixels。 行為接近 renderer core，但 storage lifetime 仍由 Win32 object 間接提供。
Rasterizer Bresenham line、triangle scan conversion。 目前依賴 RenderDevice*，這是合理的 observer relation，但型別沒有明說 non-null / non-owning。

3 每一 frame 的生命週期

現在 frame loop 的控制權在 Application::Run()： 

Application::Run()
  |
  | QueryPerformanceCounter()
  | if delta_time >= target_frame_time:
  v
screen.DispatchEvents()
  |
  v
OnUpdate(delta_time)
  |
  v
OnRender()
  |
  | user/demo code calls device/rasterizer
  v
CPU framebuffer is modified
  |
  v
screen.UpdateScreen()
  |
  | BitBlt()
  v
window presents copied image
  |
  v
SetWindowTitle(FPS / dt / frame)

這裡要注意兩個不同的 lifecycle：

  • Program/resource lifecycle：Application constructor 建資源，destructor 釋放資源。
  • Frame lifecycle：每 frame poll event、update state、render pixels、present pixels。
分離架構時最容易搞錯的點 — 不要把 frame lifecycle 的 Present 誤當成 resource lifecycle 的 owner。Present 應該只是使用一份已經存在的 framebuffer；它不應該偷偷決定 renderer target 的 lifetime。

4 CPU framebuffer 實際怎麼被寫入

目前 RenderDevice::SetPixel 的核心語意是：

void RenderDevice::SetPixel(int x, int y, uint32_t color) {
  if (x >= 0 && x < width && y >= 0 && y < height) {
    *(uint32_t*)(frame_buffer + y * pitch + x * 4) = color;
  }
}

這句話不是「呼叫 OS 畫點」，而是單純的 CPU store： 

byte address = frame_buffer + y * pitch + x * 4
write 4 bytes = color

required invariants:
  0 <= x < width
  0 <= y < height
  pitch >= width * 4
  frame_buffer points to writable memory
  pixel format agrees with present path

這也是後面測試驗證要抓住的第一個核心：software renderer 的可測性來自它先產生一份 deterministic CPU memory。只要能穩定檢查這份 memory，就不必每次都靠肉眼看 window。

5 現在為什麼還不是完整前後端分離

目前確實已經有一點分離：Rasterizer 不直接呼叫 Win32 API，RenderDevice 也只看 FramebufferConfig。但還沒完全分離，原因是核心面向仍被 ScreenManager 綁住：

分離層次 目前狀態 下一步
Rasterizer vs Win32 drawing API 已分離。DrawLine / DrawTriangle 沒有呼叫 BitBlt。 維持。
RenderDevice vs framebuffer owner 只分離一半。RenderDevice 借 pointer，但 pointer 來自 Win32 DIBSection。 抽出 owned Framebuffer。
Core headers vs platform headers screen_manager.h 直接 #include <windows.h>。 讓 Win32 type 留在 Win32 backend 的 private implementation。
Present path replacement 目前只有 ScreenManager::UpdateScreen()。 抽出 DisplayBackend::Present(const Framebuffer&)。
比較精準的現況描述是：Pixel-Renderer 已經把 rasterization logic 和 Win32 present call 隔開，但還沒有把 CPU framebuffer ownership、window/event/present backend、core header dependency 完全拆開。

6 後續章節如何接上

這章是 Implementation Track 的入口，後面章節的順序不是任意的： 

I-01  Current Architecture
  |
  +-- I-02..I-07 先處理 compiler / debugger / build graph
  |
  +-- I-08      抽出 owned Framebuffer
  |
  +-- I-09      定義 DisplayBackend interface
  |
  +-- I-10      Win32 backend 只是其中一個 Present implementation
  |
  +-- I-11      SDL backend 也是 Present implementation
  |
  +-- I-16      用 golden image / regression test 驗證 renderer core 沒壞
  |
  +-- I-17      用 header boundary / dependency hygiene 防止平台污染回流

換句話說，抽象不是「先想一個漂亮 class hierarchy」。抽象來自可檢查的切割：

  • render core 能不能在沒有 Win32 window 的情況下跑測試？
  • macOS / Ubuntu 能不能編譯 core，而不看到 windows.h？
  • Win32 backend 和 SDL backend 能不能吃同一份 Framebuffer？
  • 改 rasterizer 後，golden image 能不能告訴你畫面是否變了？
本章學到的三件事：
  1. 目前資料流是 Rasterizer -> RenderDevice -> ScreenManager-owned DIBSection -> BitBlt，不是 rasterizer 直接畫到螢幕。
  2. 現在已有初步分離，但 framebuffer ownership 和 Win32 present path 仍綁在 ScreenManager。
  3. 真正的下一步不是換 SDL 畫線，而是抽出 owned Framebuffer、DisplayBackend，再用測試驗證這條邊界。
