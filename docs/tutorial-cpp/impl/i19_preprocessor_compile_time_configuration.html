Preprocessor / Compile-time Configuration
跨平台不是到處塞 ifdef，而是把 compile-time 差異限制在正確邊界
preprocessor macro platform cmake

I-07 已經把 CMake target graph 建起來，但 C++ 還有一個更早發生的階段：preprocessor。它在 compiler 真正解析 C++ 語法之前，先處理 #include、macro expansion、conditional compilation。這也是跨平台專案最容易失控的地方。

你之後會想支援 Windows / macOS / Ubuntu，可能還會加 SDL、OpenGL、D3D、ImGui。直覺做法是到處加 #ifdef _WIN32。短期有效，但如果這些條件散進 renderer core，frontend/backend 分離就會被 compile-time 分支污染。

本章範圍 — 釐清 preprocessor 的位置：include、include guard、#pragma once、object-like/function-like macro、platform macro、feature macro、conditional compilation、CMake target_compile_definitions。目標是讓 platform/backend 選擇停在 platform layer，不污染 renderer core。

1 naive model：ifdef 可以解決跨平台

最容易寫成：

#ifdef _WIN32
  BitBlt(...);
#else
  SDL_RenderPresent(...);
#endif

如果這段只存在 main.cpp 或 backend factory，問題不大。如果它出現在 RenderDevice、Rasterizer、math type、framebuffer storage，就代表 core 的語意被平台編譯條件切碎。

核心規則 — #ifdef 應該出現在 compile-time boundary，不應到處出現在 domain logic。Renderer core 不應需要知道自己正在 Windows、macOS 還是 Ubuntu 上編譯。

2 Preprocessor 在哪一層發生

SVG：preprocessor 發生在 C++ 語意檢查之前。這就是為什麼 platform macro 失控會讓某些 path 長期沒有被編譯到。

3 include guard 與 pragma once

Header 會被多個 translation unit include，也可能透過 include chain 被重複看到。include guard 的目的不是加速，而是避免同一個 header 內容在同一個 translation unit 被重複展開。

#ifndef PIXEL_RENDERER_FRAMEBUFFER_H
#define PIXEL_RENDERER_FRAMEBUFFER_H

class Framebuffer { ... };

#endif

#pragma once 在主流 compiler 上可用、簡潔，但不是 C++ standard。對這個專案來說兩者都能接受；重要的是 header boundary 不應 include 不需要的 platform header。

4 Macro 的兩種風險

Macro 是 textual substitution，不尊重 scope、type system、namespace。它的風險分兩種：

Macro 類型用途風險
object-like macro#define WIDTH 800沒有型別、沒有 scope，容易撞名。
function-like macro#define MIN(a,b)重複 evaluate、缺型別檢查。
configuration macroPIXEL_USE_SDL若散落到 core，build matrix 變難測。
替代規則 — 常數優先用 constexpr，小函式優先用 inline/function/template，configuration macro 儘量只放在 build boundary 或 platform boundary。

5 Platform macro 與 feature macro

平台 macro 通常由 compiler/toolchain 定義，例如： 

_WIN32        Windows
__APPLE__     Apple platform
__linux__     Linux
__clang__     Clang
__GNUC__      GCC-compatible compiler
_MSC_VER      MSVC

專案自己的 feature macro 應該更語意化，例如： 

PIXEL_RENDERER_USE_WIN32_BACKEND
PIXEL_RENDERER_USE_SDL_BACKEND
PIXEL_RENDERER_ENABLE_ASSERTS
PIXEL_RENDERER_ENABLE_SANITIZER_NOTES

重點是：core code 不應到處問 _WIN32。比較好的做法是 CMake 根據平台和 option 決定 backend target，然後只在 backend source 或 factory 裡做選擇。

6 target_compile_definitions：把 macro 綁到 target

不要用全域 add_definitions() 把 macro 灑到所有 target。用 target-based definition：

target_compile_definitions(renderer_platform_sdl
  PRIVATE PIXEL_RENDERER_USE_SDL_BACKEND=1
)

target_compile_definitions(renderer_core
  PRIVATE PIXEL_RENDERER_ENABLE_ASSERTS=1
)

這會讓 macro 跟 target boundary 對齊：SDL backend 知道自己是 SDL backend；renderer core 不需要知道 SDL header 或 SDL macro。

7 compile-time selection vs runtime polymorphism

Backend 選擇有兩層：

選擇層問題工具
Compile-time 這個平台/配置要不要編 SDL backend source？ CMake option、platform detection、target source selection。
Runtime Application 持有哪個 backend instance？ std::unique_ptr<DisplayBackend>、factory。 
CMake:
  decides which backend target is built

Application:
  owns DisplayBackend interface

Renderer core:
  sees only Framebuffer and DisplayBackend contract
和 T-17 / T-16 的連接 — Compile-time selection 決定哪些 code 被編譯；runtime ownership 決定誰 owns backend object；API boundary 決定 renderer core 能看見什麼。

8 Pixel-Renderer preprocessor policy

Core:
  no platform #ifdef in RenderDevice / Rasterizer / math
  no windows.h / SDL.h through public core headers
  use constexpr / enum class / typed config where possible

Backend:
  platform macro allowed inside backend implementation
  SDL/Win32 specific compile definitions are PRIVATE

Factory / app layer:
  may choose backend based on build option
  keeps conditional compilation localized

CMake:
  uses options and target_compile_definitions
  avoids global add_definitions for platform selection
本章學到的三件事：
  1. Preprocessor 在 C++ type checking 前做 text-level selection；因此 macro 邊界失控會讓某些平台 path 長期沒有被編譯或檢查。
  2. #ifdef 應該留在 build/platform/factory boundary，不應散落到 renderer core 的 domain logic。
  3. CMake 的 target_compile_definitions 應該把 feature macro 綁到需要它的 target，避免 Win32/SDL configuration 污染全專案。
