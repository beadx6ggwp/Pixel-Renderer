IMPLEMENTATION TRACK · I-16
Verification / Golden Image / Regression Testing
software renderer 的價值不只是能畫，而是能被穩定檢查
testing golden-image determinism regression

當 renderer 還只有一條線、一個三角形時，靠眼睛看 window 似乎就夠了。但只要你開始改 Bresenham、edge function、z-buffer、shader interface、UI draw command、SDL backend，就會遇到同一個問題：畫面變了，到底是 bug、預期改動，還是平台 present path 的差異？

這章補上的不是「測試框架速查」，而是一個 renderer project 必須有的思考層：rendering result 要如何變成可比較、可回歸、可定位的 artifact。

本章範圍 — 建立 Pixel-Renderer 的 verification strategy：unit tests 檢查小規則，framebuffer tests 檢查 CPU pixels，golden image tests 檢查整張結果，debug dump 協助定位差異。重點放在 software renderer core，不依賴 Win32/SDL window 才能測。

1 naive model：畫面看起來對就是對

最直覺的測試方式是： 

run program
look at window
if triangle appears:
  good

這個模型很適合第一天確認程式有跑起來，但它很快會失效：

  • 一個像素偏一格，肉眼不一定看得出來。
  • 重構 Framebuffer ownership 後，window 仍然可能顯示正常，但 memory layout 已經錯了。
  • SDL / Win32 / macOS present path 可能有色彩格式或 pitch 差異。
  • triangle edge rule 改掉後，邊界 pixel 的差異要能被精準定位。
  • CI 或無視窗環境不能靠人工看畫面。
測試的第一性問題 — 你要測的不是「window 是否看起來像」，而是「renderer core 是否對同一份 input 產生同一份符合規格的 framebuffer」。window 是觀察手段，不是判準本身。

2 分層測試地圖

SVG：testing 應該從 renderer core 往外長。越往下越接近平台，越不適合當作主要 correctness oracle。
層級 測什麼 不該測什麼
Unit-level 小函式規則：bounds、edge function、color conversion、matrix helper。 不要開 window，不要依賴 OS event loop。
Framebuffer-level 在 CPU buffer 裡畫固定 primitive，檢查某些 pixel 或統計量。 不要把 BitBlt / SDL_RenderPresent 放進 correctness 判斷。
Golden image 固定 scene 的完整 framebuffer 影像。 不要把每次視覺變動都默默更新 golden。
Platform smoke window 是否能建立、present 是否能走完、事件是否能關閉。 不要用它判斷 rasterizer math 是否正確。

3 Golden image 的基本模型

Golden image test 的核心流程是： 

test scene input
  |
  v
headless renderer core
  |
  v
CPU framebuffer
  |
  +-- write actual image/debug dump
  |
  v
compare with golden artifact
  |
  +-- exact match for integer deterministic paths
  +-- tolerance match for floating-point or platform-sensitive paths

對目前 Pixel-Renderer 的第一版來說，最好先做 exact match，因為目前主要是 CPU rasterizer、32-bit color、固定尺寸、沒有 texture filtering、沒有 MSAA，也還沒有跨 SIMD/FMA 差異。當你進入 perspective-correct interpolation、floating depth、不同 compiler optimization 後，再引入 tolerance。

不要太早把 tolerance 當遮羞布 — 如果現在只是 SetPixel、line、flat-color triangle，結果應該高度 deterministic。這時 tolerance 太寬，會把 off-by-one、pitch 錯誤、color channel 交換都藏起來。

4 第一版測試應該長什麼樣子

等 I-08 的 owned Framebuffer 落地後，可以先做一個完全不開 window 的 test path：

Framebuffer fb(64, 64);
RenderDevice device(fb.View());
Rasterizer rasterizer(&device);

device.Clear(0x00000000);
rasterizer.DrawLine(0, 0, 63, 63, 0xffffffff);

expect_pixel(fb, 0, 0, 0xffffffff);
expect_pixel(fb, 63, 63, 0xffffffff);
expect_pixel_count(fb, 0xffffffff, 64);

這個測試很小，但它檢查了三個重要 invariant：

  • Framebuffer 可以在沒有 ScreenManager 的情況下存在。
  • RenderDevice 寫的是 project-owned CPU memory。
  • Rasterizer 的輸出可以被程式檢查，而不是只能看 window。

5 Debug dump：失敗時要能定位

Golden image test 最常見的壞味道是：它只告訴你「不一樣」，但不告訴你哪裡不一樣。第一版至少要輸出： 

artifacts/
  actual.ppm
  expected.ppm
  diff.ppm
  diff.txt

diff.txt:
  size: 800x600
  mismatched pixels: 124
  first mismatch: x=321 y=240
  expected: 0xff00ff00
  actual:   0xff000000

PPM 是很適合早期 renderer 的格式，因為它簡單、可自己寫、沒有 PNG encoder dependency。等工具鏈穩定後，再考慮 PNG。

工程判準 — 測試失敗時，artifact 要能回答「哪個 pixel、哪個顏色、差多少、是哪張 scene」。如果只得到 failed，debugger 還是會被迫回到人工猜測。

6 測試與抽象分離的關係

測試不是 refactor 之後才補的東西。它反過來定義什麼叫分離成功：

架構目標 對應測試 代表的工程意義
Renderer core 不依賴 Win32 headless framebuffer test 可在 macOS / Ubuntu build。 core header 沒有平台污染。
Win32 / SDL 只替換 present path 同一份 golden framebuffer 可交給不同 backend present。 display backend 沒有改變 rendering result。
UI / shader interface 可演進 固定 UI draw list 或 shader input 可以產生可比較 output。 未來 refactor 有 regression net。

7 建議落地順序

  1. 先讓 Framebuffer 成為 project-owned std::vector<uint32_t>。
  2. 讓 RenderDevice 接受 FramebufferView 或 Framebuffer&。
  3. 新增 headless test executable，不開 window。
  4. 先測 Clear、SetPixel、line，再測 triangle。
  5. 加入 PPM writer 和 golden compare。
  6. 最後才做 Win32 / SDL smoke test。
本章學到的三件事：
  1. software renderer 的 correctness oracle 應該是 deterministic CPU framebuffer，不是肉眼看到的 window。
  2. golden image test 要搭配 exact/tolerance policy 和 debug dump，否則只會變成難用的 failed 訊息。
  3. 測試本身能驗證抽象是否真的分離：core 能 headless 測、backend 只 present、platform header 不污染 renderer。
