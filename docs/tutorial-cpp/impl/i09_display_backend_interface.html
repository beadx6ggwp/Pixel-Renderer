IMPLEMENTATION TRACK · I-09
DisplayBackend Interface
backend interface 應該抽 present/input/window，不應抽 rasterization
interface backend lifetime

I-08 已經把最重要的 ownership 切開：renderer core owns CPU framebuffer，DisplayBackend 只 present final pixels。現在問題變成：這個 backend interface 應該長什麼樣？

介面不是把所有平台差異都藏起來就好。好的 interface 要剛好描述替換點：Win32、SDL、macOS 會不同的是 window、event/input、present path；不應該不同的是 triangle rasterization、SetPixel、depth test、shader interface。

本章範圍 — 從目前 ScreenManager 的責任拆出 DisplayBackend interface：Init、Shutdown、PollEvents、ShouldClose、input query、Present(const Framebuffer&)。同時說明什麼不該放進 interface、如何用 unique_ptr<DisplayBackend> 管 lifetime、如何避免 backend 反向 owns framebuffer，以及 Win32/SDL/macOS 實作會在哪裡分歧。

1 naive model：把 ScreenManager 改名成 DisplayBackend

最容易做錯的 refactor 是只改名字： 

class ScreenManager
  -> class DisplayBackend

如果內部仍然 owns framebuffer，仍然把 GetFrameBuffer() 給 RenderDevice，那只是命名變漂亮，架構沒有真的分離。 

bad interface:
  unsigned char* GetFrameBuffer()
  int GetPitch()
  void UpdateScreen()

why bad:
  renderer still writes backend-owned memory
  backend still decides framebuffer layout
  SDL/macOS replacement still affects RenderDevice
interface 不能保留錯誤 ownership — 只要 interface 還回傳 writable framebuffer pointer 給 renderer core，DisplayBackend 就不只是 display backend，它仍然是 render target owner。

2 從 ScreenManager 拆責任

目前 ScreenManager 同時做五件事： 

ScreenManager
  window creation/destruction
  input state
  event pump
  DIBSection resource
  framebuffer ownership
  BitBlt present

重構後應該拆成：

責任 新位置 理由
CPU framebuffer ownership Framebuffer renderer core 的 render target，不屬於平台層。
pixel write RenderDevice software renderer 的底層 write API。
window lifecycle DisplayBackend implementation Win32/SDL/macOS 各自不同。
event/input polling DisplayBackend 或 InputState 平台 event source 不同，但上層要共用 input query。
present final pixels DisplayBackend::Present Win32 BitBlt、SDL_UpdateTexture、macOS upload 的替換點。

3 interface sketch

第一版可以先設計得保守：

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;

  virtual bool Init(int width, int height, const char* title) = 0;
  virtual void Shutdown() = 0;

  virtual void PollEvents() = 0;
  virtual bool ShouldClose() const = 0;
  virtual bool IsKeyDown(KeyCode key) const = 0;

  virtual void Present(const Framebuffer& framebuffer) = 0;
};

這裡最重要的是最後一行：Present 接收 const Framebuffer&。這表示 backend 只能讀取 final pixels，不能 owns 或修改 renderer target。

核心 interface invariant — DisplayBackend owns platform resources；Framebuffer owns CPU pixels；RenderDevice writes Framebuffer；DisplayBackend only presents Framebuffer.

4 什麼不該放進 DisplayBackend

一個 interface 的清楚程度，常取決於它拒絕放什麼。

不該放 原因 應該在哪
SetPixel pixel write 是 renderer core，不是 display。 RenderDevice
DrawLine, DrawTriangle rasterization 不應被 SDL/Win32 替換。 Rasterizer
GetFrameBuffer 會讓 renderer 寫 backend-owned memory。 Framebuffer owns, RenderDevice borrows view。
shader execution software shader pipeline 是 renderer feature。 renderer/shader module。
UI drawing API UI 應輸出 draw commands，再由 software renderer 畫到 framebuffer。 UI renderer / command execution。
第二個修正 — DisplayBackend 是 present/input/window abstraction，不是 graphics API abstraction。它不是 SDL_Renderer wrapper，也不是 GPU RHI。

5 Application composition root

Application 應該在最外層選 backend，然後把 renderer core 和 display backend 組起來： 

Application
  owns backend: unique_ptr
  owns framebuffer
  owns render device
  owns rasterizer

Run loop:
  backend->PollEvents()
  OnUpdate(dt)
  framebuffer.Clear()
  OnRender()
  backend->Present(framebuffer)

概念 code：

std::unique_ptr<DisplayBackend> backend;
Framebuffer framebuffer(width, height);
RenderDevice device(framebuffer.View());
Rasterizer rasterizer(&device);

backend = std::make_unique<Win32DisplayBackend>();
backend->Init(width, height, "Pixel Renderer");

unique_ptr<DisplayBackend> 解決 ownership；virtual 解決 runtime dispatch。兩者常一起出現，但不是同一件事。

6 Win32 / SDL / macOS 實作差異

Win32DisplayBackend::Present(fb)
  copy fb pixels into DIBSection
  BitBlt(mem_dc -> window_dc)

SDLDisplayBackend::Present(fb)
  SDL_UpdateTexture(texture, nullptr, fb.Data(), fb.PitchBytes())
  SDL_RenderCopy / SDL_RenderTexture
  SDL_RenderPresent

MacOSDisplayBackend::Present(fb)
  upload/copy pixels to native surface/texture
  submit to compositor

三者的共同點是：都不決定 triangle 怎麼 rasterize，也不決定 SetPixel 寫哪裡。它們只負責把已完成的 pixels 交給平台。

7 input state：直接 query 還是獨立 InputState？

第一版可以讓 backend 提供 IsKeyDown，和目前 ScreenManager 接近。但長期更乾淨的方式是讓 backend 更新一個 platform-independent InputState。 

Option A:
  backend->IsKeyDown(KeyCode::Escape)

Option B:
  backend->PollEvents(input_state)
  app reads input_state

first version:
  Option A is simpler

future UI/game logic:
  Option B separates event source from app state

重點是不要讓上層直接依賴 VK_ESCAPE 或 SDL scancode。要先建立自己的 KeyCode 或 input abstraction。

8 virtual interface vs template/static dispatch

Display backend 每 frame 只呼叫少數幾次：poll events、present。這不是 rasterizer hot loop。使用 virtual interface 的成本通常不是瓶頸，換來的是簡單的 runtime backend selection。

方式 適合 在這裡的判斷
virtual interface runtime replacement，少量 calls。 DisplayBackend 適合。
template/static polymorphism hot loop，compile-time policy。 pixel shader/math policy 之後再考慮。
function pointer table C-style ABI/plugin boundary。 目前不需要。
和 T-11 的連接 — virtual、type erasure、variant、template dispatch 都是 polymorphism 工具。DisplayBackend 先用 virtual 是因為它在 frame boundary，不在 pixel hot loop。

9 CMake target boundary

I-07 的 target graph 在這裡應該長成： 

pixel_core
  Framebuffer
  RenderDevice
  Rasterizer
  DisplayBackend interface header
  no windows.h / SDL.h

pixel_platform_win32
  Win32DisplayBackend
  includes windows.h
  links gdi32 user32

pixel_platform_sdl
  SDLDisplayBackend
  includes SDL.h
  links SDL2

pixel_app
  composition root
  links pixel_core + selected backend

如果 pixel_core 需要 link gdi32 或 include SDL.h，就表示 interface boundary 破了。

10 verification checklist

  • DisplayBackend 是否沒有 GetFrameBuffer()？
  • Present 是否接 const Framebuffer&？
  • RenderDevice 是否不 include Win32/SDL/macOS headers？
  • Application 是否 owns backend lifetime，而不是 backend owns renderer state？
  • Win32/SDL/macOS backend 是否只在 platform target 裡編譯？
  • input key code 是否有平台無關表示法？
  • 同一個 framebuffer test pattern 是否能被不同 backend present？
本章學到的三件事：
  1. DisplayBackend 抽的是 window/event/present，不抽 SetPixel、rasterizer、shader 或 UI drawing。
  2. 正確 interface boundary 是 Present(const Framebuffer&)：backend owns platform resources，renderer owns CPU pixels。
  3. unique_ptr<DisplayBackend> 管 lifetime，virtual 管 runtime dispatch；CMake target boundary 應保證 core 不依賴 Win32/SDL/macOS。
