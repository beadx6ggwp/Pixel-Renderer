Toolchain Map
把 compiler、linker、build system、editor、debugger 分清楚，才談得上跨平台工程化
toolchain MinGW Apple-Clang

這章不是圖學教學，也不推導任何 rasterization。它只處理一件事：Pixel-Renderer 現在為什麼是 Windows/MinGW 工作流，以及要讓 macOS 也能開發時，哪些東西必須被拆清楚。

你目前看到的不是單一「編譯器問題」，而是一整條 toolchain pipeline：VSCode 呼叫 task，task 呼叫 make，make 呼叫 g++.exe，compiler 產生 object files，linker 連進 gdi32/user32，debugger 用 GDB 啟動 .exe。

本章範圍 — 讀目前專案的 makefile 與 .vscode 設定，建立 Windows MinGW、MSVC、Apple Clang 的工程地圖。這章只講工程化：build、link、debug、IntelliSense、platform-specific source。圖學推導留給 rendering notes。

1 naive model：換 compiler 就能跨平台

最直覺的想法是： 

Windows: g++.exe main.cpp
macOS:   clang++ main.cpp

如果語法都是 C++，應該換 compiler 就能跑。

這對目前專案不成立。原因不是 C++ 語法，而是工程依賴：

  • screen_manager.h include windows.h。
  • screen_manager.cpp 使用 CreateWindow、CreateDIBSection、BitBlt、PeekMessage。
  • application.cpp 使用 QueryPerformanceCounter 與 VK_ESCAPE。
  • makefile link -lgdi32 -luser32。
  • launch.json 用 GDB 啟動 Windows .exe。

所以 macOS 的問題不是「Apple Clang 不會 C++」，而是它沒有 Win32 SDK、GDI library、Windows message loop 與 .exe/GDB 這套 runtime 假設。

2 目前 Pixel-Renderer 的 toolchain

目前 repo root 有 makefile，VSCode 的 tasks.json 會用 Git Bash 執行 make debug。 

Ctrl+Shift+B
  |
  v
VSCode tasks.json
  |
  | command: make debug
  | shell: C:\Program Files\Git\bin\bash.exe
  v
makefile
  |
  | CXX = g++.exe
  | CXXFLAGS = -fdiagnostics-color=always
  | INCLUDES = -I. -Itool -Isrc
  | LIBS = -lgdi32 -luser32
  v
MinGW GCC compile + link
  |
  v
build/debug/PixelRenderer_debug.exe

Debug 時，launch.json 再用 cppdbg + MIMode: gdb 啟動這個 exe。也就是：

層 目前工具 責任
Editor VSCode 按鍵、task、debug UI、IntelliSense。
Build driver make + Git Bash 決定哪些 source 編譯、輸出到哪裡、link 哪些 library。
Compiler MinGW g++.exe 把每個 translation unit 編成 object file。
Linker MinGW linker 解析 symbols，連進 gdi32/user32。
Debugger GDB 啟動 exe、設 breakpoint、讀 debug symbols。
注意

c_cpp_properties.json 主要餵給 IntelliSense。它讓 VSCode 比較會補字、少報紅線，但它不是實際 build graph。真正編譯哪些檔案、include 哪些 path、link 哪些 library，是 makefile 決定的。

3 三種 Windows/macOS 工具鏈不是同一種東西

工具鏈 平台直覺 object/library 世界 debugger 適合用途
MinGW GCC Windows 上的 GCC-like workflow GNU/MinGW runtime，link gdi32/user32 GDB 延續目前 makefile 與 VSCode 設定。
MSVC Windows 原生 Visual Studio toolchain MSVC ABI、.lib、Windows SDK MSVC debugger Windows 原生工程、Visual Studio/CMake 整合。
Apple Clang macOS 原生 compiler Mach-O、Apple SDK、libc++ LLDB macOS 開發；不能直接編 Win32 source。

這三者都能編 C++，但它們不是同一個 binary ecosystem。T-01 說過：source API 和 binary ABI 是不同層次。跨平台工程的核心，是讓 shared core source 不依賴某個平台的 SDK，然後每個平台只編自己的 backend。

4 目前 Win32-only dependency 在哪裡

以現在架構，Win32 dependency 沒有被關在一個 implementation file 裡。 

src/core/screen_manager.h
  includes windows.h
  exposes HWND / HDC / HBITMAP as private members

src/core/screen_manager.cpp
  calls RegisterClass / CreateWindow / CreateDIBSection / BitBlt

src/core/application.cpp
  calls QueryPerformanceCounter
  checks VK_ESCAPE

makefile
  LIBS = -lgdi32 -luser32

這代表只要 macOS 嘗試編 application.cpp 或 include screen_manager.h，就會在 compile 階段失敗；即使先避開 header，也會在 link 階段找不到 Win32 symbols。

不要用 include path 解平台問題

如果錯誤是 windows.h not found，在 macOS 上不是應該去找一份假的 windows.h。真正問題是 core source 不該依賴 Win32 header。這要靠 source split 與 backend interface 解，不是靠 VSCode include path。

5 工程化方向：先分 source，再換 backend

第一個工程目標不是「讓 SDL 畫三角形」。第一個目標是讓 core 能在不同 platform compiler 下被編譯。 

Shared core target
==================
src/core/application.cpp       no windows.h
src/core/render_device.cpp     no windows.h
src/render/rasterizer.cpp      no windows.h
src/core/framebuffer.cpp       owns CPU pixels

Platform target: Windows
========================
src/platform/win32/win32_display_backend.cpp
  includes windows.h
  links gdi32/user32

Platform target: SDL
====================
src/platform/sdl/sdl_display_backend.cpp
  includes SDL.h
  links SDL2

Application chooses one backend
===============================
DisplayBackend* backend = CreatePlatformBackend();

這樣 macOS 可以先編 shared core + SDL backend；Windows 可以編 shared core + Win32 backend 或 SDL backend。這才是 engineering abstraction，而不是把 renderer 改成 SDL renderer。

6 Makefile 到 CMake 的理由

目前 makefile 對單一 Windows/MinGW workflow 很直接。但跨平台後，它會開始吃力：

  • 不同平台要選不同 source files。
  • 不同 backend 要 link 不同 libraries。
  • MSVC、MinGW、Apple Clang 的 compiler flags 不同。
  • VSCode、command line、CI 最好共享同一份 build graph。

這是後面 I-03 要引入 CMake 的原因。CMake 不是因為「業界都用」，而是因為你需要描述 target、source、include path、compile definitions、link libraries 的關係。

# future shape, not current code
add_library(pixel_core ...)
add_executable(pixel_renderer ...)

target_link_libraries(pixel_renderer PRIVATE pixel_core)

if(WIN32)
  target_sources(pixel_renderer PRIVATE win32_display_backend.cpp)
  target_link_libraries(pixel_renderer PRIVATE gdi32 user32)
endif()

7 第三方 graphics library 怎麼接進工程

實際工程裡接 OpenGL、D3D、SDL、ImGui、Vulkan 這類 library，不是「include 一個 header」就結束。你至少要分清楚五件事： 

Using a library usually means:

1. headers
   compiler needs declarations

2. link libraries
   linker needs symbols / import libraries

3. runtime binaries
   program needs DLL / dylib / framework at run time

4. platform SDK or loader
   system API, function loader, generated bindings

5. build-system ownership
   CMake target / package manager / source vendoring

如果只修 include path，compiler 可能不報錯，但 linker 仍然會炸；如果 linker 成功，runtime 也可能因為找不到 DLL 或 framework 而啟動失敗。

Library/API 常見接入方式 工程注意點
OpenGL Windows link opengl32；modern functions 通常還需要 GLAD/GLEW loader。 opengl32 只保證老 API entry points；真正使用現代 OpenGL 時，function pointer loading 是額外工程步驟。
D3D11 / D3D12 使用 Windows SDK headers，link d3d11、dxgi、d3dcompiler 或 D3D12 相關 libs。 這是 Windows platform graphics API；MSVC/CMake 整合通常順，但 MinGW 路線要特別確認 headers/libs 支援。
SDL2 / SDL3 include SDL headers，link SDL library，runtime 需要對應 DLL/dylib/framework。 適合當 cross-platform platform layer；在本專案裡應作為 display/input backend，不是取代 software rasterizer core。
Vulkan 使用 Vulkan SDK headers/libs，加上 validation layers、loader。 比 OpenGL/D3D 更明確要求你管理 instance/device/swapchain；工程設定比教學初期需要的更重。
ImGui 通常直接 vendored source，並接一個 backend：SDL+OpenGL、SDL+Vulkan、Win32+D3D 等。 ImGui 本體產生 UI draw data；真正 present/render 仍由 backend 負責。

所以如果未來 Pixel-Renderer 接 SDL 或 ImGui，build system 最好把 dependency 寫成 target relationship，而不是散落在全域 include path 和全域 linker flags：

# future shape, not current code
find_package(SDL2 CONFIG REQUIRED)

add_library(pixel_platform_sdl sdl_display_backend.cpp)
target_link_libraries(pixel_platform_sdl PRIVATE SDL2::SDL2)
target_link_libraries(pixel_renderer PRIVATE pixel_core pixel_platform_sdl)

OpenGL/D3D 也是同樣原則：不要讓 RenderDevice 或 Rasterizer 到處 include graphics API headers。把它們關在 backend target 裡，core 只依賴你自己定義的 interface。

這和目前 software renderer 目標不衝突

即使未來工程裡接 OpenGL、D3D 或 Vulkan，它們也可以只是 window/swapchain/present 或 debug visualization 的 backend。只要 triangle setup、rasterization、depth/interpolation 仍由 CPU renderer core 執行，你就沒有把學習核心交給外部 API。

8 VSCode 設定要拆成三件事看

檔案 現在用途 常見誤解
tasks.json 描述按 build 時要執行什麼 command。 它不是 compiler 本身，只是呼叫 make/CMake。
launch.json 描述 debugger 如何啟動 program。 debugger 設定錯不代表 build 錯。
c_cpp_properties.json 餵 IntelliSense include path、compiler path、standard。 它不保證真實 build 會成功。

後續工程化時，理想狀態是讓 VSCode 盡量讀 CMake configure 結果，而不是手寫一份跟 build graph 可能不同步的 include path。

9 建議的短期路線

你現在短時間用 Mac 出門，不需要立刻把所有東西 native 化。比較穩的路線是：

  1. 先把 roadmap 和章節整理好，避免邊寫 code 邊迷路。
  2. 抽出 owned Framebuffer，讓 renderer core 不再借用 Win32 DIB memory。
  3. 抽出 DisplayBackend interface。
  4. 保留 Win32 backend 作為 Windows baseline。
  5. 新增 SDL backend，讓 macOS/Windows 都能走同一條 present path。
  6. 之後再評估 macOS native backend；那是 platform backend 深化，不是第一優先。
常見陷阱
  • 把 VSCode 紅線當成 compiler error；IntelliSense 和 build 是兩條系統。
  • 只改 compiler command，卻沒有處理 Win32 source、system libraries、debugger。
  • 以為接 library 只要 include header；實際還有 link library、runtime binary、SDK/loader 與 build target。
  • 為了 macOS 能編，直接刪掉 Win32 backend；這會失去 baseline。
  • 把 SDL 當 renderer core，改用 SDL_RenderGeometry，讓 software rasterizer 的學習目標消失。
本章三個 takeaways
  1. Toolchain 是 editor、build driver、compiler、linker、libraries、debugger 的整體，不是單一 compiler 指令。
  2. 接 OpenGL、D3D、SDL、ImGui 這類 library 時，要同時處理 headers、link libraries、runtime binaries、SDK/loader 與 build-system target。
  3. 跨平台工程化的第一步是隔離 platform-specific source，讓 renderer core 不 include Win32/SDL/macOS headers。
