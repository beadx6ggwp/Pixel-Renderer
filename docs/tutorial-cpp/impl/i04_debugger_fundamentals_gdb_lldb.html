Debugger Fundamentals: GDB / LLDB
debugger 控制的是 process，不是 C++ 原始碼本身
GDB LLDB debug-symbols

前一章把 GDB 放進 build workflow：VSCode 先 build，再用 GDB 啟動 exe。但 debugger 值得單獨講，因為它不是 VSCode 的附屬功能，而是一個控制執行中程式的工具。

對 Pixel-Renderer 來說，debugger 是你理解 runtime state 的顯微鏡。你可以停在 SetPixel，看 frame_buffer 的位址；停在 DrawTriangle，看 barycentric weights；停在 crash 現場，沿著 call stack 回去找誰傳了壞資料。

本章範圍 — 從第一性原理理解 debugger：process、instruction pointer、breakpoint、stack frame、register、memory、debug symbols、-g、-O0。工具以目前 Windows/MinGW 的 GDB 為主，macOS 對應到 LLDB。

1 naive model：debugger 在讀 C++ code

直覺上你在 VSCode 點紅點，然後程式停在某一行，所以好像 debugger 是「照著 C++ 原始碼執行」。這個模型不精準。

CPU 執行的是 machine code。Debugger 控制的是 process 的執行狀態，再透過 debug symbols 把 machine address 對回 source line。 

C++ source
  |
  | compile with -g
  v
machine code + debug info
  |
  | loaded by OS
  v
process memory
  |
  | controlled by debugger
  v
breakpoint / step / inspect state

2 process 裡有什麼

當 PixelRenderer_debug.exe 被啟動，它會變成一個 process。Debugger 可以觀察或控制的主要東西包括：

Process state 意思 renderer 例子
instruction pointer CPU 下一個要執行的 machine instruction。 目前停在 RenderDevice::SetPixel 的哪一行。
call stack 目前 function call 的堆疊。 main -> Application::Run -> MyApp::OnRender -> Rasterizer::DrawLine。
stack frame 某個 function call 的 locals / parameters。 x、y、color、delta_time。
registers CPU 暫存器。 低階 crash 或 ABI/calling convention 問題才常看。
memory process address space。 frame_buffer + y * pitch + x * 4 指向的 pixel。

3 debugger 如何控制 process

先把 VSCode 拿掉，只看最底層。Debugger 要做的事情其實很暴力：啟動或 attach 到一個 process，讓它停下來，讀它的 register / memory，需要時改它的 register / memory，最後再讓它繼續跑。 

+--------------------+        OS debug interface        +----------------------+
| debugger process   |  ----------------------------->  | debuggee process     |
| gdb / lldb         |                                  | PixelRenderer.exe    |
|                    |  <-----------------------------  |                      |
| commands / UI      |     stop event / signal / trap   | code / stack / heap  |
+--------------------+                                  +----------------------+

debugger can ask OS:
  - launch this program under debug control
  - stop / continue one thread
  - read / write debuggee memory
  - read / write debuggee registers
  - be notified when trap, crash, breakpoint, watchpoint happens

在 Linux / macOS 這類 Unix 系統，這類能力概念上常透過 ptrace 或同等 debug facility 完成；在 Windows 則是 Windows Debug API 這一類機制。GDB / LLDB 不是直接「魔法讀你的程式」，而是透過 OS 允許的 debug interface 控制另一個 process。

第一性原理 — debugger 不是 C++ interpreter。它面對的是已經載入 memory 的 machine code、register state、stack、heap。C++ source line 只是 debug symbols 幫你還原出來的視圖。

停住程式時到底停了什麼

CPU 每個 thread 都有自己的 execution state。最重要的是 instruction pointer，例如 x86-64 上常見的 RIP。當 debugger 說「停在某一行」，實際意思是某個 thread 的 instruction pointer 停在某段 machine instruction 附近，而 debug symbols 告訴 GDB：這個 address 對應到某個 .cpp 的某一行。 

source view:
  RenderDevice::SetPixel(x, y, color)
  {
      frame_buffer[y * width + x] = color;   <-- VSCode highlight
  }

machine view:
  address 0x1400123a0:
      mov eax, [rbp-0x14]
      imul eax, [rbp-0x08]
      add eax, [rbp-0x10]
      mov [rcx+rax*4], edx                 <-- CPU really executes this

debug symbol line table:
  0x1400123a0 .. 0x1400123bc  => render_device.cpp:42

所以 debugger 的能力分成兩層：底層是 process control；上層才是 source-level presentation。VSCode 再包一層 UI，把 GDB/LLDB 的結果畫成紅點、call stack panel、watch panel。

4 breakpoint 背後：trap instruction

Breakpoint 看起來像是在 source line 上貼一個標記，但 CPU 不認得 C++ line。Debugger 必須把 source line 對應到 machine address，然後讓 CPU 執行到那個 address 時產生一個 trap。 

you set:
  break RenderDevice::SetPixel

debugger resolves:
  RenderDevice::SetPixel -> machine address 0x140012380

debugger patches code memory conceptually:
  original byte at 0x140012380 is saved
  instruction at 0x140012380 is replaced by a trap instruction

CPU executes:
  hits trap
  OS receives debug exception / signal
  OS notifies debugger
  debugger shows source line

在 x86/x86-64 上，software breakpoint 常見做法是把目標 address 的第一個 byte 換成 INT3，也就是 0xCC。CPU 執行到 INT3 時會觸發 trap，OS 把這個事件交給 debugger。Debugger 再把原本被換掉的 byte 暫時放回去、調整 instruction pointer、讓程式可以繼續執行。

為什麼這很重要 — breakpoint 不是免費的概念標記。它會改動 debuggee 的 code memory 或使用硬體 debug facility。這也是為什麼 self-modifying code、JIT、multi-thread、optimized code 有時會讓 breakpoint 行為變得不直覺。

step / next 也不是單純下一行

Source-level step 和 next 依賴 debug line table。Debugger 會根據目前 instruction pointer、line mapping、function boundary，安排 temporary breakpoint 或 single-step trap。

操作 表面意思 底層概念
step 進入下一個 source statement，遇到 function call 會進去。 沿著 line table / call boundary 控制 instruction execution。
next 執行下一行，但不進入 function call。 在目前行之後放 temporary breakpoint，讓 call 內部跑完。
stepi 執行下一個 machine instruction。 不看 C++ line，只看 CPU instruction。
nexti 越過下一個 machine instruction call。 instruction-level 的 step over。

當你用 -O2 debug 時，source line 會跳來跳去，常常不是 GDB 壞掉，而是 compiler 已經重排、inline、消掉或合併了原本的 source statement。

5 debug symbols 到底存什麼

-g 不是讓 binary 變慢的 debug mode；它是讓 compiler / linker 額外輸出「如何把 machine world 映回 source world」的 metadata。 

debug information answers questions like:

  address 0x140012380 belongs to which function?
    - RenderDevice::SetPixel(int, int, uint32_t)

  address 0x14001239a belongs to which source line?
    - src/core/render_device.cpp:42

  where is local variable x right now?
    - maybe stack slot [rbp-0x14]
    - maybe register eax
    - maybe optimized out

  what is the type of color?
    - uint32_t

  how do I unwind one stack frame?
    - frame pointer or unwind/debug frame information

GCC / Clang 常見 debug format 是 DWARF；MSVC 傳統上使用 PDB。MinGW + GDB 的世界比較接近 GCC/DWARF；Windows + MSVC + Visual Studio debugger 則偏向 PDB。LLDB 在 macOS 上也主要吃 Clang 產生的 DWARF。

資訊種類 debugger 用途 沒有它會怎樣
line table address 對回 file:line。 只能看到 address / assembly。
symbol name 知道 function / global variable 名稱。 backtrace 可能只剩 address。
type info print color 時知道它是 uint32_t。 只能當 raw bytes 看。
local variable location 知道 x 此刻在 stack slot 還是 register。 locals panel 會空或不準。
unwind info 從目前 frame 回到 caller。 backtrace 可能斷掉。

這也解釋了為什麼「變數 optimized out」不是玄學。Compiler 可能根本沒有把那個 source-level variable 保存在固定位置；它可能被常數折疊、被放在 register 又覆蓋、被拆成多段 lifetime，或者整個 expression 已經被重寫。

6 stack frame、register、calling convention

當你看到 call stack，不要把它想成「C++ function 名單」。更底層地說，每次 function call 都會建立一段 execution context：return address、部分 saved registers、local variables、temporary values、可能還有 spill 出來的 register。這段 context 就是 stack frame 的核心直覺。 

high address

  caller frame
  ------------------------------------------------
  return address back to caller's caller
  saved registers
  caller local variables

  callee frame
  ------------------------------------------------
  return address back to caller
  saved frame pointer, if used
  callee local variables
  spilled temporaries

low address

stack often grows downward on common desktop CPUs

以 x86-64 的直覺來看，RSP 常代表目前 stack top，RBP 在未省略 frame pointer 時常作為目前 stack frame 的穩定基準。Compiler 可以用 [rbp-0x14] 這類 offset 找 local variable；也可以完全不用 RBP，改靠 unwind info 和 RSP offset。這就是為什麼最佳化後 backtrace 有時比較難看。 

one possible unoptimized frame:

  rbp + 16   argument spill / caller data
  rbp + 8    return address
  rbp + 0    old rbp
  rbp - 4    local int x
  rbp - 8    local int y
  rbp - 16   local uint32_t color
  rsp        current stack top

Calling convention 則規定「caller 和 callee 如何交接資料」。例如在 Windows x64 ABI，前幾個整數/pointer 參數常經由 RCX、RDX、R8、R9 傳遞；在 System V AMD64 ABI，也就是多數 Linux/macOS x86-64 環境，前幾個參數常是 RDI、RSI、RDX、RCX、R8、R9。你在 C++ 看到的 SetPixel(x, y, color)，到底落在哪些 registers 或 stack slots，是 ABI 和 compiler 一起決定的。

你在 GDB 看到 底層可能代表 為什麼會影響 debug
print x GDB 依 debug info 去 register 或 stack slot 取值。 如果 x 被 optimized out，就沒有穩定位置可讀。
backtrace Debugger 根據 frame pointer 或 unwind info 往 caller 回溯。 frame pointer 被省略或 stack 被破壞時，call stack 可能不完整。
info registers 直接看 CPU thread state。 source variable 看不到時，register 是最後底層線索。
disassemble 看目前 function 的 machine instructions。 source line 對不上時，用 assembly 確認 CPU 實際在跑什麼。
Renderer debug 觀點 — 當 SetPixel crash，source-level 你看到的是 x、y、color。Machine-level 看到的是 registers、stack slots、memory address。Debug symbols 只是把這兩個世界接起來。

7 為什麼需要 -g

沒有 -g，binary 仍然可以執行，但 debugger 很難把 machine code 對回你熟悉的 source-level 名字。 

without -g:
  address 0x4012ab
  maybe function symbol
  poor local variable info

with -g:
  src/core/render_device.cpp:21
  RenderDevice::SetPixel(int, int, uint32_t)
  locals: x, y, color
  types: uint32_t, unsigned char*

-g 不代表程式變成 debug mode。它只是把 debug information 放進輸出，讓 debugger 能理解 source mapping、local variables、type information。

8 為什麼 Debug build 常用 -O0

-g 和 -O0 是不同事情。

Flag 作用 對 debug 的影響
-g 產生 debug symbols。 讓 debugger 能對回 source、function、local variables。
-O0 關閉大多數最佳化。 source line、變數、執行順序比較直覺。
-O2 開啟最佳化。 變數可能被消掉、function 可能 inline、執行順序可能重排。

所以 Release build 也可以帶 -g，但 debug 體驗不一定好，因為最佳化後的 binary 已經不再一對一對應原始碼。

10 看 memory 與 pointer

Renderer debug 很常需要看 memory。GDB 的 x 指令是 examine memory：

# 看 frame_buffer 前 16 個 32-bit word，用 hex 顯示
x/16xw frame_buffer

# 看某個 pixel address
p frame_buffer + y * pitch + x * 4
x/1xw frame_buffer + y * pitch + x * 4

x/16xw 這種語法可以拆成 x / count format unit：

部分 例子 意思
count 16 看 16 個單位。
format x 用 hex 顯示。也可以用 d decimal、u unsigned、i instruction、s string。
unit w word，通常是 4 bytes。b 是 byte，h 是 2 bytes，g 是 8 bytes。

這對 Pixel-Renderer 很有用，因為 SetPixel 的本質就是 memory write： 

address = frame_buffer + y * pitch + x * 4
*(uint32_t*)address = color

如果畫面怪掉，你可以先確認是 rasterizer 算錯座標，還是 frame_buffer 寫入位置錯，還是 present path 沒把資料送到視窗。

framebuffer address 要看 pitch，不只看 width

很多 framebuffer bug 不是顏色算錯，而是 address 算錯。真正的 row stride 常叫 pitch，意思是一列 pixel 在 memory 中跨多少 bytes。它不一定永遠等於 width * 4，因為某些 backend 或 image buffer 可能為了 alignment 在 row 尾端 padding。 

logical pixels:

  (0,0) (1,0) (2,0) ... (width-1,0)
  (0,1) (1,1) (2,1) ... (width-1,1)

memory with pitch:

  row 0: [pixel][pixel][pixel] ... [pixel][padding...]
  row 1: [pixel][pixel][pixel] ... [pixel][padding...]

address:
  base + y * pitch + x * bytes_per_pixel

如果目前 Pixel-Renderer 自己擁有 std::vector<uint32_t>，那 pitch 可以是 width * sizeof(uint32_t)。但一旦你接 SDL texture lock、image library、GPU upload staging buffer，pitch 就要被當成獨立資料，而不是從 width 硬推。

看 color 時也要知道 byte order

假設你寫入 0xff3366aa，在 little-endian CPU 的 memory byte order 會是低位 byte 先放： 

uint32_t value:
  0xff3366aa

bytes in memory on little-endian:
  aa 66 33 ff

GDB:
  x/1xw addr   => 0xff3366aa
  x/4xb addr   => 0xaa 0x66 0x33 0xff

所以當你用 x/4xb 看 pixel，看到的順序不是「人類寫 hex literal 的順序」，而是 CPU memory layout。這跟 API 要的是 ARGB、RGBA、BGRA 是另一層問題，兩者要分開。

11 watchpoint：看誰改了記憶體

Breakpoint 是「執行到某行停下」。Watchpoint 是「某塊 memory 被改時停下」。

watch *(uint32_t*)(frame_buffer + y * pitch + x * 4)

如果你懷疑某個 pixel 被後面流程覆蓋，watchpoint 比到處撒 breakpoint 更直接。不過 watchpoint 成本通常比較高，數量也有限，適合精準追單一位置。

Watchpoint 底層常靠 CPU hardware debug registers。這代表它很強，但也有現實限制：數量很少、不同平台支援程度不同、watch expression 太複雜時 debugger 可能退回比較慢的軟體檢查。

GDB 指令 停下條件 適合用途
watch expr expr 的值被寫入改變。 追某個 pixel 什麼時候被覆蓋。
rwatch expr expr 被讀取。 追誰在讀一個壞掉的 pointer。
awatch expr expr 被讀或寫。 追難以定位的資料流，但停得很頻繁。

12 Pixel-Renderer debug recipes

追一個 pixel 是否被寫入

# naive：每個 pixel 都停，通常太吵
break RenderDevice::SetPixel
run
print x
print y
print color
print frame_buffer
backtrace

這能回答：這個 pixel 是誰呼叫 SetPixel 寫的？當下座標和顏色是什麼？

但 SetPixel 是 hot path，直接 break 會停非常多次。比較實用的做法是 conditional breakpoint：

# 只停在你關心的 pixel
break RenderDevice::SetPixel if x == 400 && y == 300
run
backtrace
print /x color
print x
print y

這個技巧很重要，因為 software renderer 的核心 loop 可能每 frame 呼叫幾萬到幾百萬次。你不應該用「人工一直 continue」來 debug hot loop，而是讓 breakpoint 條件貼近你正在驗證的 invariant。

追 triangle rasterization

break Rasterizer::DrawTriangle
run
next
print min_x
print max_x
print min_y
print max_y

如果 bounding box 明顯錯，問題在 triangle setup；如果 bounding box 正確但 inside test 怪，問題才在 barycentric/edge function。

追 stack frame 與 caller 責任

當 SetPixel 收到越界座標時，不要只在 SetPixel 內修掉。你要先問：誰傳進來的？這是 call stack 的用途。

break RenderDevice::SetPixel if x < 0 || y < 0 || x >= width || y >= height
run
backtrace
frame 0
info locals
frame 1
info locals
frame 2
info locals 
frame 0: RenderDevice::SetPixel
  sees symptom: x/y invalid

frame 1: Rasterizer::DrawLine or DrawTriangle
  likely source of generated coordinate

frame 2: MyApp::OnRender
  likely source of input geometry

debug question:
  Is clipping missing?
  Is edge stepping wrong?
  Is caller passing screen-space or NDC-space by mistake?

Debugger 的價值不是只看「現在變數是多少」，而是沿著 caller/callee 邊界追責任：哪一層應該保證 invariant？哪一層只是防守？

追 present path

break ScreenManager::UpdateScreen
run
backtrace

如果 framebuffer 已經正確，但視窗不更新，就要看 BitBlt 前後的 DC/bitmap 狀態。

crash triage：從 SIGSEGV / access violation 往回推

Crash 時先不要猜。最小流程是：看停在哪個 instruction/source line、看 call stack、看 locals、看 pointer address、看周圍 memory。

run
# crash 後
backtrace
frame 0
info locals
print frame_buffer
print x
print y
print pitch
set $addr = frame_buffer + y * pitch + x * 4
print /x $addr
x/8xb $addr
info registers
x/i $pc

info registers 和 x/i $pc 是 source-level debug 失效時的底層出口。$pc 代表 program counter；在 x86-64 GDB 裡常對應 $rip。如果 debug symbols 不完整，至少還能知道 CPU 正在執行哪個 instruction，以及它碰到的 address 是否像 null、未初始化值、或越界後的奇怪位址。

13 GDB 與 VSCode 的關係

VSCode debug UI 很方便，但你要知道它背後仍然在跟 GDB 溝通。常見對應是：

VSCode GDB 概念 你應該怎麼想
紅點 breakpoint 在某個 source 對應 address 停住。
Step Into step 進入 function call。
Step Over next 執行下一行，不進入 call。
Call Stack panel backtrace 看目前 execution path。
Watch print expression 每次停住時重新 evaluate expression。

所以你可以先用 VSCode UI 操作，但遇到奇怪問題時，要知道可以回到 GDB 指令層檢查。

常見陷阱
  • 沒有用 -g 編譯，卻期待 debugger 顯示完整 source/variable。
  • 用 Release -O2 debug，看到變數 optimized out 就以為程式壞掉。
  • 只看目前停住的 line，不看 call stack，結果不知道誰傳進來壞資料。
  • 在 SetPixel 這種 hot path 下無條件 breakpoint，導致每個 pixel 都停，真正問題反而看不見。
  • 把 width * 4 當成永遠正確的 row stride，忽略 pitch / alignment / backend buffer layout。
  • 只會用 VSCode UI，不知道背後對應 GDB 的 break、next、print、backtrace。
本章核心 takeaways
  1. Debugger 控制的是 process execution；source-level debug 依賴 -g 產生的 debug symbols。
  2. Breakpoint 底層會讓 CPU 觸發 trap；watchpoint 則常依賴 CPU hardware debug registers。
  3. Stack frame、register、calling convention 是 print variable 和 backtrace 背後真正的執行模型。
  4. Pixel-Renderer debug 要沿著資料流下 breakpoint：rasterizer 算座標，RenderDevice 寫 framebuffer，DisplayBackend present。
