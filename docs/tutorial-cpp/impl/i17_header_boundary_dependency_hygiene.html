Header Boundary / Dependency Hygiene
跨平台不是從 ifdef 開始，而是從 public header 不污染 core 開始
headers dependency platform-boundary build-graph

如果你想讓同一份 renderer core 可以在 Windows、macOS、Ubuntu 上編譯，真正的敵人常常不是 #ifdef 本身，而是 header boundary 失控：某個 core header include 了 windows.h，某個 math type 偷偷 include SDL，某個 public interface 暴露了 HWND 或 SDL_Texture*。

這章把 I-01、I-08、I-09、I-11、I-16 串起來：如果 owned framebuffer 和 DisplayBackend 是真的抽象，那 core-facing header 就不能把 platform-private type 洩漏出來。

本章範圍 — 釐清 public header、private implementation、forward declaration、include path、compile dependency、link dependency 的差別。目標是讓 renderer core 可以 headless build/test，Win32/SDL/macOS backend 各自留在 platform target 裡。

1 naive model：能編譯就代表依賴沒問題

一開始很容易覺得： 

#include 

If it compiles on my Windows machine:
  dependency is fine

這個模型只檢查「當前機器」和「當前 build path」。它沒有檢查：

  • macOS 是否能看到同一個 header。
  • headless test 是否被迫 link Win32 或 SDL。
  • 修改 platform backend 是否造成 renderer core 全部重編。
  • public interface 是否把 private backend type 變成使用者必須知道的東西。
核心規則 — platform type 不應出現在 renderer core 的 public header。HWND、HDC、SDL_Window*、SDL_Renderer*、SDL_Texture* 都應該留在 backend implementation 或 backend-private header。

2 Include dependency 和 link dependency 不一樣

C++ 專案裡至少有三種依賴要分開看：

依賴種類 發生在哪裡 失控後果
Include dependency preprocessor 需要讀哪些 header。 平台 header 污染 core；編譯變慢；macOS/Ubuntu 看不到 Windows header。
Compile dependency 某個 translation unit 要知道哪些 type layout / inline function。 小改一個 backend header，整個 renderer 重編。
Link dependency 最後 executable 需要哪些 object file / library symbol。 headless test 被迫 link SDL/Win32，CI 和跨平台 build 變複雜。

所以「不要 include 太多」不是風格潔癖，而是 build graph 的控制問題。

3 理想的 header 邊界

SVG：backend 可以 include core；core 不應 include backend。這是跨平台和 headless testing 的基本方向。

4 目前專案的具體問題

目前 screen_manager.h 直接：

#define UNICODE
#define _UNICODE

#include <windows.h>

這代表任何 include screen_manager.h 的 translation unit 都被迫看到 Win32 世界。以現在 application.h 來看，它 include 了 screen_manager.h，所以 Application 的 public/protected surface 也被 Win32 污染。 

application.h
  includes screen_manager.h
    includes windows.h

result:
  code that wants Application also sees Win32
  macOS cannot compile this header
  headless test cannot reuse Application cleanly
這不表示目前 code 錯得不能用 — 早期專案直接包 Win32 是合理的。問題是當目標變成 SDL/macOS/Ubuntu/backend replacement/testing 時，這個 header boundary 會變成主要阻力。

5 Forward declaration 只能解決一部分

你已經在 rasterizer.h 看到一個好的方向：

// rasterizer.h
class RenderDevice;

class Rasterizer {
private:
  RenderDevice* device;
};

因為 header 裡只存 pointer，不需要知道 RenderDevice 的完整 layout，所以可以 forward declare。這減少 include dependency。

但 forward declaration 不是萬能：

情況 能不能 forward declare 原因
只存 T* 或 T& 通常可以 compiler 只需要知道 T 是一個 type。
value member T member; 不行 需要知道 size、alignment、destructor。
inline function 呼叫 T 的 member 不行 需要完整 definition。
std::unique_ptr<T> 可以但要小心 destructor 通常要放到 .cpp，確保看到 complete type。

6 建議的目錄與 target 邊界

未來 refactor 可以朝這種方向走： 

src/
  core/
    framebuffer.h/.cpp
    render_device.h/.cpp
    application_core.h/.cpp

  render/
    rasterizer.h/.cpp
    shader_interface.h

  platform/
    display_backend.h
    win32/
      win32_display_backend.h/.cpp   includes windows.h privately
    sdl/
      sdl_display_backend.h/.cpp     includes SDL.h privately

tests/
  test_framebuffer.cpp
  test_rasterizer_line.cpp
  test_golden_images.cpp

對應到 build target，應該是： 

renderer_core
  sources: framebuffer, render_device, rasterizer, math
  public deps: standard library only

renderer_platform_win32
  links: renderer_core
  private deps: Win32/GDI

renderer_platform_sdl
  links: renderer_core
  private deps: SDL2 or SDL3

renderer_tests
  links: renderer_core
  no window dependency
判斷是否乾淨 — 如果 renderer_tests 需要 include windows.h 或 link SDL，只是為了測 SetPixel / DrawLine，那代表 dependency boundary 還沒有切好。

7 實作時的 hygiene checklist

  • Core header 不 include windows.h、SDL.h、OpenGL、D3D header。
  • Backend header 盡量只暴露 platform-neutral interface；platform handle 放 private member 或 PImpl。
  • 能放到 .cpp 的 include 就不要放在 .h。
  • Public header 裡避免 inline 寫太多需要完整 type 的實作。
  • Build target 用 PRIVATE / PUBLIC dependency 表達誰會把 include/link requirement 傳出去。
  • 定期做 headless build/test，確認 core 沒有被 platform header 拉回去。

8 和第三方 library 的關係

SDL、OpenGL、D3D、ImGui 這些 library 不該被看成「全專案都能 include 的東西」。它們應該進入對應 layer：

Library / API 應該在哪一層 不應該污染哪裡
SDL display backend、event/backend layer。 rasterizer、math、framebuffer ownership。
OpenGL / D3D / Metal future hardware backend 或 display path。 software renderer core。
ImGui UI frontend 或 adapter layer；輸出 draw data。 RenderDevice::SetPixel 的低階 memory model。

這不是拒絕第三方 library，而是讓它們有明確邊界。你可以用 SDL 開 window、用 ImGui 產生 UI draw data、未來用 OpenGL/D3D 做 hardware backend，但不能讓這些外部 API 直接決定 software renderer core 的型別。

本章學到的三件事：
  1. 跨平台的第一步不是大量 #ifdef，而是避免 platform-private type 出現在 core public header。
  2. include dependency、compile dependency、link dependency 是三種不同問題；乾淨的 target boundary 要分別控制它們。
  3. 真正可維護的 third-party integration 是讓 SDL/OpenGL/D3D/ImGui 進入正確 layer，而不是讓它們滲進 renderer core。
