IMPLEMENTATION TRACK · I-10
Win32DisplayBackend
把 DIBSection / memory DC / BitBlt 保留在 Win32 present path，而不是讓 renderer core 依賴它
Win32 GDI BitBlt backend

I-09 已經決定 interface 邊界：RenderDevice 寫自己的 CPU framebuffer，DisplayBackend::Present 只負責把 finished pixels 顯示出去。現在要把這個抽象落到目前專案最熟的 backend：Win32 + GDI。

這章不是要把 Win32 API 全部背起來，而是要回答一個更底層的問題：如果 framebuffer 不再由 Win32 DIBSection 擁有，Win32 backend 內部到底還剩下什麼？

本章範圍 — 從目前 ScreenManager 拆出 Win32DisplayBackend：它 owns HWND、HDC、HBITMAP、DIBSection staging memory；它不 owns renderer framebuffer。每幀 Present(const Framebuffer&) 會把 project-owned pixels copy 到 DIBSection，再用 BitBlt 從 memory DC 複製到 window DC。

1 concrete case：現在 ScreenManager 做了什麼

目前 src/core/screen_manager.cpp 的流程可以簡化成這樣： 

ScreenManager::Init()
  RegisterClass()
  CreateWindow()
  GetDC(window)
  CreateCompatibleDC(window_dc)
  CreateDIBSection(mem_dc, BITMAPINFO, &ptr)
  SelectObject(mem_dc, dib_bitmap)
  frame_buffer = ptr

RenderDevice
  gets frame_buffer from ScreenManager
  writes pixels directly into DIBSection memory

ScreenManager::UpdateScreen()
  BitBlt(window_dc, 0,0,w,h, mem_dc, 0,0, SRCCOPY)

這條路能運作，因為 CreateDIBSection 給你一段 CPU 可以寫的 bitmap memory，GDI 也能把同一個 bitmap selected into memory DC。也就是說，這塊 memory 同時扮演兩種角色：

角色 目前誰使用 問題
renderer target RenderDevice::SetPixel renderer core 被 Win32-created memory 綁住。
GDI present source BitBlt 這其實是 platform backend 的 staging resource。
naive model 的失敗點 — 如果把 ScreenManager 改名成 Win32DisplayBackend，但仍然提供 GetFrameBuffer() 給 RenderDevice，那只是改名字。真正的耦合仍然存在：renderer target lifetime 由 Win32 resource 決定。

2 新模型：DIBSection 變成 present staging buffer

抽出 owned Framebuffer 後，Win32 DIBSection 不再是唯一 framebuffer。它只是 Win32 backend 內部為了 BitBlt 準備的 staging bitmap。 

Project-owned CPU memory                         Win32/GDI-owned resources
------------------------                         -------------------------

Framebuffer                                      Win32DisplayBackend
  pixels: vector<uint32_t>                         HWND window
  width/height/pitch                               HDC  window_dc
        |                                          HDC  mem_dc
        | RenderDevice::SetPixel                   HBITMAP dib_bitmap
        v                                          uint8_t* dib_pixels
  final frame pixels                                      ^
        |                                                 |
        | Present(fb) copies rows                         |
        +-------------------------------------------------+
                                                          |
                                                          | BitBlt
                                                          v
                                                   window client area

這個設計刻意多留一次 CPU copy。第一版這是可接受的，因為它讓 boundary 很乾淨：

  • Framebuffer 的 lifetime 由 project 控制，不由 Win32 控制。
  • RenderDevice 的 memory layout 不依賴 windows.h。
  • 之後 SDL backend 可以把同一份 Framebuffer upload 到 SDL_Texture。
  • 之後 macOS backend 可以把同一份 Framebuffer upload 到 Metal texture 或 CoreGraphics image path。

3 Win32DisplayBackend owns 什麼

從 RAII 角度看，Win32DisplayBackend 是一個 resource-owning type。它 owns 的不是 renderer pixels，而是 Win32/GDI handles：

class Win32DisplayBackend : public DisplayBackend {
public:
  Win32DisplayBackend() = default;
  ~Win32DisplayBackend() override { Shutdown(); }

  bool Init(int width, int height, const char* title) override;
  void Shutdown() override;
  void PollEvents() override;
  void Present(const Framebuffer& fb) override;

private:
  HWND window_ = nullptr;
  HDC window_dc_ = nullptr;
  HDC mem_dc_ = nullptr;
  HBITMAP dib_bitmap_ = nullptr;
  HBITMAP old_bitmap_ = nullptr;
  uint8_t* dib_pixels_ = nullptr;
  int width_ = 0;
  int height_ = 0;
  int dib_pitch_ = 0;
};

注意這裡沒有 GetFrameBuffer()。這是最關鍵的差異。DIBSection memory 是 backend private staging memory，不是 renderer public target。

ownership invariant — Win32DisplayBackend 可以 owns dib_pixels_ 的存取權，但不能把它借給 RenderDevice 當長期 render target。

4 Init：建立 window 與 DIBSection

Init 做的事和目前 ScreenManager::Init 很像，但語意不同：它是在建立 present path，不是在建立 renderer framebuffer。 

Win32DisplayBackend::Init(w, h)
  create OS window
  get window DC
  create compatible memory DC
  create top-down 32-bit DIBSection
  select DIBSection into memory DC
  keep dib_pixels_ as private staging pointer
BITMAPINFO bi = {};
bi.bmiHeader.biSize = sizeof(BITMAPINFOHEADER);
bi.bmiHeader.biWidth = width;
bi.bmiHeader.biHeight = -height; // top-down: row 0 is top row
bi.bmiHeader.biPlanes = 1;
bi.bmiHeader.biBitCount = 32;
bi.bmiHeader.biCompression = BI_RGB;

void* ptr = nullptr;
dib_bitmap_ = CreateDIBSection(mem_dc_, &bi, DIB_RGB_COLORS, &ptr, nullptr, 0);
old_bitmap_ = (HBITMAP)SelectObject(mem_dc_, dib_bitmap_);
dib_pixels_ = static_cast<uint8_t*>(ptr);
dib_pitch_ = width * 4;

biHeight = -height 很重要。Windows bitmap 傳統上可以是 bottom-up；如果用正高度，row 0 代表影像底部。software renderer 通常把 y = 0 當上方，所以用 negative height 建 top-down DIB 可以避免 present 時上下顛倒。

5 Present：copy rows，再 BitBlt

第一版 Present 可以很直接：檢查尺寸，逐列 copy framebuffer pixels 到 DIBSection，最後 BitBlt。

void Win32DisplayBackend::Present(const Framebuffer& fb) {
  if (fb.Width() != width_ || fb.Height() != height_) {
    // first version: reject or recreate DIBSection
    return;
  }

  for (int y = 0; y < height_; ++y) {
    const uint8_t* src = fb.Bytes() + y * fb.Pitch();
    uint8_t* dst = dib_pixels_ + y * dib_pitch_;
    std::memcpy(dst, src, width_ * 4);
  }

  BitBlt(window_dc_, 0, 0, width_, height_, mem_dc_, 0, 0, SRCCOPY);
}

這裡不要把 memcpy 視為倒退。它是架構切割的代價。等語意穩定後，才評估少 copy 的 path，例如讓 Win32 backend 用 StretchDIBits 從 project-owned memory 直接 present，或讓 backend 提供 transient mapped target。但第一版不應該為了省 copy 又把 ownership 混回去。

6 BitBlt 之後發生什麼

BitBlt 的名字容易讓人以為它等於「把東西送到螢幕」。更精確地說，它是要求 GDI 把一個 device context 的矩形像素複製到另一個 device context。 

CPU / process memory
  Framebuffer::pixels
        |
        | memcpy in Present()
        v
Win32 DIBSection memory
  selected into mem_dc
        |
        | BitBlt(mem_dc -> window_dc)
        v
GDI / display driver / DWM path
  copies or composites into the window surface
        |
        | desktop compositor picks final desktop image
        v
GPU scanout / monitor
  user sees the frame later

實際底層可能依 Windows 版本、driver、DWM compositor、視窗是否被遮擋而不同。對 Pixel-Renderer 的工程判準是：BitBlt 之後，Present 的責任已經結束。renderer core 不應該假設它立刻等於 physical monitor scanout。

這不是 GPU pipeline — 你手寫的 triangle rasterization 已經在 CPU 完成。Win32/GDI 這裡只是 present final pixels。即使 OS/driver 後面用 GPU compositing，也不代表你的 triangle 被 GPU rasterizer 重畫一次。

7 event/input：先保留簡單 query

目前 ScreenManager::DispatchEvents 用 PeekMessage、TranslateMessage、DispatchMessage pump Win32 message queue，WindowProc 更新 key/mouse state。這也可以搬進 Win32DisplayBackend： 

Win32DisplayBackend::PollEvents()
  while PeekMessage(...)
    TranslateMessage(...)
    DispatchMessage(...)

WindowProc(hwnd, msg, wParam, lParam)
  self = GetWindowLongPtr(hwnd, GWLP_USERDATA)
  update self->keys_
  update self->mouse_
  set self->should_close_ on WM_CLOSE/WM_DESTROY

這裡要分清楚兩層：

Win32 mechanism project-facing meaning
WM_KEYDOWN / WM_KEYUP IsKeyDown(KeyCode)
WM_MOUSEMOVE mouse position in client area
WM_CLOSE / WM_DESTROY ShouldClose()

第一版可以讓 Win32DisplayBackend 同時處理 input。之後如果 UI 系統長大，再把平台事件轉成 project-level InputState 或 event queue。

8 Shutdown：resource release order

Win32/GDI resource 的釋放順序不是裝飾細節。mem_dc_ 目前選入了 dib_bitmap_，所以 destroy 前要先把 old_bitmap_ select 回去。 

Shutdown()
  if mem_dc_ and old_bitmap_
    SelectObject(mem_dc_, old_bitmap_)

  if dib_bitmap_
    DeleteObject(dib_bitmap_)

  if mem_dc_
    DeleteDC(mem_dc_)

  if window_dc_
    ReleaseDC(window_, window_dc_)

  if window_
    DestroyWindow(window_)

這就是 T-06 RAII 的實作場景：一個 resource-owning type 要能在 partially initialized state 下安全 shutdown。也就是說，如果 CreateDIBSection 失敗，Shutdown 仍然不能 double free 或使用 invalid handle。

9 build boundary：Win32 檔案只在 Windows 編譯

Win32DisplayBackend.h/.cpp 會 include windows.h。因此它不應該被 macOS build target 編譯，也不應該被 renderer core header 間接 include。 

include/
  display_backend.h        no windows.h
  framebuffer.h            no windows.h

src/platform/win32/
  win32_display_backend.h  may include windows.h
  win32_display_backend.cpp

src/platform/sdl/
  sdl_display_backend.h
  sdl_display_backend.cpp

CMake 或 Makefile 之後要做的是 platform source selection，而不是在每個 core header 裡塞 #ifdef _WIN32。

工程判準 — 如果 macOS 編譯 render_device.cpp、rasterizer.cpp、framebuffer.cpp 時完全不需要知道 HWND、HDC、HBITMAP，這才算 backend boundary 有切乾淨。

10 實作 commit sequence

真的要改 code 時，不建議一次把所有東西改成最終型。比較穩的順序是：

Step Change 驗證
1 新增 Framebuffer，讓 RenderDevice 寫 project-owned pixels。 Clear / SetPixel 對 memory layout 正確。
2 保留 ScreenManager，但 UpdateScreen(const Framebuffer&) copy 到 DIB 再 BitBlt。 畫面仍能顯示，顏色與上下方向正確。
3 抽出 DisplayBackend interface。 Application 只依賴 interface。
4 把 Win32 code 移到 Win32DisplayBackend。 core headers 不再 include windows.h。
5 調整 build system，只在 Windows target 編譯 Win32 backend。 Windows build 跑 Win32 backend；macOS build 不碰 Win32 source。

11 這和 SDL backend 的關係

Win32 backend 與 SDL backend 的差異不在 SetPixel。差異只在 Present： 

same:
  Rasterizer -> RenderDevice -> Framebuffer

different:
  Win32DisplayBackend::Present(fb)
    memcpy rows to DIBSection
    BitBlt mem_dc to window_dc

  SDLDisplayBackend::Present(fb)
    SDL_UpdateTexture(texture, fb.Bytes(), fb.Pitch())
    SDL_RenderTexture / SDL_RenderCopy
    SDL_RenderPresent

這就是為什麼第一版應該先用 Framebuffer + Present 的抽象，而不是把 DrawLine 換成 SDL_RenderDrawLine 或 GDI LineTo。那會替換掉 rasterizer core，不是替換 display backend。

12 Summary

本章三件事
1. Win32DisplayBackend owns window/GDI resources；Framebuffer owns renderer pixels。
2. 第一版 Present 可以 copy project-owned pixels 到 DIBSection，再用 BitBlt 顯示。多一次 copy 換來乾淨 boundary。
3. Win32 / SDL / macOS 應替換的是 present path，不是 SetPixel、line rasterization、triangle rasterization、shader pipeline。
