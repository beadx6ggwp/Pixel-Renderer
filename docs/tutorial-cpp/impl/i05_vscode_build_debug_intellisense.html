VSCode Build / Debug / IntelliSense Setup
VSCode 不是 compiler；它只是協調 build command、debugger、editor language service
tasks.json launch.json IntelliSense

前面三章已經拆過 toolchain、build workflow、debugger。這章回到 VSCode，但不是教你「按哪個按鈕」。我們要把 VSCode 的三份設定檔放回底層模型：tasks.json 只是執行 build command，launch.json 只是啟動 debugger，c_cpp_properties.json 只是餵 IntelliSense。

這件事對 Pixel-Renderer 很重要，因為你很快會同時面對 Windows MinGW、macOS Apple Clang、可能的 SDL library、之後的 CMake。若你把 VSCode 紅線、compiler error、link error、debugger failure 混在一起，會很難判斷到底該改 editor、build system，還是程式碼。

本章範圍 — 讀目前 repo 的 .vscode/tasks.json、.vscode/launch.json、.vscode/c_cpp_properties.json，從底層解釋 VSCode 如何包住 make、GDB、IntelliSense。這章只寫教學筆記，不修改專案設定檔。

1 naive model：VSCode 會編譯 C++

最常見的直覺是： 

I press Run in VSCode
VSCode compiles C++
VSCode runs the program
VSCode shows errors

這個模型會讓你調錯地方。VSCode 本身不會編譯 C++。它呼叫外部工具，然後把結果呈現在 UI 裡。 

VSCode
  |
  +-- tasks.json
  |     runs external build command
  |     e.g. make debug
  |
  +-- launch.json
  |     starts external debugger
  |     e.g. GDB through cppdbg / MI
  |
  +-- c_cpp_properties.json
        configures editor language service
        e.g. includePath, compilerPath, intelliSenseMode

所以第一性原理的診斷問題是：

現象 先懷疑哪一層 不要先改什麼
terminal 說找不到 header 真實 compile command 的 -I。 不要只改 IntelliSense。
Problems panel 有紅線，但 make debug 成功 IntelliSense model。 不要亂改 makefile。
linker 說 undefined reference object files / libraries / link order。 不要改 includePath。
按 F5 找不到 exe launch.json 的 program 或 preLaunchTask。 不要先猜 C++ source 壞了。

2 目前 repo 的 VSCode 快照

目前 repo 有三個 VSCode 設定檔： 

.vscode/
  tasks.json
  launch.json
  c_cpp_properties.json

它們和 makefile 的關係是： 

Ctrl+Shift+B
  |
  v
.vscode/tasks.json
  |
  | command: make
  | args: ["debug"]
  v
makefile
  |
  | CXX = g++.exe
  | INCLUDES = -I. -Itool -Isrc
  | LIBS = -lgdi32 -luser32
  v
build/debug/PixelRenderer_debug.exe

F5
  |
  v
.vscode/launch.json
  |
  | preLaunchTask: make debug Pixel-Renderer
  | MIMode: gdb
  v
GDB launches build/debug/PixelRenderer_debug.exe

IntelliSense 則是另一條線： 

editor opens .cpp/.h
  |
  v
C/C++ extension reads c_cpp_properties.json
  |
  | includePath
  | compilerPath
  | cppStandard
  | intelliSenseMode
  v
parse source for autocomplete / squiggles / go-to-definition
目前有一個 drift 例子 — makefile 的 include path 是 -I. -Itool -Isrc，但 c_cpp_properties.json 目前只有 ${workspaceFolder} 和 ${workspaceFolder}/src。因為 root include path 可能讓 tool/... 仍然被找到，所以不一定立刻壞；但這就是 editor model 和 build model 可能不同步的例子。

3 tasks.json：把 build command 包起來

目前 tasks.json 的 debug task 核心是：

"label": "make debug Pixel-Renderer",
"type": "shell",
"command": "make",
"args": ["debug"],
"options": {
  "cwd": "${workspaceFolder}",
  "shell": {
    "executable": "C:\\Program Files\\Git\\bin\\bash.exe",
    "args": ["-c"]
  }
},
"problemMatcher": ["$gcc"]

這段設定可以逐層拆開：

欄位 底層意思 目前效果
label 這個 task 的名字。 launch.json 用 preLaunchTask 找它。
type: shell 透過 shell 執行 command。 不是直接 exec compiler，而是跑 shell command。
command: make 真正被呼叫的 build driver。 由 makefile 決定 compile/link 規則。
args: ["debug"] 傳給 make 的 target。 執行 make debug。
cwd command 執行目錄。 讓 make 在 repo root 找到 makefile。
problemMatcher: "$gcc" 把 GCC-style error 解析成 Problems。 不影響編譯，只影響 UI 如何定位錯誤。

注意 problemMatcher 不是 compiler，也不是 linter。它只是把 terminal output 裡像 file.cpp:line:col: error: 的文字解析成 VSCode 可以點擊的 Problems entry。

為什麼用 Git Bash

目前 task 指定：

"executable": "C:\\Program Files\\Git\\bin\\bash.exe"

原因很務實：makefile 裡使用了 Unix-like command，例如 mkdir -p 和 rm -rf build。在 Windows 上，Git Bash 提供比較接近 Unix shell 的環境，讓這些指令可以跑。 

VSCode task
  |
  v
Git Bash
  |
  v
make debug
  |
  v
makefile uses mkdir -p / rm -rf / g++.exe
小心文件漂移 — 目前 tasks.json 底部註解寫的是 mingw32-make，但實際 command 是 make。這不一定代表錯，因為你的 shell PATH 可能讓 make 指到正確工具；但教學和設定檔最好不要長期不一致。

4 launch.json：把 debugger 啟動方式包起來

Debug configuration 的核心是：

"type": "cppdbg",
"request": "launch",
"program": "${workspaceFolder}/build/debug/PixelRenderer_debug.exe",
"cwd": "${workspaceFolder}",
"externalConsole": true,
"MIMode": "gdb",
"preLaunchTask": "make debug Pixel-Renderer"

這裡的流程是： 

F5 / Start Debugging
  |
  v
VSCode reads selected launch configuration
  |
  v
run preLaunchTask
  |
  v
tasks.json: make debug
  |
  v
check program path exists
  |
  v
start C++ debug adapter / MI engine
  |
  v
start GDB
  |
  v
GDB launches PixelRenderer_debug.exe

這裡至少有四層，不要混在一起：

層 誰負責 壞掉時常見現象
build preLaunchTask + makefile F5 前就 compile/link 失敗。
program path launch.json.program build 成功但 VSCode 找不到 exe。
debugger path GDB discovery 或 miDebuggerPath 找不到 GDB、啟動 debugger 失敗。
debug symbols makefile flags，例如 -g 能跑但 breakpoint/locals/source mapping 不完整。

cppdbg、MI、GDB 的關係

cppdbg 是 VSCode C/C++ extension 提供的 debug type。當 MIMode 是 gdb，VSCode 不是直接自己解讀 GDB，而是透過 MI protocol 跟 GDB 溝通。 

VSCode Debug UI
  |
  v
C/C++ extension debug adapter
  |
  | Machine Interface commands
  v
GDB
  |
  | OS debug interface
  v
PixelRenderer_debug.exe process

你在 UI 點 Step Over，底下會變成 GDB 能理解的操作。你看到 Call Stack panel，底下是 GDB/MI 回傳的 frame 資訊。這就是 I-04 說的：VSCode 是 UI，debugger 才是真正控制 process 的工具。

5 setupCommands：debugger 啟動後先下指令

目前 launch.json 有兩個 setup commands：

"text": "-enable-pretty-printing"
"text": "-gdb-set disassembly-flavor intel"

它們不是 C++ flags，也不是 build options。它們是在 GDB 啟動後，VSCode 先送給 GDB 的命令。

Command 用途 底層意思
-enable-pretty-printing 讓 GDB 比較會顯示 STL container。 影響 debugger 顯示，不影響程式。
-gdb-set disassembly-flavor intel assembly 顯示改成 Intel syntax。 影響 disassembly 呈現，不影響 machine code。

這類設定很適合放在 launch config，因為它們屬於 debug session preference，不屬於 makefile。

6 c_cpp_properties：IntelliSense 的世界

目前 c_cpp_properties.json 是：

"name": "Win32",
"includePath": [
  "${workspaceFolder}",
  "${workspaceFolder}/src"
],
"defines": [],
"compilerPath": "g++",
"cppStandard": "c++17",
"intelliSenseMode": "windows-gcc-x64"

這份檔案餵給 editor language service。它的工作是讓 VSCode 能解析 C++，提供補字、跳轉、紅線、hover type。它不會決定 makefile 怎麼編。 

c_cpp_properties.json
  |
  +-- includePath
  |     editor can find headers
  |
  +-- compilerPath
  |     editor can infer system include paths / predefined macros
  |
  +-- cppStandard
  |     editor parses as C++17
  |
  +-- intelliSenseMode
        editor chooses Windows GCC-like parsing mode

這也是為什麼你可能遇到兩種相反情況：

情況 代表什麼 該檢查
VSCode 紅線，但 make debug 成功 IntelliSense model 不等於 build model。 c_cpp_properties.json。
VSCode 沒紅線，但 make debug 失敗 editor 解析通過，不代表真實 compiler/linker 成功。 makefile、compiler output、link libraries。
不能用 IntelliSense 修 link error — includePath 只讓 editor/compiler 找 declaration。Link error 要看 object files、library path、library name、ABI、target platform。

7 Debug / Release config 的語意差異

目前 VSCode 有 debug 和 release 兩組 build/debug flow：

模式 task target makefile flags debugger 體驗
Debug make debug Pixel-Renderer build/debug/PixelRenderer_debug.exe -g source/locals 比較可讀，但目前沒有明確 -O0。
Release make release Pixel-Renderer build/release/PixelRenderer_release.exe -O2 可以跑 GDB，但 locals/line mapping 可能不直覺。

Debug build 的核心不是「比較慢」而已，而是要讓 debugger 有足夠資訊、讓 source line 和 machine code 的對應比較接近。I-04 已經講過：-g 產生 debug symbols，-O0 降低最佳化造成的 source mismatch。

目前可改善點 — makefile debug rule 有 -g，但沒有明確寫 -O0、-Wall、-Wextra。這不是本章要直接修改的事，但後面 I-06 Make Workflow 應該整理 debug/release flags。

8 Windows、macOS、CMake 之後怎麼演進

短期內，現在的 VSCode 設定是 Windows/MinGW-oriented：

  • tasks.json 指向 Git Bash。
  • makefile 使用 g++.exe。
  • launch.json 啟動 .exe 並使用 GDB。
  • c_cpp_properties.json 使用 windows-gcc-x64。

macOS 不是把這幾個字串改成 clang 就結束。因為目前 source 仍包含 Win32/GDI dependency，makefile 也 link gdi32/user32。正確演進應該分階段： 

Stage 1: Current Windows MinGW
  VSCode -> make -> g++.exe -> Win32/GDI -> GDB

Stage 2: Make workflow cleanup
  debug/release flags explicit
  source dirs explicit
  platform-specific source separated

Stage 3: CMake target graph
  renderer_core target
  app target
  win32_backend target
  sdl_backend target

Stage 4: VSCode reads CMake
  CMake Tools configures compiler/includes/defines
  IntelliSense follows compile_commands.json
  less duplicated includePath

長期比較乾淨的 VSCode 模型不是手寫一份龐大的 c_cpp_properties.json，而是讓 build system 產生 compile_commands.json，讓 editor 跟著真實 build graph。

9 第三方 library 接入時，VSCode 只是一層外殼

之後接 SDL、ImGui、OpenGL、D3D 時，你會看到很多教學叫你「在 VSCode 加 include path」。這只解決 editor 或 compile 的一小部分。

需求 真實 build 層 VSCode 對應
找到 header -I/path/to/include 或 CMake include dirs。 IntelliSense 也需要知道 include path。
找到 library -L/path/to/lib + -lname，或 platform framework。 不是 c_cpp_properties.json 的責任。
runtime 找到 DLL/dylib PATH、rpath、bundle、copy runtime binary。 launch.json.environment 有時可輔助。
debug 第三方互動 debug symbols、source mapping、debugger config。 launch.json 和 debugger setup。

所以 VSCode 設定的正確位置是：讓你方便呼叫 build/debug，並讓 editor 理解 source。它不應成為專案唯一的 build truth。真正的 truth 應該逐步收斂到 makefile，之後再收斂到 CMake target graph。

10 verification：如何判斷是哪一層壞了

遇到錯誤時，用下面順序切問題。

先離開 VSCode UI，跑真實 build

make debug

如果 terminal 裡的真實 build 失敗，就先看 makefile / compiler / linker。VSCode 紅線不是第一優先。

確認產物路徑和 launch.json 一致

makefile:
  DEBUG_TARGET = build/debug/PixelRenderer_debug.exe

launch.json:
  program = ${workspaceFolder}/build/debug/PixelRenderer_debug.exe

這兩個要一致。否則 build 成功，F5 仍然可能找不到 executable。

確認 preLaunchTask label 完全一致

tasks.json:
  label = make debug Pixel-Renderer

launch.json:
  preLaunchTask = make debug Pixel-Renderer

這是字串匹配。差一個空格、改了 label 沒同步，都會讓 debug 前 build 失敗。

區分 includePath 與真實 -I

makefile:
  INCLUDES = -I. -Itool -Isrc

c_cpp_properties.json:
  includePath = ${workspaceFolder}, ${workspaceFolder}/src

如果未來你發現 editor 和 compiler 對 header 的理解不同，就從這裡比對。

常見陷阱
  • 把 c_cpp_properties.json 當成 build 設定，結果修了紅線但 terminal build 還是失敗。
  • 改了 makefile output path，忘記同步 launch.json.program。
  • 改了 task label，忘記同步 preLaunchTask。
  • 在 Windows task 裡依賴 Git Bash，但到 macOS/CI/PowerShell 後沒有重新定義 shell 假設。
  • 接 SDL/ImGui 時只加 include path，忘記 link library 和 runtime binary。
本章學到的三件事：
  1. tasks.json、launch.json、c_cpp_properties.json 分別控制 build command、debugger launch、editor IntelliSense，不是同一層。
  2. VSCode 的 UI 現象要回推到底層工具：compiler、linker、make、GDB、language service，不能用同一種方式修所有錯誤。
  3. 未來跨平台與第三方 library 接入，應讓 build truth 收斂到 make/CMake target graph，再讓 VSCode 跟著它，而不是手寫一份平行世界。
