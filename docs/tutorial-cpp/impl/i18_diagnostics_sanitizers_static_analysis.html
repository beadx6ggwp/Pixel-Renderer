Diagnostics / Sanitizers / Static Analysis
debugger 讓你觀察一個錯誤；diagnostics 讓錯誤在進入 review 前先暴露
warnings asan ubsan static-analysis

I-04 已經講過 debugger：breakpoint、watchpoint、stack frame、memory inspect。問題是，debugger 通常是你已經知道「這裡可能有問題」之後才拿起來用。Renderer 的錯誤常常不是這樣：一個 SetPixel 越界、一個 signed overflow、一個 stale FramebufferView，可能只在特定解析度、特定 compiler optimization、特定 backend 才爆。

這章補的是 diagnostic surface：你要讓 compiler、sanitizer、static analyzer、test runner 在不同時間點幫你抓不同層級的錯誤。它不是取代 GDB，而是把「錯誤從不可見變可見」這件事工程化。

本章範圍 — 建立 Pixel-Renderer 的 diagnostics 工作流：compiler warnings、debug symbols、ASan、UBSan、static analysis、clang-tidy、debug/release profile。重點是釐清每個工具看到哪一層事實，以及它們對 framebuffer、numeric semantics、lifetime、backend boundary 的檢查能力。

1 naive model：等 crash 再用 debugger

一個常見直覺是： 

run program
if it crashes:
  open debugger
else:
  it is probably fine

這對 C++ renderer 特別危險，因為很多錯誤不保證立刻 crash：

  • 寫出 framebuffer 一格之外，可能剛好寫到還沒被用到的 memory。
  • signed overflow 是 undefined behavior，compiler 可以用「它不會發生」來最佳化。
  • dangling reference 可能在下一 frame 才被讀。
  • SDL / Win32 backend conversion 錯誤可能只表現成顏色不對，不會 crash。
核心錯誤 — 沒 crash 不代表沒有 bug。C++ 的很多錯誤是「語意已經失效，但目前 machine state 還看似能跑」。

2 工具不是同一層東西

先把工具分層：

工具發生時間看見什麼看不見什麼
Compiler warnings compile time 可疑 conversion、shadowing、unused、可能未初始化。 實際 runtime index 是否越界。
Static analysis compile / analysis time 部分 control-flow、lifetime、null path、bug pattern。 所有跨 frame / dynamic data 問題。
Sanitizers runtime 實際發生的 memory/UB/thread 錯誤。 沒有被 test path 執行到的 bug。
Debugger runtime interactive process state、stack、memory、register、watchpoint。 沒有停下來檢查的 path。
Golden tests test runtime renderer output 是否符合 contract。 錯誤原因，需要 sanitizer/debugger 補。
SVG：diagnostics 是一組不同時間點的觀測面，不是單一工具。

3 Warnings：先讓 compiler 質疑你的假設

第一層應該是穩定的 warning profile。對 Pixel-Renderer，最重要的是 conversion、shadowing、uninitialized、format、extra warnings。

# GCC / Clang baseline
-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Wsign-conversion

# Debug build should keep symbols and avoid heavy optimization first
-g -O0

注意：-Wconversion 和 -Wsign-conversion 可能很吵，但 renderer code 真的會大量碰到 signed coordinate 與 unsigned size 的交界。這些 warning 不是要你盲目消掉，而是逼你在邊界處寫清楚：

if (x < 0 || y < 0 || x >= width || y >= height) {
  return;
}
size_t index = static_cast<size_t>(y) * stride + static_cast<size_t>(x);
不要把 warning 當成 noise — 對 renderer 來說，conversion warning 常常是在問：你到底是在做座標運算、buffer size 運算，還是在做 bit packing？

4 ASan：抓 memory access contract

AddressSanitizer 會在 runtime 幫你抓常見 memory error，例如：

  • heap buffer overflow。
  • stack buffer overflow。
  • use after free。
  • use after scope。
  • double free。

對 Pixel-Renderer，ASan 最直接的價值是抓 framebuffer 越界： 

SetPixel(x = width, y = 10)
  expected: rejected by bounds check
  bug: writes framebuffer[y * width + width]
  ASan: catches access if it crosses poisoned redzone
# GCC / Clang
-fsanitize=address -fno-omit-frame-pointer -g

-fno-omit-frame-pointer 讓 call stack 比較可讀。Debugging renderer memory bug 時，你通常需要知道錯誤是從 Rasterizer、RenderDevice、UI draw command，還是 backend conversion path 進來的。

5 UBSan：抓 C++ 語意已經失效的地方

UndefinedBehaviorSanitizer 抓的是 C++ 語意層的錯誤。這和 T-03、T-15 直接相連：

UB 類型Renderer 例子後果
signed overflowint index = y * width + x overflow。compiler 可假設不發生，最佳化後邏輯變形。
shift out of range1 << 32 或用 signed shift pack color。pixel packing 不再有定義。
misaligned access把 byte buffer cast 成需要 alignment 的 struct。某些平台會慢，某些平台會壞。
null dereferencebackend / device pointer 尚未建立就使用。不只是 crash，語意上已經 UB。
# GCC / Clang
-fsanitize=undefined -g
和 T-15 的連接 — numeric policy 不是文件好看而已。當你決定 signed coordinate、unsigned size、uint32_t packed pixel 的規則，UBSan 就能幫你檢查某些規則是否被破壞。

6 Static analysis 與 clang-tidy：抓模式，不抓真相

Static analysis 不執行程式，所以它不能知道所有 runtime 狀態。但它可以抓很多「看起來像錯」的 pattern：

  • 未初始化 member。
  • 可能 null dereference。
  • dangling pointer / reference 的局部路徑。
  • copy constructor / assignment operator 的 resource handling suspicious pattern。
  • modernize 建議，例如用 override、nullptr、std::make_unique。

但它不是 oracle。對 renderer 這種高度依賴資料流、frame loop、backend lifetime 的程式，static analyzer 只能提供線索。真正要確認，還是要搭配 sanitizer 和 test。 

static analyzer says:
  this path may use null DisplayBackend*

your job:
  is this a real state?
  should constructor enforce invariant?
  should API accept nullable pointer?
  should test create this state?

7 Debug / Sanitized / Release build profile

不要只想成 debug 和 release 兩種 build。對 renderer，至少要三種：

Profile目的典型設定
Debug 互動式 GDB/LLDB、快速單步、看變數。 -g -O0，assert 開啟。
Sanitized 跑測試，抓 memory/UB。 -g -O1 -fsanitize=address,undefined。
Release 效能測量與使用者執行。 -O2 或 -O3，保留必要 logging switch。

Sanitized build 不等於 release。Sanitizer 會插入檢查、改變 memory layout、變慢；它的任務是抓錯，不是測最終效能。

8 Pixel-Renderer 第一版 diagnostics policy

第一版可以這樣定義： 

Everyday build:
  warnings enabled
  debug symbols enabled
  asserts enabled in debug

Before refactor merge:
  run headless tests
  run sanitized tests
  inspect new warnings

Before performance discussion:
  run release build separately
  do not use sanitizer timings as performance evidence

When image mismatch happens:
  inspect golden diff first
  if memory suspicious: ASan
  if numeric suspicious: UBSan + small repro
  if state suspicious: debugger/watchpoint

這套策略會讓 I-16 的 golden image test 不再孤立。Golden test 告訴你「輸出錯了」；ASan/UBSan 告訴你「執行過程是否已經踩破語意」；debugger 幫你深入其中一條 path。

本章學到的三件事：
  1. Debugger 是互動式觀察工具；warnings、sanitizers、static analysis 是讓錯誤更早、更自動暴露的 diagnostics surface。
  2. ASan 主要抓 memory access contract；UBSan 主要抓 C++ 語意失效；golden tests 則抓 renderer output contract。
  3. Pixel-Renderer 應至少分 Debug、Sanitized、Release 三種 profile，避免把抓錯、互動 debug、效能測量混在同一個 build 裡。
