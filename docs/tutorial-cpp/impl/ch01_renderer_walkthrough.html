code-reading architecture renderer

前面 18 章在講 C++ 語意：object 怎麼存在、誰 owns resource、copy/move 代表什麼、virtual dispatch 什麼時候值得。現在把這些概念拿回 Pixel-Renderer，重新讀目前專案。

這章不是要批判現在 code 寫得好或壞，而是建立一張清楚的架構地圖：目前資料怎麼流、誰擁有 framebuffer、哪裡已經有抽象、哪裡還被 Win32 綁住。

本章範圍 — 從 main.cpp、Application、ScreenManager、RenderDevice、Rasterizer 讀出 ownership graph 與 frame flow。這章先做 architecture reading，不直接改 code。

1 naive model：畫面是 Rasterizer 畫出來的

如果只看 main.cpp，很容易得到一個太粗的模型：

device->Clear(0x000000);
RenderFrame(frame_count);
rasterizer->DrawLine(100, 100, 700, 500, 0x00FF00);

看起來像是 Rasterizer 直接把東西畫到螢幕上。但真正的資料流比較長： 

User code
  |
  | calls DrawLine / DrawTriangle / SetPixel
  v
Rasterizer
  |
  | calls RenderDevice::SetPixel()
  v
RenderDevice
  |
  | writes bytes through frame_buffer pointer
  v
ScreenManager-owned DIBSection memory
  |
  | BitBlt()
  v
Win32 window surface / compositor / display

所以 Rasterizer 不是 display backend。它只是把幾何轉成 pixels。真正把 CPU memory 顯示到 window 的，是 ScreenManager::UpdateScreen() 裡的 BitBlt。

2 main.cpp：使用端看到的 API

目前 main.cpp 定義一個 MyApp : public Application：

class MyApp : public Application {
public:
  MyApp() : Application(800, 600, L"PixelRenderer") {}

  void OnUpdate(float delta_time) override { ... }
  void OnRender() override { ... }
};

這是 Ch17/Ch18 提到的 Template Method：Application::Run() 固定 frame loop，MyApp 只 override 每一 frame 的行為。

這裡已經有一個不錯的分離

demo code 不需要自己處理 Win32 message loop、DIBSection、BitBlt、frame timing。它只要在 OnRender() 裡描述「這一幀要畫什麼」。

但也要看到它還不完整：MyApp 可以直接碰到 protected 的 screen、device、rasterizer。這讓上層方便實驗，但 boundary 還很鬆。

3 Application：目前的 owner 與 orchestrator

Application 現在做三件事：

  • owns ScreenManager screen。
  • 用 new 建立 RenderDevice 和 Rasterizer。
  • 在 Run() 裡控制 event、update、render、present、title update。

建構流程可以畫成： 

Application::Application(w, h, title)
  |
  | screen.Init(w, h, title)
  v
ScreenManager creates:
  - HWND window_handle
  - HDC window_dc
  - HDC mem_dc
  - HBITMAP dib_bitmap
  - unsigned char* frame_buffer
  |
  | Application queries framebuffer data
  v
FramebufferConfig
  - buffer = screen.GetFrameBuffer()
  - width  = screen.GetWidth()
  - height = screen.GetHeight()
  - pitch  = screen.GetPitch()
  |
  | new RenderDevice(config)
  v
RenderDevice borrows frame_buffer pointer
  |
  | new Rasterizer(device)
  v
Rasterizer borrows RenderDevice pointer

這裡最重要的 C++ 語意是：Application owns RenderDevice 和 Rasterizer，但 RenderDevice 不 owns framebuffer memory。它只是保存 ScreenManager 借出的 pointer。

Object 目前 owner 它借用誰 語意
ScreenManager Application value member Win32 OS resources window/input/framebuffer/present 都包在一起。
RenderDevice Application raw owning pointer ScreenManager::frame_buffer 用 pointer 寫入 framebuffer，但不負責釋放它。
Rasterizer Application raw owning pointer RenderDevice* 把 line/triangle 轉成 SetPixel 呼叫。
目前的 raw pointer 有兩種語意混在一起

Application::device 和 Application::rasterizer 是 owning pointer，destructor 會 delete。但 Rasterizer::device 是 observer pointer，不該 delete。這就是 Ch16 說 smart pointer 不是為了「比較新」，而是為了把 ownership policy 寫進 type。

4 ScreenManager：目前最重的 class

ScreenManager 現在同時處理很多責任：

  • 建立 Win32 window：RegisterClass、CreateWindow、ShowWindow。
  • 建立 DIBSection：CreateDIBSection，取得 CPU 可寫的 frame_buffer pointer。
  • 處理 input：keyboard、mouse state。
  • 處理 event：PeekMessage、DispatchMessage、WindowProc。
  • present framebuffer：BitBlt(window_dc, ..., mem_dc, ...)。

換句話說，它其實是： 

ScreenManager
  = Win32Window
  + Win32Input
  + Win32FramebufferOwner
  + Win32PresentBackend
  + event dispatcher

這就是為什麼現在還沒有真正做到跨平台 backend 分離。因為 ScreenManager 的 public API 看起來像抽象的 screen，但 header 裡直接 #include <windows.h>，private members 也全部是 HWND、HDC、HBITMAP。

關鍵判斷

只要一個 core-facing header 直接暴露 platform type，例如 HWND、HDC、SDL_Window*，上層就會被該 platform 的 compile dependency 綁住。這不只是檔案包含問題，而是 architecture boundary 還沒有切乾淨。

5 RenderDevice：現在是 framebuffer writer

RenderDevice 的核心非常小：

void RenderDevice::SetPixel(int x, int y, uint32_t color) {
  if (x >= 0 && x < width && y >= 0 && y < height) {
    *(uint32_t*)(frame_buffer + y * pitch + x * 4) = color;
  }
}

這段 code 的本質是 CPU memory write： 

frame_buffer: unsigned char*
pitch: bytes per row
color format: currently treated as uint32_t

address = frame_buffer + y * pitch + x * 4
store uint32_t color to that address

這是一個很好的學習點：RenderDevice 目前不是 GPU device，也不是 platform backend。它是 software renderer 的 pixel write interface。它知道 framebuffer memory layout，但不知道這塊 memory 最後會被 BitBlt、SDL_UpdateTexture、還是其他 present path 顯示。

這裡已經有一點「渲染核心 vs 顯示」分離

RenderDevice::SetPixel 不呼叫 Win32 API。它只寫 pointer。這表示 rasterization core 已經可以朝 platform-independent 方向走。但 framebuffer 的 ownership 仍在 ScreenManager，所以分離還不完整。

6 Rasterizer：geometry 到 pixel 的轉換

Rasterizer 做的是幾何到 pixel 的轉換。現在有兩個入口：

  • DrawLine：Bresenham line rasterization。
  • DrawTriangle：bounding box 掃描 + barycentric coordinates inside test。

它的 dependency 是 RenderDevice*：

class Rasterizer {
public:
  Rasterizer(RenderDevice* _device) : device(_device) {}
private:
  RenderDevice* device;
};

這個 pointer 應該被讀成 observer relation：Rasterizer 不 owns RenderDevice，只是每次畫線或三角形時需要借用它來寫 pixels。 

Rasterizer::DrawTriangle
  |
  | compute bounding box
  | for each pixel candidate
  |   compute barycentric weights
  |   if inside triangle
  v
RenderDevice::SetPixel(x, y, color)

這也說明為什麼不應該把 SDL 或 Win32 繪圖 API 直接塞進 DrawTriangle。如果 DrawTriangle 變成 SDL_RenderGeometry 或 SDL_RenderDrawLine 的 wrapper，你就不是在寫 software rasterizer，而是在把 rasterization 交還給外部 renderer。

7 每一 frame 發生什麼

目前 frame loop 的順序是： 

Application::Run()
  |
  | QueryPerformanceCounter()
  | if enough time passed:
  v
screen.DispatchEvents()
  |
  v
OnUpdate(delta_time)
  |
  v
OnRender()
  |
  | device->Clear()
  | RenderFrame()
  | rasterizer->DrawLine()
  v
CPU framebuffer memory is now updated
  |
  v
screen.UpdateScreen()
  |
  | BitBlt(window_dc, ..., mem_dc, ...)
  v
Window receives copied image
  |
  v
SetWindowTitle(FPS / dt / frame)

用 CPU/GDI 的 buffer 角度看： 

CPU side
========

RenderDevice writes:

  frame_buffer pointer
  |
  v
+-----------------------------------+
| DIBSection memory                 |
| row 0: pixels                     |
| row 1: pixels                     |
| ...                               |
+-----------------------------------+
  ^
  |
  owned by ScreenManager / Win32 GDI resource


Present side
============

ScreenManager::UpdateScreen()

  mem_dc selects dib_bitmap
  window_dc targets window

  BitBlt(window_dc, 0,0,w,h, mem_dc, 0,0, SRCCOPY)

  DIBSection pixels are copied to window DC

所以這裡沒有真正的 GPU pipeline。這是 CPU rasterizer + Win32/GDI present path。也正因為如此，它適合拿來學 software rasterization：你能完全掌握每個 pixel write。

8 目前到底分離到哪裡

可以把「分離」拆成三個層次：

層次 目前狀態 判斷
render algorithm vs pixel write Rasterizer 透過 RenderDevice 寫 pixel 已有初步分離。
pixel write vs platform present RenderDevice 不呼叫 Win32，但 framebuffer 由 ScreenManager owns 分離一半，ownership 還綁在 Win32 backend。
platform backend abstraction ScreenManager header 直接包含 windows.h 還沒完成；目前仍是 Win32-only。

所以比較精準的說法是：

目前專案已經把 rasterization logic 和 Win32 drawing call 隔開了一層，但還沒有把 framebuffer ownership 和 display backend 完全抽出來。

9 下一步理想架構圖

如果之後要支援 SDL/macOS，同時維持「我完全掌握 framebuffer」這個學習目標，理想方向不是把 DrawLine 換成 SDL API，而是把 ownership 改成： 

Application
  |
  | owns
  v
+-----------------------+
| Framebuffer           |
| vector data |
| width/height/pitch    |
+-----------------------+
  ^
  |
  | borrows / writes
  |
RenderDevice
  ^
  |
  | borrows
  |
Rasterizer


Application
  |
  | owns through interface
  v
DisplayBackend
  |
  +-- Win32DisplayBackend::Present(framebuffer)
  |       -> BitBlt or StretchDIBits
  |
  +-- SDLDisplayBackend::Present(framebuffer)
  |       -> SDL_UpdateTexture + SDL_RenderCopy + SDL_RenderPresent
  |
  +-- MacOSDisplayBackend::Present(framebuffer)
          -> platform-specific present path

這樣分離後，每一層的責任會變清楚：

Layer 責任 不該知道
Rasterizer line/triangle/edge function/interpolation/depth test Win32、SDL、window handle。
RenderDevice pixel/depth write、clear、viewport、buffer access OS event loop、present API。
Framebuffer owns CPU pixel memory and layout 誰拿它去 present。
DisplayBackend window/event/present platform implementation triangle barycentric math。

10 為什麼這對 UI / shader / FPGA 也重要

這不是單純為了 Mac 出門能跑。它會影響後面的所有擴充。

Immediate-mode UI

UI layer 最好產生 draw commands 或直接呼叫 renderer 的高階 primitive，例如 DrawRect、DrawText、DrawImage。它不應該知道 BitBlt 或 SDL_Texture。

Shader pipeline

Shader layer 應該在 geometry/rasterization pipeline 內部運作。Display backend 只負責把完成的 framebuffer 顯示出去，不應該參與 vertex shader、fragment shader、interpolation。

FPGA golden model

如果 software renderer 未來要作為硬體驗證的 golden model，最重要的是 command、state、buffer 的語意清楚。Window present 是觀察結果的手段，不應該混進 renderer core。

常見誤解
  • PixelBuffer 本身不是前後端分離；它只是資料容器。分離要看誰 owns 它、誰寫它、誰 present 它。
  • SDL 不是 renderer core；在這個專案裡它應該是 display backend。
  • 把 DrawTriangle 換成 SDL_RenderGeometry 會失去 software rasterizer 的學習核心。
  • 把所有東西都變 virtual 不是分離；正確 boundary 才是分離。
本章三個 takeaways
  1. 目前資料流是 Rasterizer -> RenderDevice -> ScreenManager-owned DIBSection -> BitBlt。
  2. 現在已有初步分離：rasterizer 不直接碰 Win32；但還沒完全分離，因為 framebuffer ownership 和 platform backend 都在 ScreenManager。
  3. 下一步應該抽出 owned Framebuffer 與 DisplayBackend::Present，讓 Win32/SDL/macOS 只替換 present path，不替換 software rasterizer core。
