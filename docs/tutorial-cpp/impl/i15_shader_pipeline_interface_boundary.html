Shader Pipeline Interface Boundary
shader 不是外部 API，而是 software renderer 內部可測、可替換的 stage policy
shader pipeline draw-call verification

I-14 把 UI draw data 接到 owned framebuffer。現在要處理 renderer pipeline 的另一條主線：如果未來 Pixel-Renderer 要有 vertex shader、fragment shader、varying interpolation、depth/blend state，C++ interface 要怎麼設計，才不會變成 OpenGL/D3D wrapper，也不會把一切塞成一個 per-pixel callback。

這章不重推圖學數學；edge function、barycentric、perspective-correct interpolation 會屬於 rendering 筆記。這裡只處理工程邊界：哪些資料在 stage 之間流動，哪些 state 屬於 draw call，shader dispatch 該用 virtual、template、variant 還是 command，失敗時又怎麼驗證。

本章範圍 — 建立 shader pipeline 的工程邊界：DrawCall、vertex input/output、varying interpolation boundary、fragment input/output、PipelineState、resource binding、depth/blend output merger、command buffer、verification dump。目標是讓 CPU software renderer 成為可測、可替換、可 debug 的 reference pipeline，而不是把 shader interface 過早綁死在某個 graphics API。

1 naive model：shader 就是一個 callback

最直接的模型是： 

for each pixel:
  color = fragment_callback(x, y)
  SetPixel(x, y, color)

這可以做出 demo，但很快會失效：

  • vertex shader 的輸出不只是 color，還有 position、varyings、depth。
  • rasterizer 需要知道哪些資料要被 interpolated。
  • fragment shader 不應直接決定是否寫 framebuffer；depth/blend/scissor 還在後面。
  • 如果 callback 裡什麼都能做，測試很難定位問題出在哪個 stage。
naive model 失敗點 — callback 只描述「某處可以呼叫一段程式」，但沒有描述 pipeline stage 的 input/output contract。shader pipeline 的重點是資料邊界，不只是 function pointer。

2 Pipeline 資料流

SVG：shader pipeline 的工程重點是 stage input/output 清楚，這樣才有辦法測試、dump、替換 policy。
DrawCall
  vertices / indices / shader / uniforms / state / resources
      |
      v
Vertex stage
  object data -> clip position + varyings
      |
      v
Primitive setup / clipping / rasterization
  covered pixels + barycentric + interpolated varyings
      |
      v
Fragment stage
  interpolated inputs -> color/depth/discard
      |
      v
Output merger
  scissor + depth test + blend -> framebuffer

3 Stage data types：先定義資料，不要先定義 API 名稱

第一版可以從幾個資料邊界開始：

struct VertexIn {
  Vec3 position;
  Color color;
  Vec2 uv;
};

struct VertexOut {
  Vec4 clip_position;
  float inv_w;
  Vec3 color;
  Vec2 uv;
};

struct FragmentIn {
  Vec3 color;
  Vec2 uv;
  float depth;
};

struct FragmentOut {
  Color color;
  float depth;
  bool discard = false;
};

這些型別不是最終答案，但它們先把每個 stage 的 contract 寫出來。後面要改 interpolation、加 texture、加 normal、加 material，都能看到是哪個 stage 的資料邊界在變。

4 Shader dispatch：hot path 和 runtime boundary 要分開

T-11 已經講過，抽象工具要看變化點。Shader pipeline 裡常有兩種需求：

需求 適合工具 理由
內層 fragment shader 呼叫 template / concept 高頻 hot path，希望 inline、避免 per-pixel virtual dispatch。
frame-level material/shader selection command / variant / registry 低頻選擇，把選到的 shader 送進具體 draw path。
tooling 或 demo callback type erasure 可接受 不在 hot loop 時，便利性可能比極限性能重要。
template <class Shader>
void DrawIndexed(const MeshView& mesh,
                 Shader& shader,
                 const PipelineState& state,
                 RenderTarget& target);

這不是說所有 shader 都永遠要 template 化。第一版可先用 concrete shader function。重點是不要把 per-fragment hot path 一開始就塞進 virtual Fragment() 或 std::function，否則後面要優化很痛苦。

5 PipelineState：不要把 state 散在 global flags

Pipeline state 應該跟 draw call 的語意一起保存，否則 command buffer replay、debug dump、golden image 測試都會失去上下文。

struct PipelineState {
  DepthMode depth_mode = DepthMode::Less;
  BlendMode blend_mode = BlendMode::Opaque;
  CullMode cull_mode = CullMode::Back;
  Rect viewport;
  Rect scissor;
};
State用途第一版建議
depth test決定 fragment 是否覆蓋舊 depth。off / less。
blend modeUI/text/alpha 合成。opaque / alpha blend。
cull mode背面剔除。none / back。
viewportclip space 到 screen mapping。先用整個 framebuffer。
scissor限制輸出區域。接 UI clip rect 時會用到。

6 Resource binding：不要讓 command 存 raw pointer

Shader 後面會需要 texture、uniform、material。第一版最容易的做法是 command 直接存 pointer，但這會把 lifetime 問題塞進 command buffer。 

fragile:
  DrawCommand {
    Texture* texture;
    Shader* shader;
  }

problem:
  command may outlive pointed object
  debug dump cannot serialize pointer meaningfully
  tests become order/lifetime dependent

比較穩的方向是先用 handle/id 或明確的 borrowed view：

struct DrawMeshCommand {
  MeshId mesh;
  MaterialId material;
  PipelineState state;
  Mat4 transform;
};

早期可以更簡單，直接把 mesh/span/shader 借進 DrawIndexed。但一旦要保存 command buffer、做 debug dump 或跨 frame replay，就要避免 command 內藏難以驗證的 raw pointer。

7 Output merger：fragment shader 不應直接寫 framebuffer

Fragment shader 產生的是 candidate output，不是最終 memory write。最終是否寫入 framebuffer，應該由 output merger 根據 state 決定： 

FragmentOut candidate
  |
  +-- scissor test
  +-- depth test
  +-- blend with old color
  +-- write color buffer
  +-- write depth buffer

這個邊界非常重要。否則 shader 可以任意呼叫 SetPixel，depth/blend/scissor 就會散落在各種 shader 裡，renderer 也很難做一致測試。

工程規則 — Shader stage 應該回傳資料，output merger 才寫 framebuffer。這讓 depth/blend/scissor 成為 pipeline state，而不是每個 shader 各自實作一次。

8 Command buffer：frontend 與 renderer 的合約

Scene frontend 不應直接進入 inner rasterizer loop。它可以產生 draw command，renderer 再把 command 轉成具體 draw： 

Scene / UI frontend builds:
  DrawMeshCommand
  DrawUiCommand
  ClearCommand

Renderer executes:
  sort / validate commands
  bind resources
  dispatch pipeline
  write framebuffer

這讓 frontend/backend boundary 變得可測：

  • 固定 command buffer，可以測 renderer output。
  • dump command buffer，可以重播 bug。
  • 不同 DisplayBackend 不會改變 command semantics。
  • UI draw list 和 scene draw call 最後都能進同一個 output merger。

9 Verification：每個 stage 都要能 dump

I-16 會講 golden image，但 shader pipeline 不能只靠最後一張圖。你還要能把 stage artifact 拿出來：

Stage 可 dump 的資料 能定位什麼錯
Vertex stageVertexOut positions / varyingstransform、matrix order、w component。
Primitive setupscreen-space triangle、area、edge equationsbackface、winding、coverage rule。
Rasterizercovered pixel set、barycentric weightsoff-by-one、top-left rule、precision。
Fragment stageFragmentIn / FragmentOutinterpolation、texture sampling、color conversion。
Output mergerdepth before/after、blend resultdepth compare、alpha blend、write mask。
 
test failure:
  final image mismatch

debug path:
  compare VertexOut
  compare covered pixels
  compare FragmentIn at x,y
  compare FragmentOut
  compare depth/blend write

這就是為什麼 stage boundary 要先清楚。沒有邊界，就沒有中間 artifact；沒有中間 artifact，golden image 失敗時只能猜。

10 最小演進順序

  1. 先定義 VertexIn、VertexOut、FragmentIn、FragmentOut。
  2. 讓 triangle rasterizer 產生 barycentric，先插值 color/depth。
  3. 加入 DepthBuffer 和 output merger。
  4. 加入 simple fragment shader，先 flat color，再 interpolated color。
  5. 加入 texture sampling 和 alpha blend。
  6. 把 draw call 收進 command buffer。
  7. 加入 stage-level dump 與 golden image test。
  8. 最後才設計 shader/material registry。
本章學到的三件事：
  1. Shader pipeline interface 要描述 stage input/output，不只是把 fragment callback 塞進 loop。
  2. Shader hot path 可用 template/concept，PipelineState/DrawCommand 則應明確保存 depth、blend、cull、resource binding。
  3. 清楚的 stage boundary 是 renderer verification 的基礎：每個 stage 都能單獨 dump、比較、回歸測試。
