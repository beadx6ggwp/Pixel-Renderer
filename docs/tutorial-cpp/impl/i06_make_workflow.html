IMPLEMENTATION TRACK · I-06
Make Workflow
make 不是 compiler；它是用 timestamp 執行 dependency graph 的工具
make dependency-graph build-flags

前面 I-03 拆過 g++ driver，I-05 拆過 VSCode 如何呼叫 make debug。這章把焦點放到 make 本身。Make 的核心不是「幫你編譯 C++」，而是：根據 target、prerequisites、recipe 和檔案時間戳，決定哪些 command 要重新執行。

Pixel-Renderer 目前的 makefile 可以跑，但它也暴露了下一階段工程化會遇到的問題：source 用 wildcard 全掃、object file 用 notdir 壓平、header dependency 沒追、debug/release flags 不夠明確、Win32 libraries 是全域設定、shell 假設依賴 Git Bash。這章先理解它，再談怎麼演進；本章不直接修改 makefile。

本章範圍 — 用目前 makefile 建立 make 的第一性模型：target/prerequisite/recipe、timestamp、automatic variables、pattern rule、VPATH、wildcard、order-only prerequisites、phony target、debug/release flags、header dependency、platform/library boundary。最後銜接到 I-07 CMake target graph。

1 naive model：make 是比較方便的 shell script

最直覺的想法是： 

makefile = a list of shell commands
make debug = run those commands in order

這不夠精準。Shell script 主要描述「照順序做什麼」。Makefile 描述的是「什麼檔案依賴什麼檔案，以及如果 target 過期要怎麼重建」。 

shell script:
  command 1
  command 2
  command 3

make:
  target depends on prerequisites
  if target is missing or older than prerequisites:
      run recipe

所以 make 的真正模型是 dependency graph executor。它會從你要求的 target 往下找 prerequisites，判斷哪些檔案需要更新。

2 target / prerequisites / recipe

Make rule 的基本形式是：

target: prerequisites
    recipe

注意 recipe 前面通常必須是 tab，不是空格。這是 make 的歷史包袱，也是很多人第一次寫 makefile 會踩的坑。

目前 makefile 的 debug executable rule 是：

$(DEBUG_TARGET): $(DEBUG_OBJECTS) | $(DEBUG_BUILDDIR)
    $(CXX) $(CXXFLAGS) $(INCLUDES) $^ -o $@ $(LIBS)

拆開看：

部分 目前內容 意思
target $(DEBUG_TARGET) 要產生 build/debug/PixelRenderer_debug.exe。
normal prerequisites $(DEBUG_OBJECTS) 所有 object files；它們比 exe 新時要重新 link。
order-only prerequisites | $(DEBUG_BUILDDIR) 先確保目錄存在，但目錄時間戳變動不應觸發 relink。
recipe $(CXX) ... $^ -o $@ ... 真正執行 link command。

3 timestamp：make 如何判斷要不要重做

Make 預設用檔案修改時間判斷 target 是否過期： 

if target does not exist:
  build it

else if any prerequisite is newer than target:
  rebuild it

else:
  target is up to date

對 C++ object file 來說： 

build/debug/application.o depends on application.cpp

if application.cpp is newer than application.o:
  run g++ -c application.cpp -o application.o
else:
  skip

這也是為什麼 make 能節省時間：只重編改過的 translation unit，再 relink executable。

timestamp 模型的盲點 — 如果 object file 只依賴 .cpp，但 .cpp include 的 header 改了，make 不一定知道要重編。這就是 C++ makefile 需要 header dependency tracking 的原因。

4 automatic variables：$@、$<、$^

目前 makefile 用了三個 automatic variables：

$(DEBUG_BUILDDIR)/%.o: %.cpp | $(DEBUG_BUILDDIR)
    $(CXX) $(CXXFLAGS) $(INCLUDES) -g -c $< -o $@
變數 意思 在這條 rule 裡
$@ target。 build/debug/application.o
$< 第一個 prerequisite。 application.cpp
$^ 所有 normal prerequisites。 link rule 裡的所有 object files。

這些變數讓 pattern rule 可以重複使用。否則每個 .cpp 都要手寫一條 compile command。

5 wildcard、VPATH、notdir：目前 source discovery 怎麼運作

目前 makefile 用這幾行找 source：

SRCDIRS = . tool src src/core src/render
VPATH = $(SRCDIRS)
SOURCES = $(foreach dir,$(SRCDIRS),$(wildcard $(dir)/*.cpp))
DEBUG_OBJECTS = $(patsubst %.cpp,$(DEBUG_BUILDDIR)/%.o,$(notdir $(SOURCES)))

概念流程是： 

SRCDIRS
  |
  | wildcard each dir/*.cpp
  v
SOURCES with paths
  |
  | notdir removes directories
  v
file names only
  |
  | patsubst %.cpp -> build/debug/%.o
  v
object files in one flat build directory

這讓目前專案很省事：新增一個 .cpp 到指定資料夾，make 會自動掃到。但這個設計也有兩個工程風險。

風險 1：同名檔案 collision

因為 notdir 把路徑拿掉，所以這兩個 source 會撞到同一個 object： 

src/core/window.cpp
src/platform/win32/window.cpp

both become:
build/debug/window.o

目前 repo 可能還沒撞，但未來拆 platform/win32、platform/sdl、platform/macos 時，同名檔案非常常見。

風險 2：全自動掃 source 會模糊平台邊界

如果 SRCDIRS 直接把所有 backend 資料夾都掃進來，Windows build 可能也嘗試編 macOS source，macOS build 可能也嘗試編 Win32 source。 

bad long-term:
  SRCDIRS = src platform/win32 platform/sdl platform/macos
  SOURCES = all *.cpp

better:
  CORE_SOURCES = renderer core math ...
  WIN32_SOURCES = platform/win32/...
  SDL_SOURCES = platform/sdl/...
  build target chooses one backend source set
不要讓 wildcard 取代架構決策 — wildcard 很方便，但平台 source 邊界應該是顯式設計。跨平台時，「哪些 source 參與這個 target」本身就是 build graph 的語意。

6 Debug / Release flags 應該成為明確 policy

目前 debug compile rule 是：

$(CXX) $(CXXFLAGS) $(INCLUDES) -g -c $< -o $@

release compile rule 是：

$(CXX) $(CXXFLAGS) $(INCLUDES) -O2 -c $< -o $@

這能工作，但 debug/release 的 policy 還不夠清楚。更可維護的方向是把共同 flags、debug flags、release flags 拆開： 

COMMON_CXXFLAGS = -std=c++17 -Wall -Wextra -fdiagnostics-color=always
DEBUG_CXXFLAGS  = -g -O0
RELEASE_CXXFLAGS = -O2 -DNDEBUG
Flag group 用途 原因
common 所有 build 都應使用。 語言標準、警告、共通 include。
debug debugger-friendly。 -g -O0 讓 source/locals 比較直覺。
release performance-oriented。 -O2 -DNDEBUG，之後可加 LTO 等。

這不是形式潔癖。當你接 sanitizer、profiling、SDL、macOS backend 時，flags 如果都混在一個 CXXFLAGS 裡，很快會變得不可推理。

7 Header dependency：目前最容易被忽略的 correctness gap

C++ translation unit 是 .cpp 加上 include 進來的 headers。可是目前 pattern rule 只寫：

$(DEBUG_BUILDDIR)/%.o: %.cpp | $(DEBUG_BUILDDIR)

這代表 make 只知道 object 依賴 .cpp。如果你改了 render_device.h，所有 include 它的 .cpp 理論上都要重編，但 make 不會自動知道。 

render_device.h changed
  |
  +-- application.cpp includes it
  +-- rasterizer.cpp includes it
  +-- main.cpp maybe indirectly includes it

make needs dependency files to know this.

GCC/Clang 常見做法是產生 .d dependency files：

-MMD -MP

概念上會產生： 

build/debug/rasterizer.o: src/render/rasterizer.cpp src/core/render_device.h ...

if render_device.h is newer than rasterizer.o:
  rebuild rasterizer.o
I-06 的工程結論 — 只靠 %.o: %.cpp 對 C++ 專案是不完整的。短期小專案可以靠 clean rebuild 避開，長期要讓 compiler 產生 dependency files，或讓 CMake/Ninja 管這件事。

8 Order-only prerequisites：為什麼目錄放在 | 後面

目前 makefile 有：

$(DEBUG_BUILDDIR)/%.o: %.cpp | $(DEBUG_BUILDDIR)

| 後面的 prerequisite 是 order-only prerequisite。意思是：先確保這個 prerequisite 存在，但不要因為它的 timestamp 變新就重建 target。 

normal prerequisite:
  if prerequisite newer than target -> rebuild

order-only prerequisite:
  must exist before recipe runs
  timestamp does not force rebuild

目錄很適合放在 order-only，因為建立或更新目錄 timestamp 不應該讓所有 object file 都重編。

9 Platform libraries：LIBS 是 link layer，不是 compile layer

目前：

LIBS = -lgdi32 -luser32

這兩個 library 是 Win32/GDI present path 需要的 link input。它們不是 C++ include path，也不是 IntelliSense 設定。

需求 makefile 位置 例子
找到 header declaration INCLUDES -Isrc, -Ithird_party/SDL/include
找到 library binary LDFLAGS / library path -Lthird_party/SDL/lib
指定要 link 的 library LIBS -lgdi32, -luser32, -lSDL2

之後接 SDL 時，不應只在 VSCode 裡加 include path；make/CMake target 必須知道 compile includes、link libraries、runtime binary。這就是為什麼 I-07 要進 CMake target-based build。

10 Makefile 的自然演進方向

不直接改目前 makefile，但可以先看它該往哪裡演進： 

current:
  one executable
  one global source scan
  one global include set
  one global LIBS

next:
  explicit source groups
  explicit debug/release flags
  generated header dependencies
  separate platform/backend sources
  separate compile flags and link flags

later:
  CMake targets
    renderer_core
    app
    win32_backend
    sdl_backend

Make 能處理這些事情，但跨平台與第三方 library 一多，手寫 makefile 會開始把「dependency graph」和「platform target model」混在一起。CMake 的價值不是比較高級，而是它用 target 把 include dirs、compile definitions、link libraries、source sets 綁到正確的 build unit。

11 verification：怎麼觀察 make 在做什麼

Make 的問題不要靠猜，可以用幾個方式觀察。

只印出要跑的 command，不真的執行

make -n debug

這可以檢查 make 打算編哪些檔案、link 哪些 object，不會修改 build output。

從乾淨狀態重建

make clean
make debug

這能排除 stale object file 干擾。若 clean build 成功但 incremental build 怪，常常是 dependency tracking 不完整。

測 header dependency 是否被追到

1. build once
2. touch src/core/render_device.h
3. run make -n debug
4. observe whether dependent .cpp files rebuild

以目前 makefile 來看，因為沒有 .d files，header 改動不一定觸發正確重編。這是 I-06 之後應該修的工程項目。

檢查 object collision 風險

if two source files share the same basename:
  src/core/window.cpp
  src/platform/win32/window.cpp

notdir maps both to:
  window.o

未來拆 platform/backend source 時，要避免這種 object path 壓平。

常見陷阱
  • 把 make 當 shell script，忽略 target/prerequisite/timestamp 模型。
  • 只讓 object 依賴 .cpp，忘記 header dependency。
  • 用 notdir 把不同資料夾的同名 source 壓成同一個 object file。
  • 讓 wildcard 自動掃進所有 platform source，導致 Windows/macOS/SDL backend 邊界不清楚。
  • 把 -I、-L、-l 混成同一種 library 設定。
本章學到的三件事：
  1. Make 的核心是 dependency graph 和 timestamp，不是「照順序跑 shell command」。
  2. 目前 makefile 能支撐 Windows/MinGW baseline，但 header dependency、object path、flags、platform source boundary 都需要在下一階段整理。
  3. 第三方 library 與跨平台 backend 不能靠 VSCode include path 解決；build graph 必須明確描述 source、include、link、runtime 的邊界。
