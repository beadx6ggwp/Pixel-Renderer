Framebuffer Ownership Refactor
RenderDevice 應該寫 renderer-owned CPU framebuffer，不該寫 platform-owned DIB memory
framebuffer ownership backend

前面幾章把 C++ 語意、toolchain、Make、CMake target graph、preprocessor configuration、third-party dependency boundary 都鋪好。現在開始碰 Pixel-Renderer 架構本體：目前 RenderDevice::SetPixel() 能畫圖，但它寫的 memory 其實是 ScreenManager 透過 Win32 CreateDIBSection() 建出來的 DIB backing memory。這讓 renderer core 和 Win32 present path 綁在一起。

這章不直接改 code，而是把 refactor 的「正確切割線」講清楚：software renderer 應該擁有自己的 CPU framebuffer；Win32、SDL、macOS backend 只負責把這份 framebuffer present 到 window。

本章範圍 — 讀目前 ScreenManager、RenderDevice、Application 的 framebuffer flow，指出 coupling；設計 Framebuffer / FramebufferView / RenderDevice / DisplayBackend::Present() 的責任；比較 Win32 DIBSection、owned std::vector<uint32_t>、SDL_UpdateTexture 三種 present path；最後給出可逐步落地的 refactor sequence。

1 naive model：現在已經有前後端分離了

目前 code 看起來已經有幾層： 

Application
  owns ScreenManager
  owns RenderDevice
  owns Rasterizer

Rasterizer -> RenderDevice::SetPixel()
ScreenManager -> UpdateScreen() -> BitBlt()

這確實比所有 code 都塞在 main.cpp 好。但「有 class 分層」不等於「rendering backend 和 platform backend 已經分離」。判斷分離要看 ownership 和 data boundary，不是看檔案數量。

問題 目前答案 分離後應該答案
誰 owns color framebuffer memory？ ScreenManager / Win32 DIBSection。 Framebuffer in renderer/core。
RenderDevice 寫哪塊 memory？ ScreenManager::GetFrameBuffer() 回傳的 platform memory。 renderer-owned CPU framebuffer。
換 SDL/macOS 要改哪裡？ 會牽動 framebuffer owner 與 present memory。 只換 DisplayBackend::Present() implementation。
第一個修正 — 抽象分離不是「有一個 RenderDevice class」就完成。只要 renderer core 寫的是 Win32-created memory，平台 display resource 仍然滲進 rendering path。

2 目前實際資料流

目前 ScreenManager::Init() 做了這件事：

dib_bitmap = CreateDIBSection(mem_dc, &bi, DIB_RGB_COLORS, &ptr, NULL, 0);
old_bitmap = (HBITMAP)SelectObject(mem_dc, dib_bitmap);
frame_buffer = (unsigned char*)ptr;
pitch = w * 4;

接著 Application 把這個 pointer 包成 FramebufferConfig：

FramebufferConfig config;
config.buffer = screen.GetFrameBuffer();
config.width = screen.GetWidth();
config.height = screen.GetHeight();
config.pitch = screen.GetPitch();
device = new RenderDevice(config);

RenderDevice::SetPixel() 直接寫這塊 memory：

*(uint32_t*)(frame_buffer + y * pitch + x * 4) = color;

最後 ScreenManager::UpdateScreen() present：

BitBlt(window_dc, 0, 0, width, height, mem_dc, 0, 0, SRCCOPY);

完整資料流如下： 

Win32 CreateDIBSection()
  |
  | returns ptr to DIB backing memory
  v
ScreenManager::frame_buffer
  |
  | GetFrameBuffer()
  v
FramebufferConfig::buffer
  |
  | injected into RenderDevice
  v
RenderDevice::frame_buffer
  |
  | SetPixel writes bytes
  v
same DIBSection memory
  |
  | selected into mem_dc
  v
BitBlt(mem_dc -> window_dc)
耦合點 — RenderDevice 看似不知道 Win32 type，但它的 framebuffer lifetime、pitch、pixel format、present path 都由 Win32 DIBSection 決定。這是 semantic coupling，不一定會出現在 include dependency 上。

3 正確切割：rendering memory vs present resource

要支援 SDL/macOS，不能把 SetPixel 換成 SDL_RenderDrawPoint，也不能讓 Rasterizer 知道 backend。正確切割是： 

Rasterizer / RenderDevice
  decides pixels and writes CPU memory

Framebuffer
  owns color buffer
  exposes data + width + height + pitch + format

DisplayBackend
  reads/copies/uploads final pixels
  presents to OS window

換句話說，renderer backend 和 display backend 不是同一層：

層 回答的問題 例子
Software renderer core 哪些 pixel 應該是什麼顏色？ DrawTriangle, depth test, shader, alpha blend。
Framebuffer final pixels 存在哪裡？layout 是什麼？ std::vector<uint32_t>, pitch, format。
Display backend 這份 pixels 怎麼交給 OS/window/compositor？ Win32 BitBlt, SDL UpdateTexture, macOS upload。
核心方向 — RenderDevice::SetPixel() 保留，但它寫入的 destination 要從 platform-owned DIB memory 改成 renderer-owned CPU framebuffer。

5 Framebuffer 的第一版 contract

第一版不要過度抽象。Pixel-Renderer 目前可以先固定 32-bit color buffer：

class Framebuffer {
public:
  Framebuffer(int width, int height)
    : width_(width), height_(height), pixels_(static_cast<size_t>(width) * height) {}

  uint32_t* Data() { return pixels_.data(); }
  const uint32_t* Data() const { return pixels_.data(); }
  int Width() const { return width_; }
  int Height() const { return height_; }
  int PitchBytes() const { return width_ * static_cast<int>(sizeof(uint32_t)); }
  void Clear(uint32_t color) { std::fill(pixels_.begin(), pixels_.end(), color); }

private:
  int width_ = 0;
  int height_ = 0;
  std::vector<uint32_t> pixels_;
};

這個 type 的 contract 很清楚：

  • Framebuffer owns pixel memory。
  • Data() 回傳的是 borrowed pointer，不應比 Framebuffer 活得更久。
  • 第一版 pitch 可以等於 width * 4，之後若要 alignment 再改。
  • pixel format 要先固定，例如 0xAARRGGBB 或目前 Win32 實際使用的 byte layout。
copy policy 要明確 — Framebuffer 是否允許 copy 是設計選擇。教學工具可以允許 copy；renderer hot path 可先 = delete copy，保留 noexcept move，避免不小心複製整張畫面。

6 RenderDevice 應該借用 Framebuffer，而不是 owns pointer

RenderDevice 的定位是 pixel write API，不是 framebuffer owner。它可以在建構時拿 Framebuffer&，或拿一個 lightweight view：

struct FramebufferView {
  uint32_t* pixels = nullptr;
  int width = 0;
  int height = 0;
  int pitch_bytes = 0;
};

class RenderDevice {
public:
  explicit RenderDevice(FramebufferView target) : target_(target) {}
  void SetPixel(int x, int y, uint32_t color);

private:
  FramebufferView target_; // borrowed view, not owner
};

這裡要用 T-04/T-05 的語意看：FramebufferView 是 cheap-copy borrowed view，不 owns lifetime。真正 owner 是 Framebuffer。這個 distinction 會避免很多後續 bug。 

Framebuffer owns:
  vector pixels

FramebufferView borrows:
  uint32_t* pixels
  width / height / pitch

RenderDevice stores:
  borrowed FramebufferView
  uses it only while Framebuffer is alive

7 Win32 backend 第一版：copy owned framebuffer into DIBSection

如果 Win32 backend 還想用目前 BitBlt path，第一版可以保留 DIBSection，但它不再是 renderer framebuffer owner。它只是 present staging resource。 

Owned Framebuffer
  |
  | memcpy row by row
  v
Win32 DIBSection memory
  |
  | BitBlt
  v
Window DC

概念 code：

void Win32DisplayBackend::Present(const Framebuffer& fb) {
  for (int y = 0; y < fb.Height(); ++y) {
    auto* dst = dib_pixels_ + y * dib_pitch_bytes_;
    auto* src = reinterpret_cast<const uint8_t*>(fb.Data()) + y * fb.PitchBytes();
    std::memcpy(dst, src, fb.PitchBytes());
  }
  BitBlt(window_dc_, 0, 0, fb.Width(), fb.Height(), mem_dc_, 0, 0, SRCCOPY);
}

這看起來多了一次 copy，但它換來一個重要能力：renderer core 不再依賴 Win32-created memory。之後 SDL backend 也可以用同一份 Framebuffer： 

SDLDisplayBackend::Present(fb)
  |
  | SDL_UpdateTexture(texture, nullptr, fb.Data(), fb.PitchBytes())
  v
SDL_Texture -> SDL_RenderCopy -> SDL_RenderPresent
第一版先選穩定 lifetime — 先用 application-owned std::vector<uint32_t> + backend copy/upload。等架構穩定後，再評估 SDL_LockTexture 或 zero-copy-ish path。不要一開始就讓 RenderDevice 的 target pointer 跟 lock/unlock lifetime 綁在一起。

8 Refactor sequence：不要一次切太多

這種改動容易把 window、input、framebuffer、present、build system 全攪在一起。比較穩的順序是：

階段 改什麼 不改什麼 驗證
Step 1 新增 Framebuffer type，暫不接入。 現有 Win32 path。 unit-like test: create/clear/check pixels。
Step 2 讓 RenderDevice 可寫 FramebufferView。 ScreenManager 還可保留舊 DIB present。 Clear、SetPixel 對 CPU buffer 正確。
Step 3 Application owns Framebuffer，不再從 ScreenManager 取 render target。 Window/input 還在 ScreenManager。 畫面仍可 present。
Step 4 ScreenManager::UpdateScreen 改成接收 const Framebuffer& 並 copy 到 DIB。 還不急著命名 DisplayBackend。 Win32 畫面和 refactor 前一致。
Step 5 把 ScreenManager 拆成 window/input + Win32DisplayBackend。 Rasterizer math。 core target 不 include Windows headers。
工程原則 — 先改 ownership，再改 interface 名字。否則很容易產生一個叫 DisplayBackend 的 class，但裡面仍然讓 renderer 寫 platform memory。

9 Resize、pitch、format：不要偷省的三件事

Framebuffer refactor 不是只有 pointer 變成 vector。還要明確三個 contract。

Resize

resize 會使 std::vector reallocate，舊的 Data() pointer 可能失效。因此 RenderDevice 如果長期保存 FramebufferView，resize 後要 rebind target，或讓 RenderDevice 每次從 Framebuffer& 取 view。 

Framebuffer::Resize()
  |
  | vector reallocates
  v
old pixels pointer invalid
  |
  v
RenderDevice target view must be refreshed

Pitch

第一版 pitch 可以是 width * 4。但不要把這個假設散落在所有 code 裡。用 PitchBytes() 集中管理，未來 alignment、subsurface、texture upload 才有位置調整。

Pixel format

Win32 DIBSection、SDL texture、你的 Color packing 不一定用同一個命名。第一版要明確說 uint32_t 裡每個 byte 的意義。否則換 SDL 後最常見 bug 不是 crash，而是紅藍顛倒或 alpha 不對。

不要讓 hidden assumptions 變成 ABI — x * 4、width * 4、uint32_t color 的 byte order、top-down/bottom-up buffer 都是 contract。若不命名，之後 backend 會各自猜。

10 build target 也要跟著反映架構

I-07 的 CMake target model 到這裡開始有用。重構後 build graph 應該長這樣： 

pixel_core
  framebuffer.cpp
  render_device.cpp
  rasterizer.cpp
  no windows.h

pixel_platform_win32
  win32_display_backend.cpp
  includes windows.h
  links gdi32 user32

pixel_renderer_app
  main.cpp
  application.cpp
  links pixel_core + selected backend

這讓 compile boundary 可以檢查 architecture boundary。如果 pixel_core 需要 include windows.h，表示 refactor 還沒完成。

11 verification：怎麼知道真的分離了

Refactor 完成不是「畫面還會動」而已。要用幾個具體檢查：

檢查 通過條件
ownership Framebuffer owns pixels；RenderDevice 不 owns，也不從 ScreenManager 取 pointer。
include boundary core/render source 不 include windows.h、SDL headers、macOS headers。
present boundary Win32/SDL/macOS code 只在 Present(const Framebuffer&) 讀/copy/upload pixels。
output refactor 前後同一個 test pattern 顯示一致。
resize resize 後不使用舊 pointer；framebuffer size、backend staging resource 一起更新。 
test pattern:
  clear black
  set four corner pixels
  draw diagonal line
  draw filled rectangle

expected:
  same pixels before/after ownership refactor
本章學到的三件事：
  1. 目前耦合點不是 RenderDevice 名字，而是它寫入 Win32 DIBSection memory；真正分離要先改 framebuffer ownership。
  2. 第一版目標是 Framebuffer owns CPU pixels，RenderDevice 借用 view 寫入，DisplayBackend::Present 讀/copy/upload final pixels。
  3. refactor 要分階段做：先 ownership，再 present signature，再 backend interface；同時明確處理 resize、pitch、pixel format、include/build target boundary。
