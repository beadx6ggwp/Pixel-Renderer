Immediate-mode UI Integration Boundary
UI 不直接畫到 SDL/Win32，而是產生 draw data，交給 software renderer 寫 owned framebuffer
UI ImGui draw-data software-backend

I-13 把 Pixel-Renderer 切成 frontend、software renderer、owned framebuffer、display backend。現在要把 UI 放進這張圖。你前面提過未來想自己寫 UI，甚至接 ImGui；這件事的關鍵不是「UI 長什麼樣」，而是 UI 的 state、draw data、font resource、clip rect、alpha blend 要站在哪一層。

Immediate-mode UI 很容易被誤解成「沒有 state」。真正比較精準的說法是：每 frame 重新呼叫 widget function 建立互動與 draw list，但有些 state 仍然存在，只是它們有明確 owner 和 lifetime。

本章範圍 — 建立 UI integration 的工程模型：input state、widget ID、hot/active/focused state、app-owned value、layout cursor、draw list、font atlas、clip rect、alpha blend。核心結論是 UI frontend 只決定 widget 與 draw data；UI software backend 把 draw data rasterize 到 owned framebuffer；DisplayBackend 仍然只做 window/input/present。

1 naive model：UI 直接呼叫平台 draw API

最直接的做法會長這樣： 

Button()
  if hovered:
    SDL_SetRenderDrawColor(...)
    SDL_RenderFillRect(...)

or:

Button()
  GDI Rectangle(...)
  TextOut(...)

這個模型看起來很快能做出畫面，但它把 UI、platform backend、renderer core 綁死在一起。結果是：

  • SDL 版 UI、Win32 版 UI、macOS 版 UI 都要各寫一套。
  • UI rasterization 繞過你自己的 software renderer。
  • UI 不能跟 scene 共用 clipping、alpha blend、debug dump、golden image 測試。
  • DisplayBackend 不再只是 present，它被迫知道 UI drawing semantics。
不要接錯層 — SDL 在這裡應該處理 window/input/present。UI 不應直接呼叫 SDL_RenderFillRect 或 SDL_RenderGeometry，否則你把 UI rasterization 交給 SDL/GPU，而不是 Pixel-Renderer。

2 正確資料流：UI frontend 到 framebuffer

SVG：UI frontend 產生 draw data；UI software renderer 寫 owned framebuffer；DisplayBackend 只負責 input/present。
InputState
  |
  v
UiContext
  widget calls, ID stack, hot/active/focused state, layout
  |
  v
UIDrawList
  rectangles, text, images, triangles, clip rects
  |
  v
UI Software Renderer
  rasterize + alpha blend into Framebuffer
  |
  v

3 Immediate-mode UI 的 state 不是不存在

Immediate-mode 的重點不是沒有 state，而是 state 被拆成幾種不同 lifetime：

State Owner 生命週期 用途
InputState DisplayBackend 填入，app/UI 借用 per frame mouse position、button down、key、text input。
hot UiContext 通常 per frame 滑鼠目前 hover 的 widget ID。
active UiContext 跨 frame 短期 正在 press、drag、text edit 的 widget ID。
focused UiContext 跨 frame 鍵盤輸入目前送到哪個 widget。
widget value application 由資料本身決定 checkbox bool、slider float、selected index。
UIDrawList UiContext 或 frame arena per frame 本幀要畫的 UI commands。
常見誤解 — Immediate-mode UI 不是「每 frame 忘記所有事情」。它是每 frame 重新描述 UI，並用穩定 ID 把跨 frame 的 interaction state 接回來。

4 Widget ID：為什麼 button 需要身份

如果 UI 裡有兩個同名 button，或 button 在 list 裡重複出現，UI system 不能只靠文字判斷誰是誰。你需要 widget ID： 

BeginPanel("Inspector")
  Button("Reset")      -> ID hash("Inspector/Reset")
  Slider("Roughness")  -> ID hash("Inspector/Roughness")
EndPanel()

BeginPanel("Material")
  Button("Reset")      -> ID hash("Material/Reset")
EndPanel()

ID 的用途不是只為了 debug；它是 hot/active/focused state 的 key：

struct UiContext {
  UiId hot = 0;
  UiId active = 0;
  UiId focused = 0;
  Rect clip_rect;
  Vec2 cursor;
  std::vector<UiCommand> draw_list;
};

5 Layout cursor：UI 不只是在畫矩形

一個簡單 button 看起來只是 rect + text，但它背後其實做了： 

Button(label):
  id = make_id(label)
  rect = layout_reserve(size)
  hovered = rect_contains(mouse)
  update hot/active based on input
  clicked = mouse_released_inside_active_item
  push draw commands:
    background rect
    border rect
    text
  return clicked

所以 UI frontend 的責任是「互動 + layout + draw data」。它不該知道 SDL_Texture 或 HDC；它只需要知道輸入狀態、目前 layout cursor、clip rect、要輸出的 draw command。

6 Draw command 設計：不要急著存任意 callback

第一版 UI draw list 可以用 value commands。這呼應 T-11：UI commands 是 per-frame artifact，適合排序、dump、回放、測試。

struct UiRectCommand {
  Rect rect;
  Color color;
  Rect clip;
};

struct UiTextCommand {
  std::string_view text;
  Vec2 pos;
  Color color;
  Rect clip;
};

struct UiImageCommand {
  Rect rect;
  TextureId texture;
  Rect uv_rect;
  Rect clip;
};

using UiCommand = std::variant<UiRectCommand, UiTextCommand, UiImageCommand>;
string_view lifetime — 如果 draw list 會延後到本 frame 後面才執行，UiTextCommand::text 不能 view 一段已經失效的 temporary string。第一版可以要求文字來源活到 frame 結束；更穩的做法是 frame string arena 或把文字複製進 UI frame storage。

7 clipping 與 alpha blending

UI 幾乎一定需要 clipping。Panel 裡的內容不能畫出 panel；scroll region 裡的 widget 不能溢出可視範圍。這可以先用每個 command 帶 clip： 

current clip = intersect(parent clip, widget/panel clip)

for command:
  visible_rect = intersect(command.rect, command.clip)
  rasterize only visible_rect

Alpha blending 則是 UI 與 scene 共用 framebuffer 時一定會遇到的問題： 

src = UI pixel color with alpha
dst = framebuffer pixel

out.rgb = src.rgb * src.a + dst.rgb * (1 - src.a)
out.a   = 1 or composited alpha policy

第一版可以先用 8-bit alpha、straight alpha，並明確定義 color format。不要讓 UI renderer 和 scene renderer 對 0xAARRGGBB / 0xRRGGBBAA 有不同理解。

8 Font atlas：文字不是平台功能

如果 UI renderer 要完全走 software path，文字也不能靠 Win32 TextOut 或 SDL_ttf 直接畫到 window。更乾淨的模型是： 

font file / bitmap font
  |
  v
FontAtlas
  glyph bitmap + metrics + texture/image storage
  |
  v
UiTextCommand
  text + position + style
  |
  v
UI Software Renderer
  glyph lookup
  blit/sample atlas
  alpha blend into framebuffer

第一版可以先用 bitmap font 或簡單 font atlas；重點不是立刻做漂亮 typography，而是讓文字也成為 renderer 可測的 draw data。

9 ImGui 接入時，你實作的是 software backend

ImGui core 的工作是產生 ImDrawData。常見 backend 是 OpenGL/D3D/Metal，但那只是把 draw data 送去 GPU。對 Pixel-Renderer，你要做的是 software backend： 

ImGui widgets
  |
  v
ImDrawData
  command lists
  indexed triangles
  clip rects
  texture ids
  font atlas
  |
  v
imgui_software_backend
  iterate cmd lists
  apply clip rect
  rasterize indexed triangles
  sample font atlas
  alpha blend
  |
  v
owned Framebuffer
ImGui concept Pixel-Renderer 對應 注意
ImDrawVertUI vertex input包含 pos、uv、color。
ImDrawIdxindexed triangles要走自己的 triangle rasterizer 或 UI rasterizer。
ClipRectscissor / clip rect必須在 CPU rasterizer 實作。
TextureIdfont atlas / image handle不能是裸平台 texture pointer。

10 Verification：UI 也要能回歸測試

UI 如果只是看 window，很快會變成難以維護。比較好的測試層次是：

  • 給固定 InputState，檢查 Button() 是否回傳 clicked。
  • 給固定 layout，檢查 draw list 裡 command 數量、rect、clip 是否正確。
  • 給固定 draw list，render 到小 framebuffer，做 golden image compare。
  • ImGui software backend 可以用固定 ImDrawData fixture 測 rasterization。
Input fixture
  -> UiContext widget calls
  -> UIDrawList
  -> UI software renderer
  -> framebuffer
  -> golden image / command dump

這也讓 UI 不只是「能畫」。它變成 renderer pipeline 的一部分，可以被 dump、比較、回歸測試。

11 最小實作順序

  1. 建立 platform-neutral InputState，由 DisplayBackend 填入。
  2. 建立 UiContext：ID stack、hot/active/focused、cursor layout、clip stack、draw list。
  3. 先支援 Button、Checkbox、Slider 的 rect drawing。
  4. 加入 clip rect stack 與 alpha blending。
  5. 加入 bitmap font atlas 與 UiTextCommand。
  6. 把 UI draw list 接到 golden image tests。
  7. 最後才接 ImGui ImDrawData。
本章學到的三件事：
  1. Immediate-mode UI 不是沒有 state，而是把 app-owned value、UiContext interaction state、per-frame draw list 分清楚。
  2. UI frontend 應產生 draw data；UI software backend 才負責 clip、rasterize、sample font atlas、alpha blend 到 owned framebuffer。
  3. ImGui 接入時應消費 ImDrawData 寫 software backend，DisplayBackend 仍然只做 window/input/present。

下一章會把 shader pipeline interface 補深。UI draw list 和 shader draw calls 最後都會碰到同一個問題：資料要如何進入 renderer pipeline，stage input/output 如何定義，錯了又如何驗證。
