Renderer Frontend / Backend Engineering Boundary
把 scene/UI/shader、software rasterizer、CPU framebuffer、Win32/SDL present path 分成可維護的工程層
architecture frontend backend framebuffer

前面已經把幾塊地基補起來：I-08 說 framebuffer ownership，I-09 定義 DisplayBackend，I-10 把 Win32 DIBSection 改看成 present staging resource，I-11 把 SDL 定位成 window/event/present backend，T-09 則把 std::vector、std::span、iterator invalidation 接到 CPU buffer。

現在要把它們組成一張架構地圖。你前面一直追問的核心其實是：如果我希望完全掌握 framebuffer，以後自己寫 UI、甚至接 ImGui，Pixel-Renderer 到底要分成哪些層？ 這章不寫新 code，而是建立 refactor 前必須先穩定的邊界。

本章範圍 — 建立 Pixel-Renderer 的 frontend/backend engineering boundary。這裡的 backend 不是只有一種：有 software rendering backend、UI rendering backend、display/present backend。章節目標是分清 Scene/UI frontend 負責決定 what to draw，SoftwareRenderer/Rasterizer 負責 how to turn primitives into pixels，Framebuffer 負責 CPU-side image storage，DisplayBackend 只負責把 finished pixels 交給 Win32/SDL/macOS/Linux present path。

1 naive model：frontend/backend 只有兩層

一開始很容易把架構想成： 

Frontend:
  app / scene / UI

Backend:
  renderer / SDL / Win32 / framebuffer / everything low-level

這個分法太粗，會把三種完全不同的問題混在一起：

混在一起的詞 其實在問什麼 如果混淆會怎樣
render backend 誰把 line/triangle/UI command 轉成 pixels？ 誤以為 SDL_Renderer 可以取代自己的 rasterizer。
display backend 誰把 finished framebuffer 顯示到 window？ 誤把 Win32 DIBSection 或 SDL_Texture 當成 renderer target owner。
UI backend 誰把 UI draw data 轉成 pixels？ 誤讓 UI 直接呼叫 SDL_RenderDrawRect，繞過自己的 renderer。

所以「前後端分離」不能只問有沒有 interface。你要先問：這個 interface 切的是哪個問題的前後端？

命名陷阱 — SDL 文件裡的 SDL_Renderer 叫 renderer，但在 Pixel-Renderer 的學習目標裡，它不應該是 renderer core。它最多是 display backend 裡用來 copy texture / present 的 platform layer。

2 四層架構：what、how、where、present

這章建議先用四層分法：

SVG：四層邊界。真正的 pixels 由 software renderer 寫進 owned CPU framebuffer；Win32/SDL/macOS/Linux 只是 present path。
Layer 1: Rendering Frontend
  decides WHAT to draw
  scene, demo, UI widgets, ImGui draw data, draw order

Layer 2: Software Rendering Backend
  decides HOW to draw
  line rasterization, triangle rasterization, depth, blend, shader calls

Layer 3: Owned Framebuffer / RenderTarget
  decides WHERE pixels live
  CPU memory, color/depth, pitch, pixel format

Layer 4: Display Backend
  decides HOW finished pixels reach the OS/window
  Win32 BitBlt, SDL_UpdateTexture, future native path

這裡的重點是：Layer 2 和 Layer 4 都可能被稱為 backend，但它們不是同一種 backend。Layer 2 是 renderer backend，Layer 4 是 display backend。Pixel-Renderer 的學習核心在 Layer 2；跨平台 window 顯示在 Layer 4。

3 目前專案為什麼還沒完全分離

現在最大的耦合不是 SetPixel 這個 function 名稱，而是 framebuffer ownership 來自 platform layer。舊模型大概是： 

ScreenManager
  owns Win32 window
  owns Win32 DIBSection
  owns pointer to DIB pixels
  handles input
  does BitBlt

RenderDevice
  borrows ScreenManager's DIB pointer
  SetPixel writes into platform-owned memory

Rasterizer
  calls RenderDevice::SetPixel

這表示 renderer core 的 output storage 其實被 Win32 present path 決定。只要 storage 仍由 ScreenManager / DIBSection owns，你就還不能說渲染核心和顯示 backend 分離。

還沒分離的判準 — 如果 RenderDevice 必須從 ScreenManager 取得 framebuffer pointer，或 core header 需要知道 HWND、HDC、SDL texture，這就不是乾淨分離。

分離後應該變成： 

Application
  owns Framebuffer / RenderTarget
  owns RenderDevice bound to that target
  owns DisplayBackend interface

RenderDevice
  writes project-owned CPU memory

DisplayBackend
  reads/copies/uploads final pixels during Present()

4 Framebuffer 不是前端，也不是 display backend

你前面問過「這樣我的 PixelBuffer 就不是做渲染前後端了？」答案是：PixelBuffer / Framebuffer 是兩邊之間的資料邊界，不是其中一個 backend 本身。

元件 問題 責任
Rendering Frontend 要畫什麼？ 建立 scene/UI commands、決定 draw order、選 material/shader。
Software Renderer 怎麼把 primitives 轉成 pixels？ transform、clip、rasterize、depth、interpolate、shade、blend。
Framebuffer pixels 存在哪？ owns CPU color/depth storage，提供 view 給 RenderDevice 寫入。
DisplayBackend 怎麼顯示已完成 pixels？ Win32/SDL/macOS/Linux window、event、present。

這樣設計的好處是，Framebuffer 成為穩定交界面。UI、triangle rasterizer、future shader pipeline 都寫同一個 CPU render target；Win32/SDL/macOS/Linux 都只消費 final image。

5 frame loop：每一層在一幀裡做什麼

一幀可以拆成這個順序： 

while (running) {

  1. DisplayBackend::PollEvents(input)
       platform events -> project InputState

  2. Application::Update(dt, input)
       update scene state / UI state

  3. RenderTarget::Clear()
       clear color/depth in owned CPU memory

  4. RenderingFrontend::BuildFrame()
       decide draw calls / UI commands / scene primitives

  5. SoftwareRenderer::Execute()
       rasterize into owned Framebuffer

  6. DisplayBackend::Present(framebuffer)
       copy/upload final pixels to window
}

這個順序保證了兩件事。第一，platform event 先被轉成 project-level input，不讓 SDL scancode 或 Win32 key code 污染 core。第二，Present 永遠在 rasterization 完成之後，backend 拿到的是 final image，不是半成品 render target。

6 interface sketch：先保守，不急著過度抽象

第一版可以先用很保守的 interface。重點是讓 ownership 清楚，不是一次做出完整 engine。

struct Framebuffer {
  int width = 0;
  int height = 0;
  int pitch_pixels = 0;
  std::vector<uint32_t> color;
};

struct FramebufferView {
  std::span<uint32_t> color;
  int width = 0;
  int height = 0;
  int pitch_pixels = 0;
};
class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;

  virtual bool Init(int width, int height, const char* title) = 0;
  virtual void PollEvents(InputState& input) = 0;
  virtual bool ShouldClose() const = 0;
  virtual void Present(const Framebuffer& framebuffer) = 0;
};

注意這裡沒有 DisplayBackend::GetFrameBuffer()。只要 backend 開始回傳 writable framebuffer pointer，上層就會再次把 renderer output 綁到 platform-owned memory。

第一版夠用就好 — 先不要把 framebuffer format、multi-surface、swapchain、GPU resource、tile buffer 全塞進 interface。先讓 CPU owned framebuffer + present-only backend 跑通，後面的複雜度才有乾淨位置可以放。

7 UI / ImGui 應該接在哪一層

如果你要自己寫 immediate-mode UI，或未來接 ImGui，最重要的是不要讓 UI 直接依賴 display backend。 

Correct direction:

UI frontend
  creates widgets / layout / draw commands
      |
      v
UI software renderer
  rectangles / text / triangles / scissor / alpha blend
      |
      v
Owned Framebuffer
      |
      v
DisplayBackend::Present


Avoid:

UI -> SDL_RenderDrawRect
UI -> SDL_RenderGeometry
UI -> Win32 GDI drawing
UI -> platform framebuffer pointer

ImGui 也是同一個概念。ImGui core 產生 ImDrawData；你可以寫自己的 software ImGui backend，把 ImGui 的 indexed triangles、font atlas、clip rect、blend 轉成 CPU framebuffer pixels。這樣 display backend 仍然只 present final framebuffer。

系統 Frontend Rendering backend Display backend
自製 UI Widget tree / immediate calls 你的 UI software renderer Win32 / SDL / native
ImGui ImGui widgets -> ImDrawData 你的 ImGui software backend Win32 / SDL / native
Scene renderer Scene objects / draw calls 你的 triangle rasterizer Win32 / SDL / native
前向連結 — I-14 會專門講 immediate-mode UI integration boundary：UI state、layout、draw command、font atlas、clip rect、alpha blend 應該如何進入 software renderer，而不反向依賴 SDL/Win32。

8 shader pipeline 未來應該接在哪裡

Shader interface 不是 display backend 的事。它屬於 software rendering backend 內部，介於 draw command 和 rasterizer hot loop 之間。 

Rendering Frontend
  builds DrawCall:
    mesh
    transform
    material/shader object
    render state
      |
      v
Software Renderer
  vertex stage
  clipping / setup
  rasterization
  fragment stage
  depth / blend
      |
      v
Framebuffer

這也說明為什麼前面 T-08 的 templates/concepts 重要。Shader hot loop 可能適合 template functor 或 concept，因為它發生在每 vertex / per fragment 的高頻路徑；DisplayBackend 則每 frame 呼叫少數幾次，virtual interface 很合理。

邊界 建議抽象 理由
DisplayBackend virtual interface 每 frame 少量呼叫，runtime platform selection 清楚。
Shader hot loop template / concept / concrete functor 高頻呼叫，compiler inline 和 type requirement 更重要。
Command buffer value type / variant later 需要排序、batch、replay、debug dump。
UI draw data plain structs + spans 清楚表達 per-frame data ownership 和 borrowed views。

9 build graph 也要跟架構一致

工程邊界不能只存在腦中，build target 也要表達同一件事。大致應該長這樣： 

pixel_math
  Vec, Matrix, Color

pixel_core
  Framebuffer
  RenderDevice
  Rasterizer
  SoftwareRenderer
  InputState
  no windows.h
  no SDL.h

pixel_ui             future
  UI widgets
  UI draw commands
  UI software renderer
  no SDL_RenderDrawRect

pixel_platform_win32
  Win32DisplayBackend
  may include windows.h
  links user32/gdi32

pixel_platform_sdl
  SDLDisplayBackend
  may include SDL.h
  links SDL2 or SDL3

pixel_app
  links pixel_core + selected platform backend

這張 build graph 是 I-07 的 CMake target-based build 在架構上的應用。判準很簡單：如果 pixel_core 可以在 macOS / Ubuntu 編譯，而且不需要假裝有 windows.h 或 SDL headers，邊界就開始正確。

10 refactor sequence：不要一次大爆改

實作時不要一次把 UI、SDL、CMake、shader interface 全部一起改。比較穩的 sequence 是：

Step 改動 驗證
1 新增 owned Framebuffer / FramebufferView。 可以建立 test pattern，不依賴 Win32 DIBSection。
2 讓 RenderDevice bind/write owned framebuffer。 SetPixel 不再寫 ScreenManager owned memory。
3 把 ScreenManager 收斂成 Win32DisplayBackend。 Win32 backend 只在 Present copy/upload final pixels。
4 加入 DisplayBackend interface。 Application 只持有 unique_ptr<DisplayBackend>。
5 加入 SDL backend。 同一份 framebuffer 能走 SDL_UpdateTexture present。
6 整理 CMake target / VSCode 設定。 Windows Win32、Windows SDL、macOS SDL、Ubuntu SDL 的 source selection 合理。
7 在 owned framebuffer 上接 UI / ImGui software backend。 UI 不呼叫 SDL/Win32 drawing API，只產生或消費 renderer-level draw data。

11 判斷分離是否成功的檢查表

  • RenderDevice 是否完全不知道 Win32 / SDL / macOS API？
  • Framebuffer 是否由 renderer/application owns，而不是 display backend owns？
  • DisplayBackend::Present 是否只接收 const Framebuffer& 或短生命週期 view？
  • UI 是否能在不呼叫 SDL drawing API 的情況下畫到 framebuffer？
  • 同一個 test pattern 是否能被 Win32 backend 和 SDL backend present？
  • macOS / Ubuntu build 是否能完全避開 Win32 source？
  • SDL backend 是否沒有提供 SetPixel、DrawLine、DrawTriangle？
  • shader / rasterizer hot loop 是否沒有 virtual display backend call？
反向依賴警訊 — 如果某天 Rasterizer include 了 sdl.h，或 UI widget 直接呼叫 BitBlt / SDL_RenderGeometry，代表 display layer 又往上污染了 renderer core。

12 Summary

本章學到的三件事：
  1. Pixel-Renderer 的「前後端」不是兩層，而至少要分成 rendering frontend、software rendering backend、owned framebuffer、display backend。
  2. Framebuffer 是 renderer core 與 display backend 之間的 CPU-side image storage 邊界；它不是 UI frontend，也不是 Win32/SDL backend。
  3. Win32、SDL、macOS、Ubuntu/Linux 的差異應停在 display/present backend；line、triangle、shader、UI rasterization 應留在自己的 software renderer。

下一章回到 Theory Track：當這些 layer 開始有 interface 後，真正需要補的是 contract。哪些條件用 assert？哪些錯誤要回傳？哪些 invariant 應該由 type 保證？哪些 function 應該 noexcept？這會決定你的 abstraction 是可維護的工程邊界，還是只是一堆名字漂亮的 classes。
