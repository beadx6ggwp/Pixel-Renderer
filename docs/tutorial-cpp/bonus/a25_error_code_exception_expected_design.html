APPENDIX · A-25
Error code / exception / expected 的設計分歧
從 backend init、asset loading、programmer bug、recoverable failure 分出錯誤語意
error-handling exception expected contracts

C++ 錯誤處理常被吵成風格問題：有人說不要用 exception，有人說 error code 很醜，有人說 expected 才現代。這樣討論會卡住，因為它把不同種類的 failure 混成同一種東西。

前置閱讀 — 如果你現在還不熟 g++ diagnostic、compiler error、linker error、runtime failure 的差異，建議先看 A-32 Error Literacy。本篇假設你已經能分辨「誰在報錯」，再討論程式跑起來後要如何設計 failure channel。

真正該先問的是：這個 failure 是 programmer bug、可恢復的 runtime failure、跨平台 API boundary failure，還是 partial construction cleanup 問題？ 不同答案會導向不同工具。

本篇範圍 — 從 naive「回傳 bool 就好」開始，拆出 assert/contract、error code、exception、optional、expected-style result 的不同語意。最後回到 Pixel-Renderer：backend init、SDL/Win32 failure、asset loading、shader/resource creation、golden test mismatch、programmer invariant violation 應該各自如何表達。

1 naive model：失敗就 return false

bool InitSdlBackend() {
  if (SDL_Init(...) != 0) return false;
  if (!CreateWindow()) return false;
  return true;
}

bool 最大問題是資訊太少。失敗了，但為什麼？是 SDL 沒初始化、window 建立失敗、pixel format 不支援、還是 programmer 傳了非法尺寸？caller 能不能恢復？要不要顯示給使用者？

第一個修正 — 錯誤處理工具不是先選語法，而是先分類 failure。分類錯了，無論用 exception、error code 或 expected 都會難用。

2 四種 failure 類型

failure 類型例子常見工具
programmer bug / contract violationframebuffer width 為負、index 越界、null borrowassert、precondition、debug check。
recoverable runtime failureasset file 不存在、window 建立失敗、format 不支援error code / result / expected。
exceptional failure無法局部處理、需要跨多層 unwindexception + RAII cleanup。
absence is normal查不到 optional resource、沒有 hover widgetoptional / nullable borrow。

這裡的重點是：programmer bug 不應包裝成 recoverable error。負尺寸 framebuffer 不是叫使用者重試，而是 code 違反 invariant；asset missing 則是 runtime failure，caller 可能能提示路徑或 fallback。

3 Error code：適合 boundary，但容易被忽略

Error code 的優點是明確、不需要 exception runtime、跨 C API boundary 自然。缺點是 caller 可以忽略它，並且錯誤資料常被壓扁。

enum class BackendError {
  SdlInitFailed,
  WindowCreateFailed,
  TextureCreateFailed,
};

[[nodiscard]] BackendError InitBackend();

如果只回傳 enum，成功值怎麼表示？如果需要錯誤訊息或 OS error code，又要再加欄位。這會自然推到 result / expected 形狀。

4 expected-style result：把 success value 和 error 放在型別裡

struct BackendInitError {
  BackendError code;
  std::string message;
};

[[nodiscard]] Expected<SdlDisplayBackend, BackendInitError>
CreateSdlDisplayBackend(WindowConfig config);

這種設計把 failure 放進 function signature：caller 不能假裝不知道它可能失敗。它適合 backend init、asset loading、resource creation 這類「可預期失敗，而且 caller 應該決定怎麼處理」的流程。

5 Exception：適合跨層 unwind，但需要 RAII 配合

Exception 的強項是：中途失敗時可以離開多層 call stack，並讓已建構 object 自動 destructor。它和 RAII 是一組設計。

但 exception 不適合所有地方。Hot loop、C ABI boundary、需要明確列出每種 recoverable failure 的 API，可能更偏向 expected/error code。

和 RAII 的關係 — exception 不是 cleanup 策略；RAII 才是 cleanup 策略。Exception 只是在失敗時選擇離開路徑，RAII 保證離開路徑上的 resource 會被釋放。

6 optional：不是錯誤，只是沒有值

optional 適合表示 absence 是正常狀態：

std::optional<TextureId> FindTexture(std::string_view name);
Widget* HoveredWidget(); // nullable borrow is also possible

但如果你需要知道 asset loading 為什麼失敗，optional 不夠。沒有值不等於錯誤原因。

8 設計檢查表

Before choosing an error mechanism, ask:

1. Is this a programmer bug or runtime failure?
2. Can caller recover locally?
3. Does caller need a detailed reason?
4. Is absence a normal result?
5. Is this API crossing C / DLL / plugin boundary?
6. Is this in a hot loop?
7. Do partially acquired resources need cleanup?
8. Can [[nodiscard]] prevent ignored errors?
本篇學到的三件事：
  1. Error code、exception、expected、optional 的分歧來自 failure semantics：bug、recoverable failure、unwind failure、normal absence 不是同一種問題。
  2. Pixel-Renderer 的 backend init / asset loading 適合 structured result；internal invariant 違反適合 assert；hot loop 應靠 pre-validation；exception 必須和 RAII cleanup 一起看。
  3. 好的錯誤 API 要讓 caller 看見「可能失敗、為什麼失敗、能不能恢復」，而不是只回傳 false 或把所有東西都 throw。
