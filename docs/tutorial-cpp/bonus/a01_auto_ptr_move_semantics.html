auto_ptr 與 Move Semantics 的誕生
從 destructive copy 看懂 C++11 為什麼需要 rvalue reference、deleted copy 與 unique_ptr
auto_ptr move-semantics unique_ptr regular-type

你問的問題很準：std::auto_ptr 看起來也能做到 ownership transfer，那 C++11 為什麼還需要 move semantics？這個問題不是「舊 API 怎麼用」，而是 C++ 語意演化的核心：如果一個 operation 會破壞 source object，它還能不能叫 copy？

這篇 Appendix 不把 auto_ptr 當成現代工具來教。它是一個歷史反例：它成功暴露了 unique ownership transfer 的需求，也同時暴露了 C++98 缺少語言層 move channel 的代價。

本篇範圍 — 從 C++98 的限制開始，拆解 auto_ptr 如何用 copy syntax 做 destructive transfer，為什麼這破壞 Copyable / Regular type 假設，為什麼 STL container 和 algorithm 會因此危險，以及 C++11 如何用 rvalue reference、std::move、deleted copy、move constructor、unique_ptr 把這個語意修正回來。
先給分層答案 — 如果只看效果，auto_ptr 確實能把 ownership 交出去；如果看語意，它把 move 偽裝成 copy；如果看語言設計，它證明 C++98 缺少一條「explicit transfer」通道；如果看現代 C++，正確答案是 non-copyable + movable type，也就是 unique_ptr 代表的模型。

0 先把真正問題講清楚：不是 pointer，而是 ownership transfer

auto_ptr、unique_ptr、move semantics 表面上都和 pointer 有關，但底層問題其實是：某個 resource 只能有一個 owner 時，owner 要怎麼被交出去？ 

Resource:
  Texture memory / SDL_Texture / Win32 HBITMAP / CPU framebuffer pixels

Possible policies:
  duplicate  -> copy creates another independent resource
  share      -> copy points to same resource with shared lifetime policy
  transfer   -> source gives ownership to target
  borrow     -> non-owner observes resource temporarily

對 unique ownership 來說，copy 不是自然操作。你不能把一個只能被一個 owner 釋放的 resource 同時交給兩個 owner，否則會 double delete；你也不能讓 copy 偷走 source，否則 copy 的語意變成消耗 source。

Figure 1：unique ownership 的核心操作分類。auto_ptr 的失敗點是把 transfer 放進 copy channel。

1 naive model：auto_ptr 既然能轉移，那 move 只是語法改良？

先看一段 C++98 風格的程式：

std::auto_ptr<Texture> a(new Texture{});
std::auto_ptr<Texture> b = a; // looks like copy

直覺上你可能會說：這不就把 texture ownership 從 a 轉到 b 了嗎？所以 std::move 和 unique_ptr 只是把語法變漂亮？

問題就在 b = a。這個 expression 的表面形狀是 copy：source 是 lvalue，語法沒有任何「我願意放棄 a」的標記。但執行後，a 失去 ownership。 

Before:
  a -> Texture
  b -> null

Expression:
  b = a     // copy-looking syntax

After:
  a -> null
  b -> Texture
核心問題 — auto_ptr 做到的是 transfer，但它走的是 copy channel。這叫 destructive copy：語法看起來像複製，實際卻破壞 source。

2 C++98 卡在哪：沒有「只接受 temporary / 將亡物件」的 channel

C++98 有 copy constructor：

T(const T& other);

這能從 lvalue 或 temporary 建構新 object，但它的語意預設是 copy。對 unique ownership 來說，真正想要的是： 

I can transfer from a temporary:
  MakeTextureOwner() -> TextureOwner

I can transfer when caller explicitly gives up ownership:
  std::move(owner)

I should reject normal lvalue copy:
  TextureOwner b = a;  // no, this would duplicate unique ownership

可是 C++98 沒有 T&&。你無法在 function signature 裡清楚寫出「我只接受 rvalue / transferable source」。於是 auto_ptr 用 library 技巧硬做 ownership transfer。

為什麼 C++98 不能只把 copy 禁掉？

你可能會問：那 C++98 時代為什麼不直接把 copy constructor 設成 private，禁止 copy？答案是：可以禁止，但這樣 owner 就很難被當成 value 交出去。

class TextureOwner98 {
public:
  TextureOwner98(Texture* p) : ptr(p) {}

private:
  TextureOwner98(const TextureOwner98&); // no copy
  TextureOwner98& operator=(const TextureOwner98&);
  Texture* ptr;
};

TextureOwner98 LoadTexture(); // hard to express transfer by value in C++98

今天你會寫 return TextureOwner{};，讓 prvalue / move / elision 處理 transfer。但 C++98 沒有 rvalue reference，也沒有 C++17 的 guaranteed copy elision 模型。禁止 copy 後，很多「從 factory 回傳 owner」、「把 owner 放進某個結構」、「暫時交給另一個 object」的寫法都會卡住。 

C++98 choices for unique owner:
  allow copy      -> may duplicate ownership, dangerous
  forbid copy     -> hard to return/pass/store by value
  auto_ptr hack   -> transfer via copy-looking syntax

C++11 answer:
  forbid copy
  allow move
  let return-by-value use move/elision path

3 auto_ptr 的本質：把 copy constructor 改造成 ownership transfer

概念上，auto_ptr 的 copy constructor 像這樣：

template<class T>
class auto_ptr {
public:
  auto_ptr(auto_ptr& other) // not const!
    : ptr(other.release()) {}

  T* release() {
    T* p = ptr;
    ptr = nullptr;
    return p;
  }

private:
  T* ptr = nullptr;
};

注意 constructor 接的是 non-const reference。因為它必須修改 source：把 source 裡的 pointer 取走並清空。這也是為什麼它和正常 copy 的語意不同。 

normal copy:
  source remains usable and represents same value after copy

auto_ptr copy:
  source is modified and loses ownership

這個 non-const reference 又帶來另一個怪味道：正常 copy constructor 通常長這樣：

T(const T& other);

它能從 const source copy，因為 copy 不應改 source。但 auto_ptr 要偷 ownership，所以不能從 const auto_ptr& 轉移。這代表它連「copy constructor 應該接受 const source」這個常見期待都破壞了。

const std::auto_ptr<Texture> ca(new Texture{});
// std::auto_ptr<Texture> b = ca; // cannot transfer from const owner

這裡可以看到 auto_ptr 卡在兩個相反需求中間： 

To look copy-like:
  it should accept const T& and not modify source.

To transfer ownership:
  it must modify source and clear pointer.

These two requirements cannot both be true.
這不是單純命名問題 — 如果一個 type 的 copy 會改變 source，那 generic code 只要假設「copy 不破壞 source」就可能出錯。這不是 programmer 忘記看文件，而是 type 破壞了語言與 library 共同依賴的語意模型。

4 為什麼 container / algorithm 會受傷？

Generic container 和 algorithm 不知道你的 domain。它們依賴的是 type requirements。對一個 Copyable type，copy 之後 source 應該仍然是原本的 value。

假設 vector 重新配置 storage： 

old storage:
  [ p0 ][ p1 ][ p2 ]

vector grows:
  allocate new storage
  copy old elements to new storage
  destroy old storage

If element is auto_ptr:
  copying p0 transfers ownership out of old p0
  old p0 becomes null
  operation order now has surprising side effects

更一般地說，algorithm 也會出問題。某些排序、交換、暫存、複製 range 的流程，可能只是想搬動 value，但 auto_ptr 的 copy 會讓 source 失去 ownership。這使得它很難滿足 container/algorithm 對 value type 的期待。

具體壞在哪：不是「會變 null」這麼簡單

真正危險的是 generic code 的局部推理被破壞。假設某個 algorithm 為了重排元素，暫時 copy 一份：

template<class T>
void AlgorithmStep(T& x) {
  T tmp = x; // generic code expects x still represents same value
  Use(x); // with auto_ptr, x may now be empty
  Use(tmp);
}

對 std::string、Vec3、PipelineState 這類正常 value，這個 mental model 成立。對 auto_ptr，T tmp = x 已經把 x 消耗掉。Algorithm 不需要真的做錯什麼；它只是相信 Copyable type 的一般語意。 

Generic algorithm's assumption:
  T tmp = x;
  x is still a usable T with same semantic value.

auto_ptr reality:
  T tmp = x;
  x has been modified by the copy operation.

Problem:
  The operation name and type requirement say "copy",
  but the effect is "consume".
語意正常 Copyable typeauto_ptr
copy sourcesource 保持同一個 valuesource 被清空
copy targettarget 取得一份等價 valuetarget 取得唯一 ownership
generic assumptioncopy 不消耗 sourcecopy 會消耗 source
container suitability可作為一般 value type破壞很多泛型假設
所以問題不是 container 寫得不夠小心。 如果 generic library 必須到處特判「這個 type 的 copy 會不會偷走 source」，那 Copyable 這個 requirement 就失去工程價值了。

5 auto_ptr_ref：library hack 的味道

auto_ptr 還有一個歷史上很有代表性的設計：auto_ptr_ref proxy。你不需要記住細節 API，但要理解它暴露的問題。

因為 C++98 沒有 rvalue reference，auto_ptr 想支援一些 conversion / return / assignment 情境，就需要 proxy object 協助把 ownership 暫時包起來再交出去。這代表問題已經超出普通 library class 能自然表達的範圍。 

C++98 wants:
  distinguish "copy from stable lvalue"
  from       "transfer from temporary / explicit source"

But language has no T&&.

auto_ptr workaround:
  non-const copy
  release()
  proxy conversion helper
  surprising generic behavior
這是語言設計的訊號 — 當 library 需要 proxy trick 才能表達一個普遍需求，通常代表語言缺少一個語意通道。C++11 的 rvalue reference 就是在補這條通道。

6 C++11 的修正：move 是另一條語意通道

C++11 加入 rvalue reference 後，type 可以同時說清楚：

class TextureOwner {
public:
  TextureOwner(const TextureOwner&) = delete; // no copy
  TextureOwner& operator=(const TextureOwner&) = delete;

  TextureOwner(TextureOwner&& other) noexcept; // transfer allowed
  TextureOwner& operator=(TextureOwner&& other) noexcept;
};

現在 call site 也變誠實：

TextureOwner a = LoadTexture();
TextureOwner b = a; // compile error: copy disabled
TextureOwner c = std::move(a); // explicit transfer intent

std::move(a) 本身不搬東西。它只把 expression 轉成 xvalue，讓 overload resolution 選到 move constructor。真正搬 ownership 的還是 type 的 move constructor。 

copy channel:
  T(const T&)        -> source remains semantically intact

move channel:
  T(T&&)             -> source may become moved-from but valid

call site:
  b = a              -> copy attempt
  b = std::move(a)   -> explicit transfer attempt
Figure 2：std::move 的完整資料流。它只是把 expression 變成 xvalue；resource transfer 發生在被選中的 move constructor / move assignment 裡。

moved-from state 和 auto_ptr 的 empty source 有什麼不同？

兩者表面上都可能讓 source 變成「空」。差異在於：C++11 move 把這件事放進 move channel，並要求 moved-from object 仍然是一個可 destruct、可 assign、滿足 type invariant 的有效 object。 

auto_ptr copy:
  b = a;
  a becomes empty as a side effect of copy-looking syntax.

unique_ptr move:
  b = std::move(a);
  a becomes empty/null-like as a documented moved-from state.
  The call site explicitly marked transfer intent.

所以「source 可能變空」不是問題本身。真正問題是這個變化發生在哪條語意通道、call site 是否明確、generic code 是否能從 type requirements 推理。

7 為什麼 move assignment 仍然用 =，卻不是 auto_ptr 的問題？

你提出的疑問很關鍵：如果我們寫

Buffer& operator=(Buffer&& other) noexcept;

那 call site 仍然長得像：

b = std::move(a);

它不也是用 = 嗎？是不是只是把 destructive transfer 藏在 assignment syntax 裡？

答案是：= 只是 assignment expression syntax，不等於 copy semantics。真正決定 copy 或 move 的，是右邊 expression 的 value category，以及 overload resolution 選到哪個 operator=。

Buffer a;
Buffer b;

b = a; // RHS a is lvalue -> cannot bind to Buffer&&
b = std::move(a); // RHS is xvalue -> can bind to Buffer&& 
b = a
  syntax: assignment
  RHS expression: lvalue
  selected operation: operator=(const Buffer&) if available
  transfer intent: not expressed

b = std::move(a)
  syntax: assignment
  RHS expression: xvalue
  selected operation: operator=(Buffer&&)
  transfer intent: explicit at call site

C++11 沒有把 = 拆成 copy-assignment token 和 move-assignment token。它選擇更一般的設計：同一個 assignment expression，透過 RHS 的 value category 進入不同 overload。

Figure 3：assignment syntax 不等於 copy semantics。b = a 和 b = std::move(a) 的 RHS value category 不同，因此 overload resolution 選到不同 operator=。

這正好把前面幾章串起來： 

T-04 value category:
  std::move(a) produces xvalue.

T-23 overload resolution:
  xvalue chooses operator=(Buffer&&) over copy assignment.

T-24 forwarding:
  generic wrapper must use std::forward to preserve RHS intent.

T-05 object transfer:
  operator=(Buffer&&) defines how resource is actually transferred.
判斷重點 — move assignment 使用 = 沒問題，因為 transfer intent 由 RHS 的 xvalue 明確表達；auto_ptr 的問題是 b = a 中 RHS 仍是 lvalue，call site 沒有表達 transfer intent，卻發生 destructive transfer。

8 unique_ptr：auto_ptr 需求的正確現代表達

std::unique_ptr 不是「比較新的 auto_ptr」。它是同一個需求在 C++11 語意模型下的正確表達：

std::unique_ptr<Texture> a = std::make_unique<Texture>();
std::unique_ptr<Texture> b = a; // error
std::unique_ptr<Texture> c = std::move(a); // ok
設計點auto_ptrunique_ptr
copy syntax會 destructive transferdeleted，不能 copy
move syntax沒有語言層 move channel用 T&& 與 std::move
container interaction破壞 Copyable 假設MoveConstructible / MoveAssignable
source statecopy 後 source 被清空move 後 source valid but unspecified/null-like
現代使用不應使用；C++17 已移除unique ownership 的標準工具

9 和 Regular type 的關係

這個故事也解釋了為什麼後面值得補 T-26 Regular Types / Equality / Hashing / Ordering。C++ generic code 很依賴 type law。所謂 Copyable 不只是「有 copy constructor」；它還暗示 copy 不消耗 source，並且 copy 後兩者在語意上等價。 

Regular-ish value intuition:
  T b = a;
  // a is still usable
  // b represents the same value
  // algorithms can reason locally

auto_ptr:
  T b = a;
  // a changed
  // b owns what a owned
  // generic reasoning breaks

所以 move semantics 的價值不只是 performance。它讓 type 能誠實宣告自己不是 Copyable，但可以 Movable。這把 ownership transfer 從「偽裝成 copy」改成「獨立語意」。

10 Pixel-Renderer：哪些東西像 auto_ptr 的問題？

Pixel-Renderer 之後會有很多 unique ownership： 

DisplayBackend:
  owns window / SDL renderer / platform texture

Framebuffer:
  owns CPU pixel storage

TextureResource:
  owns decoded image or uploaded backend object

CommandBuffer:
  owns commands for this frame or replay

UI font atlas:
  owns glyph bitmap and texture handle

這些 type 如果不能 copy，就應該直接說不能 copy；如果能 move，就應該走 move constructor / move assignment。不要設計出「copy 看起來可以，但其實偷 ownership」的 API。

class SdlDisplayBackend {
public:
  SdlDisplayBackend(const SdlDisplayBackend&) = delete;
  SdlDisplayBackend& operator=(const SdlDisplayBackend&) = delete;

  SdlDisplayBackend(SdlDisplayBackend&&) noexcept = default;
  SdlDisplayBackend& operator=(SdlDisplayBackend&&) noexcept = default;
};
設計檢查 — 看到任何 owner type 都問：copy 是 duplication 嗎？copy 會共享嗎？copy 會偷走嗎？如果答案是第三個，這個 API 就在重演 auto_ptr 的問題。

Pixel-Renderer owner type checklist

把這個歷史教訓落到專案，可以用下面這張表檢查每個 type：

Type 類型copy policymove policy檢查重點
Vec3, RGBA8, Rectcopyable valuemove 沒特殊意義copy 後 source 和 target 都是同一個 semantic value。
Framebuffer明確選擇：deep copy 或 delete copymove 可轉移 pixel storage不能讓 copy 偷走 pixels；如果 copy 成本高，寧可 delete 或提供 Clone()。

如果你真的需要「複製一份 resource」，名字應該把成本和語意說出來，例如 Clone()、CopyPixels()、DuplicateTextureData()。不要讓普通 copy constructor 同時有時候 deep copy、有時候 share、有時候 transfer。 

Good:
  Framebuffer b = a.Clone();      // explicit deep duplication
  Framebuffer c = std::move(a);   // explicit ownership transfer
  FramebufferView v = b.View();   // explicit borrow

Bad:
  Framebuffer b = a;              // if this secretly steals pixels, auto_ptr problem returns

11 分層結論：本質差異在哪？

層次auto_ptrmove semantics + unique_ptr
語法表面b = a 看起來像 copyb = std::move(a) 明確標出 transfer intent
底層機制copy constructor 呼叫 release()overload resolution 選 T(T&&)
語意destructive copycopy disabled，move explicit
invariantsource copy 後失去 ownershipsource move 後仍 valid，但內容可被轉移
泛型相容性破壞 Copyable type 假設清楚滿足 Movable / non-Copyable
設計哲學library hack 補語言缺口語言提供 transfer channel，library 正常表達 ownership
本篇學到的三件事：
  1. auto_ptr 的問題不是做不到 ownership transfer，而是用 copy syntax 做 destructive transfer，破壞 Copyable / Regular type 的假設。
  2. C++11 move semantics 的核心價值不是只有效能，而是新增一條語言層 transfer channel：同一個 = assignment syntax 可以依 RHS value category 進入 copy 或 move overload。
  3. unique_ptr 是 unique ownership 的現代表達：copy deleted、move explicit，這正是 Pixel-Renderer 的 backend handle、framebuffer owner、texture resource 應該採用的語意方向。

回到主線時，T-05 負責建立 copy/move/object transfer 的基本模型；T-17 會把 unique_ptr、shared_ptr、weak_ptr、custom deleter、resource handle 這些 ownership adapter 系統化。
