Error Literacy：從 g++ 訊息到錯誤分層
先學會讀 compiler / linker / runtime / sanitizer 在說什麼，再談 error handling 設計
error-literacy g++ diagnostics linker

你會挑出 A-25，是因為你現在對「error」還沒有一張底層地圖。這很正常：C++ 的 error 不只一種。Compiler error、linker error、runtime failure、assertion、sanitizer report、undefined behavior 都會被口語上叫「錯」，但它們不是同一層。

所以在談 exception / error code / expected 前，應該先學會一件更基礎的能力：error literacy，也就是看懂誰在報錯、它在哪個階段報錯、訊息裡哪一行是 root cause、哪些只是附帶 note。

本篇範圍 — 先建立錯誤分層：compile-time error、link-time error、runtime failure、programmer bug、undefined behavior、sanitizer report。接著拆 g++ 訊息：file:line、error、warning、note、candidate、required from、undefined reference。最後回到 Pixel-Renderer：你遇到 build 失敗時，如何先判斷是 C++ 語意、header/linker、build command、還是程式執行期 failure。

1 naive model：error 就是程式錯了

這句話太粗。你要先問：誰說錯？在什麼階段說錯？

階段誰報它在檢查什麼例子
compile timecompiler語法、型別、overload、template 規則cannot bind int& to rvalue
link timelinkersymbol 是否找得到、是否重複undefined reference
runtime程式 / OS / library外部世界是否成功檔案不存在、SDL init failed
debug contractassert程式自己的 invariantindex out of range
sanitizerASan / UBSanruntime 偵測 memory/UB 類 bugheap-buffer-overflow
undefined behavior可能沒人報語言規則已破壞dangling reference、out-of-bounds write
第一個修正 — 不要看到 error 就直接改 code。先分類：這是 compiler 不接受 source，linker 找不到 symbol，還是程式跑起來後外部資源失敗？

2 g++ 訊息的基本結構

file.cpp:line:column: error: main message
  source line
      ^
file.cpp:other-line: note: extra context
candidate: possible function...
required from here...

你讀 g++ 訊息時，不要從最上面一路被淹沒。先找第一個真正的 error:，再看它附近的 note:。如果有很多 template trace，通常最底層或最靠近你自己檔案的位置才是要看的點。

詞意思讀法
errorcompiler 無法產生 object file必須修。
warning可編譯，但可能有 bug 或可疑寫法專案應盡量修，尤其開 -Wall -Wextra。
note補充上下文不是新的錯，而是解釋 error 為何發生。
candidateoverload resolution 考慮過的 function看為什麼每個 candidate 不符合。
required fromtemplate instantiation chain追哪個 template 使用觸發了錯誤。

3 範例：cannot bind int& to rvalue

void f(int& x) {}

int main() {
  f(3);
} 
error: cannot bind non-const lvalue reference of type 'int&'
       to an rvalue of type 'int'

拆法： 

cannot bind
  reference binding failed

non-const lvalue reference of type 'int&'
  f wants a mutable lvalue int object

to an rvalue of type 'int'
  you passed 3, a temporary value

real meaning:
  int& means "I need an existing modifiable int object."
  3 is not an existing named object you can mutate.

這不是 runtime error。程式根本還沒跑。Compiler 在 type/value category 規則那一層拒絕了 source code。

4 範例：no matching function 與 candidate

void Draw(Line line);
void Draw(Triangle tri);

Draw(Rect{}); 
error: no matching function for call to 'Draw(Rect)'
note: candidate: void Draw(Line)
note:   no known conversion from 'Rect' to 'Line'
note: candidate: void Draw(Triangle)
note:   no known conversion from 'Rect' to 'Triangle'

這種訊息要讀成 overload resolution 的失敗報告： 

Compiler tried to find Draw(...)
  candidate 1: Draw(Line)      rejected
  candidate 2: Draw(Triangle)  rejected

Root cause:
  You called Draw with Rect,
  but no overload accepts Rect and no conversion exists.

5 Linker error：undefined reference 不是語法錯

如果 compiler 已經產生 object file，但 linker 找不到 function definition，你會看到類似： 

undefined reference to `Framebuffer::Clear(unsigned int)'

這通常代表：

  • 你宣告了 function，但沒有定義。
  • 定義在另一個 .cpp，但 build command 沒有把那個 object file link 進來。
  • signature 不一致：宣告和定義的 parameter / namespace / const 不一樣。
  • library link flag 缺失，例如 SDL library 沒有被 linker 加進來。
和 A-03 的關係 — A-32 先教你辨認「這是 linker error」；A-03 可以再深挖 declaration、definition、translation unit、ODR、symbol、ABI 的問題生成史。

6 Template error：先找你自己的 call site

Template error 常爆很長，因為 compiler 會列出 instantiation chain。不要先試圖理解每一行 STL 內部訊息。先找兩種位置： 

1. first error that states the failed rule
2. line in your code that triggered instantiation

例如 std::vector<TextureOwner> 如果需要 copy，但 TextureOwner deleted copy constructor，你可能會看到很多 allocator / uninitialized_copy 相關訊息。真正意思可能只是： 

vector operation tried to copy TextureOwner
TextureOwner copy constructor is deleted
Therefore this operation is not allowed

這時你要回到 type semantics：這個 container operation 是否真的需要 copy？能不能改成 move？move constructor 是否 noexcept？還是這個 type 本來就不該放進需要 copy 的 container operation？

7 Runtime failure 和 error handling 設計是下一層

等程式跑起來後，才會進入 A-25 講的問題：SDL init 失敗、asset path 找不到、window create failed、golden image mismatch，這些 failure 要怎麼回傳給 caller？ 

Build-time diagnostics:
  compiler/linker rejects program before it runs

Runtime failure:
  program runs, external condition fails

API error handling:
  program decides how to report runtime failure to caller

所以 A-32 是 A-25 的前置篇：先能讀懂「誰在報錯」，再決定「程式內部該怎麼設計錯誤通道」。

Figure 1：先判斷錯誤在哪個階段發生，才知道該看 type rule、symbol/link command、runtime failure，還是 sanitizer report。

8 讀錯誤的固定流程

When you see an error:

1. Identify the phase:
   compiler / linker / runtime / assert / sanitizer

2. Find the first real error:
   ignore repeated notes first

3. Locate your source line:
   not only STL/header internals

4. Translate the message:
   what rule failed?

5. Map it to concept:
   reference binding / overload / symbol / lifetime / ownership

6. Fix the cause, not the nearest symptom.
本篇學到的三件事：
  1. Error literacy 的第一步是分層：compiler error、linker error、runtime failure、assertion、sanitizer report、undefined behavior 不是同一種錯。
  2. g++ 訊息要拆成 error、note、candidate、required from：先找第一個真正 error，再回到你自己的 source line。
  3. A-32 處理「怎麼讀錯誤」；A-25 才處理「程式跑起來後，runtime failure 要怎麼透過 API 傳給 caller」。
