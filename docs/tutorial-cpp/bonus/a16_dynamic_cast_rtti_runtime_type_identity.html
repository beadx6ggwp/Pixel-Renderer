dynamic-cast RTTI runtime-polymorphism type-identity

T-11 已經比較 virtual、template、type erasure、std::variant。但有一個很容易讓人困惑的工具還沒拆深：dynamic_cast。它看起來像「從 base class 偷偷摸回 derived class」，所以很多 C++ 文章會直接說它是 design smell。

這句話有一半對。dynamic_cast 常常代表你切錯抽象，但它不是毫無存在理由。它真正回答的是：當你手上只有 base pointer / reference，而你需要在 runtime 檢查 object 的 actual dynamic type 時，語言能不能給你一條 type-safe 的路？

本篇範圍 — 從 naive downcast 開始，拆出 static type / dynamic type 的差異、RTTI 和 polymorphic base 的關係、dynamic_cast<T*> 與 dynamic_cast<T&> 的失敗語意、multiple inheritance / cross-cast 為什麼需要 runtime type metadata。最後回到 Pixel-Renderer：哪些地方不應該用 dynamic_cast，哪些 debug/editor/plugin boundary 可以接受，以及 variant、visitor、virtual function、type erasure 如何替代它。

1 naive model：我知道它其實是 Derived，所以 cast 回去就好

想像你有一個 backend interface：

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
  virtual void Present(const Framebuffer&) = 0;
};

class SdlDisplayBackend : public DisplayBackend {
public:
  SDL_Window* Window() const;
};

某個地方拿到 DisplayBackend*，突然想呼叫 SDL 專屬 API：

void DebugSdlWindow(DisplayBackend* backend) {
  auto* sdl = static_cast<SdlDisplayBackend*>(backend);
  Inspect(sdl->Window());
}

這段的危險點是：static_cast 在這裡只相信你。若實際 object 是 Win32DisplayBackend，這個 pointer 會被當成錯的 object layout 使用，後面可能直接 undefined behavior。

核心 failure point — downcast 不是一般型別轉換。你不是把資料換一種表示法，而是在宣稱「這個 base subobject 背後的 complete object 是某個 derived type」。如果宣稱錯了，object layout 和 member access 都會錯。

理解 dynamic_cast 前，先分清兩個 type：

Virtual dispatch 使用 dynamic type 選 function；dynamic_cast 使用 dynamic type 檢查「能不能安全轉到目標 derived/base/cross type」。

3 dynamic_cast 做了什麼

最常見的安全 downcast 是 pointer 版本：

if (auto* sdl = dynamic_cast<SdlDisplayBackend*>(backend)) {
  Inspect(sdl->Window());
} else {
  Log("not an SDL backend");
}

如果成功，它回傳調整後的 pointer；如果失敗，它回傳 nullptr。Reference 版本沒有 null 可以回傳，所以失敗時丟 std::bad_cast：

SdlDisplayBackend& sdl = dynamic_cast<SdlDisplayBackend&>(backendRef);

這要求 source type 是 polymorphic class，也就是通常要有至少一個 virtual function。原因不是語法任性，而是 runtime 需要有可追蹤 dynamic type 的 metadata 路徑。沒有 virtual 的 class，語言通常不保證 object 裡有 RTTI dispatch 所需的資訊。

4 RTTI 是什麼：不是魔法，是 runtime type metadata

RTTI 是 Run-Time Type Information。在 C++ 裡，你會透過 typeid、dynamic_cast 間接接觸它。實作細節不由標準固定，但常見模型是 polymorphic object 透過 vptr 連到 vtable，vtable 附近能找到 type metadata 與 class hierarchy 資訊。 

Polymorphic object (rough model):

object memory:
  [ vptr ] [ base members ] [ derived members ]
       |
       v
  vtable / RTTI metadata:
    virtual function entries
    type_info
    hierarchy / offset info
不要過度具體化 — vptr/vtable/type_info 是理解用的常見模型，不是你可以在 portable C++ 裡手動依賴的 layout contract。真正可依賴的是語言規則：polymorphic hierarchy 上的 dynamic_cast 會做 runtime checked conversion。

RTTI 不只處理單一繼承的 downcast。它也要處理 multiple inheritance 下的 pointer adjustment 和 cross-cast：

struct IResizable { virtual ~IResizable() = default; };
struct IPresentable { virtual ~IPresentable() = default; };
struct SdlWindowBackend : IResizable, IPresentable {};

IResizable* r = new SdlWindowBackend;
IPresentable* p = dynamic_cast<IPresentable*>(r); // cross-cast

這不是簡單「改一下型別名稱」。因為 IResizable base subobject 和 IPresentable base subobject 在 complete object 裡可能有不同 offset。dynamic_cast 需要 runtime metadata 來找正確位置。

5 RTTI 不是完整 reflection

這裡很容易和 C# reflection 混在一起。兩者都屬於 runtime type information 的大範圍，但解的問題不同。

C++ RTTI 解的是比較窄的問題：給我一個 polymorphic base pointer / reference，我能不能在 runtime 問它的 dynamic type，並安全地轉成某個 derived/base/cross type？ 

C++ RTTI / dynamic_cast can answer:
  Is this Base* actually pointing to a Derived object?
  Can this interface pointer cross-cast to another interface?
  What is the type_info identity of this polymorphic object?

C++ RTTI usually cannot answer:
  What fields does this struct have?
  What are the field names?
  Can I enumerate methods?
  Can I set property "x" by string?
  Can I auto-generate an inspector UI without extra metadata?

Reflection 解的是更寬的問題：把 type 本身當成資料來查詢、列舉、操作。 Serializer、debug UI、editor inspector、dependency injection、asset pipeline 會想知道一個 type 有哪些 fields、每個 field 的名字與型別、能不能讀寫、是否有 attribute/default value。

// Normal code: programmer knows the structure at source level.
v.x = 1.0f;

// Reflection/metadata-style code wants this shape:
for (auto field : FieldsOf<Vec3>()) {
  DrawEditorControl(field.name, field.Get(v));
}

標準 C++ 目前沒有 C# 那種完整內建 reflection。C++ codebase 若需要 editor/serializer/debug dump，通常會建立 explicit metadata layer：用 macro、template table、code generation、Clang tooling 或第三方 library 登記欄位。

和 Pixel-Renderer 的關係 — dynamic_cast 可幫 debug tool 問「這個 backend 是不是 SDL backend」；reflection/metadata 則會幫 future UI inspector 問「這個 Material 有哪些 shader parameters 可以畫成控制項」。這是相鄰問題，但不是同一層。

6 那為什麼很多 codebase 避免 dynamic_cast？

不是因為 dynamic_cast 一定慢或一定錯，而是它常暴露出更大的設計問題：

症狀背後問題更常見的設計方向
每次拿到 base 都先 cast 成 derivedbase interface 沒有表達真正需要的 operation把 operation 放進 virtual interface，或改成 capability interface。
根據 derived type 寫一串 if/elsetype set 其實是 closed setstd::variant + visitor。
hot loop 裡大量 castruntime type query 進入效能敏感路徑template policy、pre-dispatch、data-oriented layout。
跨 module / plugin cast 失敗或不穩ABI / RTTI boundary 複雜穩定 C ABI、explicit interface、capability query。
cast 後存 derived pointer 到長生命週期ownership/lifetime 不清楚owner / borrow / id 分清楚。
更精確的說法 — dynamic_cast 不是原罪。它是 runtime type query。問題是：如果核心架構一直需要問「你到底是哪個 derived type」，那 polymorphic boundary 可能沒有提供正確的抽象行為。

7 替代方案不是同一種東西

避免 dynamic_cast 時，不是機械地改成某個唯一工具。要先判斷你真正的問題是哪一類。

你真正想做的事可能工具例子
所有 derived 都應支援同一操作virtual functionDisplayBackend::Present()。
種類有限，而且要依型別做不同處理std::variant + visitorRenderCommand：Clear、DrawLine、DrawTriangle。
hot path policy 需要 inlinetemplate / conceptfragment shader、blend/depth policy。
需要保存未知 callable / object，但提供 value-like wrappertype erasuretool callback、small command adapter。
只想問 optional capabilitycapability interface / query APIbackend 是否支援 resize、clipboard、native window handle。

把 cast 改成 virtual function

// Cast-heavy style
if (auto* sdl = dynamic_cast<SdlDisplayBackend*>(backend)) {
  sdl->SetTitle(title);
}

// Better if every backend should support title
backend->SetWindowTitle(title);

把 cast 改成 variant visitor

using Command = std::variant<Clear, DrawLine, DrawTriangle>;

std::visit([](auto& cmd) {
  Execute(cmd);
}, command);

這裡不是因為 variant 比 dynamic_cast 高級，而是 command set 對 early renderer 來說通常是 closed set，而且 command 是資料，不一定需要 object identity。

8 什麼時候 dynamic_cast 是合理的？

有些情境它是務實工具：

  • Debug inspector / editor tooling：核心系統不依賴 cast，但 debug 面板想顯示 concrete object 的額外資訊。
  • Legacy API boundary：既有 polymorphic hierarchy 無法立刻改 interface。
  • Optional capability query：某些 backend 額外支援 native handle，但不是所有 backend 的核心 contract。
  • Test / assertion：測試想確認 factory 真的產生某個 derived type。
  • Multiple inheritance cross-cast：需要在 runtime 從一個 interface 切到同一 complete object 的另一個 interface。

關鍵是：cast 不應該是 renderer core 的日常控制流。它可以存在於 boundary、debug、migration、capability query，但不應讓 rasterizer 或 command execution 每一步都問 concrete type。

9 Pixel-Renderer 判斷地圖

場景建議理由
Application 呼叫 backend presentvirtual function，不需要 castPresent() 是所有 backend 的共同 contract。
Renderer core 想拿 SDL_Window*通常不要這代表 platform detail 洩漏進 core。
Debug panel 顯示 SDL native info可接受 dynamic_cast 或 capability querydebug/tooling 不應污染 core path。
RenderCommand 執行不同 commandvariant / visitorcommand 是 closed data set，適合 value semantics。
Fragment shader 每 pixel dispatchtemplate / concept避免 runtime type query 進 hot loop。
Backend optional feature明確 capability interface比到處 cast concrete backend 更可控。
class NativeWindowProvider {
public:
  virtual ~NativeWindowProvider() = default;
  virtual void* NativeWindowHandle() const = 0;
};

if (auto* native = dynamic_cast<NativeWindowProvider*>(backend)) {
  InspectNativeWindow(native->NativeWindowHandle());
}

這裡比直接 cast 成 SdlDisplayBackend 好一些，因為 caller 問的是 capability，不是 concrete class。但如果這個 capability 變成核心需求，仍應提升到 DisplayBackend 的正式 interface。

10 設計檢查表

Before using dynamic_cast, ask:

1. Am I in debug/tooling/migration code, or renderer core?
2. Do all derived types actually need this operation?
3. Is my type set closed enough for variant?
4. Is this in a hot loop?
5. Am I using cast to bypass a missing virtual function?
6. Is this really a capability query rather than a concrete type query?
7. Does this cross dynamic library / plugin ABI boundary?
8. Who owns the object after the cast, and how long does the borrow live?
Pixel-Renderer 原則 — 不要用 dynamic_cast 讓 renderer core 依賴 SDL/Win32 concrete backend。若 core 需要某個能力，把它放進 interface；若只有 debug tool 需要，可以把 cast 限制在 tooling boundary。

11 一張圖：dynamic_cast 在抽象選擇中的位置

Figure 2：dynamic_cast 是 runtime query 路線。它不是唯一多型工具，也不應取代 interface/data/policy 的設計。
本篇學到的三件事：
  1. dynamic_cast 解的是 runtime type identity 問題：從 polymorphic base pointer/reference 安全檢查 dynamic type，失敗時 pointer 回傳 nullptr、reference 丟 std::bad_cast。
  2. RTTI 讓 runtime checked downcast、cross-cast、typeid 成立；它不是 C# reflection 那種可列 field/method/property 的完整 metadata system。
  3. Pixel-Renderer core 不應靠 dynamic_cast 找 SDL/Win32 concrete type；核心能力應進 interface，closed command set 用 variant，hot path 用 template，debug/tooling/capability boundary 才可有限使用 RTTI。

回到主線時，T-11 會把 dynamic dispatch 的選擇拉回更大的地圖：virtual、type erasure、variant、template 各自處理不同的變化點。這篇 A-16 則是專門補上 runtime type identity 這個分支。
