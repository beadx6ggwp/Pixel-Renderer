為什麼 C++ 沒有真正的 interface keyword
從 abstract class、pure virtual、concepts、type erasure、ABI boundary 比較不同 interface 意義
interface abstract-class concepts ABI

從 C# / Java 回頭看 C++，你可能會問：為什麼 C++ 沒有真正的 interface keyword？C++ 不是只能用 abstract class 模擬嗎？

這個問題的陷阱是：interface 不是單一概念。 有時它表示 runtime polymorphic object；有時表示 compile-time requirement；有時表示 binary/plugin boundary；有時只是 data protocol。C++ 沒有把其中一種形式命名成唯一的 interface，因為它同時保留了多種抽象路線。

本篇範圍 — 從 Java/C# interface 的直覺開始，拆解 C++ 裡的 abstract class、pure virtual、concept、type erasure、free function protocol、C ABI boundary 各自解什麼問題。最後回到 Pixel-Renderer：DisplayBackend 可用 abstract class；shader policy 可用 concept；RenderCommand 可用 value protocol；plugin / third-party boundary 可能要穩定 C-style interface。

1 naive model：interface 就是一組 methods

最直覺的 interface 模型是： 

interface = list of functions a type must implement

在 C++ 裡可以用 pure virtual class 表達：

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
  virtual void PollEvents(InputState&) = 0;
  virtual void Present(const Framebuffer&) = 0;
};

這看起來就是 interface。但它同時選擇了很多東西：runtime dispatch、object identity、pointer/reference use、virtual destructor、vtable ABI、可能的 heap ownership。

第一個修正 — pure virtual class 是一種 interface，但不是所有 interface。它是 runtime polymorphism 形式的 interface。

3 Concepts：compile-time interface

如果 shader type 在 compile time 已知，你不需要 virtual interface。你需要的是「這個 type 能不能被當成 shader 使用」。

template <class S>
concept FragmentShader = requires(S s, FragmentInput in) {
  { s.Fragment(in) } -> std::same_as<Color>;
};

這也是 interface，只是它不產生 runtime vtable。它是 compile-time contract：不符合就編譯失敗，符合就讓 compiler 看到 concrete type，可能 inline。

4 Type erasure：給使用者一個穩定 wrapper

std::function 也是 interface：它讓任何滿足 R(Args...) call signature 的 callable 變成同一個 wrapper type。 

Caller sees:
  std::function

Hidden behind wrapper:
  function pointer
  lambda
  functor
  bind expression

這不是 abstract base class，也不是 concept，但它同樣提供了一層 interface：外部只知道怎麼 call，不知道內部 concrete type。

5 Binary boundary：interface 也可能是 ABI 問題

如果你要跨 DLL、plugin、不同 compiler、不同 standard library ABI，C++ class interface 可能不是最穩定的選擇。Virtual class 涉及 vtable layout、name mangling、allocator ownership、exception boundary。

這時候很多工程會退回 C-style interface：

struct RendererPluginApi {
  void* userData;
  void (*present)(void* userData, const Framebuffer* fb);
  void (*destroy)(void* userData);
};

這看起來比較低階，但它把 ABI contract 寫得更明確。這也是 interface，只是不是 OO 語法。

Figure 1：C++ 的 interface 不是單一語法，而是依 boundary 類型選 abstract class、concept、type erasure 或 ABI table。

不要把所有 interface 都做成 virtual class — 這會把 runtime dispatch、heap ownership、object identity、ABI 風險帶到不需要它們的地方。Interface 先問 boundary，再選語法。
本篇學到的三件事：
  1. C++ 沒有單一 interface keyword，因為 interface 可能是 runtime object boundary、compile-time contract、value protocol、type-erased wrapper 或 ABI boundary。
  2. Pure virtual class 是 runtime polymorphism 的 interface；concept 是 compile-time interface；type erasure 和 C ABI table 也各自提供不同形式的 interface。
  3. Pixel-Renderer 應讓 DisplayBackend 用 abstract class、shader policy 用 concept、command 用 value/variant、plugin boundary 才考慮 C ABI function table。

回到主線時，T-11 會比較 runtime polymorphism、type erasure、variant；T-08 會整理 template/concept。A-22 的目的，是把「interface」拆成多種 boundary 問題。
