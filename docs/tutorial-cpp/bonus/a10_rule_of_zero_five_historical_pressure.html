APPENDIX · A-10
Rule of Zero / Five 的歷史壓力
從 raw resource owner、double free、copy bug、move-only type 推到現代 C++ 的預設設計
rule-of-zero rule-of-five ownership move-only

Rule of Three、Rule of Five、Rule of Zero 很容易被背成口訣：「寫 destructor 就要寫 copy constructor、copy assignment；C++11 後還要 move constructor、move assignment；最好什麼都不要寫。」但這樣背會倒果為因。

真正問題是：當一個 class 擁有 resource 時，compiler 自動生成的 copy / assign / destroy 是否仍然符合 ownership invariant？ 如果答案是否定的，這些 rule 才會出現。

本篇範圍 — 從 raw pointer owner 的 double free 開始，推導 Rule of Three；再加入 C++11 move semantics，說明為什麼 Rule of Five 出現；最後解釋現代 C++ 為什麼偏好 Rule of Zero：把 ownership 交給 std::vector、std::unique_ptr、RAII wrapper，讓你的 domain type 不需要手寫特殊 member functions。

1 naive model：class 裡有 pointer，compiler 幫我 copy 就好

class Image {
public:
  Image(int count) : pixels_(new uint32_t[count]), count_(count) {}
  ~Image() { delete[] pixels_; }
private:
  uint32_t* pixels_ = nullptr;
  int count_ = 0;
};

如果你沒有寫 copy constructor，compiler 會做 memberwise copy。這對 int 沒問題，對 owning pointer 卻是錯的：兩個 Image object 會指向同一塊 pixel memory，兩個 destructor 都會 delete 它。 

Image a:
  pixels_ -> heap block

Image b = a;   // compiler-generated shallow copy

Image a:
  pixels_ ----+
              +-> same heap block
Image b: -----+

Destruction:
  ~b deletes heap block
  ~a deletes same heap block again
Rule 的起點 — 問題不是 pointer 本身，而是 compiler 不知道這個 pointer 是 owner。memberwise copy 對 value member 合理，對 unique owner 會破壞 ownership invariant。

2 Rule of Three：如果你管理 resource，就要定義三件事

C++98 的壓力是：有 destructor 表示你可能擁有某種 resource；那 copy constructor 和 copy assignment 也不能再讓 compiler 自動猜。

特殊 member要回答的問題
destructorobject 死亡時如何釋放 resource？
copy constructor從另一個 owner 建立新 object 時，是 deep copy、禁止 copy，還是 share？
copy assignment已經擁有 resource 的 object 被賦值時，舊 resource 如何處理？self-assignment 怎麼辦？

如果 resource 可以複製，copy constructor 應該 deep copy；如果不能複製，copy 應該禁止。但 C++98 沒有 = delete，常見做法是把 copy 宣告成 private 且不定義。這就是早期 C++ resource owner 很難寫對的原因。

3 C++11：move 讓「禁止 copy 但可轉移」變成一等語意

很多 resource 不能 copy，但可以 transfer。例如 SDL_Texture、file handle、Win32 handle、GPU resource。C++11 加入 move constructor / move assignment 後，Rule of Three 擴張成 Rule of Five。

新增 member要回答的問題
move constructor從即將被搬走的 source 建立新 owner，source 留在可 destruct state。
move assignment釋放目前持有的 resource，再接收 source 的 resource，source 留在可 destruct state。
class TextureOwner {
public:
  ~TextureOwner() { Destroy(handle_); }
  TextureOwner(const TextureOwner&) = delete;
  TextureOwner& operator=(const TextureOwner&) = delete;
  TextureOwner(TextureOwner&& other) noexcept
    : handle_(other.handle_) { other.handle_ = nullptr; }
private:
  TextureHandle handle_ = nullptr;
};

這段程式表達的是：這個 type 是 unique owner。不能 copy，但可以 move。move 後 source 不再 owns resource，但仍然能安全 destruct。

4 Rule of Zero：把 resource owner 拆到正確的 member

Rule of Zero 的意思不是「不要管 ownership」。它的意思是：讓專門的 RAII member 管 ownership，讓你的 domain type 只組合這些 members。

class Framebuffer {
public:
  Framebuffer(int w, int h)
    : width_(w), height_(h), pixels_(w * h) {}
private:
  int width_ = 0;
  int height_ = 0;
  std::vector<uint32_t> pixels_;
};

這個 Framebuffer 不需要寫 destructor / copy / move。std::vector 已經把 allocation、copy、move、destruction 寫對了。你的 class 剩下的是 domain invariant：width、height、pixels size 要一致。

Figure 1：Rule of Five 是低階 owner wrapper 的代價；Rule of Zero 是把代價隔離後，讓大多數 domain type 回到普通 value。

設計檢查 — 如果你開始手寫 destructor，立刻停下來問：這個 class 是不是低階 owner wrapper？如果不是，能不能把 resource 放進 vector、unique_ptr 或另一個小 RAII type？
本篇學到的三件事：
  1. Rule of Three/Five 不是口訣，而是 raw resource ownership 破壞 compiler-generated copy/assign/destroy 後逼出的修補規則。
  2. Rule of Zero 不是忽略 resource，而是把 resource management 放進專門 RAII member，讓 domain type 回到普通 value/invariant 設計。
  3. Pixel-Renderer 應讓 math/command/framebuffer domain type 盡量 Rule of Zero，把 SDL/Win32/file/resource handle 的 Rule of Five 壓縮在小 wrapper 裡。
