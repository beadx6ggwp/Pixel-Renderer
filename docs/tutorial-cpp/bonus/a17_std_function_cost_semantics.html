std::function 的成本與語意
從 lambda capture、type erasure、small buffer optimization、allocation、callback lifetime 解析
std-function type-erasure lambda callback-lifetime

std::function 很方便：你可以把 function pointer、lambda、functor 都塞進同一個型別。於是 naive model 會變成：只要需要 callback，就用 std::function。

但 Pixel-Renderer 後面會有 shader callback、UI event callback、tool command、job system task。這些 callback 的頻率、ownership、capture lifetime 都不一樣。std::function 解的是「保存任意 callable」的 type erasure 問題，不是所有 callable 問題的預設答案。

本篇範圍 — 從 callable type 各不相同開始，拆解 std::function 如何用 type erasure 保存 callable、為什麼可能 allocation、為什麼有 indirect call、copyability 對 captured state 有什麼要求，以及 lambda reference capture 延後執行時如何 dangling。最後回到 Pixel-Renderer：UI callback 可以用，fragment hot loop 不該預設用，command buffer 要小心 ownership。

1 naive model：callback 就用 std::function

std::function<void(Event&)> onClick;
onClick = [](Event& e) { HandleClick(e); };

這很合理。但如果你把 fragment shader 也寫成：

std::function<Color(FragmentInput)> fragmentShader;

每個 pixel / fragment 都透過 erased callable 呼叫，可能就把 indirect call、無法 inline、capture storage 的成本塞進 hot loop。

第一個分界 — std::function 適合低頻、保存、替換、工具層 callback；不適合預設放進 per-pixel/per-fragment hot path。

2 為什麼 lambda 型別需要被 erase

每個 lambda expression 都有獨一無二的 closure type：

auto a = [](int x) { return x + 1; };
auto b = [](int x) { return x + 1; };
// a and b have different compiler-generated types.

Template 可以保留 concrete type：

template <class F>
void CallNow(F&& f) { f(); }

但如果你要把 callback 存進 class member 或 vector，member 需要一個固定型別。std::function 就是把各種 callable 包成同一個 runtime wrapper。 

Different callable types:
  function pointer
  lambda without capture
  lambda with capture
  functor object
  bind expression

std::function:
  one wrapper type

4 成本不是只有「慢」

成本發生原因影響
indirect callcallable type 被 erased，call site 不一定能 inlinehot loop 可能受影響。
possible allocationcapture 太大或 implementation 不放進 small buffer建立 callback 時可能碰 allocator。
copyabilitystd::function 傳統上要求 stored callable 可 copymove-only capture 不適合直接塞。
empty statestd::function 可以沒有 target呼叫空 function 會丟 std::bad_function_call。
lifetime opacitycapture 內容被藏在 wrapper 裡reference capture 延後執行可能 dangling。

這些成本有些是 runtime，有些是語意成本。對 UI button callback，這通常可以接受；對 fragment shader，成本位置完全不同。

5 capture lifetime：最容易忽略的 bug

std::function<void()> MakeCallback() {
  int local = 42;
  return [&] { Use(local); }; // dangling reference
}

std::function 會保存 closure object，但不會延長 reference capture 指向的 local variable。這類 bug 在 UI callback、job task、deferred command 裡很常見，因為 callback 可能晚於建立它的 scope 執行。

不要讓 type erasure 掩蓋 lifetime — 當 callback 會被保存或延後執行，capture by reference 必須特別審查。你需要明確 value capture、shared ownership、resource id，或禁止跨 frame 保存。

6 Pixel-Renderer 選擇表

場景建議工具理由
fragment shader hot looptemplate callable / concept讓 compiler 看到 concrete type，有機會 inline。

deferred job taskmove-only task wrapper / explicit ownership避免 copyability 與 dangling capture 問題。

7 設計檢查表

Before using std::function, ask:

1. Do I need to store the callable, or just call it now?
2. Is this in a hot loop?
3. Can the callable allocate?
4. Does it capture references?
5. Can the callback outlive the captured objects?
6. Do I need copyable callable, or move-only task?
7. Would variant/plain command data be more inspectable?
8. Is this API boundary trying to hide too much ownership?
本篇學到的三件事：
  1. std::function 解的是保存任意 callable 的 type erasure 問題，不是所有 callback 的預設答案。
  2. 它的成本包含 indirect call、possible allocation、copyability requirement、empty state、capture lifetime opacity；其中 lifetime 常比速度更危險。
  3. Pixel-Renderer 應把 std::function 留給低頻 UI/tooling callback；shader hot loop 用 template，command buffer 優先用 plain data / variant。
