APPENDIX · A-02
Pointer 語意地圖：address、borrow、owner、handle
pointer 不是一種語意；它只是很多 lifetime / ownership policy 共用的表面形狀
pointer-semantics borrow ownership resource-id

A-01 用 auto_ptr 說明一件事：問題不在「指標能不能被搬」，而在 ownership transfer 走錯語意通道。這自然引出更大的疑問：C++ 裡這麼多像 pointer 的東西，T*、T&、span、unique_ptr、shared_ptr、weak_ptr、TextureId，它們到底差在哪？

這篇 Appendix 的目的不是教你背 API，而是建立一張語意地圖：同樣是「可以指到某個東西」，它可能只是 address、可能是 borrow、可能是 unique owner、可能是 shared lifetime、可能只是 resource registry 的 id。

本篇範圍 — 從 naive pointer model 開始，拆出 address / access path / borrow / unique ownership / shared ownership / weak observer / span-view / resource handle / generational id。最後回到 Pixel-Renderer：Framebuffer pointer、SDL handle、DisplayBackend owner、TextureId、Command payload 應該各自用哪種語意表達。

1 naive model：pointer 就是指向一個東西

最直覺的模型是： 

pointer = address of object

這個模型在 machine level 有用，但在 C++ design level 太粗。因為同樣是 T*，可能有完全不同的 contract：

Texture* FindTexture(TextureId id); // nullable borrow?
Texture* CreateTexture(); // caller owns and must delete?
void Draw(Framebuffer* fb); // borrow? optional? mutable?
void DestroyTexture(Texture* texture); // consumes ownership?
核心修正 — raw pointer 表面只說 address / access path，不足以說明 ownership、nullability、lifetime、mutation、release responsibility。這些語意如果不寫進 type 或 API contract，就會被註解和慣例承擔。

2 語意地圖：先問它是哪一種關係

Figure 1：pointer semantic map。先問關係是 borrow、owner、shared lifetime、registry handle，才決定用哪個 C++ 型別。

3 raw pointer：address / nullable access path

T* 最適合表達低階 access path，尤其是 C API boundary、nullable borrow、array data pointer。但它自己不說誰負責釋放。

Raw pointer 用法語意是否推薦
uint32_t* pixels指向 pixel storage 的 access path可以，但要搭配 size/pitch/lifetime owner。
SDL_Window*C library opaque handle應包進 RAII owner，外部不要裸露 ownership。
Texture* Find(...)nullable observer/borrow可以，但要明確不 owns。
Texture* Create()caller owns?不推薦；ownership 太隱晦。

4 reference：non-null borrow，不是 ownership

T& 表達「這裡需要一個已存在的 object，而且不接受 null」。它是 borrow，不是 owner。

void Clear(Framebuffer& fb, RGBA8 color);
void Present(const Framebuffer& fb);

這兩個 signature 很清楚： 

Framebuffer&:
  non-null mutable borrow

const Framebuffer&:
  non-null read-only borrow

Neither one owns the framebuffer.
Caller must keep owner alive.

5 span / view：borrowed range 要帶 size/pitch

Framebuffer pixels 不是只有 pointer。你還需要 width、height、pitch、format。這時 view 比裸 pointer 更誠實：

struct FramebufferView {
  std::span<uint32_t> pixels;
  int width = 0;
  int height = 0;
  int pitchPixels = 0;
};

View 通常 cheap copy，但不 owns lifetime。它的 invariant 是：owner 必須活得比 view 久。

View 不是比較安全的 owner — span / view 只是把 pointer + size 包在一起。它不延長 underlying storage lifetime。

6 unique_ptr：unique owner，不是比較安全的 pointer

unique_ptr 表達唯一擁有者。它的重點不是「會自動 delete」而已，而是 copy disabled、move explicit、destructor owns cleanup。

std::unique_ptr<DisplayBackend> display = CreateDisplayBackend();
display->Present(framebuffer);

如果 resource 不是用 delete 釋放，就用 custom deleter 或 RAII wrapper：

struct SdlWindowDeleter {
  void operator()(SDL_Window* w) const noexcept { SDL_DestroyWindow(w); }
};
using SdlWindowPtr = std::unique_ptr<SDL_Window, SdlWindowDeleter>;

7 shared_ptr / weak_ptr：control block policy

shared_ptr 不是「比較安全的 pointer」。它表示共享 ownership，背後有 control block、strong count、weak count。這個 policy 很重，不該用來逃避 owner 設計。 

shared_ptr:
  multiple owners
  object destroyed when last strong owner disappears
  can create cycles
  reference count update cost

weak_ptr:
  observes a shared ownership group
  does not keep object alive
  lock() may fail

Pixel-Renderer hot path、command buffer、shader resource binding 不應預設用 shared_ptr。如果你需要 stable identity，resource id / registry 通常更可控。

8 Resource id / handle：不是 pointer，是 lookup key

TextureId 不應該是假裝 pointer。它是 resource registry 的 key：

struct TextureId {
  uint32_t index = 0;
  uint32_t generation = 0;
};

Texture* Lookup(TextureId id);

這種設計把 lifetime 放在 registry，call site 拿的是 id。它可以避免 dangling pointer，也能讓 cache/reload/debug dump 有更明確的 identity。 

Pointer:
  direct address
  can dangle if owner dies

Resource id:
  indirect lookup
  can validate generation
  can represent missing/unloaded resource
  registry owns actual storage

本篇學到的三件事：
  1. Pointer-like 型別不是同一種語意；raw pointer 只表達 access path，reference/span 表達 borrow，unique_ptr/shared_ptr 表達 ownership policy，resource id 表達 registry lookup。
  2. 選 pointer 型別前要先問 lifetime：誰 owns？誰可以 mutate？是否 nullable？是否可跨 frame？是否需要 registry validation？
  3. Pixel-Renderer 的核心 hot path 應偏好明確 borrow / value / id；platform resource 用 RAII owner；不要用 shared_ptr 或 raw owning pointer 掩蓋架構問題。
