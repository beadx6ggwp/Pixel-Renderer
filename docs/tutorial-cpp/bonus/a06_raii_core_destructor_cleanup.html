APPENDIX · A-06
RAII 為什麼是 C++ 的核心，不只是 destructor 技巧
從 C cleanup path、goto fail、exception unwinding、partial initialization 推出 type-level lifetime
RAII destructor exception-unwinding resource-lifetime

T-06 已經把 RAII 放進主線：resource lifetime 應該綁到 object lifetime。但如果只把 RAII 記成「destructor 會自動釋放」，你會錯過它真正解決的問題：在多出口、多 resource、建構失敗、exception unwinding 都存在的語言裡，cleanup 要怎麼變成局部且可組合的規則？

這篇 Appendix 不是再教一次 unique_ptr。它要往回追：如果你從 C 的 acquire/release、goto cleanup、manual flag 開始，一步一步撞到 control flow 的複雜度，就會比較自然地理解為什麼 C++ 把 destructor、constructor、stack unwinding、member destruction order 設計成一組語意。

本篇範圍 — 從 naive cleanup model 開始，拆解 C 的 goto cleanup 為什麼合理但不夠，把 exception / early return / partial initialization 納入模型，推到 RAII 的核心 invariant：只要 object lifetime 結束，對應 resource release path 就必須執行一次。最後回到 Pixel-Renderer：Framebuffer pixel storage、Win32 handle、SDL_Window / SDL_Texture、temporary state change、profiling scope 應如何用 RAII 表達。

1 naive model：記得在函式結尾 release 就好

最直覺的 resource 管理模型是線性的： 

Acquire resource
Use resource
Release resource

如果函式只有一條路徑，這個模型沒有問題：

SDL_Texture* tex = SDL_CreateTexture(renderer, fmt, access, w, h);
RenderSomething(tex);
SDL_DestroyTexture(tex);

但真實程式不會這麼單純。你會有 validation failure、early return、第二個 resource 建立失敗、debug branch、exception、resize、hot reload。問題不是你不知道要 release，而是 control flow 一分岔，release responsibility 就開始散落。 

Acquire A
  if fail -> return
Acquire B
  if fail -> release A, return
Acquire C
  if fail -> release B, release A, return
Use A/B/C
  if early return -> release C/B/A
Normal exit -> release C/B/A
第一個 failure point — manual cleanup 的真正成本不是多打一行 Release()，而是每新增一條離開路徑，就要重新驗證所有已取得 resource 是否都被正確釋放，而且釋放順序是否正確。

2 C 的 goto cleanup 其實不是笨方法

在 C 裡，成熟工程常用 goto cleanup 把多個 failure path 收斂到同一個出口：

Texture* texture = NULL;
Buffer* staging = NULL;
bool ok = false;

texture = CreateTexture();
if (!texture) goto cleanup;

staging = CreateStagingBuffer();
if (!staging) goto cleanup;

ok = Upload(texture, staging);

cleanup:
if (staging) DestroyBuffer(staging);
if (texture && !ok) DestroyTexture(texture);

這不是低級寫法。它是在沒有 destructors、沒有 stack unwinding 的語言裡，把 release path 集中化的一種 discipline。它比每個 return 前手動釋放可靠。

但它仍有幾個結構限制：

  • cleanup policy 仍然活在 function body 裡，不活在 type 裡。
  • caller 看不到 Texture* 到底是 owner、borrow，還是 optional output。
  • 如果 resource 被傳到別的 function，ownership transfer 需要靠命名或註解。
  • 只要中途插入新 resource，就要手動修改 cleanup 區塊與成功條件。
重要視角 — RAII 不是「C 寫得爛所以 C++ 自動清」。更精確地說，RAII 是把 C 的 cleanup discipline 從 function-level convention 升級成 type-level invariant。

3 為什麼 exception 讓問題變尖銳

如果 C++ 只有 return path，manual cleanup 還能勉強靠規範撐住。但 C++ 有 exception：control flow 可以從任意深處跳出目前 scope。

void LoadTexture() {
  SDL_Texture* tex = SDL_CreateTexture(...);
  DecodePng(); // may throw
  UploadPixels(tex);
  SDL_DestroyTexture(tex);
}

如果 DecodePng() throw，最後那行 destroy 不會執行。你可以說「那我們專案不用 exception」，但即使你自己不 throw，standard library allocation、third-party library wrapper、constructor failure、test harness 都可能讓 unwinding 進入你的 code boundary。

C++ 的語言答案是：stack unwinding 會呼叫已建構完成的 local objects 的 destructor。

Figure 1：manual cleanup 依賴 control flow 到達 release line；RAII 依賴 object lifetime 結束時 destructor 被語言呼叫。

4 partial initialization：建構失敗時誰負責釋放？

更深的問題是 constructor 中途失敗。如果一個 object 有三個 member，前兩個建好了，第三個建構時 throw，整個 object 的 destructor 不會被呼叫，因為完整 object 尚未建構完成。

但 C++ 會 destruct 已經成功建構的 member，順序和建構順序相反： 

class RendererBackend {
  WindowOwner window;       // constructed 1st
  RendererOwner renderer;   // constructed 2nd
  TextureOwner texture;     // throws while constructing 3rd
};

If TextureOwner construction throws:
  ~RendererOwner()
  ~WindowOwner()
  // ~RendererBackend() does not run

這件事非常關鍵：如果每個 member 自己都是 RAII owner，那 partial initialization 自然安全；如果你把三個 raw handle 放在同一個 class 裡，然後期待 class destructor 統一釋放，建構失敗時 destructor 可能根本不會跑。

嚴重錯誤 — 不要在 constructor 裡取得多個 raw resource，然後只靠最外層 destructor 清。constructor 中途失敗時，完整 object destructor 不會執行；你需要讓每個已取得的 resource 立刻進入自己的 RAII member。

5 RAII 的正式語意：resource lifetime 是 object invariant

現在可以把 RAII 從口號整理成 invariant： 

For an owning RAII type T:

1. Constructed T owns either a valid resource or a well-defined empty state.
2. Destroying T releases the owned resource exactly once.
3. Moving T transfers ownership and leaves source destructible.
4. Copying T is disabled unless the resource can be meaningfully duplicated.
5. All control-flow exits use the same destruction rule.

這就是為什麼 RAII 會和 copy/move、Rule of Zero/Five、noexcept move、custom deleter、smart pointer 接在一起。它不是獨立技巧，而是 C++ object model、lifetime model、exception model 的交會點。

6 destructor 不應該 throw

如果 destructor 在 stack unwinding 期間又 throw，程式通常會進入 std::terminate。原因很直接：語言已經在處理一個 exception，如果 cleanup path 又丟出第二個 exception，runtime 很難知道哪個 failure 應該繼續傳播。 

Throw from work:
  DecodePng() throws
  stack unwinding starts
  ~TextureOwner() runs
  destructor throws again
  -> terminate

所以 RAII release operation 應該設計成不 throw。對 OS / SDL / Win32 handle 這類 release API，通常 destroy function 本身也不該用 exception 表達失敗。若 release 有錯誤狀態，應記 log、assert debug invariant，或提供顯式 Close() / Flush() 讓 caller 在 destructor 前處理可回報錯誤。

7 scope guard：不是所有 cleanup 都是 resource owner

有些 cleanup 不是「擁有一個 resource」，而是「離開 scope 時恢復某個狀態」。例如暫時切換 renderer state、暫時 push debug marker、暫時改 working directory、暫時 lock mutex。這時可以用 scope guard 或 specialized RAII type。

class ScopedBlendMode {
public:
  ScopedBlendMode(Renderer& r, BlendMode next)
    : renderer_(r), old_(r.BlendMode()) {
    renderer_.SetBlendMode(next);
  }
  ~ScopedBlendMode() { renderer_.SetBlendMode(old_); }
private:
  Renderer& renderer_;
  BlendMode old_;
};

這和 unique_ptr 不同，但抽象形狀一樣：constructor 建立暫時 invariant，destructor 回復 invariant。

8 Pixel-Renderer：哪些東西需要 RAII？

場景RAII 型別重點 invariant
CPU framebuffer pixelsFramebuffer owns std::vector<uint32_t>valid object 代表 pixel storage 與 width/height/pitch 一致。
Win32 HBITMAP / HDCDibSectionOwner / DcOwnerdestructor 呼叫對應 Win32 release；copy disabled，move allowed。
SDL window / renderer / textureSdlWindow / SdlRenderer / SdlTextureopaque handle 不外洩 ownership；release 順序由 member order 表達。
temporary UI clippingScopedClipRect離開 scope 時恢復上一個 clip rect。
profiling markerScopedTimerconstructor 記起點，destructor 記終點並提交 sample。
mutex lockstd::lock_guard所有離開路徑都 unlock。

對未來的 SDL backend，第一版可以長這樣：

struct SdlTextureDeleter {
  void operator()(SDL_Texture* p) const noexcept {
    if (p) SDL_DestroyTexture(p);
  }
};

using SdlTexture = std::unique_ptr<SDL_Texture, SdlTextureDeleter>;

這個型別很小，但語意很重：它讓 SDL_Texture* 從「裸 handle」變成 move-only owner。呼叫端不需要記得 destroy；container、return by value、early return、exception unwinding 都會走同一個 cleanup rule。

9 RAII 不是把所有東西都包成 class

RAII 的反面不是「不用 class」。真正的設計判準是：這個東西是否有需要被保證結束的 lifetime boundary？

問題如果答案是 yes如果答案是 no
是否需要 release / restore / unlock？應考慮 RAII owner 或 scope guard。普通 value / view 即可。
是否跨 early return / exception？RAII 幾乎是預設答案。單純 local computation 不需要包裝。
是否有 ownership transfer？copy/move policy 必須明確。borrow/reference/span 可能更合適。
destruction 是否有順序依賴？用 member declaration order 表達。不要製造多餘 wrapper。
不要過度抽象 — RAII 是 lifetime invariant，不是「看到 resource 就包一層 class」的儀式。若只是暫時 borrowed view，應用 reference/span；若是 registry lookup，應用 id/handle；若是真正 owns/release，才需要 owner type。

10 一張結構圖：RAII 在 C++ 語意森林裡的位置

Figure 2：RAII 不是孤立語法。它把 object model、move semantics、exception model、API design 合在同一個 lifetime invariant 裡。
本篇學到的三件事：
  1. RAII 的本質不是 destructor 技巧，而是把 C 的 cleanup discipline 從 function-level convention 升級成 type-level lifetime invariant。
  2. Exception unwinding 和 partial initialization 讓 RAII 變成 C++ 核心語意：已建構完成的 local object / member 會在離開 scope 或建構失敗時自動走 destructor。
  3. Pixel-Renderer 應把 CPU framebuffer、Win32/SDL handles、temporary UI state、profiling scope 都用 owner / scope guard / borrow 的語意分清楚，而不是靠 raw pointer 和註解記憶 cleanup。
