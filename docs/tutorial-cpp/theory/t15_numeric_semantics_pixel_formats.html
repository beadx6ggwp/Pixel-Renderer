Numeric Semantics / Pixel Formats
renderer 不是只在算數字，而是在承諾每個 bit、range、rounding 和 pixel layout 的語意
integer floating-point bit-ops pixel-format

Pixel-Renderer 很快會開始處理 color、depth、screen coordinate、texture coordinate、alpha blend。直覺上這些都是「數字」，但 C++ 裡的數字不是抽象數學物件。它們有 width、signedness、overflow rule、rounding、conversion、endianness、bit layout。

這章補的是目前 C++ 主線的一個洞：renderer numeric semantics。它不重推圖學數學，而是問：當你把 Color{255, 128, 0} pack 成 uint32_t，當你把 float depth 轉成 fixed-point，當你算 x + width * y，C++ 到底保證了什麼？哪些地方如果想錯，畫面可能不只是錯，還可能進入 undefined behavior？

本章範圍 — 建立 software renderer 需要的 numeric model：fixed-width integer、signed/unsigned conversion、overflow、bit shift、packed pixel format、endianness、float precision、rounding、epsilon、deterministic comparison。目標是讓 framebuffer、color、depth、texture coordinate、golden image test 的資料語意可檢查。

1 naive model：數字就是數學數字

最容易形成的模型是： 

int is integer
float is real number
uint32_t color is just 4 bytes
overflow means value wraps around
pixel format is just naming RGBA

這些都只對一半。C++ 的 numeric types 是 machine-level value representation 加上一套語意規則。Renderer 又特別常把「數值」和「bit layout」混在一起，所以你必須先分清：

  • numeric value：例如 red channel 的值是 255。
  • object representation：這個 value 在 memory 裡的 bytes。
  • semantic format：例如 pixel 被定義成 RGBA8、BGRA8、ARGB8。
  • API interpretation：Win32 / SDL / image writer 會如何解讀這些 bytes。
第一個錯誤 — 不要把「我看到 memory 裡有四個 byte」直接等同於「平台會把它當成 RGBA」。Packed pixel 的問題是 semantic contract，不只是 storage 問題。

2 Fixed-width integer：先決定你要幾個 bit

Renderer code 應該少用裸 int 表示 pixel storage 或 binary format。int 的大小雖然在主流平台通常是 32-bit，但 C++ 語意上它不是「檔案格式 / framebuffer format」的精準承諾。

Type適合表達注意
intloop index、一般 arithmetic。不要用來承諾 pixel format width。
uint8_tcolor channel、byte buffer。運算時常會 integer promotion。
uint32_tRGBA8/BGRA8 packed pixel。要明確定義 channel bit position。
size_tcontainer size、memory size、byte count。不要和 signed coordinate 隨便混算。
floatnormalized coordinate、depth、interpolation。不是精確實數，comparison 要有策略。

例如 framebuffer index 常見寫法：

size_t index = static_cast<size_t>(y) * width + x;

這行看似簡單，但它暗含前提：x、y 已經通過 bounds check，而且轉成 size_t 前不是負數。如果 y 是 -1，轉成 unsigned 後會變成巨大值，接著 index 可能越界。

座標用 signed，size 用 unsigned — screen coordinate 可能暫時落在 viewport 外，所以常用 signed integer；buffer size 不可能是負數，所以用 size_t。兩者交界處必須有明確 bounds check。

3 Overflow：signed 和 unsigned 不是同一種錯

C++ 對 overflow 的規則非常重要：

情況C++ 語意Renderer 後果
unsigned overflowmodulo 2^N wrap。適合 bit mask、hash、packing，但不一定適合 color math。
signed overflowundefined behavior。compiler 可假設不發生，導致最佳化後結果不可預期。
narrowing conversion可能截斷或 implementation-defined。float/int/channel 轉換要 clamp。

Color blending 不能靠 uint8_t 自己 wrap：

// wrong semantic: 250 + 20 wraps to 14 if stored in uint8_t
uint8_t r = a.r + b.r;

// better: widen, compute, clamp, then narrow
int r32 = static_cast<int>(a.r) + static_cast<int>(b.r);
uint8_t r = ClampToByte(r32);

這裡的重點不是「哪個比較快」，而是語意。Wrap、saturate、clamp、normalize 是不同操作，不能混在一起。

4 Packed pixel：RGBA 是語意，不是自然定律

假設你定義：

uint32_t pixel = (r << 24) | (g << 16) | (b << 8) | a;

這只是你定義的 bit layout。Memory 裡的 byte order 還會受 endianness 影響；而 SDL / Win32 / image writer 又可能要求另一種 channel order。最好把它明確命名：

struct RGBA8 {
  uint8_t r, g, b, a;
};

uint32_t PackRGBA8(RGBA8 c) {
  return (static_cast<uint32_t>(c.r) << 24) |
         (static_cast<uint32_t>(c.g) << 16) |
         (static_cast<uint32_t>(c.b) << 8) |
         static_cast<uint32_t>(c.a);
}

注意每個 channel 都先轉成 uint32_t 再 shift。這讓 shift 的 width、promotion、overflow 更可控。

SVG：packed pixel 至少有三層：你定義的 bit layout、CPU 實際 byte order、外部 API 對 bytes 的解讀。不要把三者混成同一件事。

5 Bit shift：先轉型，再位移

Bit operation 很像底層，但 C++ 仍然有規則：

  • shift count 不能大於等於 promoted type 的 bit width。
  • 對 signed negative value 做 shift 會有危險語意。
  • uint8_t 參與運算時通常會先被 promoted 成 int。
  • mask、shift、or 的每一步最好都在明確的 unsigned width 裡做。
uint32_t r32 = static_cast<uint32_t>(r);
uint32_t pixel = (r32 & 0xffu) << 24;

這種 code 看起來比一行 expression 煩，但它把你的意圖寫進型別：這是 unsigned 32-bit packing，不是一般 arithmetic。

6 Floating point：不是 real number

Depth、texture coordinate、barycentric interpolation 常用 float。但 float 是 finite precision value，有 rounding、NaN、infinity、denormal、不同最佳化選項下的差異。

問題naive model更好的規則
比較a == b 應該可用。幾何/顏色 tolerance 要由測試規格定義。
累加加法順序無關。float addition 不保證 associative。
轉整數cast 只是拿整數部分。先定義 rounding / floor / ceil / clamp policy。
跨平台同一份 code 必定 bit-identical。golden test 要分 exact 與 tolerance。

例如 viewport transform 把 float x 變成 pixel x，不能只靠隨手 cast：

int px = static_cast<int>(x); // truncates toward zero

你要先問：這裡想要 floor、round-to-nearest、還是 top-left fill rule 的固定規則？這是 renderer convention，不是 C++ cast 可以替你決定的事。

7 Determinism：測試要先定義容忍範圍

I-16 會講 golden image。這裡先建立 numeric 連接：renderer 測試不是只有「完全一樣」或「隨便差一點」。你要按資料類型分 policy：

資料比較方式理由
packed clear colorexact match。純 integer write，應該完全一樣。
Bresenham line maskexact match。若演算法是 integer deterministic，差一點就是規則不同。
float interpolated colorper-channel tolerance。rounding path 可能不同，除非你固定所有步驟。
depth bufferexact 或 epsilon，取決於 representation。integer fixed-point 和 float depth 的策略不同。
和 I-16 的連接 — golden image test 的核心不是「截圖存起來」。它要求你先定義 numeric contract：哪些值必須 bit-exact，哪些值允許 tolerance，哪些差異代表演算法規則改變。

8 Pixel-Renderer numeric policy

第一版可以採取保守策略： 

Coordinates:
  use signed int for screen-space pixel coordinates
  check bounds before converting to size_t index

Framebuffer:
  store pixels as uint32_t
  define one canonical PixelFormat in code
  backend converts only if platform/API requires it

Color:
  keep channels as uint8_t at storage boundary
  widen to int or float during blending/interpolation
  clamp before narrowing

Depth:
  choose float or fixed-point intentionally
  define comparison and clear value

Tests:
  exact match for integer-only output
  tolerance for float-derived output

這些規則的價值是把「我覺得這樣應該對」變成可以寫進 code review、test case、debug dump 的 contract。

9 Verification：怎麼檢查 numeric bug

  • 用 small image test：例如 4x4 framebuffer，手算每個 pixel 的 packed value。
  • 針對 pack/unpack 寫 round-trip test：Unpack(Pack(c)) == c。
  • 針對 bounds check 測 x = -1、x = width、y = -1、y = height。
  • 用 sanitizer 找 signed overflow、out-of-bounds、undefined behavior。
  • golden image mismatch 時同時輸出 numeric diff，不只輸出「圖片不同」。
auto c = RGBA8{255, 128, 0, 255};
uint32_t p = PackRGBA8(c);
assert(UnpackRGBA8(p).r == 255);
assert(UnpackRGBA8(p).g == 128);
本章學到的三件事：
  1. Renderer numeric types 不是抽象數學；每個 integer、float、packed pixel 都有 width、range、conversion、rounding 與 representation contract。
  2. Pixel format 要分清 semantic channel order、bit layout、memory byte order、API interpretation；不要讓 SDL/Win32 backend 反過來污染 renderer core 的 color model。
  3. Golden tests 必須建立 numeric policy：integer output 可 exact match，float-derived output 要明確 tolerance，所有 narrowing 都應先 clamp 或定義 rounding。

下一章回到 implementation：UI draw data 和 shader draw calls 都會用到這些 numeric policy。顏色、clip rect、alpha blend、depth、interpolation 如果沒有共同規則，未來 debug image mismatch 會很難判斷是 C++ bug、renderer rule bug，還是 backend conversion bug。
