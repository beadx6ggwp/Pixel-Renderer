PART 2 · CHAPTER 04
struct 與 class
自訂型別、封裝、與責任邊界
struct class encapsulation

前三章你已經知道程式如何被編譯、object 住在哪裡、pointer/reference 如何搬運地址。現在要問下一個問題：為什麼 C++ 要讓你自己定義型別？

這不是為了把幾個變數包起來而已。C++ 的型別系統是在描述語意規範：哪些值必須一起存在？哪些狀態不能被外面任意修改？誰負責維持 invariant？誰代表一段 resource lifetime？這也是 C++ 相比 C 多出來的核心能力之一：把約束從註解和習慣，推進到語言結構裡。

本章範圍 — 不把 struct 和 class 當成語法表，而是把它們看成「語意邊界」。你會看到什麼時候一個型別應該只是資料，什麼時候應該把資料藏起來，什麼時候成員函式是在保護 invariant。Pixel-Renderer 會作為例子，但目標是恢復你對 C++ type semantics 的判斷力。

1 自訂型別不是語法糖

假設沒有 Vertex，你畫三角形可能會寫成這樣：

rasterizer->DrawTriangle(
  400, 100, 0, 0xFF0000,
  500, 300, 0, 0x00FF00,
  300, 300, 0, 0x0000FF);

這不是只是醜而已，而是沒有語意。編譯器不知道哪四個數字是一個 vertex，你自己過兩天也會忘記參數順序。struct Vertex 的作用是把「一組必須一起移動的資料」命名成一個概念：

struct Vertex {
    float x, y, z;
    uint32_t color;
};

這時候 DrawTriangle 的簽名就開始像人話：

void Rasterizer::DrawTriangle(const Vertex& v1,
                                const Vertex& v2,
                                const Vertex& v3,
                                uint32_t color);
第一原則 — 當幾個值在概念上必須一起存在、一起傳遞、一起被理解時，就應該考慮定義一個型別。型別是在幫程式碼建立語意，不是只是在整理變數。

2 struct：公開的資料形狀

在 C++ 裡，struct 和 class 的能力幾乎一樣。真正的差異只有預設存取權限：

關鍵字預設權限常見語意
structpublic資料包、數學向量、設定值、跨模組資料協議
classprivate有 invariant、有生命週期、有操作規則的物件

例如你的 FramebufferConfig 很適合是 struct：

struct FramebufferConfig {
    unsigned char* buffer;
    int width;
    int height;
    int pitch;
};

它的角色不是保護資料，而是作為一份明確的資料協議：

ScreenManager     -> fills FramebufferConfig
    -> RenderDevice reads FramebufferConfig

這裡用 struct 很合理，因為呼叫者本來就需要看見每個欄位。它像一張表格，告訴 RenderDevice：buffer 在哪裡、寬高是多少、每一列跨多少 bytes。

3 class：把資料和規則綁在一起

再看 RenderDevice。它也有資料：

unsigned char* frame_buffer;
int pitch;
int width;
int height;

但這些資料不能隨便讓外面改。假設外面把 pitch 改錯，SetPixel() 就會寫到錯誤的記憶體位置。這時候 class 的價值出現了：把資料藏起來，只暴露安全操作。

class RenderDevice {
public:
    RenderDevice(const FramebufferConfig& config);
    void Clear(uint32_t color);
    void SetPixel(int x, int y, uint32_t color);
    int GetWidth() const;
    int GetHeight() const;

private:
    unsigned char* frame_buffer;
    int pitch;
    int width;
    int height;
};

外面不能直接改 frame_buffer、pitch、width、height，只能透過 SetPixel() 和 Clear() 操作。這不是為了「物件導向感覺比較高級」，而是為了保護一個 invariant：

RenderDevice 的 invariant — frame_buffer、pitch、width、height 必須彼此一致，否則 address = frame_buffer + y * pitch + x * 4 就會寫錯位置。

4 public / private 不是權限遊戲

很多人第一次學 public / private，會以為它只是「能不能從外面點到」。更根本的理解是：public 是承諾，private 是實作細節。

public

外部模組可以依賴的穩定介面。例如 RenderDevice::SetPixel()，Rasterizer 可以放心呼叫。

private

這個 class 自己可以改的內部細節。例如未來你把 unsigned char* 換成 owned framebuffer，只要 SetPixel() 語意不變，外面不用知道。

這跟你最近討論的 display backend 抽象是同一個思想。真正穩定的是介面，不是裡面的 Win32 DIBSection、SDL_Texture 或 CPU vector。

5 Pixel-Renderer 的型別地圖

型別建議角色原因
Vertexdata struct位置與顏色是一組頂點資料，外部需要直接建立與讀取。
FramebufferConfigdata struct跨模組傳遞 framebuffer pointer/size/pitch 的資料協議。
RenderDevicebehavior class封裝 pixel write 規則，保護 pitch/width/height 的一致性。
Rasterizerbehavior class持有 RenderDevice pointer，把線段與三角形轉成 SetPixel。
Applicationlifecycle class管理 init/run/close 與 frame loop。

6 何時該從 struct 變成 class

一個簡單判斷：

  • 如果外部可以自由讀寫欄位，而且不會破壞語意，使用 struct。
  • 如果欄位之間有一致性規則，外部亂改會讓物件壞掉，使用 class。
  • 如果物件需要負責資源生命週期，例如開 window、配置 buffer、關閉 handle，使用 class。

未來你把 framebuffer ownership 拉回 renderer/core 時，也會遇到同樣問題。FramebufferConfig 可以是公開資料包，但真正擁有 pixels 的 Framebuffer 可能就應該是 class，因為它要保證 buffer size、pitch、format 一致。

本章學到的三件事

1. struct 和 class 的語法差異很小，但設計語意不同：struct 偏資料，class 偏規則與生命週期。

2. private 不是為了刁難使用者，而是為了保護 invariant。

3. 好的 C++ 型別會讓錯誤變難：你不再傳一長串裸參數，而是傳 Vertex、FramebufferConfig、RenderDevice 這些帶有語意的東西。
