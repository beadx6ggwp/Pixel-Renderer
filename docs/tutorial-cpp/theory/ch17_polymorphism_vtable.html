PART 6 · CHAPTER 17
繼承、virtual 與 vtable
runtime dispatch 讓行為可替換，但不是免費抽象
virtual vtable interface

上一章說 unique_ptr 可以管理 lifetime。但如果你希望同一個 pointer 背後可以是 Win32DisplayBackend、SDLDisplayBackend、或 MacOSDisplayBackend，只靠 smart pointer 還不夠。

這章要回答的是：C++ 怎麼讓你透過同一個 interface 呼叫不同 implementation？代價是什麼？什麼時候值得？

本章範圍 — 建立 runtime polymorphism 的實用模型：base class pointer/reference、virtual function、vtable、virtual destructor、interface ownership。重點會放在 Pixel-Renderer 的 Application hooks、DisplayBackend abstraction、Shader/UI renderer 這些真實擴充點。

1 naive model：繼承就是共用程式碼

很多人第一次學 inheritance 會把它想成：

class B inherits A
so B can reuse A's code

這只是 inheritance 的一種用途，而且常常不是最重要的用途。對 renderer architecture 來說，更重要的是：

caller knows the interface
runtime object decides the implementation

也就是 dynamic dispatch。你不是為了少打 code 才使用 virtual，而是為了在 runtime 替換行為。

2 目前專案已經有 virtual hook

Application 現在已經有一組 virtual hooks：

class Application {
protected:
  virtual void OnInit() {}
  virtual void OnUpdate(float delta_time) {}
  virtual void OnRender() {}
  virtual void OnClose() {}
};

Application::Run() 不需要知道你的 demo app 實際怎麼 render。它只知道「每一 frame 要呼叫 OnUpdate 和 OnRender」。真正行為由 derived class override。

這是 Template Method pattern 的雛形

Base class 固定流程：init、loop、update、render、close。Derived class 填入某些步驟。這不是 backend abstraction，但它已經是 runtime polymorphism 的一種使用方式。

3 DisplayBackend interface

未來如果要把 Win32 繪製替換成 SDL 或 macOS，適合出現另一種 interface：

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;

  virtual bool Init(int width, int height, const char* title) = 0;
  virtual void PollEvents() = 0;
  virtual bool ShouldClose() const = 0;
  virtual void Present(const Framebuffer& fb) = 0;
};

然後 Application 可以 owns 一個 interface pointer：

std::unique_ptr<DisplayBackend> backend;
backend = std::make_unique<SDLDisplayBackend>();
backend->Present(framebuffer);

這裡有兩層語意：

  • unique_ptr：Application owns backend lifetime。
  • virtual Present：實際呼叫 SDL / Win32 / macOS implementation。

4 vtable 的最低限度模型

只要 class 有 virtual function，多數實作會在 object 裡放一個隱藏的 vptr，指向一張 vtable。vtable 裡放的是 virtual functions 的實際 function pointers。

所以 dynamic dispatch 的概念流程是：

backend->Present(fb)
  -> read object vptr
  -> find Present slot in vtable
  -> call SDLDisplayBackend::Present or Win32DisplayBackend::Present

5 virtual destructor 不是可選裝飾

如果你會透過 base pointer delete derived object，base class destructor 必須是 virtual：

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
};

原因是：

std::unique_ptr<DisplayBackend> backend = std::make_unique<SDLDisplayBackend>();
// unique_ptr destructs through DisplayBackend*

如果 destructor 不是 virtual，derived destructor 可能不會正確執行，SDL window / texture / platform handle 就可能 leak 或以錯誤方式釋放。

規則 — 只要一個 class 是 polymorphic base，並且可能透過 base pointer/reference 擁有或刪除 derived object，就給它 virtual destructor。

6 virtual 的成本與邊界

Virtual dispatch 的成本通常包含：

  • object 多一個 vptr。
  • 呼叫時多一次間接跳轉。
  • compiler 比較難 inline 和最佳化。
  • CPU branch prediction / instruction cache 可能受影響。

這不代表不能用 virtual。它代表你要把 virtual 放在正確的邊界，而不是放在每個 pixel 或每個 inner loop。

位置 是否適合 virtual 理由
DisplayBackend::Present 適合 每 frame 呼叫一次，換取平台可替換性。
Rasterizer::SetPixel per pixel 通常不適合 inner loop 內呼叫數百萬次，dispatch 成本與 inline 會重要。
Application::OnRender 適合 frame-level hook，成本可忽略，架構清楚。
Shader interface 視階段而定 教學版可以 virtual；高效版可能改 template、function object、JIT 或 command data。

7 不是所有 polymorphism 都要 inheritance

C++ 有很多替代方式：

  • virtual interface：runtime 可替換，適合 backend / platform boundary。
  • template：compile-time polymorphism，適合 hot path 或 math code。
  • function pointer / std::function：把 callable 當資料傳遞。
  • variant：有限集合的 alternatives，避免 heap allocation 和 virtual base。
  • command buffer：把行為轉成資料，再由 backend 解讀。

選擇時先問：你需要 runtime 替換嗎？型別集合是否固定？呼叫是否在 hot loop？ownership 在哪裡？這些問題比「OOP 好不好」更具體。

8 Pixel-Renderer 的落點

對 Pixel-Renderer，目前最合理的 virtual 邊界是：

  • DisplayBackend：Win32 / SDL / macOS 替換點。這是最值得 virtual 的地方。
  • Application hooks：目前已經存在，用來讓 demo app 填入 update/render 行為。
  • Shader interface：早期可以用 virtual 幫助理解 programmable pipeline；後續再評估 hot path 成本。
  • UI renderer backend：UI draw commands 可以被不同 backend consume，interface 可以放在 frame-level，不要放在 per-pixel inner loop。
架構判斷 — 用 virtual 隔離「低頻、平台相依、需要替換」的邊界；避免在「高頻、資料密集、可靜態決定」的內層演算法濫用 virtual。
常見陷阱

1. 把 inheritance 當 code reuse 工具 — renderer 架構裡更常需要的是 interface substitution，不是共享幾行 helper code。

2. 忘記 virtual destructor — base pointer owns derived object 時，沒有 virtual destructor 會讓 teardown 語意壞掉。

3. 把 virtual 放進 hot loop — per-pixel / per-vertex inner loop 應先考慮 static dispatch、data-oriented design 或 command data。

4. 混淆 lifetime 與 dispatch — unique_ptr 解決 ownership；virtual 解決 runtime method selection。它們常一起用，但不是同一件事。
本章學到的三件事

1. Virtual function 讓 base pointer/reference 在 runtime 呼叫 derived implementation，核心是 substitutable behavior。

2. vtable 的代價是間接呼叫與較差 inline 機會；適合放在 frame/platform boundary，不適合濫放在 hot loop。

3. Polymorphic base class 通常需要 virtual destructor，尤其當它會被 unique_ptr<Base> 擁有時。
