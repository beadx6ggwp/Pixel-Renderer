抽象不是只有 virtual；不同 dispatch 工具對應不同 lifetime、ownership 與 hot path 成本
virtualvarianttype-erasuredispatch

T-10 把 interface 看成 contract。下一個問題是：如果一個 interface 有多種實作，C++ 要用什麼機制表達？Pixel-Renderer 裡會遇到 Win32/SDL display backend、shader object、UI command、render pass、resource loader。它們都像「多型」，但不該全部用同一種工具。

這章的核心不是背 vtable，而是建立 dispatch decision：這個變化點發生在 runtime 還是 compile time？在 hot loop 還是 frame boundary？型別集合是 open 還是 closed？資料需要 value semantics 還是 reference semantics？

本章範圍 — 比較 virtual interface、template/static polymorphism、type erasure、std::variant、visitor、command buffer。最後回到 Pixel-Renderer：DisplayBackend 為什麼適合 virtual，shader hot loop 為什麼偏向 template，UI/RenderCommand 為什麼常適合 value command 或 variant。

1 naive model：多型就是 virtual

最常見的想法是：只要未來可能替換，就做 base class + virtual function。 

class Thing { virtual void Do() = 0; };
class A : public Thing {};
class B : public Thing {};

這解決了一種問題：runtime 有一個 object，你不知道它的 concrete type，但你知道它滿足一組 virtual operations。可是它也帶來 pointer/reference lifetime、heap allocation、indirection、object identity、vtable ABI、ownership 這些問題。

更底層地說，virtual 不是「比較抽象」的魔法。它是在 object 裡放入一條 runtime dispatch 路徑，讓 call site 透過 object 的 dynamic type 找到真正要呼叫的 function。這個模型很有用，但它不是免費，也不是唯一的表達方式。 

Base* p = new Win32DisplayBackend;
p->Present(framebuffer);

rough machine model:
  p points to object
  object carries or points to vtable
  call site loads function pointer from vtable
  indirect call to Win32DisplayBackend::Present
naive model 失敗點 — 「用 virtual 就分離」這句話不夠精準。virtual 只解決 runtime dispatch；它不自動解決 ownership、lifetime、thread safety、hot-loop cost、header dependency，也不保證 interface 切在正確邊界。

3 virtual interface：適合 frame boundary

Display backend 是 virtual 的好例子。它不在 pixel hot loop，而且真的需要 runtime 替換。

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
  virtual void PollEvents(InputState&) = 0;
  virtual void Present(const Framebuffer&) = 0;
};

這裡 virtual cost 通常不是瓶頸，因為每 frame 呼叫少數幾次。它換來的是清楚的 runtime platform boundary。

virtual 真正在保證什麼

用 DisplayBackend 做 virtual interface，重要的不是「Win32 和 SDL 都有同名 function」，而是 Application 可以只依賴一組 platform-neutral contract： 

Application owns:
  std::unique_ptr display;

Application knows:
  display->PollEvents(input)
  display->Present(framebuffer)

Application does not know:
  HWND
  HDC
  SDL_Window
  SDL_Texture
  BitBlt
  SDL_RenderPresent

這種 virtual boundary 適合放在低頻、粗粒度、platform-facing 的位置。它不適合放在 per-pixel / per-fragment 的內層 loop。

4 template/static polymorphism：適合 shader hot path

Shader 或 pixel operation 如果每 fragment 呼叫一次，virtual dispatch 就可能進入 hot loop。這時候 template 讓 compiler 看到 concrete callable，有機會 inline。

template <class Shader>
void DrawTriangle(Triangle tri, Shader& shader) {
  // fragment loop can call shader.Fragment(...) directly
}

代價是型別在 compile time 決定，binary 可能有多份 instantiation，error message 也可能更重。所以它適合 renderer core 的高頻 policy，不適合 platform backend selection。

template 不是 runtime plugin system

Template 的精神是讓 compiler 在 compile time 看到具體型別。這很適合：

  • fragment shader 被內層 loop 大量呼叫。
  • blend policy / depth policy 需要 inline。
  • math type 希望用 concept 約束 operations。

但它不適合處理「使用者啟動程式時選 Win32 或 SDL」這種 runtime platform selection。因為那不是 hot-loop policy，而是 executable 啟動後選擇哪個 backend object。

5 variant：closed set 的 value polymorphism

如果種類有限，而且你想把 commands 放進 vector、排序、replay、debug dump，std::variant 很適合。

using RenderCommand = std::variant<
  ClearCommand,
  DrawLineCommand,
  DrawTriangleCommand,
  DrawUiCommand
>;

Variant 的缺點是新增 command type 要更新 visitor，屬於 closed set design。但對早期 renderer command buffer，這反而是優點：你能明確知道有哪些命令。

Variant 的底層模型可以想成：一塊足以容納最大 alternative 的 storage，再加上一個 tag 記錄目前是哪一種型別。 

std::variant

storage:
  enough bytes for the largest command

tag:
  0 = Clear
  1 = DrawLine
  2 = DrawTriangle

visit:
  switch(tag)
    call correct overload

所以它和 virtual 最大差別是：variant object 自己就是 value，它不需要每個 command 都獨立 heap allocate，也不需要透過 base pointer 追蹤 object identity。對 command buffer 來說，這通常更接近你想要的資料模型。

6 type erasure：保留值語意的 runtime wrapper

std::function 是常見 type erasure：它可以包不同 callable，但暴露同一個 call interface。你不知道 concrete type，但可以像 value 一樣保存它。

Type erasure 適合 plugin-like API、工具層 callback、少量 runtime customization。不適合把每 pixel shader 都包成 std::function，因為可能有 indirection、allocation、inline 困難。

Type erasure 可以看成「把 concrete type 藏起來，但保留一組可呼叫 operations」。它和 virtual 很像，但包裝方式常常是 value-like wrapper： 

std::function
  can hold:
    function pointer
    lambda object
    functor object

caller sees:
  callback(event)

caller does not see:
  concrete closure type
lambda capture 與 lifetime — type erasure 會讓 callable 很方便保存，但不會替你延長 reference capture 的 lifetime。UI callback 或 render command 若捕捉 stack reference，等 command 被延後執行時就可能 dangling。

7 command buffer：為什麼 value semantics 很重要

Renderer command buffer 的目標通常不是保存一群有 identity 的 polymorphic objects，而是保存一串可重播、可排序、可 dump 的資料： 

Frame N:
  ClearCommand
  DrawTriangleCommand
  DrawUiRectCommand
  DrawTextCommand

Need:
  replay
  sort by layer/material later
  dump to debug file
  compare in tests
  move/copy as frame artifact

如果每個 command 都是 std::unique_ptr<ICommand>，你會得到 open-ended polymorphism，但也會得到 heap allocation、pointer chasing、debug dump 較麻煩、copy semantics 消失。對早期 renderer，closed set 的 plain struct/variant 常更可控。

Command representation優點代價
virtual ICommand新增 command 不用改 variant list。heap allocation、indirection、ownership 複雜、dump 較麻煩。
std::variantvalue semantics、cache 友善、可 visit、可測試。新增 command 要更新 visitor；closed set。
plain arrays by type最容易批次處理、cache 佳。排序與跨類型 ordering 要額外設計。

不要統一成單一抽象工具 — 把 DisplayBackend、shader、UI command、resource handle 全部做成 virtual class，會把不同頻率、不同 lifetime、不同 ownership 的問題混在一起。

9 如何驗證抽象選對了

抽象不是只看圖好不好看，而是要能被工程行為檢查：

檢查問題如果答案是 yes如果答案是 no
renderer core 是否能不 include platform header？DisplayBackend boundary 可能切對。virtual interface 可能放錯層，或 header 洩漏 backend type。
shader hot loop 是否能被 inline？template/concept 方向合理。每 pixel virtual/type-erased callback 可能過重。
command buffer 是否容易 dump 和 golden test？value command / variant 方向合理。command 被 object identity 和 pointer lifetime 綁住。
新增 backend 是否不需要改 renderer math？platform boundary 乾淨。display/present 和 rasterizer 還混在一起。

10 Summary

本章學到的三件事：
  1. Polymorphism 不是 virtual 的同義詞；它是在處理變化點如何 dispatch、如何 owns、如何表達 type set。
  2. DisplayBackend 適合 virtual，shader hot loop 適合 template/concept，command buffer 常適合 value type 或 std::variant。
  3. 選抽象工具前先問 open/closed type set、hot path 頻率、value/reference semantics、ownership lifetime。
