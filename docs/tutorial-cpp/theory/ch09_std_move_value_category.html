std::move 與 value category
std::move does not move; it changes the expression
std::move xvalue prvalue

上一章說 move 的本質是 ownership transfer。但 C++ 不是看到一個 type 有 move constructor，就自動從所有地方把 resource 搬走。Compiler 必須先知道：這個 expression 能不能被當成「source 可以被放棄」。

所以這章要處理一個名字很誤導的工具：std::move。最重要的句子先放在前面：std::move(x) 本身不會 move x。

本章範圍 — 建立 std::move 與 value category 的最小模型。你會區分 object lifetime、expression category、move constructor 這三層，並知道為什麼 std::move 只是讓後續 operation 有機會選到 move path。

1 naive model：std::move(x) 會 move x

先看這段 code：

#include <utility>

Buffer x(1024);
std::move(x);

很多人的直覺是：

std::move(x)
  -> move x
  -> x 變成 moved-from

這是錯的。上面這段 code 沒有建立另一個 Buffer，也沒有把 x assign 給另一個 Buffer。所以沒有 move constructor，也沒有 move assignment。

第一個分離 — std::move 不是 move operation。真正 transfer resource 的 operation 是 T(T&&) 或 T& operator=(T&&)。

2 用 logging type 看發生什麼

用一個會印出 copy / move 的 type 觀察：

struct Trace {
  Trace() { print("default"); }
  Trace(const Trace&) { print("copy"); }
  Trace(Trace&&) noexcept { print("move"); }
};
Trace x;
std::move(x);        // 沒有建立新 object，通常不印 move
Trace y = std::move(x); // 初始化 y，這裡才可能印 move

差別不是 std::move 這個 token 有魔法，而是第二行多了一個需要建構 y 的語境。初始化 y 時，overload resolution 看見 source expression 是可以 move-from 的形式，於是選到 move constructor。

3 std::move 真正做的是 cast

概念上，std::move(x) 幾乎就是：

static_cast<T&&>(x)

也就是把 expression x 轉成 xvalue expression。這裡要小心：改變的是 expression，不是 object 本身。

第二個分離 — object 有 storage 和 lifetime；expression 有 type 和 value category。x 和 std::move(x) 指向同一個 object，但它們是不同 category 的 expression。

4 最小 value category 模型

完整 C++ taxonomy 有 glvalue、rvalue、lvalue、xvalue、prvalue。這章先不背表，先抓住 renderer 開發最常碰到的三種：

Expression Category 直覺
x lvalue 有名字、可辨識的 existing object；compiler 不會假設你願意放棄它。
std::move(x) xvalue 仍然指向 existing object，但表達「可以把它當成 expiring source」。
T{} prvalue 直接產生一個 T value，用來初始化 object；C++17 後不能再簡單想成先 temporary 再 move。

這三個 category 會影響 overload resolution：

void use(const T&);
void use(T&&);

T x;
use(x);            // x 是 lvalue，通常選 const T&
use(std::move(x)); // xvalue，可以選 T&&
use(T{});         // prvalue，可以選 T&&

5 type 和 value category 是兩個軸

一個最容易卡住的 case 是 named rvalue reference：

void consume(Buffer&& b) {
  use(b);            // b 這個 expression 是 lvalue
  use(std::move(b)); // 這裡才是 xvalue
}

b 的 declared type 是 Buffer&&，但 expression b 有名字，所以它是 lvalue expression。這就是為什麼 move constructor 裡如果要把 member 繼續 move 到下一層，也常常要寫 std::move(member)。

常見陷阱 — 不要把 T&& 這個 type spelling 和 expression 的 value category 混成同一件事。C++ overload resolution 看的是 expression category，不只是 variable 宣告時的 type。

6 value category 不是 lifetime

再看一次：

Buffer x(1024);
std::move(x);

std::move(x) 是 xvalue expression，但 object x 沒有死掉，也沒有自動進入 moved-from state。它的 lifetime 還是到 scope 結束。

只有當後續 operation 真的使用這個 xvalue source，例如：

Buffer y = std::move(x);

move constructor 才可能 transfer resource，然後把 x 留在 safe-to-destroy 的 moved-from state。

第三個分離 — value category 描述 expression；lifetime 描述 object；moved-from state 是 move operation 執行後的 object state。這三件事相關，但不是同一件事。

7 Pixel-Renderer 的落點

這章的目的不是讓你背 value category 表，而是讓你之後能讀懂 renderer 裡的 ownership API：

  • Framebuffer owner：如果 function 接收 Framebuffer&&，它不是借用 pixels，而是在表達「callee 可以接手 ownership」。
  • Resource cache：loader 產生 TextureData 後，可以用 std::move(data) 交給 cache；真正搬的是 TextureData 的 move constructor。
  • DisplayBackend：window handle / SDL texture 這種 resource 通常 move-only。你要看清楚 API 是 borrow &，還是 consume &&。
  • Return path：return obj; 和 return std::move(obj); 不只是風格不同，它們會改變 return expression category。下一章會處理 RVO / NRVO。
對 renderer API 的警告 — 不要為了看起來 modern 到處加 std::move。如果 object 後面還要用，就不要 move；如果 return local object，先不要直覺寫 return std::move(obj);，因為它可能破壞 NRVO shape。
常見陷阱

1. 以為 std::move(x); 會改變 x — 沒有後續 construction / assignment，就沒有 transfer。

2. 以為 xvalue 等於 object 快死了 — std::move(x) 指向的 x 還活著；只是 expression 被標成可 move-from。

3. 以為 T&& name 在 expression 中仍是 rvalue — named rvalue reference expression 是 lvalue，需要 std::move(name) 才變 xvalue。
本章學到的三件事

1. std::move 本身不 move；它大致是 static_cast<T&&>，把 expression 變成 xvalue。

2. object 有 lifetime；expression 有 value category。x 和 std::move(x) 可以指向同一個 object，但 category 不同。

3. 真正 transfer resource 的是 move constructor / move assignment；moved-from state 是 operation 執行後才出現的 object state。
