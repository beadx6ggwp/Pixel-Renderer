Move 是 ownership transfer
copy means duplication, move means transfer
move ownership moved-from

前面幾章把 resource-owning type 的壓力一路推到 Rule of 5。現在要正式回答：move 到底解決什麼問題？

最常見的錯誤直覺是「move 是比較快的 copy」。這句話在某些 case 看起來對，但它抓錯了本質。Move 的核心不是快，而是：當 source object 可以被放棄時，destination 可以接手 source 擁有的 resource。

本章範圍 — 建立 move semantics 的正確心智模型：copy 是 duplication，move 是 ownership transfer。你會看到為什麼 inline value type 的 move 可能沒什麼意義，而 Buffer / texture / backend handle 這種 resource owner 的 move 才是關鍵。

1 naive model：move = faster copy

很多人第一次學 move 會得到這個簡化模型：

copy = slow
move = fast

這個模型有一點實用直覺，但它不夠精準。真正的區別是：

Copy vs Move

copy：A 和 B 都要一份。語意是 duplication。
move：A 不再需要原本 resource，B 接手。語意是 transfer。

如果 A 之後還要保留原本內容，你不能 move from A。如果 A 是 temporary、即將離開 scope、或是 container reallocation 中即將被替換的位置，move 才有語意上的合理性。

2 Inline value data：沒有東西可以偷

先看一個 renderer 會常見的數學型別：

struct Matrix4x4 {
  float m[16];
};

這個 object 的資料就在 object 本體裡。沒有 heap block、沒有 handle、沒有外部 resource。copy 時要複製 16 個 float；move 時也沒有什麼 ownership 可以轉移，多半還是要把 16 個 float 放到 destination。

關鍵修正 — move 對每個 type 都比較快，這個說法是錯的。對 inline value data，move 可能和 copy 幾乎一樣，因為沒有外部 resource 可以接手。

3 Resource owner：move 開始有意義

現在回到 Buffer：

class Buffer {
public:
  explicit Buffer(size_t n)
    : ptr(new char[n]), size(n) {}

  ~Buffer() { delete[] ptr; }

private:
  char* ptr = nullptr;
  size_t size = 0;
};

Buffer 本體只有 pointer 和 size，但真正大的資料在 heap 上。deep copy 要 allocate 新 block 並複製內容；move 可以只轉移 pointer ownership。

4 Move constructor 的核心

Buffer 的 move constructor 可能長這樣：

Buffer(Buffer&& other) noexcept
  : ptr(other.ptr), size(other.size) {
  other.ptr = nullptr;
  other.size = 0;
}

前兩行把 resource handle 複製到 destination；後兩行把 source 放進 empty-but-valid state。這不是可有可無的清理，而是 move semantics 的核心。如果 source 還以為自己 owns 那塊 memory，兩個 destructor 還是會 double delete。

moved-from state — move 後的 source object 還活著。它不需要保留原本值，但必須仍然 valid，至少能安全 destruct，也應該能被重新 assign。標準庫通常說 moved-from object 是 valid but unspecified。

5 Move 不是 object teleportation

Move 沒有讓 source object 消失，也不是把整個 object teleport 到另一個位置。它只是一個 function call，負責用 source 建構或賦值 destination。source 的 lifetime 仍然照常走完。

Buffer a(1024);
Buffer b = move_from(a); // 概念上：b 接手 a 的 resource

// a 還活著，只是處於 moved-from state

真正的 C++ 語法會用 std::move(a) 讓 move constructor 有機會被選到；但 std::move 本身不搬東西。這個坑很重要，所以下一章會專門處理。

6 Pixel-Renderer 的落點

Move 對 Pixel-Renderer 有幾個很實際的設計落點：

  • Framebuffer owner：如果 framebuffer owns CPU pixels，copy 可能很貴或應該禁止；move 可以讓 temporary framebuffer 轉移到 renderer state。
  • DisplayBackend：Win32 window、SDL texture、macOS layer 這些 handle 通常不能 copy。它們要嘛固定 lifetime，要嘛 move-only。
  • Texture / image / mesh：載入器產生 temporary resource，再交給 resource cache 或 renderer，這正是 ownership transfer。
  • UI command buffer：每 frame 產生 draw commands，可能用 vector 直接重建；若 command buffer owns memory，move 可以避免 expensive duplication。
設計提醒 — 不是所有 renderer type 都該 move。Vec3、Matrix4x4 這類 inline math value copy 就很好。move 主要服務 resource owner，而不是替所有型別加上「現代 C++ 感」。
常見陷阱

1. 以為 move 一定更快 — 對 inline value data，move 可能和 copy 一樣。

2. move 後繼續依賴 source 原本內容 — moved-from object 是 valid，但內容通常 unspecified。你可以 destruct 或 assign，不該假設原值還在。

3. 忘記清空 source ownership — move constructor 如果只 copy pointer，不重設 source，就是 shallow copy bug 的變形。
本章學到的三件事

1. move 不是 faster copy。copy 是 duplication，move 是 ownership transfer。

2. move 對 resource-owning type 最有意義；對 inline value type，可能沒有明顯優勢。

3. move 後 source object 仍然存在，必須保持 valid moved-from state，讓 destructor 和 assignment 安全。
