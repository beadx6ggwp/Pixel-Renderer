Callable / Overload / Lambda
callable 是 behavior as value；capture lifetime 決定它能不能跨 frame 保存
lambda std::function overload

Pixel-Renderer 之後會需要 callback：window event、UI button handler、shader function、debug tool、render command execution。最直覺的想法是：「我把一段 lambda 存起來，之後呼叫就好。」可是只要這段 lambda capture 了 local reference，或被放進跨 frame command buffer，它就可能在下一幀指向已經死亡的 object。

這章要把 callable 當成一種資料來看：它不只是可以被呼叫的語法，而是帶有 type、storage、capture state、lifetime、dispatch cost、overload resolution 的 object。

本章範圍 — 建立 C++ callable 的分層模型：function pointer、member function pointer、functor、lambda closure object、generic lambda、overload set、std::function、template callable parameter、capture by value/reference、default argument、inline function。最後回到 Pixel-Renderer：shader callback、UI command、event handler、render command buffer 應該何時存 behavior，何時存 data。

1 naive model：lambda 只是匿名 function

最常見的簡化說法是： 

lambda = anonymous function

這只對不 capture 的 lambda 有一點直覺幫助。真正重要的是：lambda expression 會產生一個 closure object。如果它 capture 了資料，那些 captured state 就是這個 object 的 data members。

int exposure = 2;
auto shader = [exposure](Color c) {
  return c * exposure;
};

你可以用這個近似模型理解：

struct Closure {
  int exposure;
  Color operator()(Color c) const {
    return c * exposure;
  }
};
第一個修正 — lambda 不是純語法糖成 function pointer。Capture 會產生 state；state 有 lifetime、copy/move cost、dangling 風險。

2 callable 家族地圖

C++ 裡「可以被呼叫」的東西很多。先把森林畫出來：

Callable kind 有沒有 state 可否 inline/最佳化 常見用途
function pointer 沒有。 間接呼叫，compiler 較難看穿。 C API callback、簡單 dispatch。
member function pointer 不包含 object 本身。 需要 object + member pointer 一起呼叫。 呼叫某 class method。
functor 可以有。 template 下通常很好 inline。 policy、shader object、predicate。
lambda closure capture 決定。 auto/template 下通常很好。 local behavior、UI callback、small shader。
std::function type-erased，可以有。 通常有 type-erasure overhead。 低頻事件、editor tool、heterogeneous callback storage。 
callable
  |
  +-- no state
  |     function pointer
  |
  +-- stateful concrete type
  |     functor / lambda closure
  |
  +-- type-erased wrapper
        std::function

3 function pointer：只有 address，沒有 context

function pointer 是最接近 C 的 callable：

using PixelShaderFn = Color(*)(Color in);

Color Invert(Color in) { return Color{255 - in.r, 255 - in.g, 255 - in.b}; }

PixelShaderFn shader = &Invert;

它的限制是沒有 state。若 shader 需要 exposure、texture pointer、uniforms，你就得另外傳 context：

using ShaderFn = Color(*)(void* context, Color in);

這就是 C API 常見的 callback + user data 模式。它很明確，但型別安全弱，context cast 要自己保證。

4 functor / lambda：behavior 可以帶 state

Functor 是有 operator() 的 object：

struct TintShader {
  Color tint;
  Color operator()(Color in) const {
    return in * tint;
  }
};

Lambda closure 是 compiler 幫你生成的 functor。這讓 shader/predicate/tool callback 很自然：

auto shader = [tint](Color in) { return in * tint; };

如果 function template 接 callable，compiler 知道 callable 的具體型別，通常能 inline：

template <class Shader>
void ShadePixels(Framebuffer& fb, Shader shader) {
  // shader(pixel) may be inlined if Shader is concrete
}
callable 的結構等價 — stateful lambda 在工程模型上等價於一個有 data members 和 operator() 的 anonymous class。這不是比喻，而是很接近 C++ 的實際語意模型。

5 capture lifetime：UI command 最容易踩的坑

真正危險的不是 lambda，而是 capture 的 lifetime。

void BuildUi(CommandList& commands) {
  std::string label = "OK";
  commands.push([&] { DrawText(label); });
}

// later frame / later phase
commands.execute(); // label is already dead

這不是 lambda 語法問題，而是 T-03/T-04/T-06 的綜合問題：closure object 活得比 captured reference 指向的 object 更久。 

BuildUi stack:
  label = "OK"
  lambda captures reference to label
  command stored in command list
return from BuildUi:
  label destroyed
later:
  lambda tries to use dangling reference
跨 frame / 跨 phase command 不應 capture local reference — command buffer 應該擁有需要的資料，例如 copy string、copy rect、copy color，或保存明確 lifetime 的 resource handle。

6 behavior vs data：command buffer 應該存什麼？

遇到 UI/render command 時，有兩種設計：

設計 存什麼 優點 風險
store behavior std::function<void()> / lambda。 彈性高，適合 low-frequency tool callback。 capture lifetime、allocation、type erasure、debug 困難。
store data DrawRectCmd, DrawTextCmd, ClipCmd。 self-contained、可序列化、可重播、接近 GPU command stream。 需要設計 command schema。

對 Pixel-Renderer 的 UI draw list，第一版更適合存 data：

struct DrawTextCmd {
  Rect bounds;
  std::string text;
  Color color;
};

這樣 command 自己 owns 文字資料，不依賴 UI build function 的 local variable。

類比，不是結構等價 — UI command list 類似 GPU command buffer，因為它們都把「要做什麼」變成資料流。但 GPU command buffer 有嚴格硬體/driver ABI；你的 UI command list 是教學用 software data structure。

7 std::function：type erasure 的方便與成本

std::function<R(Args...)> 可以裝不同 concrete callable：

std::function<void(Event)> callback;
callback = [](Event e) { Handle(e); };

它的價值是 type erasure：caller 不需要知道裡面是 function pointer、lambda、functor。代價是可能有間接呼叫、可能配置、失去 inline 機會。

位置 是否適合 std::function 理由
button click callback 可以。 低頻，彈性比 raw performance 重要。
per-pixel fragment shader hot loop 通常不適合第一選擇。 每 pixel 間接呼叫成本可能被放大。
debug menu action 可以。 低頻、可讀性和彈性重要。
render command stream 通常 prefer data command。 可排序、可重播、可檢查 lifetime。

8 overload set：名字不是 function object

overload 會讓 callable 變得更微妙：

void Draw(Line);
void Draw(Triangle);

auto f = Draw; // error: Draw is an overload set, not one function

你必須給 compiler 足夠 context：

using DrawLineFn = void(*)(Line);
DrawLineFn f = static_cast<DrawLineFn>(&Draw);

這會在 shader API、tool callback、operator overload 裡出現。當你把 function 當資料傳遞時，名字解析必須收斂成一個具體 signature。

9 member function pointer：還需要 object

member function pointer 不是普通 function pointer。它描述「某個 class 的 method」，但不包含 object instance：

struct Tool {
  void OnClick(Event);
};

using Method = void (Tool::*)(Event);
Method m = &Tool::OnClick;

呼叫時還需要 object：

Tool tool;
(tool.*m)(event);

如果 callback 存的是 member function pointer + object pointer，就要保證 object pointer 的 lifetime。這和 lambda capture reference 是同一類問題。

10 Pixel-Renderer policy：哪裡用哪種 callable

場景 建議 理由
software shader hot loop template callable / concrete functor。 可 inline，避免 per-pixel type erasure。
UI click callback std::function 可以接受，但 capture 要 owns 或保證 lifetime。 低頻、彈性重要。
UI draw command prefer data command，不存 arbitrary lambda。 避免 dangling、可排序、可重播。
DisplayBackend virtual interface。 frame-level dispatch，不在 hot loop。
debug tool action std::function<void()>。 低頻且方便。 
hot path:
  prefer concrete callable / template / data-oriented dispatch

frame boundary:
  virtual interface is acceptable

low-frequency UI/tool:
  std::function is acceptable

cross-frame command buffer:
  prefer self-contained data, not borrowed behavior
本章學到的三件事：
  1. Lambda 是 closure object；capture 會變成 state，state 的 lifetime 和 copy/move cost 必須被設計。
  2. std::function 是 type-erased callable，適合低頻 callback，不適合作為 per-pixel hot loop 的預設工具。
  3. Pixel-Renderer 的 UI/render command 應優先存 self-contained data；只有低頻事件和工具 callback 才適合存 behavior。
