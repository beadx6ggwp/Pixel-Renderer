THEORY TRACK · T-01
Compilation / Linkage / ABI
C++ 程式不是一次變成 exe，而是經過 translation unit、symbol、linker 與 ABI contract
translation-unit linkage ABI

你在 Pixel-Renderer 裡看到 #include "core/application.h"，然後打 make debug，最後得到一個 .exe。直覺上好像是「compiler 讀完整個專案，產生程式」。

但 C++ 不是這樣運作。C++ 的編譯模型是把每個 .cpp 分開變成 object file，再由 linker 把 symbols 接起來。這就是為什麼 header 寫錯會造成重複定義，source 沒編進去會造成 unresolved symbol，換 compiler 又可能遇到 ABI 問題。

本章範圍 — 建立 C++ program model：preprocess、compile、link、load；理解 declaration/definition、ODR、linkage、symbol、relocation、ABI。這章是語意模型，下一章 I-02 才會把它映射到 Pixel-Renderer 的 MinGW GCC、MSVC、Apple Clang 與 VSCode 設定。

1 naive model：include 把專案組起來

初學時最常見的模型是： 

#include copies header
compiler sees all code
compiler makes exe

這個模型只對第一步有一點點對。#include 確實是文字展開，但 compiler 不是一次讀完整個專案。更精準的流程是： 

main.cpp  ──preprocess/compile──> main.o
app.cpp   ──preprocess/compile──> application.o
screen.cpp──preprocess/compile──> screen_manager.o
rast.cpp  ──preprocess/compile──> rasterizer.o

main.o + application.o + screen_manager.o + rasterizer.o + libraries
  |
  v
linker resolves symbols
  |
  v
PixelRenderer.exe

每個 .cpp 加上它 include 進來的 headers，形成一個 translation unit。Translation unit 是 C++ 編譯的基本單位。

2 declaration vs definition

Header 最常放 declaration，source 最常放 definition。差別是：

類型 意思 例子
declaration 告訴 compiler「這個名字存在，長這樣」。 void DrawLine(int, int, int, int, uint32_t);
definition 真的提供 entity 的內容或 storage。 void DrawLine(...) { ... }

這就是為什麼 rasterizer.h 可以只宣告：

class RenderDevice;
class Rasterizer {
public:
  void DrawLine(int x1, int y1, int x2, int y2, uint32_t color);
private:
  RenderDevice* device;
};

因為 header 只需要知道 RenderDevice 是一個 type，足以宣告 pointer。真正呼叫 device->SetPixel() 的 rasterizer.cpp 才需要 include render_device.h 看到完整 definition。

3 ODR：一個程式裡只能有一個定義

ODR 是 One Definition Rule。簡化版是：

同一個 program 裡，大多數具有 external linkage 的 function/object 只能有一個 definition；但可以在很多 translation units 裡看到相同 declaration。

所以這樣放在 header 裡通常會出事：

// bad_header.h
int global_counter = 0; // definition

如果三個 .cpp include 它，就會產生三份 global_counter definition。Compiler 各自編譯時不一定知道問題，linker 合併 symbols 時才炸。

正確方式通常是 header 放 declaration，source 放 definition：

// config.h
extern int global_counter;

// config.cpp
int global_counter = 0;

4 linkage：名字能不能被別的 translation unit 看見

Declaration/definition 解釋「名字與實體」；linkage 解釋「這個名字是否能跨 translation unit 連起來」。

Linkage 意思 常見寫法
external linkage 別的 translation unit 可以 link 到同一個 symbol。 一般非 static global function / variable。
internal linkage 只在目前 translation unit 內可見。 static global function、anonymous namespace。
no linkage 不能被其他 scope 以 linkage 找到。 local variable。

對 renderer 工程化來說，internal linkage 很有用：如果 win32_display_backend.cpp 裡有只給本檔使用的 helper，不要讓它污染整個 program 的 symbol namespace。

namespace {
  uint32_t ConvertColor(uint32_t rgba) { ... }
}
// ConvertColor has internal linkage

5 object file、symbol、relocation

當 application.cpp 呼叫 Rasterizer::DrawLine，compiler 不一定知道那個 function 的機器碼在哪。它只要根據 declaration 產生「我要呼叫這個 symbol」的紀錄。 

application.o
  contains:
    machine code for Application::Run
    undefined reference to Rasterizer::DrawLine

rasterizer.o
  contains:
    symbol definition: Rasterizer::DrawLine

linker:
  matches undefined reference to definition
  patches final address

這就是為什麼常見錯誤分成兩種：

錯誤階段 典型訊息 代表什麼
compile error unknown type, no member named 這個 translation unit 看不到足夠 declaration/type information。
link error undefined reference, unresolved external symbol compiler 看過 declaration，但 linker 找不到 definition 或 library。

6 ABI：binary 層級的約定

API 是 source-level contract；ABI 是 binary-level contract。你可以把 ABI 想成： 

How compiled code agrees on:
  - function names after mangling
  - calling convention
  - parameter passing
  - object layout
  - vtable layout
  - exception model
  - standard library binary layout

這就是為什麼「都是 C++ compiler」不代表 object files 可以隨便混用。MSVC 的 object file、MinGW GCC 的 object file、Apple Clang 的 object file，在格式、ABI、standard library 上都可能不同。

重要

Header 是 source code 層級的共享；library/object file 是 binary 層級的共享。當你跨 compiler 或跨 platform 時，真正麻煩的常常不是語法，而是 ABI、library format、runtime、debugger、system SDK。

7 回到 Pixel-Renderer

目前 Pixel-Renderer 的 ScreenManager header 直接 include windows.h。這件事有兩個後果：

  • 任何 include screen_manager.h 的 translation unit 都需要 Windows SDK type。
  • macOS Apple Clang 不可能直接編譯這條 source path，因為沒有 Win32 headers、symbols、libraries。

所以跨平台不是把 g++.exe 換成 clang++ 就好。你需要把 platform-specific source 分開： 

core/
  application.cpp
  render_device.cpp
  framebuffer.cpp

platform/win32/
  win32_display_backend.cpp  -> includes windows.h

platform/sdl/
  sdl_display_backend.cpp    -> includes SDL.h

public interface:
  display_backend.h          -> no windows.h, no SDL.h

這個拆法背後就是本章模型：讓 core translation units 不需要看見 platform declarations，讓 linker 只在目標平台連入對應 backend definition 與 system libraries。

常見陷阱
  • 把 function definition 放進 header，然後被多個 .cpp include，造成 ODR/link error。
  • 看到 undefined reference 就改 include path；但 include path 只解 compile，看不到 definition 是 link 問題。
  • 把 VSCode IntelliSense 設定當成真正 build 設定；它們是兩條路。
  • 混用不同 compiler 產生的 object/library，卻沒有檢查 ABI 與 runtime。
本章三個 takeaways
  1. C++ 以 translation unit 為單位編譯，再由 linker 解析 symbols；不是一次讀完整個專案。
  2. Declaration 讓 compiler 檢查 type；definition 讓 linker 找到實體；ODR/linkage 決定同名實體能不能跨檔案存在。
  3. 跨 compiler/platform 的核心問題不是只有語法，而是 ABI、system SDK、library format、runtime 與 build graph。
