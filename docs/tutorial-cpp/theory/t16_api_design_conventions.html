THEORY TRACK · T-16
API Design Conventions
好的 C++ interface 不是把函式包漂亮，而是把 ownership、lifetime、mutability、failure mode 寫進型別
api-design ownership const contracts

前面章節已經分別講過 object lifetime、RAII、value category、copy/move、span/view、contract、polymorphism、header boundary。問題是，真正寫專案時你不會把這些概念分開遇到。你會遇到一個更實際的問題：一個函式簽名到底應該長什麼樣子？

例如 DisplayBackend::Present 應該吃 Framebuffer、Framebuffer*、std::vector<uint32_t>&，還是 uint32_t* 加 width/height？Rasterizer 要不要擁有 RenderDevice？UI draw list 要回傳 by value 還是借用內部 vector？這些不是命名風格問題，而是 API 是否把語意邊界說清楚。

本章範圍 — 建立 Pixel-Renderer 的 C++ API design conventions：owner/view/borrow、const-correctness、reference/pointer/span/value return、out parameter、error return、namespace/header boundary、naming policy。目標是讓 abstraction boundary 不只存在於架構圖，也存在於函式簽名。

1 naive model：API 就是可呼叫的函式

如果只把 API 看成「外部能呼叫的函式」，你可能會寫出：

void Present(uint32_t* pixels, int width, int height);
void SetDevice(RenderDevice* device);
Command* GetDrawCommands(int* count);

這些函式可能能用，但它們沒有回答：

  • 這個 pointer 可不可以是 null？
  • 呼叫者能不能修改 pixels？
  • pixels 的 lifetime 要活到什麼時候？
  • width/height 和 memory pitch 的關係在哪裡？
  • 函式失敗時怎麼回報？
  • 回傳 pointer 指向誰擁有的資料？下一 frame 還有效嗎？
第一原則 — C++ API 不只是 call surface，而是 semantic boundary。簽名應該盡量表達 ownership、borrow、mutability、validity、failure mode 和 cost。

2 Owner / View / Borrow 先分清楚

Pixel-Renderer 最需要的三個角色是：

角色語意適合型別Renderer 例子
Owner 負責建立、維持、釋放 resource。 value type、RAII class、unique_ptr。 Framebuffer owns pixel storage；backend owns SDL/Win32 handle。
View 不擁有，只描述一段可存取資料。 std::span、string_view、自訂 FramebufferView。 UI draw list view、read-only framebuffer view。
Borrow 暫時借用一個已存在 object。 T&、const T&、nullable T*。 Rasterizer borrows RenderDevice during draw。 
Owner:
  Framebuffer framebuffer(800, 600);

Borrow:
  RenderDevice device(framebuffer);

View:
  DisplayBackend::Present(const Framebuffer& framebuffer);

這裡的重點不是一定要用哪個 class 名稱，而是不要讓一個 type 同時看起來像 owner 又像 borrowed pointer。

SVG：架構邊界要落到型別角色。Owner、borrow、view 混在一起時，抽象圖再漂亮也容易失效。

3 Reference / Pointer / Span / Value 的判準

一個實用的判準：

形式語意適合時機
T by value 擁有一份值，或明確轉移/複製。 small value、command object、return result。
const T& 不擁有、不修改、不可 null。 present read-only framebuffer、draw call descriptor。
T& 不擁有、會修改、不可 null。 renderer 寫入 framebuffer、builder 填資料。
T* 可能 nullable，或 C API interop。 optional dependency、platform handle boundary。
std::span<T> 連續資料 view，不擁有。 vertex buffer、index buffer、pixel row view。
簡單規則 — 不可 null 就先考慮 reference；需要表達 optional/null 才用 pointer；連續資料不要拆成 raw pointer + count，先考慮 std::span 或 domain-specific view。

4 const-correctness 是 capability design

const 不只是「我不想改」。它是在 API 上限制 capability：

回指 T-22 — 如果只說 const 是 read-only 還不夠。T-22 已把 const 和 value category 接起來：std::move(const T&) 通常不能真正 move，member function 後面的 const 會改變 this 的型別，mutable 也不是繞過 API contract 的後門。
class DisplayBackend {
public:
  virtual void Present(const Framebuffer& framebuffer) = 0;
};

class RenderDevice {
public:
  explicit RenderDevice(Framebuffer& framebuffer);
  void SetPixel(int x, int y, RGBA8 color);
};

這個設計表達：

  • DisplayBackend 可以讀 framebuffer 來 present，但不應修改 renderer output。
  • RenderDevice 借用可修改的 framebuffer，因為它的責任就是寫 pixel。
  • 如果某個 backend 需要 format conversion，應該寫到 backend-owned staging texture，不應改 core framebuffer。

5 Out parameter 與 return value

C++ 裡 out parameter 很常見，但它會讓 ownership 和 initialization 變模糊：

// less clear
bool LoadImage(const Path& path, Image* outImage);

// clearer if Image is movable
Result<Image> LoadImage(const Path& path);

對 modern C++ 來說，return by value 常常是好的 default，因為 move、copy elision、RAII 能把 ownership transfer 表達得更清楚。Out parameter 比較適合：

  • 填入已存在的 buffer，避免 allocation。
  • C API interop。
  • 函式需要回傳多個大物件，而且你明確定義誰先建立 storage。
Renderer 連接 — UI draw commands 可以 by value 回傳一個 command buffer；但 rasterizer 寫 pixels 應該寫進已存在 framebuffer。這不是風格差異，而是 ownership 與 allocation policy 不同。

6 Error return 要表達 failure category

T-10 已經說過 error handling 的核心是分類。API design 時要讓 failure mode 不含糊：

情況建議例子
caller 違反 precondition assert / contract；不當成 recoverable error。 用 invalid framebuffer size 建立 object。
外部環境可能失敗 回傳 Result / error code / exception policy。 SDL window 建立失敗、image file 讀不到。
值可能不存在 std::optional。 查不到 optional resource。
不該失敗的 invariant constructor 建立後維持 invariant。 Framebuffer width/height/pixels size 一致。

這會影響 backend interface。CreateWindow 類操作可以失敗；Present(const Framebuffer&) 是否可以失敗，要看你想要同步回報錯誤，還是把 backend error 變成 application-level fatal error。重點是不要讓每個函式都用 bool，但 bool 代表什麼又沒寫。

7 Namespace / header boundary 也是 API

I-17 講 header hygiene，這裡從 API 語意補一層：namespace 和 header 不只是整理檔案，而是告訴使用者哪些東西屬於 stable surface，哪些是 implementation detail。 

renderer/
  framebuffer.h          stable core API
  render_device.h        stable core API
  rasterizer.h           rendering API

platform/
  display_backend.h      stable platform-neutral API
  win32/*.h              backend-private or app layer
  sdl/*.h                backend-private or app layer

detail/
  internal helpers       no stable guarantee

如果你讓 renderer/detail 的 type 出現在 public function signature，它就不再是 detail。API boundary 一旦公開，就會變成後續 refactor 的成本。

8 Pixel-Renderer API policy 第一版

第一版可以採取下面規則： 

Ownership:
  resource owner is a RAII value type or unique_ptr-owned object
  borrowed non-null dependency is T& / const T&
  nullable dependency is T* and must be documented by name or contract

Framebuffer:
  Framebuffer owns storage
  RenderDevice borrows Framebuffer&
  DisplayBackend::Present borrows const Framebuffer&

Buffers:
  use span/view for non-owning contiguous data
  avoid raw pointer + count unless crossing C API boundary

Errors:
  assert for violated internal preconditions
  Result/optional for recoverable external failures
  do not use bool without naming the failure meaning

Headers:
  stable API headers expose platform-neutral types
  platform handles stay private to backend layer

這些規則可以隨專案成熟調整，但第一版要先有共同語言。否則每次新增 shader、texture、UI widget、backend 都會重新爭論同一件事。

本章學到的三件事：
  1. API signature 應該表達 ownership、borrow、view、mutability、failure mode 和 cost，而不只是讓函式可呼叫。
  2. T&、T*、std::span、by value、const 都是在設計 capability 與 lifetime contract。
  3. Pixel-Renderer 的第一版邊界應讓 Framebuffer owns storage、RenderDevice mutably borrows、DisplayBackend read-only presents，平台 handle 不進 core API。
