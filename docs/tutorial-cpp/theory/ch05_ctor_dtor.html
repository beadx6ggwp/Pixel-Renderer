PART 2 · CHAPTER 05
建構與解構
object lifetime、initialization、RAII
constructor destructor RAII

上一章把 struct 和 class 看成語意邊界：哪些資料應該被公開，哪些 invariant 應該被保護。現在要問更深的一層：一個 object 什麼時候真的開始存在？什麼時候結束？

這不是哲學問題。renderer 會碰到 window handle、framebuffer memory、texture、file、thread、GPU resource。這些東西都有一個共同特徵：拿到之後必須釋放。C 的做法通常是「記得呼叫 close/free/delete」。C++ 的答案更強：把取得和釋放綁進 object lifetime，讓語言自動替你收尾。

本章範圍 — 建立 constructor、destructor、member initializer list、destruction order、RAII 的心智模型。重點不是背語法，而是理解 C++ 如何把「資源必須被釋放」這件事變成型別語意。

1 naive 方法：初始化和清理分散在外面

先看最直覺的 C-style 寫法。你建立一個物件，然後另外呼叫 Init()；不用時，再記得呼叫 Close()：

Window window;
window.Init(800, 600);

// render loop...

window.Close();

這看起來合理，但它有一個致命問題：正確性依賴人的記憶。如果中間有 early return、錯誤分支、exception、或未來某個人忘了呼叫 Close()，resource 就 leak 了。

Window window;
window.Init(800, 600);

if (!load_assets()) {
  return; // 忘了 Close()，window handle leak
}

window.Close();
naive 方法的問題 — resource lifetime 被拆成兩段，取得在一個地方，釋放在另一個地方。中間任何控制流分支都可能破壞配對。C++ 的 constructor/destructor 就是為了把這種配對變成語言層級的規則。

2 Constructor：讓 object 從一開始就有效

Constructor（建構函式）的語意是：object 建立時自動執行，負責把 object 放進有效狀態。這句話比「一個特殊函式」重要得多。

class Framebuffer {
public:
  Framebuffer(int width, int height)
    : width(width), height(height), pixels(width * height) {}

  uint32_t* Data() { return pixels.data(); }

private:
  int width;
  int height;
  std::vector<uint32_t> pixels;
};

一旦 Framebuffer fb(800, 600) 成功，呼叫端就能相信它已經有正確尺寸的 pixel storage。這就是 constructor 的核心價值：把「使用前必須初始化」改成「能拿到 object 就代表它已經初始化」。

Member initializer list

Constructor 後面的 : width(width), height(height), pixels(width * height) 叫 member initializer list。它不是語法裝飾，而是成員真正被初始化的地方。

初始化
Framebuffer(int w, int h)
  : width(w), height(h), pixels(w * h) {}

成員一出生就是正確狀態。

先預設建構，再賦值
Framebuffer(int w, int h) {
  width = w;
  height = h;
  pixels = std::vector<uint32_t>(w * h);
}

成員先被預設建立，進入 body 後才改值。

對 int 這種簡單型別差異不大，但對 const 成員、reference 成員、沒有 default constructor 的成員，initializer list 是必要的。更重要的是，它表達了正確語意：初始化不是事後補值。

重點 — Constructor body 執行前，所有 member 已經被初始化完了。Initializer list 決定「成員怎麼出生」，constructor body 只是在出生後執行額外邏輯。

3 Destruction：離開 scope 時自動收尾

Destructor（解構函式）的語意是：object lifetime 結束時自動執行。如果 object 住在 Stack，它離開 scope 時 destructor 自動跑；如果 object 住在 Heap，對它 delete 時 destructor 會先跑，然後記憶體才歸還。

class Window {
public:
  Window(int width, int height) {
    handle = create_window(width, height);
  }

  ~Window() {
    if (handle) destroy_window(handle);
  }

private:
  WindowHandle handle = nullptr;
};

現在呼叫端不用記得 Close()：

void run() {
  Window window(800, 600);

  if (!load_assets()) {
    return; // window 離開 scope，~Window() 自動執行
  }

  // render loop...
} // 正常結束也會自動執行 ~Window()

4 RAII：C++ 對 resource 的核心回答

RAII 是 Resource Acquisition Is Initialization。名字有點怪，但意思很直接：

RAII — 在 constructor 取得 resource，在 destructor 釋放 resource。resource 的 lifetime 綁定到 object 的 lifetime。

這是 C++ 相對 C 最重要的語意升級之一。C 只能靠 convention：「記得配對呼叫」。C++ 可以把配對封裝進型別：

ResourceConstructor 取得Destructor 釋放
Heap memorynew / allocatedelete / free
Filefopenfclose
WindowCreateWindowDestroyWindow
DIB bitmapCreateDIBSectionDeleteObject
Mutex locklockunlock

這也是為什麼 std::vector 好用。你不需要手動釋放它內部的 array；vector object 消滅時，它的 destructor 會自動釋放 owned memory。std::unique_ptr 也是同一個思想：把 delete 放進 destructor。

5 建構與解構順序

一個 class 裡如果有多個 member，它們的建構順序不是 initializer list 寫的順序，而是成員在 class 中宣告的順序。解構順序則剛好相反。

class RendererApp {
public:
  RendererApp()
    : rasterizer(device), device(framebuffer), framebuffer(800, 600) {}

private:
  Framebuffer framebuffer; // 1. 先建構
  RenderDevice device; // 2. 再建構
  Rasterizer rasterizer; // 3. 最後建構
};

即使 initializer list 寫成 rasterizer(device), device(framebuffer), framebuffer(...)，真正順序仍然是：framebuffer、device、rasterizer。這很重要，因為 later member 可能依賴 earlier member。

常見陷阱 — 不要用 initializer list 的視覺順序判斷建構順序。請用 class member declaration order 判斷。實務上，initializer list 最好也照 declaration order 寫，避免誤導自己。

解構順序反過來

如果建構順序是 framebuffer -> device -> rasterizer，解構順序就是 rasterizer -> device -> framebuffer。這很合理：最後建好的通常依賴前面建好的，所以要最先拆掉。

6 Rule of Zero：先讓成員自己管理資源

你可能聽過 Rule of Three / Five：如果你自己寫 destructor，通常也要思考 copy constructor、copy assignment、move constructor、move assignment。原因是：一旦 class 手動管理 resource，copy/move 語意就變危險。

class BadBuffer {
public:
  BadBuffer(int n) { pixels = new uint32_t[n]; }
  ~BadBuffer() { delete[] pixels; }

private:
  uint32_t* pixels;
};

這個 class 看起來有 RAII，但還不完整。預設 copy 只會複製 pointer value，兩個 BadBuffer 會指向同一塊 array，最後 destructor 跑兩次，造成 double delete。

所以現代 C++ 的第一選擇通常是 Rule of Zero：不要自己寫 destructor，讓 std::vector、std::unique_ptr 這些已經會 RAII 的 member 幫你管理。

class Framebuffer {
public:
  Framebuffer(int width, int height)
    : width(width), height(height), pixels(width * height) {}

private:
  int width;
  int height;
  std::vector<uint32_t> pixels; // vector 自己負責釋放
};
工程判斷 — 如果 resource 已經能被標準庫型別表達，先用標準庫型別。只有在包裝 OS handle、C API handle、GPU resource 這類標準庫不認識的東西時，才自己寫 destructor。
常見陷阱

1. 在 constructor body 才補初始化 — member 其實已經先被初始化過了。需要精確控制出生狀態時，用 initializer list。

2. 忘記 destruction order — member 按宣告順序建構，反向解構。依賴關係要寫在 member order 裡。

3. 寫了 destructor 卻沒思考 copy/move — 手動管理 pointer resource 很容易 double delete。優先用 Rule of Zero。
本章學到的三件事

1. Constructor 的核心語意是讓 object 從一開始就進入有效狀態，不是事後呼叫 Init() 補救。

2. Destructor 會在 object lifetime 結束時自動執行，所以能把 resource release 綁到 scope exit。這就是 RAII 的基礎。

3. 建構順序由 member declaration order 決定，解構順序反過來。手動管理 resource 時要思考 copy/move；能用標準庫管理就用 Rule of Zero。
下一步 — 你現在知道 object lifetime 可以保護 resource lifetime。下一章會進入 Copy 語意：如果一個 object 擁有 resource，那複製它到底應該代表「複製 bytes」、「複製 ownership」、「建立另一份獨立 resource」，還是根本禁止 copy？
