THEORY TRACK · T-09
STL / Containers / Iterators / Allocator
STL 不是一包方便工具，而是一組 ownership、lifetime、invalidation、algorithm contract
std::vector std::array std::span iterator allocator

T-08 把 template 和 concepts 講成 compile-time contract。這一章接到 STL：std::vector、std::array、std::span、iterator、allocator 不是零散 API，而是 C++ 對「一群 objects 如何被擁有、排列、拜訪、搬移、失效」的標準化語意。

對 Pixel-Renderer 來說，這不是語法複習。Framebuffer、depth buffer、vertex buffer、UI command buffer、texture memory、draw list，全部都是「一段 storage + 一組 lifetime rule + 一套 access pattern」。Container 選錯，問題不會只是不優雅，而是 dangling pointer、cache miss、reallocation cost、backend lifetime 被污染。

本章範圍 — 從 std::vector 的 pointer/size/capacity 模型開始，拆 owner container、non-owning view、iterator invalidation、erase-remove、contiguous storage、allocator basics。目標是建立 Pixel-Renderer 的 container policy：什麼資料應該 owns memory，什麼資料只應該 view，什麼地方可以暫時用 vector，什麼地方未來可能需要 arena/frame allocator。

1 naive model：STL 是比較好用的 array

最常見的直覺模型是： 

C array:     T data[N]
std::vector: dynamic T data[N]
std::array: safer T data[N]
std::span:  pointer + length

所以 STL = 一堆方便包裝

這個模型只能幫你寫小程式，無法支撐 renderer。因為 renderer 真的會問：

  • 誰 owns framebuffer memory？
  • 當 std::vector reallocate 時，舊的 pointer、reference、iterator、std::span 還能不能用？
  • shader 或 UI command 可以存 std::string_view 嗎？它 view 的字串活到什麼時候？
  • command buffer 每 frame 清空時，要釋放 memory，還是保留 capacity？
  • future frame allocator 是換掉 container，還是換掉 allocation policy？

所以更精準的模型是： 

Container is not just "data structure".
Container is:

  storage ownership
  + object lifetime management
  + element layout guarantee
  + invalidation rules
  + algorithm access contract
  + allocation policy hook
關鍵轉換 — 不要先問「我要用哪個 STL class」。先問這段資料的 owner 是誰、元素是否需要 contiguous、會不會 grow、會不會把 pointer/view 借出去、借出去之後 container 會不會改變 storage。

2 三種角色：owner、view、cursor

STL container 的第一層不是 API，而是角色分工：

角色 典型型別 語意 Pixel-Renderer 例子
owner container std::vector, std::array, std::string 負責 element lifetime，也負責釋放 storage。 Framebuffer owns color buffer；model owns vertices；UI frame owns commands。
non-owning view std::span, std::string_view 只描述「我可以看這段 memory」，不延長 lifetime。 FramebufferView 借看 pixels；backend present 借看 CPU framebuffer。
cursor / iterator iterator, pointer, index 指向 container 裡某個位置，通常依賴 container storage 沒有失效。 掃描 command buffer、erase invisible widgets、遍歷 vertices。

這個分法會直接影響 interface。如果 function 需要拿走資料，就接 owner；如果只是暫時讀寫，就接 view；如果只是遍歷某段範圍，就接 iterator/range。把這些混在一起，後面一定會變成 lifetime bug。

3 contiguous storage：為什麼 renderer 特別在意

std::vector 最重要的保證不是「會自動變大」，而是它的 elements 連續排列。這件事對 renderer 非常重要，因為 framebuffer、vertex buffer、command buffer 的 hot path 通常都是線性掃描。

SVG 1：std::vector object 本身通常只是一組 metadata；真正的 elements 在另一段 contiguous storage。

這裡的「連續」不是抽象好看而已。它讓以下事情成立： 

framebuffer pixel address:
  index = y * width + x
  address = data + index

scanline fill:
  for x in [x0, x1):
    data[y * width + x] = color

linear command replay:
  for cmd in commands:
    execute(cmd)

如果你改用 linked list，單次插入可能便宜，但掃描會變成 pointer chasing。對 hot loop 來說，memory layout 往往比單一 operation 的 Big-O 更重要。

結構類比，不是完全等價 — std::vector 很像 CPU 端的 vertex buffer / command buffer：一段連續 memory，加上目前有效長度。但 GPU buffer 還牽涉 driver、VRAM、同步與 mapping，所以不能把 STL vector 直接等同成 GPU buffer。

4 std::vector：owner + growable contiguous storage

std::vector<T> owns 一段可 grow 的 contiguous storage。它管理三件事：

  • data()：目前 elements 的起點。
  • size()：已經存在、lifetime 已開始的 elements 數量。
  • capacity()：已配置但尚未全部用來建構 elements 的 storage 數量。
std::vector<uint32_t> pixels;
pixels.resize(width * height); // creates elements

std::fill(pixels.begin(), pixels.end(), clear_color);
pixels[y * width + x] = color;

如果 vector 還有 capacity，push_back 只需要在尾端建構新 element。如果 capacity 不夠，它必須： 

1. allocate larger storage
2. move or copy existing elements into new storage
3. construct the new element
4. destroy old elements
5. deallocate old storage
6. update data/size/capacity metadata

這就是為什麼 reserve 很重要。對 command buffer 或 vertex buffer，如果你大概知道上限，先 reserve 可以避免 frame 中途 reallocation。

std::vector<DrawCommand> commands;
commands.reserve(4096); // allocate storage, does not create 4096 commands

commands.clear(); // destroy elements, keep capacity
commands.push_back(cmd); // construct one command
resize vs reserve — resize(n) 讓 size() 變成 n，elements 真的存在；reserve(n) 只確保 capacity 至少是 n，不改變 size，也不讓那些位置變成可讀寫的 elements。

5 reallocation：為什麼 pointer/span/iterator 會失效

這章最容易出 bug 的點是：只要 vector reallocate，舊 storage 就不再是 vector 的 elements。你以前借出去的 pointer、reference、iterator、std::span 全部可能 dangling。

SVG 2：reallocation 不是原地變大；vector 可能搬到新 block，因此舊 view 不會自動指向新 storage。

這個問題在 framebuffer abstraction 裡特別危險。假設 backend 保存了 pixels.data()，但 render core 下一 frame 對 pixels 做了 resize，backend 手上的 pointer 就可能變成 dangling。

std::vector<uint32_t> pixels;
pixels.resize(width * height);

uint32_t* raw = pixels.data();

pixels.push_back(0xff00ff00); // may reallocate

raw[0] = 0; // bug if reallocation happened
規則 — 不要把 vector 的 pointer/span 長期存在另一個 object 裡，除非你能保證 vector 的 storage 不會改變。更穩的做法是每次 Present 或 BeginFrame 重新取得 view。

6 noexcept move：vector reallocation 會看你的 type 設計

當 vector 需要搬移 elements 到新 storage 時，它要決定用 move 還是 copy。如果 T 的 move constructor 可能 throw，而 copy 可以用，標準庫實作常會傾向 copy，因為這樣比較容易維持 strong exception guarantee。 

vector grows
  |
  +-- if T can be moved safely enough:
  |      move elements to new storage
  |
  +-- otherwise if T is copyable:
         copy elements to new storage

這就是 T-06 的 noexcept move 會回來影響 T-09 的原因。Container 不是只看元素語法上能不能 move，也看 move 的 exception contract。

class TextureHandle {
public:
  TextureHandle(TextureHandle&& other) noexcept;
  TextureHandle& operator=(TextureHandle&& other) noexcept;
  TextureHandle(const TextureHandle&) = delete;
};

std::vector<TextureHandle> textures; // reallocation depends on move correctness

對 Pixel-Renderer 第一階段的 DrawCommand、Vertex、Color，最好保持 value-like、cheap-to-move，避免 command buffer 裡塞入複雜 resource handle。Resource handle 可以在別處 owns，command 只存 id 或 stable reference。

7 std::span：borrowed view，不是小 vector

std::span<T> 是 pointer + size 的 non-owning view。它不 allocate、不 copy elements、不延長 lifetime。

void Clear(std::span<uint32_t> pixels, uint32_t color) {
  std::fill(pixels.begin(), pixels.end(), color);
}

std::vector<uint32_t> framebuffer(width * height);
Clear(framebuffer, clear_color); // vector borrows as span

對 framebuffer 來說，單純 std::span<uint32_t> 只描述一維 contiguous pixels。如果你需要 pitch、width、height、format，就應該包成 domain-specific view：

struct FramebufferView {
  std::span<uint32_t> pixels;
  int width = 0;
  int height = 0;
  int pitch_pixels = 0;
};

uint32_t& At(FramebufferView fb, int x, int y) {
  return fb.pixels[y * fb.pitch_pixels + x];
}
span 的危險 — std::span 看起來像 container，但它不是 owner。它的正確性完全依賴外部 storage 活著，而且 storage 沒被 reallocate。

8 std::array：固定大小 owner，size 是 type 的一部分

std::array<T, N> owns N 個 elements，但不動態配置。它通常直接把 elements 放在 object 裡。這適合固定維度的小資料，例如 matrix row、compile-time size 的 coefficients，或一些小 lookup table。

template <std::size_t N>
struct Vec {
  std::array<float, N> e{};
};

Vec<3> position;
Vec<4> clip_space;

但 T-08 已經提醒過：representation 相同不代表語意相同。Position3、Normal3、ColorRGB 都可能是三個 float，但 valid operations 不一定一樣。std::array 可以作為 storage，但不要讓它吃掉 domain type 的 invariant。

9 string_view：文字 view 的 lifetime 陷阱

std::string_view 是字元序列的 borrowed view。它非常適合 parse、logging、查表、短時間讀取，但不適合長期存進 command buffer，除非你能保證背後字串活得更久。

struct LabelCommand {
  std::string_view text;
};

std::vector<LabelCommand> commands;

{
  std::string tmp = "FPS";
  commands.push_back({tmp});
}

// commands[0].text now dangles

Immediate-mode UI 很容易踩到這個坑。第一版可以保守一點：UI draw command 存 std::string owner，或讓 text storage 有明確 frame lifetime，再用 string_view 指向那個 frame arena。

10 iterator invalidation：algorithm contract 的底層代價

Iterator 是「某 container 某位置」的 cursor。不同 container 對 iterator 的 invalidation rule 不一樣。這裡先抓 Pixel-Renderer 最常用的 vector：

vector operation 可能失效的東西 工程後果
push_back 且沒有 reallocate end() 與尾端相關 iterator 不要在 loop 中假設舊 end 永遠代表新尾端。
push_back 且 reallocate 所有 pointer/reference/iterator/span backend 或 command system 不能保存舊 data()。
erase erase 位置與其後所有 iterator/reference 刪除 UI commands 時要用回傳的新 iterator。
clear 所有 elements 的 reference/iterator capacity 可保留，但 elements lifetime 結束。
reserve 造成 reallocate 所有 pointer/reference/iterator/span 借出的 view 必須重新取得。

這也是為什麼有些系統會用 index 而不是 pointer。Index 不是萬靈丹，因為元素刪除或排序後 index 的語意也可能改變；但如果 vector reallocate，index 至少不會變成 dangling memory address。

11 erase-remove：刪除不是單一動作

std::remove_if 不會真的縮短 container。它只把要保留的 elements 往前搬，回傳新的 logical end。真正縮短 vector 要再呼叫 erase。

commands.erase(
  std::remove_if(commands.begin(), commands.end(), [](const DrawCommand& cmd) {
    return cmd.clip_rect.IsEmpty();
  }),
  commands.end()
); 
before:
  [keep][remove][keep][remove][keep]

remove_if result:
  [keep][keep][keep][?][?]
                    ^
                    new_end

erase(new_end, end):
  [keep][keep][keep]

這個 pattern 在 UI draw list、visibility culling、command compaction 都會用到。但如果順序有語意，例如 painter's algorithm 或 UI z-order，就不能隨便用會改變 order 的 partition 類演算法。

12 allocator basics：先理解 hook，再決定要不要用

Allocator 是 container 用來取得與釋放 raw storage 的 policy hook。它不是「更快 vector」的同義詞。你可以先用這個模型： 

std::vector
  |
  +-- asks Allocator for raw storage
  +-- constructs T objects in that storage
  +-- destroys T objects
  +-- returns storage to Allocator

第一版 Pixel-Renderer 不需要急著自訂 allocator。更重要的是先把 ownership boundary 切乾淨，讓未來真的需要 frame allocator 時，有地方可以換。

需求 第一版做法 未來可能演進
framebuffer color/depth std::vector owns stable storage 固定尺寸 buffer pool，或 platform-specific mapped memory。
UI command buffer std::vector + reserve + per-frame clear frame arena / monotonic buffer resource。
temporary raster data stack object 或 local vector scratch allocator。
resource handles stable owner container + id/reference generational handle table。
判斷順序 — 先用 ordinary containers 寫出正確 ownership 和 invalidation rule；等 profiler 或架構需求指出 allocation 是問題，再引入 allocator / memory resource。

13 Pixel-Renderer container policy

把這章收束到專案，可以得到一張暫時 policy map：

資料 建議型別 原因 注意事項
color framebuffer std::vector<uint32_t> CPU owns pixels，contiguous，方便 upload/present。 resize 後所有 view 失效；backend 每次 present 重新借 view。
depth buffer std::vector<float> 和 color buffer 同尺寸、線性掃描。 clear 要注意 NaN / far value policy。
framebuffer view FramebufferView wrapping std::span 借用 pixels，不 owns memory。 必須帶 width/height/pitch/format contract。
vertex buffer std::vector<Vertex> model data 或 generated primitives 通常連續遍歷。 如果 view 借給 pipeline stage，期間不要 reallocate。
matrix/vector internals std::array<float, N> 固定尺寸、無 heap allocation。 不要讓 storage type 取代 domain type。
UI draw commands std::vector<DrawCommand> 每 frame 建立、排序、replay、清空。 command 內不要藏 dangling string_view 或 raw resource pointer。
resource name / text std::string owner 或受控 std::string_view 文字 lifetime 常比想像複雜。 跨 frame 保存時優先 owner；view 只用在明確短生命週期。

14 實作規則：先讓 lifetime 可檢查

這章不是要你把每個地方都換成 STL，而是建立幾條工程規則：

  1. 如果資料 owns memory，型別名稱要看得出 ownership，例如 Framebuffer、Mesh、CommandBuffer。
  2. 如果資料只是借用，型別名稱要看得出 view，例如 FramebufferView、VertexSpan。
  3. 不要把 vector.data() 長期存到 backend 或 subsystem；跨 frame 保存 raw pointer 要非常警覺。
  4. hot path 預期會 grow 的 vector 先 reserve，但不要把 reserve 誤當成 elements 已經存在。
  5. command buffer 盡量存 value 或 stable id，不要存指向 temporary object 的 reference/view。
  6. allocator/frame arena 是後續優化，不是第一版抽象分離的前提。

用這種方式看，PixelBuffer 或 Framebuffer 並不是「render frontend/backend」本身；它是 renderer core 和 display backend 之間共享的 CPU-side image storage。真正的 frontend/backend 邊界會在後續 I-13 講：誰產生 render commands、誰執行 rasterization、誰負責 present。

15 Summary

本章學到的三件事：
  1. Container 是 ownership、lifetime、layout、invalidation、algorithm contract 的組合，不只是比較方便的 array。
  2. std::vector 的 contiguous storage 很適合 framebuffer、vertex buffer、command buffer，但 reallocation 會讓舊 pointer/span/iterator 失效。
  3. Pixel-Renderer 第一版應先用 std::vector owns CPU buffers、用 view 借出短生命週期 access；allocator/frame arena 等到 ownership boundary 穩定後再評估。

下一章會從 type invariant 和 error handling 回來看：container 和 view 的規則要如何用 assert、precondition、return type、exception/noexcept、contract style 寫進 interface，而不是只靠使用者記得。
