THEORY TRACK · T-14
Modern C++ Boundaries
不是越新越好，而是讓語言特性服務 renderer 的語意邊界
C++20 C++23 toolchain boundaries

這份 C++ 複習已經用到不少 Modern C++：RAII、move、std::vector、std::span、concepts、variant、thread。最後要補一個工程判斷：哪些新特性應該現在用，哪些先知道概念但延後導入？

Modern C++ 的危險不在新，而在它很容易讓專案從「renderer 學習」變成「語言特性展示」。Pixel-Renderer 的主線仍然是：framebuffer ownership、software rasterization、backend abstraction、testing verification。語言特性只有在讓這些邊界更清楚時才值得引入。

本章範圍 — 整理 Pixel-Renderer 的 Modern C++ 採用邊界：C++17/20/23 的 span、concepts、optional、variant、ranges、expected、modules、coroutines、pmr、filesystem。重點是避免把專案變成語言特性展示，而是讓每個工具對應明確問題、toolchain 成本與 verification 需求。

1 naive model：用越新的 C++ 越現代

常見的想法是： 

C++20 > C++17
C++23 > C++20
newer feature = better architecture

這個模型不夠。新語法可能讓程式更清楚，也可能讓 build、debug、跨平台、學習主線變重。對 Pixel-Renderer，語言版本選擇應該服務三件事：

  • 可理解：概念是否幫你看清 ownership、lifetime、contract？
  • 可移植：Windows / macOS / Ubuntu toolchain 是否穩定支援？
  • 可驗證：它是否讓測試、debug dump、benchmark 更清楚？
不要讓語言特性替代架構思考 — ranges、concepts、variant、coroutine 都不能自動修好錯誤的 ownership boundary。先有清楚資料流，再選工具。

2 Feature adoption map

SVG：Modern C++ 採用不是按年份，而是按問題、toolchain 成本與核心邊界決定。

3 現在值得用：讓語意更清楚的工具

工具解決的問題Pixel-Renderer 用途
std::vector, std::array穩定 owner container / fixed storage。framebuffer、depth buffer、vertex buffer、math storage。
std::span清楚表達 borrowed contiguous view。FramebufferView、MeshView、backend present view。
std::optional合法 absence，不含錯誤原因。lookup、optional resource、hit test result。
std::variantclosed-set value polymorphism。RenderCommand、UiCommand、debug event。
concepts讓 template requirement 可讀。shader callable、pixel policy、math type requirement。
std::unique_ptr明確 single ownership。DisplayBackend runtime object、platform resource wrapper。

這些工具的共同點是：它們直接讓 ownership、borrow、type set、generic requirement 更清楚。這符合 Pixel-Renderer 的學習主線。

4 先理解，但不要急著放進 core

工具延後理由未來可能用途
rangesdebug/error message 和編譯成本可能增加；hot loop 中也可能遮住實際資料流。工具層資料轉換、asset processing。
std::expectedC++23 支援度與專案標準需確認。backend init、resource loading、file parse result。
pmr allocator需要先穩定 ownership 和 allocation hotspot，否則只是提前複雜化。frame arena、UI command buffer、temporary geometry。
modules跨 compiler、CMake、IDE 支援仍會增加工程成本。大型專案 compile-time 管理。
coroutines抽象成本高，與 renderer core hot path 關聯低。asset streaming、async IO、editor tooling。
std::format很好用，但 toolchain 支援和 binary/code size 要確認。debug dump、log、test reports。
延後不等於排斥 — 先不用 modules/coroutines/pmr，不是因為它們不好，而是因為目前最關鍵的問題是 renderer core boundary、testing、backend abstraction。工具要等痛點明確再進場。

5 語言版本也是跨平台 boundary

I-12 講 cross-platform development 時，toolchain 是重要邊界。語言版本也一樣： 

Windows:
  MSVC / MinGW GCC

macOS:
  Apple Clang + libc++

Ubuntu:
  GCC / Clang + libstdc++ or libc++

Question:
  Which standard library features are consistently available?
  Which debugger understands the generated code well?
  Which CMake settings keep all platforms aligned?

所以「用 C++23」不是單一選項。你要同時問 compiler version、standard library version、CMake preset、VSCode IntelliSense、debugger，以及 CI 環境。

6 Project standard 建議

Short-term:
  Prefer C++20 if the local toolchain is stable.
  Use span, concepts, optional, variant, unique_ptr.
  Avoid requiring C++23-only features in renderer core.

Mid-term:
  Add an expected-like Result type for backend/resource init.
  Consider pmr only after allocation profiles show a real problem.
  Use ranges carefully in tools, not blindly in hot loops.

Long-term:
  Consider modules only if project size and build time justify it.
  Consider coroutines only for asset streaming or editor workflows.

這樣的策略比較無聊，但對 renderer 專案是優點：核心資料流保持直接，語言工具只在能提升語意清晰度時引入。

7 Feature-by-layer policy

Layer推薦風格避免
rasterizer hot loopplain loops、simple structs、span/view、concept-constrained shader。heavy ranges、per-pixel type erasure、allocation。
renderer core APIRAII、span、optional/result、variant for closed commands。platform header、unclear lifetime callback。
display backendunique_ptr + virtual interface、platform-private implementation。SDL/Win32 types in core headers。
tools/testsfilesystem、format、ranges can be useful。forcing tool dependencies into core。
future editor/assetexpected/result、possibly coroutine for async IO。coroutine in rasterizer core before need exists。

8 Adoption checklist

每次想引入一個新特性，先問：

  • 它是否讓 ownership/lifetime/contract 更清楚？
  • 它是否讓 Windows/macOS/Ubuntu toolchain 更難同步？
  • 它是否進入 rasterizer hot loop？如果是，成本是否可測？
  • 它是否改善 debug dump、golden image、benchmark、CI？
  • 它是否只是為了看起來 modern？
Layered Structural Inquiry 的用法 — 先從具體痛點開始，例如「command buffer pointer lifetime 很亂」。再問 naive model 為何失敗，拆到底層 lifetime/build/debug 成本，最後才選 variant、span、pmr 或其他工具。
本章學到的三件事：
  1. Modern C++ 的價值不在新，而在它是否讓 ownership、contract、generic requirement、resource lifetime 更可檢查。
  2. Pixel-Renderer 短期可穩定使用 C++20 的 span、concepts、optional、variant；C++23 expected、pmr、modules、coroutines 可先理解再延後。
  3. 語言版本是工程邊界的一部分，必須和 Windows/macOS/Ubuntu toolchain、VSCode、CMake、debugger、CI 一起考量。
