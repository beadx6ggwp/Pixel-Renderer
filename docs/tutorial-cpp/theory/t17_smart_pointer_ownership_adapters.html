THEORY TRACK · T-17
Smart Pointer / Ownership Adapters
smart pointer 不是「比較安全的 pointer」，而是幾種完全不同的 ownership adapter
unique_ptr shared_ptr weak_ptr handle

T-06 已經建立 RAII 的核心：resource lifetime 應該被 object lifetime 管起來。但 smart pointer 值得獨立成章，因為它很容易被誤解成「把 raw pointer 換成比較安全的版本」。這個模型會害你在 renderer 裡到處塞 shared_ptr，最後 lifetime 看似不會爆，實際上 ownership 變得更難推理。

這章要把 smart pointer 拆成幾種不同的 ownership adapter：unique_ptr 表達唯一擁有者；shared_ptr 表達共享擁有與 control block；weak_ptr 表達非擁有觀察並打斷 cycle；raw pointer/reference/span 表達 borrow；renderer resource handle 則常用 id 取代 pointer lifetime。

本章範圍 — 深入 unique_ptr、shared_ptr、weak_ptr、custom deleter、incomplete type、virtual destructor、aliasing、cycle、raw observer、std::span、generational handle。目標是回答 Pixel-Renderer 裡 backend、texture、mesh、material、command buffer、UI resource cache 到底該用哪種 ownership 表達。

1 naive model：smart pointer 比 raw pointer 安全

這句話只對一半。Smart pointer 主要安全的地方，是它讓某些 release path 自動化；但它不能自動讓 ownership 正確。 

wrong mental model:
  raw pointer bad
  smart pointer good
  shared_ptr safest

better model:
  pointer form must match ownership semantics
  wrong smart pointer can hide a wrong architecture

例如 command buffer 裡如果存 std::shared_ptr<Texture>，表面上 texture 不會 dangling；但你也把「draw command 可以延長 texture lifetime」這個能力交給了每個 command。這可能讓 frame replay、resource unload、debug dump 都更難判斷。

核心修正 — smart pointer 不是安全等級，而是 ownership 語意。選錯 smart pointer，通常比 raw pointer 更難 debug，因為錯誤被 reference count 或隱性 ownership 蓋住。

2 unique_ptr：唯一擁有者與 runtime polymorphism

std::unique_ptr<T> 表達一件事：這個 object 有唯一 owner。它不能 copy，只能 move。

std::unique_ptr<DisplayBackend> display;
display = std::make_unique<SdlDisplayBackend>();
display->Present(framebuffer);

這很適合 Application owns 一個 runtime-selected backend：Windows 用 Win32 backend，macOS/Ubuntu 用 SDL backend。這裡需要 runtime polymorphism，但不需要 shared ownership。

歷史補充 — C++98 曾有 std::auto_ptr，它也能轉移 ownership，但它用 copy syntax 做 destructive transfer，破壞 Copyable type 的假設。這正是 unique_ptr 要 delete copy、只允許 explicit move 的原因。完整拆解見 A-01 auto_ptr 與 Move Semantics 的誕生。
Base class destructor — 如果你用 unique_ptr<Base> 刪除 derived object，Base 應該有 virtual destructor。否則透過 base pointer delete derived object 會出問題。
class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
  virtual void Present(const Framebuffer&) = 0;
};

3 make_unique：避免裸 new 暴露在 call site

優先用 std::make_unique：

auto backend = std::make_unique<SdlDisplayBackend>(config);

它讓 allocation 和 construction 綁在同一個 expression，避免 call site 出現裸 new，也讓 ownership 從一開始就是 unique_ptr。對 renderer code review 來說，看到裸 new 應該立刻問：owner 是誰？為什麼不用 RAII member 或 make_unique？

4 custom deleter：把 C API release function 接進 RAII

SDL / Win32 / OpenGL 這類 API 的 resource 常不是用 delete 釋放。這時 unique_ptr 可以搭配 custom deleter：

struct SdlTextureDeleter {
  void operator()(SDL_Texture* p) const noexcept {
    if (p) SDL_DestroyTexture(p);
  }
};

using SdlTexturePtr = std::unique_ptr<SDL_Texture, SdlTextureDeleter>;

但是 custom deleter 也有 cost model：如果 deleter 是 empty stateless type，常可被 empty base optimization 壓掉；如果 deleter 是 function pointer 或帶狀態 object，unique_ptr 本身可能變大。

deleter 形式語意成本注意
stateless functor固定 release function。通常最適合 C API wrapper。
function pointerruntime 選 deleter。unique_ptr 需要存 pointer。
stateful deleterrelease 需要額外 context。會增加 object size，要明確理由。

5 incomplete type：PImpl 與 header hygiene

unique_ptr<T> 可以持有 incomplete type，這對 I-17 的 header boundary 很重要：

// sdl_display_backend.h
class SdlDisplayBackend {
public:
  SdlDisplayBackend();
  ~SdlDisplayBackend();
private:
  struct Impl;
  std::unique_ptr<Impl> impl_;
};

但 destructor 通常要放在 .cpp，讓 compiler 在刪除 Impl 時看得到 complete type。這是 PImpl 能把 SDL/Win32 header 藏在 implementation 的原因之一。

和 I-17 的連接 — PImpl 不是一開始就必用，但當 platform header 污染 core header 時，unique_ptr<Impl> 是一種把 private dependency 藏進 .cpp 的工具。

6 shared_ptr：共享擁有與 control block

std::shared_ptr<T> 不是「可以安全傳來傳去的 pointer」。它表示多個 owner 共同延長同一個 object 的 lifetime。每個 shared_ptr 共享一個 control block，裡面通常有 strong count、weak count、deleter 等資訊。

SVG：shared_ptr 的重點是 control block 與 reference count。copy shared_ptr 不是免費借用，而是改變 object lifetime。

這也代表它有成本：

  • copy 需要更新 reference count，通常涉及 atomic。
  • object lifetime 由最後一個 owner 決定，不容易從架構圖看出。
  • cycle 會讓 strong count 永遠不歸零。
  • hot path 裡大量 copy shared_ptr 會把 ownership 成本帶進 performance path。
renderer policy — shared_ptr 適合真的需要共享 ownership 的高層 resource cache；不適合每個 draw command、每個 pixel、每次 shader call 都 copy 一次。

7 weak_ptr：觀察 shared object，打斷 cycle

如果 A owns B，B 又用 shared_ptr owns A，就會形成 cycle： 

Material --shared_ptr--> Texture
Texture  --shared_ptr--> Material

strong count never reaches zero
destructor never runs

std::weak_ptr<T> 不增加 strong count。它可以觀察 shared object 是否還活著：

std::weak_ptr<Texture> weakTexture;

if (auto texture = weakTexture.lock()) {
  Use(*texture);
} else {
  // texture was destroyed
}

但這仍然不是 renderer command buffer 的第一選擇。Command buffer 若需要 deterministic replay，通常更適合存 resource id，而不是 weak pointer。

8 aliasing constructor：shared_ptr 可以指向子物件

shared_ptr 有一個容易被忽略的能力：aliasing constructor。它可以共享某個 owner 的 control block，但 pointer 指向另一個地址。

std::shared_ptr<Mesh> mesh = LoadMesh();
std::shared_ptr<VertexBuffer> vb(mesh, &mesh->vertices);

這代表 vb 的 lifetime 會讓 mesh 活著，但 vb.get() 指向的是 mesh 裡的 member。它很強，但也很容易讓 ownership 變隱晦。除非你很明確需要這種 shared subobject lifetime，否則不要把它當一般工具。

9 raw pointer / reference / span：borrow 不是 ownership

不是所有 pointer 都該變 smart pointer。Renderer hot path 裡，大量物件只是暫時借用：

形式語意例子
T&不可 null 的暫時 borrow。RenderDevice borrows Framebuffer。
const T&read-only borrow。DisplayBackend::Present(const Framebuffer&)。
T*nullable borrow 或 C API interop。optional debug overlay pointer。
std::span<T>borrowed contiguous range。vertices、indices、pixels row view。

Borrow 的 contract 是：owner 必須比 borrower 活得久。這件事 smart pointer 不會自動幫你想清楚；它只會改變 lifetime policy。

和 T-16 的連接 — API signature 應該先說 ownership。需要擁有才用 owner type；只是借用就用 reference/span；可能不存在才用 pointer/optional；跨 frame 保存才要重新設計 resource identity。

10 Resource handle：renderer 常用 id 取代 pointer

當你開始有 texture、mesh、material、shader resource，draw command 不一定要保存 pointer：

struct TextureId {
  uint32_t index;
  uint32_t generation;
};

struct DrawCommand {
  MeshId mesh;
  MaterialId material;
};

這種 generational handle 的想法是：resource registry owns 真正資料，command 只保存可驗證 id。當 resource 被刪除再重用 slot，generation 可以幫你抓 stale handle。 

ResourceRegistry owns:
  textures[]
  generations[]

DrawCommand stores:
  TextureId { index, generation }

lookup:
  if generation mismatch -> stale handle

這比把每個 command 都塞 shared_ptr<Texture> 更接近 renderer / engine 的資料模型，因為 command 是可序列化、可 dump、可 replay 的 value data。

11 Pixel-Renderer ownership policy

第一版可以採取： 

Application:
  owns DisplayBackend via unique_ptr
  owns Framebuffer as value member

Backend:
  owns SDL/Win32 handles via RAII wrappers or unique_ptr with custom deleter
  does not expose platform handles to renderer core

Renderer hot path:
  use references / spans / views for borrowed data
  avoid shared_ptr copy inside pixel / triangle loops

Command buffer:
  prefer value commands
  for future resources, prefer ids / handles over owning pointers

Resource cache:
  shared_ptr only when shared ownership is intentional
  weak_ptr only to observe shared lifetime or break cycles
本章學到的三件事：
  1. Smart pointer 是 ownership adapter，不是安全等級；unique_ptr、shared_ptr、weak_ptr 表達的是不同 lifetime policy。
  2. shared_ptr 有 control block、reference count、cycle、aliasing、atomic cost；它不應被拿來掩蓋不清楚的 renderer resource ownership。
  3. Pixel-Renderer 的 hot path 多半應用 reference/span/value/id；platform/backend resource 用 RAII wrapper 或 unique_ptr，future texture/material 則適合 resource handle。

下一章先補另一個底層邊界：casts。理解 smart pointer 之後，你還需要知道 pointer type 改變、object representation copy、platform buffer interpretation 之間的差別。這會直接影響 framebuffer、pixel format、C API callback context 的安全性。
