Static Storage / Global State / Initialization Order
global 不是只是變數放外面；它會改變 lifetime、dependency direction、testability 與 deterministic replay
static-storage global-state initialization-order composition-root

T-03 說 object lifetime 不等於 raw storage 存在；T-06 說 RAII 讓 resource 跟 object lifetime 綁在一起；T-16 說 API signature 應該表達 owner / borrow / view。現在要問一個工程現場很常見的問題：如果某個東西大家都會用，為什麼不直接做成 global？

Pixel-Renderer 很快會遇到這些誘惑：global renderer device、global framebuffer、global resource cache、global logger、global config、global font atlas、global default texture。它們一開始都看起來省事，但它們會把 dependency、lifetime、test isolation、multi-window、headless render、deterministic replay 全部綁在一起。

本章範圍 — 用 problem genealogy 拆 static/global state：static storage duration、constant initialization、dynamic initialization、static initialization order fiasco、function-local static、inline variable、internal linkage、singleton-style API、thread_local、composition root，以及 Pixel-Renderer 哪些狀態可以 global、哪些必須由 Application / RendererContext 顯式擁有。

1 naive model：大家都會用，所以放 global

naive 設計通常長這樣：

Framebuffer gFramebuffer(800, 600);
Renderer gRenderer(gFramebuffer);
std::unordered_map<TextureId, Texture> gTextureCache;

void DrawScene() {
  gRenderer.DrawTriangle(...);
}

這看起來省掉了 parameter passing。但你其實沒有消除 dependency，只是把 dependency 從 function signature 藏到 global namespace。之後測試想建立兩個 framebuffer、headless render 想不用 window、macOS backend 想替換 Win32 backend、replay 想重置 resource cache，都會被 global state 擋住。

核心修正 — global state 的問題不是「語法不乾淨」，而是它把 lifetime 和 dependency direction 變成隱性規則。隱性 lifetime 最後會反過來控制架構。

2 static storage duration：object 不是在 main 開始才存在

具有 static storage duration 的 object，其 storage 在 program 開始時就準備好，直到 program 結束才釋放。常見來源包含 namespace-scope variable、class static data member、function-local static、thread_local。

這裡要分開看兩件事： 

storage lifetime:
  memory for static object exists for the whole program/thread.

object initialization:
  object may be zero-initialized, constant-initialized, then dynamically initialized.

resource readiness:
  even if object exists, SDL/window/file/logger may not be ready yet.

3 static initialization order fiasco：跨 translation unit 的時間問題

同一個 translation unit 裡，namespace-scope object 的 dynamic initialization 大致照定義順序。但跨 translation unit 的初始化順序不能靠直覺依賴。

// renderer.cpp
Renderer gRenderer(gConfig);

// config.cpp
Config gConfig = LoadConfig("renderer.toml");

如果 gRenderer 在 gConfig dynamic initialization 完成前被建構，結果就依賴 link / object file / toolchain 細節。這就是 static initialization order fiasco。

Figure 1：static initialization order 的問題不是「compiler 笨」，而是跨 translation unit 的 dependency 不該用隱性啟動順序表達。

4 function-local static：解決順序，沒有解決 global state

function-local static 常被拿來解決初始化順序：

Logger& GetLogger() {
  static Logger logger;
  return logger;
}

它的好處是 lazy initialization：第一次呼叫時才建構，而且現代 C++ 保證 initialization 是 thread-safe。但它仍然是 global state，只是藏在 function 後面。

解決了什麼仍然沒解決什麼
跨 TU dynamic initialization order。測試之間重置困難。
用到才建構。dependency 不在 signature 裡。
避免裸 global object 暴露。destruction order / program shutdown 仍可能踩到其他 static。
適合少數 process-wide service。不適合 renderer core 狀態、framebuffer、resource cache。
判斷方式 — function-local static 可以用於 immutable table、debug-only registry、process-wide diagnostic sink。若它持有 renderer state、platform handle、resource cache，就要先問為什麼不是由 Application 或 RendererContext 擁有。

5 linkage 和 inline variable：可見性不是 ownership

global state 還會和 linkage 混在一起：

// internal linkage: each translation unit can have its own copy
static int gCounter = 0;

// C++17 inline variable: one entity across TUs, defined in header
inline Config gConfig{};

Internal linkage 可以降低 symbol 污染，但不會自動讓 state 正確。Inline variable 可以解決 header definition 的 ODR 問題，但也可能把 mutable global 帶進所有 include 它的地方。 

Linkage answers:
  Is this name visible across translation units?
  Is there one entity or one per TU?

Ownership answers:
  Who controls lifetime?
  Who is allowed to mutate?
  How do tests reset it?
  Can two renderers coexist?

6 thread_local：不是 global 的安全版本

thread_local 讓每個 thread 有自己的 instance。這對 worker-local scratch buffer 可能很有用，但它不是 global state 的萬能解法。

thread_local std::vector<Fragment> scratchFragments;

在 future tile-based renderer 或 job system 裡，thread-local scratch 的確能降低 allocation 和 lock。但要注意：

  • thread pool worker 的 lifetime 可能比一個 frame 長很多。
  • thread-local cache 可能讓 memory usage 依 thread 數膨脹。
  • deterministic replay 需要知道哪些 state 會跨 frame 留下來。
  • 如果 job 可以在不同 worker 跑，thread-local state 不能承載 job semantic state。
和 T-13 的連接 — thread_local 是 memory layout / allocation policy 的工具，不是 synchronization。它可以避免 shared mutation，但也會製造 per-thread hidden state。

7 composition root：讓 lifetime 由一個地方組裝

比較穩的做法是把 process startup / backend selection / resource owner 放在一個明確的 composition root，例如 main 或 Application constructor：

int main() {
  AppConfig config = LoadConfig();
  Framebuffer framebuffer(config.width, config.height);
  RenderDevice device(framebuffer);
  auto display = CreateDisplayBackend(config);
  Application app(config, framebuffer, device, *display);
  return app.Run();
}

這樣不是比較囉嗦，而是把 dependency 寫成可以檢查的順序： 

Config
  -> Framebuffer owns pixels
  -> RenderDevice borrows framebuffer
  -> DisplayBackend owns platform/window resources
  -> Application coordinates frame loop

Tests can build:
  Config
  -> offscreen Framebuffer
  -> RenderDevice
  -> NullDisplayBackend or no display at all

8 global cache / singleton 為什麼特別危險？

Resource cache 很容易被寫成 singleton：

TextureCache& Textures() {
  static TextureCache cache;
  return cache;
}

但 renderer 的 cache 通常不是 process-wide truth。它可能依 backend、pixel format、sampler policy、test case、asset root、hot reload mode 而不同。 

Global texture cache makes these hard:
  - run two tests with different asset roots
  - compare two renderers side-by-side
  - reset all resources between golden image cases
  - run headless renderer without SDL/Win32 backend
  - record/replay deterministic command streams

更好的第一版通常是顯式 owner：

struct RendererContext {
  Framebuffer& framebuffer;
  TextureRegistry textures;
  ShaderRegistry shaders;
  FrameStats stats;
};

9 Pixel-Renderer static/global policy

項目建議理由
constexpr math constants可放 header，prefer inline constexprimmutable、無 dynamic initialization、無 hidden lifetime。
lookup table若 immutable，可 function-local static 或 constexpr data避免重建，但不要持有 platform resource。
Logger可 process-wide，但核心 code 用 interface / sinkdebug service 可以集中，test 仍要可替換。
Framebuffer不要 globaltests、headless render、多 framebuffer、resize 都需要明確 owner。
DisplayBackend不要 globalowns window/platform resource，lifetime 應由 Application 控制。
Resource cache不要 process-wide singleton 作為第一版asset root、backend、test isolation、replay 都需要 context。
Default texture / font atlas由 UI/ResourceContext 擁有可以 lazy 建立，但 owner 必須明確。
不要用 global 隱藏架構問題 — 如果傳 parameter 很痛，通常代表缺少 context object、owner boundary 或 command/data flow，而不是 global 是正確答案。

10 設計檢查表

Before adding static/global state, ask:

1. Is it immutable after initialization?
2. Does it own OS/backend/file/resource handles?
3. Can tests reset or replace it?
4. Can two renderer instances coexist?
5. Does initialization depend on another translation unit?
6. Does destruction depend on another static object?
7. Is the dependency visible in function/class signatures?
8. Would a composition root or context object express it better?

如果 2、3、4、5、6 任一題讓你猶豫，就不要先做成 global。先讓 owner 明確，再觀察是否真的需要 process-wide service。

本章學到的三件事：
  1. Static storage duration 只保證 storage 活得久，不代表 resource readiness、initialization order、dependency direction 都正確。
  2. Function-local static 可以避開部分 initialization order 問題，但仍然是 hidden global state；它不應預設承載 renderer core、backend handle 或 resource cache。
  3. Pixel-Renderer 的 framebuffer、DisplayBackend、resource registry 應由 composition root / context 顯式擁有，讓 tests、headless render、multi-backend、deterministic replay 可以成立。
