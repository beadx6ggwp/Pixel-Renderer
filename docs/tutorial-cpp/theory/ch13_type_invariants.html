PART 5 · CHAPTER 13
Type = states + operations + invariants
type 不是 data layout，而是一組語意規範
type-semantics invariant operation

前面 12 章其實都在繞同一個核心：C++ 的 type 不只是幾個 bytes 怎麼排。當一個 object 會被 construct、copy、move、destroy、放進 container、從 function return，它真正重要的是：哪些狀態合法？哪些操作存在？操作後規則是否仍然成立？

這章要把這些收束成一個設計模型：Type = valid states + operations + invariants + cost model。

本章範圍 — 從 Buffer、math type、framebuffer、shader data 出發，建立 C++ type design 的檢查方式。你會看到為什麼「有欄位」不等於「有型別語意」，以及為什麼 C++ 會用 constructor / destructor / copy / move / delete operation 把 C convention 提升成 type rules。

1 naive model：type = memory layout

如果從 C 的角度看，type 很容易被想成 layout：

typedef struct {
  char* ptr;
  size_t size;
} Buffer;

這個定義確實描述了 bytes 怎麼排：一個 pointer，一個 size。但它沒有告訴 compiler：

  • ptr 是 owner 還是 borrower？
  • 誰負責 free？
  • copy 這個 struct 是 shallow copy 還是 deep copy？
  • ptr == nullptr 是合法空狀態，還是 bug？
  • size 和 ptr 之間有什麼 invariant？

所以 layout 是必要資訊，但它不是完整 type semantics。

2 Buffer 的真正語意

對一個 resource-owning Buffer，真正的 type 應該包含：

面向 Buffer 的例子
Valid states ptr == nullptr && size == 0，或 ptr 指向一塊此 object owns 的 buffer。
Operations construct、destroy、copy、move、assign、read/write。
Invariants 同一塊 owned memory 不能被兩個 owners 重複 delete；size 必須描述 buffer 容量。
Cost model deep copy 是 O(n)；move 只轉移 pointer，應該是 O(1) 且 noexcept。

這就是為什麼前面會一路講到 Rule of 0/3/5、move ownership、noexcept。它們不是零散語法，而是 type semantics 的不同面。

3 Operation set 本身就是設計

有些 type 可以 copy：

struct Color {
  float r, g, b, a;
};

Copy 一個 Color 很合理，因為它是 inline value。複製後兩邊都代表同一個抽象值。

但有些 type 不應該 copy：

class DisplayBackend {
public:
  DisplayBackend(const DisplayBackend&) = delete;
  DisplayBackend& operator=(const DisplayBackend&) = delete;

  DisplayBackend(DisplayBackend&&) noexcept;
  DisplayBackend& operator=(DisplayBackend&&) noexcept;
};

這不是少實作功能，而是在 type 的 operation set 裡明確說：copy 沒有合法語意；move 才代表 ownership transfer。

核心觀念

一個 operation 不存在，也是一種語意。= delete 不是語法上的失敗，而是把「這件事不合法」寫進 type system。

4 Invariant：操作後仍然是合法 object

Invariant 是 type 在任何 public operation 前後都必須維持的規則。例如：

Image img(width, height);
img.SetPixel(x, y, color);

SetPixel 不只是把某個 address 寫入 bytes。它必須維持：

  • x / y 在 bounds 內，或明確定義 out-of-bounds 行為。
  • pixel format 仍然符合 Image 宣告的格式。
  • stride / pitch 計算仍然不越界。
  • object 仍然可被後續 present、save、copy、move、destroy。

這就是為什麼 C++ class 不只是把 data 包起來。它把「哪些操作能改變 state，以及改變後仍然合法」這件事集中到 type 裡。

5 用抽象代數作為類比，但不要過度

你之前提過抽象代數的類比，這裡很有用。像 group 不是只有一堆元素，還有 operation、identity、inverse、laws。C++ type 也不能只看 object set，還要看 operations 和 laws。

數學類比

carrier set、operation、closure、laws。

C++ 校準

valid states、construct/copy/move/destroy、invariants、lifetime、exception safety、cost model。

但不要把所有 C++ type 都硬說成 group / ring / field。C++ object 有 destructor side effect、mutation、resource ownership、allocation failure、undefined behavior、ABI representation。數學類比的價值是提醒你：操作和規則比資料形狀更重要。

6 Pixel-Renderer 的 type 設計檢查

未來擴充 Pixel-Renderer 時，可以用這張 checklist 看每個 type：

Type 應該先問的問題
Vec3 / Matrix4x4 是不是 regular value？copy 是否便宜？operator 是否維持數學語意？
Framebuffer 誰 owns pixels？copy 是 deep copy 還是禁止？move 是否 O(1) / noexcept？
DisplayBackend handle 是否 unique ownership？copy 是否應 delete？moved-from backend 能否 safely destroy？
Shader 是 value object、interface、還是 stateful resource？可否 copy？dispatch 成本在哪裡？
UICommand 是否只是 POD-like draw data？是否適合放進 vector？reallocation 時 move/copy 成本如何？

7 C++ 相對於 C 的 semantic lifting

C 當然能寫 renderer。你可以用 struct、pointer、function、manual init/free 完成同樣事情。但很多規則會留在 programmer convention 裡：

// convention: call Init before use
// convention: call Destroy exactly once
// convention: do not shallow-copy this struct
// convention: after Move, source is empty

C++ 的方向是把這些 convention 往 type 裡提升：

  • constructor 建立 valid state。
  • destructor 結束 lifetime 並釋放 resource。
  • copy / move 定義 object delivery。
  • = delete 禁止沒有語意的操作。
  • RAII member 讓 Rule of 0 成為可能。

這就是 C++ 複雜度的來源之一：它既要保留 C 的底層控制，又想把 lifetime / ownership / operation semantics 變成可組合的 type-level 規則。

常見陷阱

1. 只看欄位，不看 operation — char* ptr 是 owner 還是 view，不能只從 layout 判斷。

2. 以為所有 type 都應該 copyable — unique ownership type 禁止 copy 反而是正確語意。

3. 忽略 cost model — 兩個 operation 語意都正確，也可能因為 O(n)、allocation、throw behavior 不適合某個 container/API。
本章學到的三件事

1. C++ type 不只是 data layout；它包含 valid states、operations、invariants、cost model。

2. copy、move、destroy、= delete 都是在定義 type 的合法操作集合，而不是單純補語法。

3. Pixel-Renderer 的 framebuffer、backend、shader、UI command 都應該先定義語意，再決定用 value、reference、pointer、RAII handle 或 interface。
