PART 4 · CHAPTER 12
vector reallocation 與 noexcept move
container 搬元素時，效能不是唯一問題，rollback 也很重要
vector noexcept move_if_noexcept

上一章說 emplace_back(args...) 可以讓新 element 直接在 vector slot 裡建構。但它也留下一個邊界：如果 vector capacity 不夠，舊 elements 必須被搬到新的 buffer。

這時問題不是「move 是否比較快」而已。對 standard library 來說，更根本的問題是：如果搬到一半失敗，原本的 vector 還能不能保持原狀？

本章範圍 — 建立 vector reallocation 的 exception-safety 模型。你會看到為什麼 move constructor 是否 noexcept 會影響 container 選 copy 還是 move，以及 std::move_if_noexcept 背後的語意。

1 naive model：vector 長大就 move 舊元素

很多人會先這樣想：

move 比 copy 快
vector reallocation 要搬舊 elements
所以 vector 一定會 move

這個模型漏掉一件事：move 不是純讀取。Move 可能修改 source object，讓它進入 moved-from state。如果 move constructor 執行到一半丟 exception，舊 buffer 裡的 source element 可能已經被改變。

2 reallocation 實際要做什麼

當 vector capacity 不夠，概念流程是：

allocate larger raw storage
construct old elements into new storage
construct the new element
destroy old elements
release old storage

最麻煩的是中間這步：

construct old elements into new storage

對每個舊 element，library 要決定：用 copy constructor 還是 move constructor。

3 為什麼 throwing move 會麻煩

假設 vector 正在把 A、B、C 搬到新 buffer：

move A -> new A' // A 可能變 moved-from
move B -> new B' // 如果這裡 throw?

如果 B 的 move constructor 丟 exception，library 想 rollback。可是 A 可能已經被 move-from，舊 vector 不一定還能回到原本完整狀態。

相對地，如果 copy constructor 失敗，舊 elements 通常還沒被改變。Library 可以 destroy 已經在 new storage 建好的 copies，釋放 new storage，然後保留 old vector。

核心差別 — copy 通常不改 source；move 可能改 source。當 move 可能 throw，container 要維持 strong exception guarantee 就會變困難。

4 noexcept move 是給 generic library 的承諾

對 resource-owning type，move constructor 常常只是偷 pointer / handle：

class Buffer {
public:
  Buffer(Buffer&& other) noexcept
    : ptr(other.ptr), size(other.size) {
    other.ptr = nullptr;
    other.size = 0;
  }

  Buffer& operator=(Buffer&& other) noexcept;

private:
  char* ptr = nullptr;
  size_t size = 0;
};

這個 noexcept 不是裝飾。它是在對 generic library 說：

你可以用 move relocation 我。
這條 path 不會 throw。
如果後面別的 element 失敗，你仍然可以用已知規則 cleanup。
noexcept move 的意義

它不是單純的效能標記，而是 exception-safety contract。Container 需要這個資訊來決定 reallocation 時能不能安全選 move。

5 move_if_noexcept 的決策

這個選擇可以用 std::move_if_noexcept 的 mental model 表示：

if T is nothrow move constructible:
  use move
else if T is copy constructible:
  use copy
else:
  use move anyway

最後一個 case 很重要：如果 type 不能 copy，那 library 沒有 copy fallback，只能 move。只是這時 exception guarantee 可能變弱，或標準容器的行為條件會更受限。

你可以用 type trait 檢查自己的 type：

static_assert(std::is_nothrow_move_constructible_v<Buffer>);

6 和 Rule of 5 的關係

如果一個 type owns resource，你寫 move constructor 時不只是在追求速度。你其實要同時維護：

  • ownership transfer：destination 接手 resource。
  • moved-from state：source 仍然 valid，可以安全 destruct / assign。
  • exception guarantee：如果 move 理論上不配置、不會 throw，就應該標成 noexcept。
  • copy policy：copy 要 deep copy、delete，還是交給 RAII member 自己處理。

這也回到 Rule of 0/3/5：special member functions 不是 checklist，而是同一個 ownership invariant 在 copy、move、destroy、container relocation 上的壓力。

7 Pixel-Renderer 的落點

這章對 Pixel-Renderer 的實際影響：

  • Framebuffer / ImageBuffer：如果它 owns std::vector<uint32_t>，Rule of 0 可能就足夠；標準容器會處理 move/noexcept trait。
  • 手寫 CPU buffer owner：如果自己 owns raw pointer，move constructor / assignment 應該通常是 noexcept，因為它只轉移 pointer。
  • UI command buffer：如果 command type 裡有 resource-owning member，vector grow 時會受 move/copy policy 影響。
  • Backend handles：SDL texture、Win32 handle、macOS object wrapper 多半是 move-only resource owner；要讓 container 或 wrapper 能安全管理，就要定義清楚 move/noexcept/destructor。
工程判斷 — 如果你能用 std::vector、std::unique_ptr、std::string 這些 RAII member 組出 type，就優先 Rule of 0。只有真的 owns raw resource 時，才手寫 Rule of 5，並嚴格檢查 move 是否能 noexcept。
常見陷阱

1. 以為 vector reallocation 一定 move — 如果 move 可能 throw 且 type 可 copy，library 可能選 copy 來維持 exception guarantee。

2. 手寫 move 卻忘記 noexcept — 對 generic containers 來說，這會讓你的 type 看起來不適合安全 move relocation。

3. 把 noexcept 當成保證效能 — 它真正表達的是不會 throw；效能改善是 container 因此能選 move path 的結果。
本章學到的三件事

1. vector reallocation 要把舊 elements 建到新 storage；這一步可以 copy，也可以 move。

2. 如果 move 可能 throw，container 可能選 copy，因為 copy 比較容易保留 old vector 以便 rollback。

3. noexcept move 是 resource-owning type 對 generic library 的 exception-safety contract，不只是效能提示。
