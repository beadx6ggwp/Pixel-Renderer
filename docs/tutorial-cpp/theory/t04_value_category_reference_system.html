THEORY TRACK · T-04
Value Category / Reference System
expression 有 type，也有 value category；reference 是 access policy，不是 ownership
lvalue xvalue std::forward

你之前已經理解過 move、xvalue、prvalue，但一段時間沒寫 C++ 後，最容易忘的是：T&& 不是「一定會 move」，std::move 不會自己搬東西，operator 回傳 reference 可能直接製造 dangling。這些問題未來會出現在 renderer math types、shader interface、UI callback、command buffer 裡。

這章要把 value category 和 reference system 重新完整接回來。核心不是背名詞，而是回答一個工程問題：當某段 code 拿到一個 expression，它到底是在借用 existing object、消耗 expiring object，還是建構一個新 value？

本章範圍 — 建立 expression 的雙軸模型：type 與 value category。你會分清 lvalue、xvalue、prvalue、T&、const T&、T&&、forwarding reference、reference collapsing、std::move、std::forward、temporary materialization、lifetime extension、operator return category，以及它們對 Pixel-Renderer UI/shader/math type 的後果。

1 naive model：左邊是 lvalue，右邊是 rvalue

很多人第一次學 value category 時會得到這種說法： 

lvalue: can appear on the left side of =
rvalue: can appear on the right side of =

這個模型有歷史直覺，但對現代 C++ 太粗。看這些例子：

int x = 1;
x = 2; // x 是 lvalue

get_ref() = 3; // function call 也可能是 lvalue
Vec3{} = Vec3{}; // class prvalue 也可能可被賦值，但不代表它是 lvalue

更精準的問法不是「能不能放等號左邊」，而是：

  • 這個 expression 是否指向一個可識別的 existing object？
  • 這個 expression 是否表示那個 object 可以被消耗？
  • 這個 expression 是否只是用來初始化一個 result object 的 value？
第一個修正 — value category 是 expression 的性質，不是 object 的性質。Object 有 lifetime；expression 有 type 和 value category。

2 expression 的兩個軸：type 與 value category

每個 C++ expression 都有 type，也有 value category。這兩個軸要分開。 

expression
  |
  +-- type
  |     e.g. Buffer, const Buffer, int, Vec3
  |
  +-- value category
        e.g. lvalue, xvalue, prvalue
SVG：expression 不是只有 type。value category 是第二軸，決定它在 overload resolution 中像 borrowed object、expiring object，還是 result value。

同一個 object 可以被不同 expression 指到：

Buffer b;

b; // expression type: Buffer, category: lvalue
std::move(b); // expression type: Buffer, category: xvalue

這兩個 expression 都指向同一個 object b。不同的是你給 compiler 的語意訊號：

Expression 指向誰 value category 語意訊號
b existing object b lvalue 這個 object 還有身份，不要假設我願意放棄它。
std::move(b) same object b xvalue 這個 object 可以被當作 expiring source。
Buffer{} result object prvalue 產生一個 value，用來初始化目標 object。

3 lvalue、xvalue、prvalue 的工作模型

完整標準 taxonomy 還有 glvalue/rvalue，但工程上先抓住三個分類就能解大多數問題：

Category 是否有 identity 是否可被消耗 例子 renderer 直覺
lvalue 有 通常否 fb, device, pixels[i] 借用一個還會繼續存在的 object。
xvalue 有 是 std::move(texture) 把 resource handle 交出去，source 之後只剩 valid-but-unspecified state。
prvalue 不是以 identity 為主 用來初始化 result Vec3{1,2,3}, a + b 產生新的 math value 或 command value。

這裡的 aha moment 是：std::move 不改 object lifetime，也不搬 resource。它只把 expression 轉成 xvalue，讓後續 overload resolution 可以選到 T&& overload。

std::move(b); // no move operation by itself
Buffer c = std::move(b); // move constructor may run here

4 reference binding：access policy

Reference 不是 ownership。Reference 是「這個 function 可以怎麼接觸一個 object/value」的語意限制。

Parameter 能綁到 語意 常見用途
T& non-const lvalue 借用並可能修改 existing object。 void Clear(Framebuffer& fb)
const T& lvalue、xvalue、prvalue 只讀借用；可延長 temporary lifetime 到 reference lifetime。 void DrawMesh(const Mesh& mesh)
T&& xvalue、prvalue 接收可被消耗的 source。 Texture(Texture&&)
T copy/move into parameter function 擁有自己的 local object。 small value、sink parameter、command value。

對 renderer API，這張表很實用：

  • 你只是畫一個 mesh，不拿走它：用 const Mesh& 或 view。
  • 你要讓 function 修改 framebuffer：用 Framebuffer&。
  • 你要把 texture/resource handle 交給 cache：可以用 Texture&& 或 by value sink。
  • 你要存一份 UI command：通常 by value，讓 command buffer 擁有資料。

5 named rvalue reference 是 lvalue expression

這是很多 move bug 的來源：

void Consume(Texture&& tex) {
  Store(tex); // tex expression is lvalue
  Store(std::move(tex)); // now xvalue
}

雖然 tex 的 declared type 是 Texture&&，但 expression tex 本身有名字，所以是 lvalue。這不是語法陷阱，而是保護機制：只要一個 object 有名字，compiler 不會自動假設每次使用都可以消耗它。 

declared type of tex:
  Texture&&

expression:
  tex             -> lvalue
  std::move(tex)  -> xvalue
第二個修正 — 不要用 variable declared type 直接推 expression category。Named variable expression 幾乎都是 lvalue，即使它的 type 是 T&&。

6 forwarding reference 與 reference collapsing

Template 裡的 T&& 有一個特殊情況：

template <class T>
void Push(T&& value);

當 T 是被推導出來的 template parameter，而且寫成 T&&，這通常叫 forwarding reference。它可以接 lvalue，也可以接 rvalue，因為 reference collapsing 規則會把結果折疊。

Call T deduced as Parameter type after collapsing value category preserved?
Push(x) Buffer& Buffer& lvalue 被保留。
Push(Buffer{}) Buffer Buffer&& rvalue 被保留。

Reference collapsing 規則可以簡化記成： 

&  + &  -> &
&  + && -> &
&& + &  -> &
&& + && -> &&

any lvalue reference wins

這就是 std::forward 的用途：在 template wrapper 裡保留 caller 傳進來時的 category。

template <class T>
void Push(T&& value) {
  commands.emplace_back(std::forward<T>(value));
}

如果這裡寫 std::move(value)，你會把所有東西都當成 expiring source，包括 caller 傳進來的 lvalue。這在 wrapper / factory / emplace API 裡通常是錯的。

std::move vs std::forward

std::move(x)：無條件把 expression 轉成 xvalue。
std::forward<T>(x)：根據 deduced T 還原 caller 的 value category。

7 temporary materialization 與 lifetime extension

C++17 之後，prvalue 的模型更像是「用來初始化 result object 的 value」，不要再簡單想成每個 prvalue 都先生成一個 temporary 再 move。看這個：

Vec3 MakeVec() {
  return Vec3{1, 2, 3};
}

Vec3 v = MakeVec();

這不應該被想成「先做一個 temporary Vec3，再 move 到 v」。更好的模型是：MakeVec() 的 prvalue 直接初始化 v 的 storage。

但當 prvalue 需要被當成 glvalue 使用時，會發生 temporary materialization。最常見是綁到 reference：

const Vec3& ref = Vec3{1, 2, 3};
// temporary lifetime is extended to ref's lifetime

這個 lifetime extension 很有用，但不要過度依賴。尤其不要回傳 reference 指向 local temporary：

const Vec3& Bad() {
  return Vec3{1, 2, 3}; // dangling
}
不要把 lifetime extension 當成 ownership 策略 — 它是局部語言規則，不是跨 frame、跨 callback、跨 command buffer 的資料生命週期管理方式。

8 operator return category：數學型別最容易踩坑

Operator 不是只讓語法漂亮，它也決定 expression category。對 math value type，最常見正確方向是：non-mutating operator 回傳 by value。

inline Vec3 operator+(const Vec3& a, const Vec3& b) {
  return Vec3{a.x + b.x, a.y + b.y, a.z + b.z};
}

這讓 a + b 是一個新的 value，適合拿來初始化下一個 object：

Vec3 c = a + b; // prvalue initializes c

危險的是為了「避免 copy」亂回傳 reference：

const Vec3& operator+(const Vec3& a, const Vec3& b) {
  Vec3 result{a.x + b.x, a.y + b.y, a.z + b.z};
  return result; // dangling reference
}

這種 bug 很適合拿 T-03 的 object lifetime 來看：result 的 lifetime 在 function return 時結束，回傳的 reference 只剩一個指向已死亡 object 的 access path。

math type 預設 — 小型 value type，例如 Vec2、Vec3、Color、Rect，non-mutating operator 通常 by value 回傳。不要為了省 imaginary copy 製造 dangling reference；copy elision 和 register passing 通常已經很有效。

9 Pixel-Renderer 後果：UI command、shader、backend view

Value category 不是語言冷知識。它會直接影響你之後幾個設計。

UI command buffer

Immediate-mode UI 常會每 frame 產生 draw commands。Command buffer 應該擁有 command data，不應保存指向 temporary widget state 的 reference。

struct DrawTextCmd {
  Rect bounds;
  std::string text; // owns text if command outlives caller expression
};

commands.emplace_back(bounds, std::string(label));

如果改成保存 const char* 或 std::string_view，就要非常清楚 referenced storage 的 lifetime 是否跨過 render phase。

Shader interface

如果 shader object 是 expensive resource 或持有 state，你要決定 API 是借用、複製，還是消耗：

API 語意 適合情況
SetShader(const Shader&) 只借用，不取得 ownership。 shader lifetime 由 caller 或 registry 管。
SetShader(Shader) by value sink，function 取得自己的 copy/move 版本。 小型 value shader 或 type-erased wrapper。
AddShader(Shader&&) 消耗 caller 的 shader resource。 resource cache 接手 ownership。

Framebuffer/backend view

Framebuffer&、std::span<uint32_t>、raw pointer + pitch 都是 borrowed view，不是 ownership。不要把 borrowed view 存到比 owner 更長壽的地方。 

good:
  RenderDevice renders using Framebuffer& during a frame

risky:
  RenderDevice stores pixels.data()
  Framebuffer resizes
  stored pointer now dangles

10 verification：怎麼檢查 category / dangling 問題

這類 bug 不一定會馬上 crash，所以要用幾個工程手段防守。

開警告

-Wall -Wextra -Wreturn-local-addr

GCC/Clang 可以抓到一些回傳 local reference/address 的問題。後面 I-06 Make Workflow 應該把 debug warnings 明確整理進 makefile。

在 API 名字上標出 ownership

名字 暗示
SetX(const X&) 借用或觀察，不一定取得 ownership。
TakeX(X&&) 明確消耗 caller 的 object。
CreateX() 通常 return by value，讓 caller 擁有 result。

用 debugger 看 move 後 state

break Texture::Texture(Texture&&)
run
backtrace
print other.handle
next
print other.handle

對 move-only resource handle，move constructor 應該把 source 放回 destructible state，例如 handle 變成 null / invalid sentinel，避免 double release。

本章學到的三件事：
  1. Expression 有 type 和 value category；value category 是 expression 的性質，不是 object 本身的性質。
  2. std::move 是無條件 xvalue cast，std::forward 是在 forwarding reference 裡保留 caller 的 value category。
  3. Renderer API 要用 reference/value category 表達 ownership 和 lifetime：UI command 應擁有跨 frame 資料，math operator 預設 by value，backend/framebuffer view 不能比 owner 活得更久。
