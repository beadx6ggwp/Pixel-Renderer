THEORY TRACK · T-10
Error Handling / Contracts / Invariants
錯誤處理不是選語法，而是先分類「誰違反了哪個 contract」
contract invariant assert noexcept

T-09 講到 std::vector、std::span、iterator invalidation 時，其實已經碰到錯誤處理：如果 FramebufferView 的 width、height、pitch 不一致，SetPixel 應該怎麼辦？如果 SDL window 建立失敗，是 bug、使用者環境問題，還是可以恢復的 runtime failure？

I-13 又把 architecture 切成 rendering frontend、software renderer、owned framebuffer、display backend。只要開始切 interface，就一定要回答：每個 interface 的合法輸入是什麼？失敗時誰負責？哪些錯誤代表 programmer bug，哪些錯誤是正常世界會發生的情況？

本章範圍 — 從 contract 出發整理 C++ error handling：valid state、invariant、precondition、postcondition、assert、return value、std::optional、expected-like result、exception、noexcept、constructor vs Init。目標不是規定 Pixel-Renderer 一定要用 exception 或不用 exception，而是建立一張判斷表：不同錯誤應該在哪一層被攔下，並如何寫進 type/interface。

1 naive model：錯誤處理就是選一種風格

常見討論會很快變成： 

C style:
  return false / error code

C++ style:
  throw exception

debug style:
  assert

modern style:
  optional / expected

這樣比較太早了。因為它先選工具，卻還沒問錯誤本身是哪一類。x 超出 framebuffer bounds、檔案不存在、SDL 沒有可用 video driver、depth buffer 尺寸不一致，這些不是同一種問題。

現象 本質 適合工具
SetPixel(-1, y) caller 違反 precondition，或 API 本來應該定義 clipping。 assert / 分成 checked 與 unchecked API。
SDL window 建立失敗 外部環境 runtime failure。 Result / error message / exception。
Framebuffer width 與 pixels size 不一致 object invariant broken。 constructor/type design 防止；debug assert 檢查。
查不到 optional resource 合法 absence。 std::optional。
destructor 失敗 cleanup path 不能合理恢復。 destructor 不 throw；log 或 best-effort cleanup。
第一個修正 — 不要先問「用不用 exception」。先問這是 programmer bug、可預期的 runtime failure、合法 absence，還是 invariant 破掉。

2 contract vocabulary：precondition、postcondition、invariant

把 function 看成 contract，它至少有三層：

SVG：error handling 先從 contract 邊界開始。工具選擇要跟違反的是哪一層有關。
詞 意思 Framebuffer 例子
Valid state object 目前是一個可安全使用或 destruct 的狀態。 空 framebuffer，或尺寸與 storage 一致的 framebuffer。
Invariant public operation 前後都應成立的規則。 pixels.size() >= pitch_pixels * height。
Precondition caller 呼叫前必須保證的條件。 0 <= x < width，如果 API 是 unchecked SetPixel。
Postcondition function 成功回來後保證的條件。 指定 pixel 已被寫成 color；其他 pixels 不變。

Contract 的價值是把責任分清楚。不是每個 function 都要檢查所有事情；但每個 function 都應該清楚「我期待 caller 保證什麼，我自己保證什麼」。

3 assert：抓 programmer bug，不是處理世界失敗

assert 適合檢查「如果發生，代表程式寫錯」的條件。它常在 debug build 啟用，release build 可能被移除，所以不能拿來處理正常 runtime failure。

void SetPixelUnchecked(FramebufferView fb, int x, int y, uint32_t color) {
  assert(x >= 0 && x < fb.width);
  assert(y >= 0 && y < fb.height);
  fb.color[y * fb.pitch_pixels + x] = color;
}

這裡的 function 名字故意寫 Unchecked。它的 contract 是：caller 必須先 clipping 或 bounds check。這種 API 適合 rasterizer hot loop，因為每 pixel 都做完整錯誤處理會太重。

但如果 API 面向外層 draw call，就可以提供 checked 版本：

void SetPixelClipped(FramebufferView fb, int x, int y, uint32_t color) {
  if (x < 0 || x >= fb.width || y < 0 || y >= fb.height) {
    return;
  }
  fb.color[y * fb.pitch_pixels + x] = color;
}
assert 的邊界 — assert 可以抓 invariant 和 precondition violation，但不應拿來處理「檔案不存在」、「SDL init 失敗」、「使用者關閉 window」這類正常 runtime event。

4 optional：合法沒有值，但不說明原因

std::optional<T> 表達「可能有 T，也可能沒有 T」。它適合 absence 本身就是合法狀態，而且 caller 不需要詳細失敗原因。

std::optional<TextureId> FindTexture(std::string_view name);

if (auto texture = FindTexture("font_atlas")) {
  DrawText(*texture);
}

但 optional 不適合需要錯誤原因的地方。例如 SDL backend 初始化失敗時，你通常想知道是 video subsystem、window、renderer、texture 哪一步失敗。 

optional:
  no value

expected-like Result:
  no value because SDL_CreateRenderer failed: "Metal renderer unavailable"

5 expected-like Result：可恢復失敗要帶原因

C++23 有 std::expected。如果專案還沒用 C++23，也可以先用自己的簡化 Result 或回傳 error enum + message。重點不是追最新標準，而是把 failure reason 寫進 interface。

struct Error {
  std::string message;
};

template <class T>
struct Result {
  std::optional<T> value;
  std::optional<Error> error;
};

Result<std::unique_ptr<DisplayBackend>> CreateDisplayBackend(BackendKind kind);

上面這個 Result 不是最終設計，因為 value/error 同時存在或同時不存在的 invariant 還沒被 type 強制保護。真正實作時可以用更嚴謹的 variant、expected polyfill、或 C++23 std::expected。

和 T-11 的關係 — 更嚴謹的 Result<T, E> 常會用 std::variant 或 type erasure 技巧。T-11 會把 variant/type erasure/runtime polymorphism 的取捨補上。

6 exception：跨多層傳播失敗，但要靠 RAII 保護

Exception 的強項是：錯誤可以跨多層 stack 傳播，而不用每一層都手動檢查 return code。它的代價是 control flow 不再只從 return statement 看得出來，且專案需要一致的 exception policy。 

LoadScene()
  LoadMesh()
    LoadFile()
      throws FileError

stack unwinds:
  local RAII objects destruct automatically

這就是為什麼 T-06 的 RAII 是 exception safety 的前提。如果每個 resource 都靠 destructor 清理，exception 傳播時 cleanup 仍然會發生。如果你靠手動 Destroy()，exception 可能跳過 cleanup path。

Pixel-Renderer 第一版可以選擇不用 exception，改用 explicit Result。這是合理的。但即使不用 exception，你仍然要懂 exception safety，因為 standard library、allocation、constructor、container operation 都可能牽涉失敗語意。

工程判斷 — 不要在 hot loop 裡把 exception 當分支機制。Rasterizer per-pixel / per-fragment 路徑應該依賴 precondition、clipping、assert、plain control flow。Exception 或 Result 更適合 resource loading、backend init、file IO、top-level application setup。

7 noexcept：不是優化開關，是不會 throw 的承諾

noexcept 的意思是：這個 function 承諾不讓 exception 離開。如果它真的 throw，程式會呼叫 std::terminate。所以 noexcept 不是單純效能提示，而是 contract。

位置 是否常見 noexcept 原因
destructor 通常應該不 throw destructor 可能在 stack unwinding 中執行。
move constructor for handle 通常應該 noexcept 只搬 pointer/handle；也幫助 vector reallocation。
hot-path math 可考慮 noexcept 如果只做 arithmetic 且不配置，語意上可承諾。
function 會 allocate 不要亂標 allocation 可能失敗；標錯會 terminate。
class SdlTexture {
public:
  ~SdlTexture() noexcept;
  SdlTexture(SdlTexture&&) noexcept;
  SdlTexture& operator=(SdlTexture&&) noexcept;

  SdlTexture(const SdlTexture&) = delete;
  SdlTexture& operator=(const SdlTexture&) = delete;
};

8 constructor vs Init：誰建立 invariant？

理想上，constructor 應該讓 object 一出生就進入 valid state。但 platform backend 常會遇到外部失敗：window 建立失敗、SDL renderer 不可用、DPI / display / permission 問題。這時候有兩種設計：

設計 優點 代價
constructor throws 成功建構後 object 一定 ready。 專案要接受 exception policy。
default construct + Init() 回傳 Result 不需要 exception；錯誤原因顯式。 type 必須定義 uninitialized state，所有 public operation 要避免未 init 使用。
factory function 回傳 Result<Backend> 成功才交出 ready object。 需要 movable backend 或 heap allocation。

對 Pixel-Renderer 的第一版，factory + unique_ptr 通常很清楚：

Result<std::unique_ptr<DisplayBackend>>
CreateSdlDisplayBackend(int width, int height, std::string_view title);

成功時 caller 拿到 initialized backend；失敗時拿到 error。這比先建立一個半初始化 object，再到處檢查 IsInitialized() 更不容易污染整個 codebase。

9 Pixel-Renderer contract map

把這章收束到目前架構，可以得到這張 contract map：

Interface Precondition / Invariant Failure policy
Framebuffer(width, height) 尺寸非負；storage size 與 pitch/height 一致。 invalid dimension 可 assert 或回傳 factory Result；allocation failure 依專案 policy。
FramebufferView borrowed storage 活著；span 足夠覆蓋 pitch/height。 debug assert；不要長期保存 view。
SetPixelUnchecked x/y already clipped。 debug assert；release 不處理。
DrawLine 可以接受畫面外座標，或明確要求 caller clip。 如果接受畫面外座標，內部必須 clip；不要靠 UB。
Present(framebuffer) framebuffer valid；pixel format match backend expectation。 通常回傳 bool/Result 或 debug assert，取決於 present failure 是否可恢復。
CreateDisplayBackend backend kind 支援、平台 library 可用。 Result with error message。
LoadTexture path/string lifetime valid。 Result，因為檔案不存在是正常 runtime failure。

10 一個實用決策樹

寫 interface 時可以用這個順序判斷： 

Is this condition a programmer bug?
  yes -> assert / static_assert / stronger type
  no  -> continue

Is absence a normal valid result without extra reason?
  yes -> optional
  no  -> continue

Is failure expected and caller can react?
  yes -> Result / expected-like / error code with message
  no  -> continue

Is failure exceptional and should cross layers?
  yes -> exception, if project policy allows it
  no  -> log + fail fast / terminate / top-level error

再加一條 C++ 專用規則： 

Can this function truly promise not to throw?
  yes -> noexcept may be part of its contract
  no  -> do not mark noexcept just for style

11 Summary

本章學到的三件事：
  1. Error handling 不是先選 assert、return code、exception 或 optional，而是先分類：programmer bug、runtime failure、合法 absence、invariant violation。
  2. Interface 應明確寫出 precondition、postcondition、invariant；assert 抓 contract violation，optional 表達合法沒有值，expected-like Result 表達可恢復且需要原因的失敗。
  3. noexcept 是不會 throw 的語意承諾，不是裝飾；Pixel-Renderer 應讓 hot path 保持簡單，讓 backend init / file IO / resource loading 用明確 failure channel。
