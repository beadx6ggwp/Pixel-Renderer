Low-level Performance / Memory Layout
performance 不是最後加速，而是資料如何被 CPU 讀寫的結果
cachelayoutAoS/SoASIMD

到目前為止，我們已經把 ownership、container、backend boundary 講清楚。可是 software renderer 最後會撞上一個非常實際的問題：即使演算法是對的，為什麼畫大一點的 triangle 就慢？為什麼同樣是 O(n)，有些 loop 明顯比較快？

答案通常不在 C++ 語法，而在 memory layout。CPU 不只是執行指令，它也在不斷從 memory 把 cache line 搬進來、猜 branch、等待 dependency、嘗試 vectorize。

本章範圍 — 建立 Pixel-Renderer 需要的低階性能模型：cache locality、row-major framebuffer、pitch/alignment、padding、false sharing、branch prediction、AoS vs SoA、SIMD intro。這章不做微優化清單，而是讓你能判斷資料結構是否適合 rasterizer hot loop。

1 naive model：演算法 Big-O 決定速度

Big-O 很重要，但 renderer hot loop 常被常數、memory bandwidth、cache locality 主導。 

Two loops both write N pixels:

good:
  write consecutive pixels
  predictable branch
  simple address pattern

bad:
  jump around memory
  unpredictable branch
  many cache misses

Both are O(N), but not equally fast.

Big-O 描述的是 input 很大時成長率的分類，但 renderer inner loop 常常在固定解析度、固定 format、固定 hardware 上跑。這時候你更常被以下東西限制：

  • 每個 pixel 要讀寫多少 bytes。
  • 資料是否連續，能不能被 prefetch。
  • branch 是否可預測。
  • compiler 是否能看懂 alias relation 並做 vectorization。
  • inner loop 是否有 allocation、virtual call、除法或難以 inline 的 callback。
性能的第一性問題 — 不要先問「我要不要 SIMD」。先問 CPU 這一幀要搬多少資料、按什麼順序搬、每個 cache line 被用掉多少、inner loop 是否規律到 compiler 和 CPU 都能利用。

2 cache line：CPU 通常不是拿一個 byte

CPU cache 一次載入的是一段連續 memory，常見 mental model 是 64 bytes 一條 cache line。對 uint32_t framebuffer 來說，一條 cache line 大約可以放 16 pixels。 

uint32_t pixels:

cache line 0:
  p0 p1 p2 ... p15

scanline write:
  p0 -> p1 -> p2 -> ...  uses loaded line well

random write:
  p0 -> p80000 -> p17 -> p5000  wastes cache lines

這就是為什麼 framebuffer 用 contiguous std::vector<uint32_t> 是好的第一版選擇。

SVG：CPU 寫 framebuffer 時，真正昂貴的常是 memory hierarchy 和 cache line 使用率，不只是 C++ 一行 store。

把 cache line 換成 pixel 數量思考

如果一條 cache line 是 64 bytes，uint32_t pixel 是 4 bytes，那一條 line 可以放 16 個 pixels。掃描式寫入時： 

write x=0  -> fetch cache line containing pixels 0..15
write x=1  -> same cache line
...
write x=15 -> same cache line
write x=16 -> next cache line

相反地，如果你每次跳到完全不同 row 或不同 tile，CPU 可能一直搬新的 cache line，卻只用其中一個 pixel。這會讓 memory bandwidth 變成瓶頸。

3 pitch / alignment：row stride 不是永遠 width * bytes

Framebuffer 的 address 計算應該用 pitch，而不是到處寫死 width * 4。

uint32_t& At(FramebufferView fb, int x, int y) {
  return fb.color[y * fb.pitch_pixels + x];
}

第一版 pitch 可以等於 width；但 interface 保留 pitch，未來接 SDL lock texture、aligned rows、image file buffer、GPU staging memory 時不會整個 API 破掉。

這裡也連回 I-11 的 SDL：SDL_LockTexture 借出的 pointer 會搭配 pitch，因為 driver/backend 不保證每一 row 剛好是 width * 4 bytes。若 renderer core 從第一天就用 pitch-aware view，之後切換 memory owner 時比較不會破壞整個 rasterizer。 

bad assumption:
  address = base + (y * width + x) * 4

pitch-aware:
  address = base + y * pitch_bytes + x * bytes_per_pixel

first version:
  pitch_bytes == width * 4

future version:
  pitch_bytes may include padding / alignment / backend constraints

4 AoS vs SoA

Array of Structs 把一個 vertex 的所有欄位放一起；Struct of Arrays 把同欄位集中。 

AoS:
  Vertex { pos, normal, uv, color }
  [V0][V1][V2]...

SoA:
  pos[]    = p0 p1 p2...
  normal[] = n0 n1 n2...
  uv[]     = u0 u1 u2...
Layout適合情境代價
AoS每次處理完整 vertex，API 直覺，第一版簡單。只需要 position 時也會把其他欄位拉進 cache。
SoA大量同欄位運算、SIMD、transform pipeline。API 較複雜，資料聚合成本較高。

判斷 AoS / SoA 不能靠口號。你要看 access pattern：

Renderer stage常讀欄位可能 layout
transform verticesposition，可能還有 normal。SoA 或分離 position stream 可能較好。
rasterize triangle三個 vertex 的 screen position、depth、varying。AoS/simple struct 對第一版較清楚。
fragment shadeinterpolated color/uv/depth。內層 loop 應避免追很多 pointer。
UI command replaycommand type、rect、color、clip。variant 或 command structs，避免 per-command heap allocation。
結構化推理 — AoS / SoA 不是誰比較先進，而是你要讓 hot loop 讀到剛好需要的資料。資料越貼近 access pattern，cache locality 和 vectorization 機會越好。

5 branch prediction：熱路徑不只看 if 數量

Branch 如果很容易預測，成本可能很低；如果結果亂跳，CPU pipeline 會常常猜錯。 

predictable:
  most pixels are inside triangle
  most depth tests pass in batches

unpredictable:
  alternating pass/fail pattern
  random alpha discard
  per-pixel virtual dispatch

Rasterizer 優化常會把 clipping、bounding box、edge tests 前移，讓內層 loop 更規律。

把 branch 從內層移到外層

目前 SetPixel 裡有 bounds check，這對早期 correctness 很重要。但如果 triangle rasterizer 已經在外層把 bounding box clip 到 screen，內層每個 pixel 再檢查一次 bounds 就可能變成重複成本。 

safe early version:
  for each candidate pixel:
    SetPixel(x, y) checks bounds

optimized later version:
  clipped bounding box guarantees x/y in range
  inner loop uses unchecked write or row pointer

contract changes:
  checked API for public/safe calls
  unchecked API for proven hot loop

這就是 T-10 contract 的性能版本：你不是任意刪掉檢查，而是把 precondition 搬到更外層，讓 hot loop 可以依賴已驗證的 invariant。

6 aliasing：compiler 不知道兩個 pointer 是否重疊

C++ optimizer 的很多能力來自它能證明某些 memory 不會互相影響。如果你把大量 raw pointer 傳來傳去，compiler 可能不敢重排或 vectorize，因為它不知道寫 color buffer 會不會同時改到 depth buffer。 

possible alias problem:
  uint32_t* color;
  float* depth;

compiler must consider:
  could these point into overlapping storage?
  will writing color affect later depth load?

對 Pixel-Renderer 的第一版，不需要急著寫平台特定的 restrict extension；更重要的是先讓資料 ownership 清楚：

  • Framebuffer owns color buffer。
  • DepthBuffer owns depth buffer。
  • RenderTargetView 明確借用兩段不同 storage。
  • hot loop 內部盡量用 row pointer / local reference，減少反覆查 object。

當資料邊界清楚，之後才有空間談更進階的 alias annotation、SIMD intrinsic、tile memory layout。

7 SIMD：不是先寫 intrinsic，而是先整理資料

SIMD 的意思是一次對多個 lane 做相同操作。它要求資料和控制流夠規律。對 software renderer 來說，常見方向是一次處理 4/8 個 pixels 的 edge function、depth test 或 color blend。

不要過早 SIMD — 如果 scalar code 的 ownership、pitch、clipping、invariant 還不穩，先寫 intrinsic 只會把錯誤放大。先讓資料 layout 和 hot loop 清楚，再做 SIMD。

8 measurement：沒有基準就沒有優化

性能討論最容易變成猜測。Pixel-Renderer 後面應該建立小型 benchmark scene，而不是只看主程式 FPS： 

benchmark cases:
  clear 800x600
  draw 10k short lines
  draw 1k small triangles
  draw 100 large triangles
  alpha blend UI rectangles
  depth test dense scene

record:
  build mode
  compiler
  resolution
  scene seed
  average / min / max frame time

這裡的核心不是追求漂亮數字，而是讓改動可比較。你改了 data layout、edge function、branch position、command buffer allocation 之後，必須知道它讓哪個 workload 變快或變慢。

測量陷阱 — Debug build、視窗 present、FPS title update、OS compositor、printf 都可能污染測量。renderer hot loop benchmark 應盡量 headless，直接測 CPU framebuffer 產生時間。

9 Pixel-Renderer performance checklist

  • Framebuffer 是否連續掃描，而不是隨機跳寫？
  • Hot loop 是否避免 per-pixel virtual call / allocation / string operation？
  • Bounds check 是否在外層 clipping 解決，而不是每 pixel 重複做？
  • Vertex / fragment data layout 是否符合實際 access pattern？
  • Command buffer 是否保留 capacity，避免每 frame allocation？
  • 測試優化前後是否用同一個 scene、解析度、build mode？

10 Summary

本章學到的三件事：
  1. Renderer hot loop 的速度常由 memory layout、cache locality、branch predictability、allocation pattern 決定，不只由 Big-O 決定。
  2. Framebuffer、vertex buffer、command buffer 應優先保持 contiguous、pitch-aware、lifetime 清楚，讓後續 SIMD 或 parallelization 有基礎。
  3. AoS/SoA、branch hoisting、SIMD 都是 access pattern 問題；先看資料如何被讀寫，再談技巧。
