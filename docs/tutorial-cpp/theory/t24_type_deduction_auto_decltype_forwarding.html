THEORY TRACK · T-24
Type Deduction / auto / decltype / Forwarding
auto 不是只是少打型別；它會決定 reference、const、copy、borrow、dangling 是否被保留下來
auto decltype template-deduction forwarding

T-04 建立 value category，T-22 建立 const/cv capability，T-23 說 call site 會經過 overload 與 conversion 規則。現在還缺一個很容易被低估的問題：當你寫 auto、decltype(auto)、auto&&、range-for 的 auto，compiler 會保留哪些語意？又會丟掉哪些語意？

這會直接影響 Pixel-Renderer 的實作。你遍歷 command buffer 時，auto cmd 可能複製 command；auto view = framebuffer.Pixels() 可能把 view/value 的語意藏起來；generic shader helper 用 decltype(auto) 回傳 reference，可能把 local object 的 dangling reference 暴露出去。

本章範圍 — 建立 type deduction 的工作模型：auto、auto&、const auto&、auto&&、decltype(expr)、decltype(auto)、template argument deduction、forwarding reference、range-for，以及 Pixel-Renderer 裡 command/view/shader helper 的 deduction policy。

1 naive model：auto 會幫我寫出正確型別

很多時候 auto 很舒服：

auto width = framebuffer.Width();
auto color = SampleTexture(uv);
auto it = commands.begin();

問題是：auto 不是「把右邊完整型別照抄到左邊」。它套用一套 deduction 規則，通常會丟掉 top-level const、丟掉 reference，並把 expression 的某些語意轉成一個新的 object。

const Command& ref = commands[0];
auto a = ref; // Command copy, not const Command&
auto& b = ref; // const Command&
const auto& c = ref; // const Command&
核心修正 — auto 不是語意保真器。它是 deduction 規則的入口。你必須在 call site 明確選擇：我要 copy、一個 mutable borrow、一個 const borrow，還是一個 forwarding reference？

2 deduction 不是一套規則，而是幾套相鄰規則

C++ 最容易卡住的地方是：auto、template deduction、decltype 看起來都在「推導型別」，但它們不是完全同一套規則。

Figure 1：type deduction 的三個常見入口。不要把 auto、decltype、forwarding reference 當成同一個工具。

3 auto 的基本行為：先像 template T 一樣推

最常見的 auto x = expr; 很像 function template 的 by-value parameter deduction：

template<class T>
void f(T x);

auto x = expr; // roughly: deduce T from f(expr)

所以 auto by value 會建立一個新 object。Reference 不會自動保留；top-level const 通常也不會保留。

寫法語意Renderer 直覺
auto x = expr;copy/move 成新 object；丟掉 reference 與 top-level const我要自己的值，允許複製 command 或 descriptor
auto& x = expr;mutable/const borrow；能不能改取決於 expr我要操作 container 裡原本那個元素
const auto& x = expr;read-only borrow；可綁 temporary 並延長 lifetime 到 reference scope我要讀 command，不想 copy，也不想改
auto* p = expr;pointer deduction；pointer 本身 copy，指向原物件C API / optional borrow boundary
auto&& x = expr;在 deduction context 是 forwarding referencegeneric wrapper 或 range-for 中保留 value category
注意 — auto by value 不是錯，它只是很明確地表示「我要一份新值」。錯的是你原本需要 borrow，卻因為懶得打 & 而不小心 copy。

4 range-for：一個字元決定 copy 還是 borrow

Command buffer 是最好的具體例子。假設每個 command 可能帶 vertex range、pipeline id、texture id、clip rect：

for (auto cmd : commands) {
  cmd.Execute(device); // copies every command
}

for (const auto& cmd : commands) {
  cmd.Execute(device); // reads original commands
}

for (auto& cmd : commands) {
  cmd.debugVisited = true; // mutates original commands
}

這不是 performance 小細節而已。它是 ownership/borrow 語意： 

auto cmd        -> value copy, local mutation does not affect buffer
const auto& cmd -> read-only borrow, no copy, cannot mutate command
auto& cmd       -> mutable borrow, no copy, mutates original command
auto&& cmd      -> generic/range adapter use, preserves reference category

5 decltype：觀察 expression，而不是建立新 object

decltype 的目的不是「幫我省略型別」，而是「根據 expression 形式取得型別」。這和 auto 的精神不同。

Command cmd;
Command& ref = cmd;

decltype(cmd) a = cmd; // Command
decltype(ref) b = cmd; // Command&
decltype((cmd)) c = cmd; // Command&, because (cmd) is lvalue expression

最反直覺的是 decltype(x) 和 decltype((x)) 不同。前者如果是 unparenthesized id-expression，取得 declared type；後者是一般 expression，會看 value category。

Expressionvalue categorydecltype(expr)
x as id-expression看起來是 lvalue，但規則特判declared type

連回 T-04 — decltype 是 value category 章節的實用出口。它把 lvalue 變成 T&、xvalue 變成 T&&、prvalue 變成 T。

6 decltype(auto)：強大，因為它可能保留 reference

auto return 會做一般 auto deduction；decltype(auto) return 會用 decltype 規則。這使它很適合寫 forwarding wrapper，但也容易把 reference 原封不動傳出去。

auto GetCommandCopy(CommandBuffer& b) {
  return b[0]; // returns Command by value
}

decltype(auto) GetCommandRef(CommandBuffer& b) {
  return (b[0]); // returns Command&
}

危險案例：

decltype(auto) Bad() {
  Command local{};
  return (local); // returns Command& dangling
}
不要把 decltype(auto) 當成「更精準的 auto」隨便用。 它的精準代表它會保留 reference。只有在你真的想做 forwarding wrapper、view access、proxy access，而且能證明 lifetime 安全時才使用。

7 forwarding reference：保留 caller 的意圖

在 template deduction context 裡，T&& 如果 T 需要被推導，就是 forwarding reference。它不是單純的 rvalue reference。

template<class F, class... Args>
decltype(auto) InvokeShaderHelper(F&& f, Args&&... args) {
  return std::forward<F>(f)(std::forward<Args>(args)...);
}

這段在語意上是：「我不決定 caller 傳進來的東西應該被 copy 還是 move；我只把 caller 原本的 lvalue/rvalue intent 往下一層傳。」如果你少了 std::forward，所有 named parameter 都是 lvalue，rvalue intent 會被吃掉。 

Caller passes lvalue command  -> wrapper receives as lvalue intent -> forward as lvalue
Caller passes temporary       -> wrapper receives as rvalue intent -> forward as rvalue

Inside wrapper:
  args has a name, so args itself is lvalue expression.
  std::forward(args) restores the original intent recorded in Args.
判斷方式 — 如果 function 不是 generic wrapper，而是 domain API，例如 DisplayBackend::Present、RenderDevice::DrawTriangle、CommandBuffer::Push，通常不要先用 forwarding reference。先用明確的 value/reference/const-reference 表達 ownership。

8 auto&& 在 range-for 和 generic code 的位置

auto&& 常讓人誤解成「我要 rvalue」。在 deduction context，它更像「不預設 value category」。這在 generic range adapter 很有用：

for (auto&& cmd : MakeCommandRange()) {
  Visit(std::forward<decltype(cmd)>(cmd));
}

但在一般 renderer code 裡，const auto& 或 auto& 通常更清楚。你要先問的是：這段程式是不是 generic library layer？如果不是，清楚表達 read-only borrow / mutable borrow 通常比使用 auto&& 更好。

9 braced initializer 和 auto 的特殊坑

auto 遇到 braces 時還有一個容易讓人誤會的地方：

auto a = 1; // int
auto b = {1}; // std::initializer_list<int>
auto c{1}; // int in modern C++ rules
// auto d = {1, 2.0}; // error, cannot deduce one element type

對 renderer 來說，這常出現在 small math type 或 descriptor 初始化附近。若你想建立 Vec3、Color、FramebufferSize，不要讓 braces + auto 把語意藏成 initializer_list。Domain type 應明確寫出來：

FramebufferSize size{width, height};
RGBA8 clearColor{0, 0, 0, 255};

10 Pixel-Renderer deduction policy

Use auto by value when:
  - the type is obvious from the right side
  - a copy/value is intended
  - the object is small and semantic value-like
  Examples: iterator, enum result, local scalar, small descriptor copy

Use const auto& when:
  - iterating command buffers / draw lists read-only
  - reading large descriptors / pipeline states
  - avoiding copy while preserving read-only contract

Use auto& when:
  - mutating commands, vertices, pixels, cached state in-place
  - the mutation is part of the function contract

Use decltype(auto) when:
  - writing a forwarding accessor/wrapper
  - preserving reference is explicitly required and lifetime is proven

Avoid generic auto&& in domain APIs:
  - DisplayBackend::Present should say const Framebuffer&
  - RenderDevice should say what it owns/borrows
  - PushCommand should decide value vs move vs borrow deliberately

可以把這章和 T-16 API Design Conventions 接起來看：function signature 表達的是 ownership / borrow / mutation capability；deduction 則是 call site 和 generic layer 會不會把這些語意保留下來。

本章學到的三件事：
  1. auto by value 會推導出新 object，通常丟掉 reference 與 top-level const；如果你要 borrow，要明確寫 auto& 或 const auto&。
  2. decltype(expr) 觀察 expression category，decltype(auto) 可能保留 reference；它強大但也能製造 dangling reference。
  3. Forwarding reference 是 generic wrapper 的工具，不是 domain API 的預設寫法；Pixel-Renderer 的核心 interface 應先用明確 signature 表達 ownership、borrow、mutation。
