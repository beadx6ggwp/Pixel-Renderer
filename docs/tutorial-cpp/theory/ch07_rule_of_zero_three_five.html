PART 2 · CHAPTER 07
Rule of 0/3/5
special member functions and ownership invariant
special-member RAII invariant

上一章看到：如果 Buffer owns heap memory，預設 copy 可能把 pointer value 複製出去，最後 double delete。現在問題變得更尖銳：如果一個 type 自己管理 resource，到底哪些函式必須一起設計？

Rule of 0/3/5 不是背誦清單。它是 ownership 壓力的結果：只要一個 type 有「我擁有這個 resource，死掉時我要釋放它」這種語意，copy、move、destroy 就不能各自獨立。

本章範圍 — 從 resource-owning Buffer 推出 Rule of 3、Rule of 5、Rule of 0。重點是理解 special member functions 如何共同維護同一個 ownership invariant，而不是背規則名稱。

1 Destructor 不是孤立的

先看這個很常見的 RAII wrapper：

class Buffer {
public:
  explicit Buffer(size_t n)
    : ptr(new char[n]), size(n) {}

  ~Buffer() {
    delete[] ptr;
  }

private:
  char* ptr = nullptr;
  size_t size = 0;
};

你寫下 destructor 的瞬間，其實宣告了一個 invariant：

Ownership invariant — 如果 ptr != nullptr，那它指向一塊由這個 Buffer 擁有、並且會在 destructor 中釋放的 heap memory。

那下一個問題自然出現：如果 Buffer b = a;，兩個 object 誰擁有 ptr？如果 compiler 自動產生 memberwise copy，a.ptr 和 b.ptr 會指向同一塊 memory，但兩個 destructor 都會 delete[] 它。

naive model 失敗 — 「我只要寫 destructor 管好釋放」是不夠的。Destructor 表示 ownership；ownership 一旦存在，copy operation 就必須回答 ownership 如何被複製、共享、轉移，或禁止。

2 Rule of 3：destroy、copy construct、copy assign

Rule of 3 說：如果你需要自訂其中一個，通常也要思考另外兩個：

Function何時發生要維護什麼
destructorobject lifetime 結束釋放 owned resource
copy constructor用另一個 object 建立新 object新 object 如何取得自己的 valid resource
copy assignment把既有 object 改成另一個 object 的內容先處理舊 resource，再取得新語意

如果 Buffer 要支援 deep copy，copy constructor 可能像這樣：

Buffer(const Buffer& other)
  : ptr(new char[other.size]), size(other.size) {
  std::memcpy(ptr, other.ptr, size);
}

copy assignment 更麻煩，因為左邊已經是一個活著的 object，可能已經 owns 一塊 memory：

Buffer& operator=(const Buffer& other) {
  if (this == &other) return *this;

  char* new_ptr = new char[other.size];
  std::memcpy(new_ptr, other.ptr, other.size);

  delete[] ptr;
  ptr = new_ptr;
  size = other.size;
  return *this;
}

注意順序：先配置新 memory，成功後才釋放舊 memory。這是 exception safety 的基本直覺：不要先把自己拆掉，再去做可能失敗的事。

3 Rule of 5：move 也加入 ownership invariant

C++11 加入 move semantics 後，resource-owning type 又多了兩個 operation 要思考：

Function語意
move constructor用一個即將被放棄的 source 建立 destination，轉移 ownership
move assignment釋放 destination 原本 resource，再接手 source resource

move constructor 的核心不是「更快 copy」，而是「把 ownership 搬給 destination，並讓 source 留在可安全 destruct 的狀態」：

Buffer(Buffer&& other) noexcept
  : ptr(other.ptr), size(other.size) {
  other.ptr = nullptr;
  other.size = 0;
}

如果沒有把 other.ptr 清掉，source object 的 destructor 仍然會釋放同一塊 memory，destination 就拿到 dangling pointer。move 後的 source 不必保留原本內容，但必須仍然是 valid object，可以被 destruct、可以被 assign。

noexcept 為什麼常出現？ — 對 resource owner 來說，move 通常只是搬 pointer 和 size，不應該 throw。標成 noexcept 也讓 std::vector 這類 container 在 reallocation 時更敢選 move path。這個細節會在 Ch12 再完整展開。

4 Rule of 0：更現代的第一選擇

Rule of 3/5 告訴你：手動管理 resource 時，special member functions 會成組出現。但現代 C++ 的優先方向其實是 Rule of 0：

Rule of 0 — 讓專門的 RAII member 管理 resource，讓你的 domain type 不需要自己寫 destructor/copy/move。

例如與其手寫 char* owner：

class Buffer {
  char* ptr;
  size_t size;
};

更好的第一版通常是：

class Buffer {
public:
  explicit Buffer(size_t n) : bytes(n) {}

private:
  std::vector<char> bytes;
};

現在 std::vector 自己負責 memory ownership。你的 Buffer 沒有手寫 destructor，copy/move/destroy 都可以交給 compiler-generated special members，因為每個 member 自己都已經有正確語意。

5 Pixel-Renderer 的落點

這章不是純 C++ 課。它會直接影響 Pixel-Renderer 未來的設計：

  • Owned framebuffer：如果 framebuffer owns pixels，copy 是 deep copy、deleted copy，還是 move-only？這會決定 Framebuffer 的 special members。
  • DisplayBackend：Win32/SDL/macOS backend 可能 owns window handle、texture、DIB section。這類 object 通常應該禁止 copy，支援 move 或完全固定 lifetime。
  • Shader object：如果 shader 只是 stateless function object，copy 可能很便宜；如果 shader owns compiled data/cache，copy policy 就要重新設計。
  • Immediate UI：UI command buffer 可能適合 Rule of 0：用 std::vector 管理 draw commands，讓 container 負責 resource。
設計檢查表 — 只要你準備寫 destructor，先停下來問：這個 resource 能不能交給 std::vector、std::unique_ptr、或某個小型 RAII handle？如果可以，優先 Rule of 0。只有 wrapper OS/GPU/C API handle 時，才自己承擔 Rule of 3/5。
常見陷阱

1. 只寫 destructor — 這通常留下 compiler-generated copy，導致 double delete。

2. move 後 source 不安全 — moved-from object 必須仍然 valid，至少能安全 destruct。

3. 過早手寫 Rule of 5 — 能用 Rule of 0 就不要手動管理 resource；手寫越多，越容易漏掉 exception safety、自我賦值、move 後狀態。
本章學到的三件事

1. Destructor 表示 ownership，ownership 會逼你一起思考 copy constructor 和 copy assignment。這就是 Rule of 3 的壓力來源。

2. C++11 後 move constructor / move assignment 加入同一組 invariant：destination 接手 resource，source 留在 valid moved-from state。

3. 現代 C++ 優先 Rule of 0：把 resource 放進已經會 RAII 的 member，讓 compiler-generated special members 自然正確。
