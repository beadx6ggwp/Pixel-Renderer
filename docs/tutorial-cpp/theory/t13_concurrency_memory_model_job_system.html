Concurrency / Memory Model / Job System Intro
多執行緒不是把 for loop 丟給 thread，而是切分 ownership、同步與可見性
thread data-race atomic job-system

T-12 說 renderer hot loop 受 memory layout 影響。下一個自然問題是：既然 software rasterizer 會慢，能不能把畫面切成多塊，丟給多個 CPU core 同時畫？可以，但這不是「開幾個 thread」這麼簡單。

Concurrency 的第一性問題不是 thread API，而是：每個 thread 寫哪段 memory？誰可以讀 shared state？什麼時候結果對另一個 thread 可見？如果兩個 thread 同時碰同一個 pixel 或 depth，C++ 語意是否仍然定義良好？

本章範圍 — 建立 C++ concurrency 的最小模型：thread、join、mutex、atomic、data race、happens-before、memory visibility、tile-based jobs、frame synchronization。目標不是馬上實作完整 job system，而是讓未來 parallel rasterizer 不會從 shared framebuffer race condition 開始。

1 naive model：把每個 triangle 丟給一個 thread

最直覺的 parallel renderer 可能長這樣： 

for each triangle:
  launch thread:
    rasterize triangle into framebuffer

這個模型很快會失敗。不同 triangle 可能覆蓋同一個 pixel，還要同時讀寫 depth buffer： 

Triangle A thread:
  if depth_A < depth[p]:
    depth[p] = depth_A
    color[p] = color_A

Triangle B thread:
  if depth_B < depth[p]:
    depth[p] = depth_B
    color[p] = color_B

這裡不是單純「誰最後寫入就贏」。兩個 thread 同時讀寫同一個 memory location，沒有 synchronization，就可能形成 C++ data race。data race 在 C++ 裡是 undefined behavior。

重要修正 — Parallel rendering 的第一步不是增加 thread 數量，而是先讓每個 thread 的 write ownership 清楚。沒有 ownership partition，就很容易把 depth/color buffer 變成 race condition。

2 data race：不是結果隨機而已

C++ 對 data race 的定義可以先抓住這個版本： 

Two threads access the same memory location.
At least one access is a write.
There is no synchronization establishing ordering.

=> data race
=> undefined behavior

Undefined behavior 的意思不是「結果隨機但程式還算在跑」。Compiler 和 CPU 都可以基於「well-defined C++ program 不會有 data race」做最佳化。當你違反這個前提，觀察到的錯誤可能不是單純少畫一個 pixel，而是整段邏輯被最佳化成你看不懂的結果。

共享資料 危險模式 比較穩的方向
color buffer多 thread 同時寫同一 pixel。tile ownership 或 per-thread intermediate buffer。
depth bufferread-modify-write 沒有同步。tile ownership，或非常謹慎的 atomic/lock 設計。
command bufferrender 時另一個 thread push command。frame command build 完成後 freeze，再分派 jobs。
resource cacheworker 同時新增 texture/material。resource loading 和 render execution 分階段。

3 happens-before：可見性需要同步點

Thread A 寫完資料，不代表 Thread B 在 C++ 語意上就能安全看見。你需要 synchronization 建立 happens-before relation。 

worker thread:
  write framebuffer tile
  finish job

main thread:
  wait/join workers
  Present(framebuffer)

join / wait establishes the synchronization boundary

常見同步工具的角色：

工具建立什麼適合位置
std::thread::joinworker 完成 happens-before join return。簡單 worker 或測試用平行階段。
std::mutexunlock / lock 之間的同步。保護低頻共享結構，例如 job queue。
std::atomic依 memory order 建立同步或原子操作。counter、flag、work index。
condition variable等待工作到來或階段完成。worker pool sleep/wake。
不要只記 API — mutex、atomic、join 的共同問題是：它們在哪裡建立 ordering？誰寫的資料，在什麼同步點之後，誰可以讀？

4 Tile-based thinking：先切 ownership

對 software rasterizer，第一個比較可控的 parallel model 是 tile-based jobs：把 framebuffer 切成不重疊 tiles，讓每個 job owns 一塊 pixel region。

SVG：tile-based jobs 的核心是 ownership partition。每個 job 寫不重疊 framebuffer region，降低 color/depth race。

Triangle 需要被分派到它覆蓋到的 tiles。這會增加 command binning 的成本，但換來寫 framebuffer 時 ownership 清楚。 

build frame commands
  |
  v
bin triangles/commands into tiles
  |
  v
submit tile jobs
  |
  v
each job:
  rasterize commands clipped to tile
  write only tile color/depth pixels
  |
  v
wait all jobs
  |
  v
Present(framebuffer)

5 mutex 與 atomic 不是萬用加速器

同步工具有它們的位置，但它們不是讓任何 shared data 都安全又快速的魔法。

工具適合不適合
std::mutex保護低頻共享結構，例如 job queue、resource registry。每 pixel lock depth buffer。
std::atomiccounter、flag、work index、簡單狀態。複雜 pixel/depth/color transaction。
thread-local storage每 thread temporary buffer、scratch allocator。需要全域 ordering 的結果。
barrier / wait group階段同步，例如所有 tile 完成後 present。把所有小操作都變成同步點。

高效 parallel renderer 通常先減少共享，再設計同步點，而不是先把共享資料全部加鎖。

6 false sharing：不同 pixel 也可能互相拖慢

即使兩個 thread 沒有寫同一個 pixel，也可能寫到同一條 cache line。這稱為 false sharing。從 C++ correctness 看可能沒 data race，但從 performance 看會讓 cache line 在 core 之間來回失效。 

cache line contains pixels 0..15

Thread A writes pixels 0..7
Thread B writes pixels 8..15

No same pixel.
But same cache line may bounce between cores.

Tile 設計時要避免太細碎的交錯寫入。Tile 大小、row alignment、job scheduling 都會影響這件事。第一版不需要立刻極限調校，但要知道「不重疊 memory location」和「不互相拖慢 cache」不是同一層問題。

7 Frame job system 的最小形狀

一個最小 job system 不需要一開始就有 work stealing、fiber、task graph。它先需要清楚的 frame boundary： 

main thread:
  poll input
  build frame commands
  freeze command buffer
  divide framebuffer into tiles
  submit tile jobs
  wait for all jobs
  present framebuffer

worker threads:
  wait for job
  pop tile job
  read frozen commands/resources
  write owned tile pixels
  mark job complete

這裡有兩個重要 invariant：

  • Worker 讀的是 frozen frame commands，不是被 UI/frontend 同時修改的 vector。
  • DisplayBackend 在所有 jobs 完成後才讀 framebuffer 做 present。

8 Determinism：平行化不能破壞 golden test

I-16 會把 golden image 放進 testing 主線。Parallel renderer 也要能對同一份 input 產生可預期 output。這不代表所有內部 job 執行順序都相同，但最後 framebuffer 應該符合定義好的規則。

問題如果沒定義需要的規則
兩個 triangle 同 depth不同 thread order 造成不同顏色。定義 tie-breaker，或接受並測試 tolerance/非決定性範圍。
alpha blending order透明物件順序不同結果不同。先排序，再分派；或限制第一版透明路徑。
floating-point reduction不同 order 有些微差異。避免 parallel reduction 進入 golden path，或定義 tolerance。
測試觀點 — 先讓 single-thread renderer 有 golden output，再把 tile jobs 用 single-thread replay 跑一次，最後才加 worker pool。這樣平行化 bug 才能被定位。

9 Pixel-Renderer 平行化順序

  1. 先讓單執行緒 renderer 的 framebuffer ownership、pitch、depth invariant 穩定。
  2. 建立 deterministic test scene，確保單執行緒輸出可比對。
  3. 把畫面切成 tiles，但先用單執行緒依序跑 tile jobs。
  4. 加入 worker pool，保證每個 job 寫不重疊 region。
  5. 加入 command binning，減少每個 tile 需要掃描的 primitives。
  6. 測量 false sharing、tile size、job granularity。
  7. 最後才考慮 work stealing、SIMD + threads、task graph。
不要先平行化未穩定的架構 — 如果 renderer core 還依賴 platform-owned framebuffer，或 UI/display backend 可能同時碰 framebuffer，多執行緒只會把 ownership bug 放大。
本章學到的三件事：
  1. C++ data race 是 undefined behavior，不是單純結果不穩；parallel renderer 要先切清楚每個 thread 寫哪段 memory。
  2. Tile-based jobs 的核心是 ownership partition：每個 job owns 不重疊 framebuffer region，present 等所有 jobs 完成。
  3. Mutex/atomic 是同步工具，不是加速保證；高效 parallel renderer 先減少共享，再定義 happens-before 與 deterministic verification。
