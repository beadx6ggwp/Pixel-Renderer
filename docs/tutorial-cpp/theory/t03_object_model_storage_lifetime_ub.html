THEORY TRACK · T-03
Object Model / Storage / Lifetime / UB
raw bytes 不是 object；地址存在不代表 lifetime 已經開始
object-lifetime storage undefined-behavior

Pixel-Renderer 接下來會遇到一個核心 refactor：把 Win32 DIBSection 裡的 memory，改成 renderer 自己擁有的 CPU framebuffer，再把 present path 抽成 DisplayBackend。表面上這像是 class 拆分；底層其實是在問：誰擁有 storage？哪個 object 的 lifetime 在那塊 storage 裡開始？誰有權把它結束？

如果你只用「記憶體四區」理解 C++，會很快卡住。Stack / heap / static / code 只能回答「storage 大概從哪裡來、活多久」。它不能完整回答「這些 bytes 什麼時候開始是一個 Framebuffer object」、「把 bytes cast 成 Vec4* 可不可以」、「destroy 後 pointer 還能不能拿來讀」。這章就是把那層補完整。

本章範圍 — 建立 C++ object model：storage、object lifetime、storage duration、alignment、object representation、strict aliasing、placement new、std::construct_at、undefined behavior。最後把這些規則接回 Pixel-Renderer 的 framebuffer ownership、backend memory、resource handle。

1 naive model：有一塊 memory 就有一個 object

最直覺的 C 模型是： 

memory is bytes
type is how I interpret those bytes
if I have an address, I can cast it to T* and use it

這個模型在 C++ 裡不夠。C++ 不只規範 bytes，也規範 object lifetime。同一塊 storage 在不同時間可以沒有 object、可以有一個 T object、也可以在 destroy 後回到 raw storage。 

raw storage
  |
  | construction begins lifetime
  v
object T exists
  |
  | destructor ends lifetime
  v
raw storage again
SVG：storage duration 和 object lifetime 是不同層。bytes 可以存在，但 object 尚未開始或已經結束。

所以 C++ 裡要分清楚兩件事：

概念 意思 renderer 例子
storage 一段可以放 object 的 bytes。 std::vector<uint32_t> 背後的 contiguous memory。
object 某個 type 的實體，具有 lifetime、value、可能有 invariant。 Framebuffer、RenderDevice、Color。
lifetime object 從開始存在到結束存在的期間。 window backend 釋放後，對應 DIBSection memory 就不能再被 framebuffer view 使用。
第一個關鍵修正 — address 存在不代表 object 存在。Pointer value 也不等於 permission。C++ 允許 compiler 假設你遵守 object lifetime 與 type aliasing 規則，違反時常常不是 crash，而是 undefined behavior。

2 storage duration：storage 大概活多久

stack / heap / static 仍然重要，但它只是 storage duration 的入口。Storage duration 說的是 storage 的可用期間，不直接等於 object semantic ownership。

Storage duration 典型來源 結束時機 注意
automatic local variable 離開 scope 不是「一定在 stack」，但可先用 stack 直覺理解。
static global / static local program 結束 初始化順序和跨 translation unit 依賴要小心。
thread thread_local thread 結束 未來 job system / worker thread 會遇到。
dynamic new、allocator、container delete / deallocate / container release storage 和 object lifetime 常被 allocator/container 分開管理。

這裡最容易混淆的是 dynamic storage。operator new 可以只分配 raw storage；constructor 才開始 object lifetime。delete 則是先呼叫 destructor 結束 lifetime，再釋放 storage。 

new T(args...)
  |
  +-- allocate storage
  +-- construct T in that storage
  +-- return T*

delete p
  |
  +-- destroy T object
  +-- deallocate storage

這就是為什麼 RAII 不是「自動 delete」這麼淺。RAII 的本質是讓 object lifetime 綁住 resource lifetime：constructor 建立有效狀態，destructor 釋放 resource，type invariant 在中間期間成立。

3 object lifetime：何時開始，何時結束

一般 local object 很直覺：

void Render() {
  Color clear{0, 0, 0, 255}; // lifetime begins
  device.Clear(clear);
} // lifetime ends

但底層 storage API 會把事情拆開。例如：

alignas(Framebuffer) unsigned char storage[sizeof(Framebuffer)];

// storage exists, but no Framebuffer object exists yet
Framebuffer* fb = new (storage) Framebuffer(width, height);
// now a Framebuffer object exists in storage

fb->~Framebuffer();
// lifetime ended; storage remains, but Framebuffer no longer exists

這叫 placement new。現代 C++ 也常用 std::construct_at 表達同一件事：

Framebuffer* fb = std::construct_at(reinterpret_cast<Framebuffer*>(storage), width, height);
std::destroy_at(fb);
跟 in-place construction 的關係 — emplace_back、optional::emplace、placement new、std::construct_at 都在處理同一件事：final storage 已知時，讓 object 直接在那塊 storage 開始 lifetime。差別只是誰管理 storage、誰負責 destroy。

4 alignment：不是任何 address 都能放任何 T

就算 storage bytes 足夠，也不代表 address 合法。每個 type 都有 alignment requirement。簡化說：某些 type 的 object 必須放在特定倍數的 address 上。 

address:
  0x1000  ok for align 16
  0x1004  ok for align 4, not ok for align 8 or 16
  0x1008  ok for align 8
  0x1010  ok for align 16

這就是為什麼剛剛 placement new 的 storage 要寫：

alignas(Framebuffer) unsigned char storage[sizeof(Framebuffer)];

如果你未來做 SIMD-friendly math type，例如 Vec4 希望 16-byte aligned，alignment 就會從語言細節變成效能與 correctness 問題。對 framebuffer 而言，row pitch 也有類似直覺：一列資料可能為了 alignment 多出 padding，不應永遠假設 pitch == width * 4。

問題 語言層 renderer 層
object 起始 address 合法嗎？ alignof(T)、alignas(T) SIMD vector、matrix、arena allocation。
row 起始 address / stride 合理嗎？ array layout / padding framebuffer pitch、SDL locked texture pitch。

5 object representation：object 可以被看成 bytes，但不是任意 type

C++ 允許你用 unsigned char、std::byte 這類 byte view 觀察 object representation。這是 serialization、debug memory dump、image buffer inspection 的基礎。

uint32_t color = 0xff3366aa;
auto* bytes = reinterpret_cast<const unsigned char*>(&color);
// bytes[0..3] shows object representation

但反過來不等於「任意 bytes 都可以當任意 object」。一段 bytes 是否代表合法 T value，取決於 type。對 uint32_t 這類整數通常直覺；對 pointer、floating point、class with invariant、resource handle 就不能亂來。

區分兩種操作

讀 object representation：把現有 object 當 bytes 看，通常可行。
把 bytes 當 object：需要確定 object lifetime、alignment、type rule、value representation 都合法。

Pixel-Renderer 的 framebuffer 常用 uint32_t pixels，這相對安全，因為 pixel 本身是 plain value。但如果你把一段 raw bytes reinterpret 成 Texture、Framebuffer、std::vector<uint32_t>，那就不是單純改 type label，而是在假裝 object lifetime 和 invariants 已經成立。

6 strict aliasing：compiler 會相信 type 規則

Naive C 模型會說： 

same address
  |
  +-- as uint32_t*
  +-- as float*
  +-- as Color*
whatever I cast to is what it is now

C++ compiler 不用這樣想。它可以根據 type-based aliasing rules 假設某些不同 type 的 pointer 不會指到同一個 object，進而做最佳化。如果你違反這個假設，程式可能在 -O0 看似正常，在 -O2 錯掉。

void Bad(uint32_t* pixel, float* f) {
  *pixel = 0xff0000ff;
  *f = 1.0f;
  // compiler may assume pixel and f do not alias
}

這不是 compiler 小氣，而是效能模型的一部分。若 compiler 必須假設任何 pointer 都可能 alias，很多 load/store optimization 都會被迫保守。

Renderer consequence — framebuffer 應該有清楚的 element type，例如 std::vector<uint32_t> 或 std::span<uint32_t>。如果需要 byte-level upload 給 SDL/Win32，就在 present boundary 轉成 void* / bytes view，而不是在 rasterizer hot loop 裡到處 reinterpret cast。

7 undefined behavior：不是「會丟 exception」

Undefined behavior 簡稱 UB。它不是「結果不一定」這麼溫和，也不是「runtime 會報錯」。UB 的意思是：程式已經離開 C++ abstract machine 的規則範圍，compiler 不需要保證任何結果。

例子 為什麼是問題 renderer 後果
use-after-lifetime object 已 destroy，還透過舊 pointer/reference 使用。 RenderDevice 持有已關閉 backend 的 framebuffer pointer。
out-of-bounds array/vector element 範圍外不是合法 object。 SetPixel 寫到 framebuffer 外，破壞其他 state。
misaligned access address 不滿足 type alignment。 SIMD math type 或 packed buffer 讀寫出問題。
strict aliasing violation 用不相容 type 存取同一 object。 debug build 正常，release build 顏色或 depth 值怪掉。
uninitialized read object value 未初始化卻被讀取。 未清 depth buffer，triangle 深度比較隨機。

UB 對 renderer 特別危險，因為 renderer hot loop 充滿 pointer arithmetic、array indexing、manual buffer layout。畫面錯不一定是圖學公式錯，有時是 C++ 語意已經被破壞。

不要用「我測起來可以」判斷 UB — UB 可能在目前 compiler、目前 optimization、目前資料大小下看似正常。換成 Release、換成 Apple Clang、換成不同 CPU alignment，就可能出現完全不同結果。

8 Pixel-Renderer：framebuffer ownership 應該怎麼想

目前最重要的工程問題是：renderer core 應該寫誰的 memory？

如果 RenderDevice 直接借用 Win32 DIBSection 的 pointer，那 object/lifetime 邊界會混在一起： 

Win32 ScreenManager
  owns DIBSection resource
  exposes raw pixel pointer
        |
        v
RenderDevice
  borrows pointer
  writes pixels
        |
        v
BitBlt presents same storage

這能跑，但它把 renderer core 的 framebuffer lifetime 綁到 Win32 backend resource。要做 SDL/macOS backend 時，就會開始問：SDL texture lock 的 pointer 何時有效？UpdateTexture path 又是誰 owns CPU buffer？這些問題其實都是 object model 問題。

比較乾淨的方向是： 

Framebuffer
  owns CPU pixel storage
  vector<uint32_t> pixels
  width / height / pitch
        |
        v
RenderDevice
  borrows Framebuffer view during rendering
  SetPixel writes CPU-owned storage
        |
        v
DisplayBackend::Present(framebuffer)
  Win32: copy/update to DIBSection then BitBlt
  SDL:   SDL_UpdateTexture then RenderCopy/Present
  macOS: upload/copy to platform surface

這個設計讓 object lifetime 更明確：

物件 owns what 借出什麼 不應該做什麼
Framebuffer CPU pixel storage。 std::span<uint32_t> 或 pointer + pitch view。 不應該知道 Win32/SDL/macOS。
RenderDevice 通常不 owns pixels，只借用 framebuffer。 SetPixel、Clear 等 write API。 不應該 close window 或 destroy backend resource。
DisplayBackend platform window / texture / surface resource。 present API。 不應該成為 rasterizer 的 pixel storage owner。
工程判準 — 如果一個 pointer 的有效期間取決於 Lock/Unlock、window resource、backend callback、或 OS handle，它就不應該成為 renderer core 長期保存的 framebuffer pointer。Renderer core 應該面對穩定 owned storage 或明確 lifetime 的 borrowed view。

9 verification：怎麼檢查 lifetime / storage 問題

這章不是只講抽象。你在專案裡可以用幾種方法檢查 object model 問題。

檢查 pointer 是否越界

break RenderDevice::SetPixel if x < 0 || y < 0 || x >= width || y >= height
run
backtrace
info locals

這能抓到 out-of-bounds write 的 source，而不是只在 crash 後猜。

檢查 framebuffer storage 是否穩定

break RenderDevice::SetFramebuffer
run
print framebuffer
print framebuffer->pixels.data()
continue
print framebuffer->pixels.data()

如果你設計成 RenderDevice 長期持有 pixels.data()，就要確定 vector 不會 reallocate。更好的方式是讓 RenderDevice 每次 frame 拿一個 view，或者讓 Framebuffer resize 後重新 bind。

使用 sanitizer

在 Clang/GCC 環境，後面 toolchain 章節可以加入 sanitizer build：

# conceptual flags; later chapters will wire this into build system
-fsanitize=address,undefined -fno-omit-frame-pointer -g

AddressSanitizer 能抓很多 out-of-bounds / use-after-free；UBSan 能抓一部分 undefined behavior。它們不是語意理解的替代品，但很適合當工程防線。

注意 — sanitizer 不是所有 UB 都能抓，也不是所有平台/編譯器設定都一樣好用。它是驗證工具，不是語言規則本身。
本章學到的三件事：
  1. Storage 和 object 不同：bytes 存在不代表某個 T object 的 lifetime 已經開始。
  2. C++ compiler 會依賴 lifetime、alignment、aliasing、initialization 規則做最佳化；違反規則就是 UB，不是可預期的 runtime error。
  3. Pixel-Renderer 的正確方向是讓 renderer core 寫穩定 CPU-owned framebuffer，backend 只負責 present，避免把 platform-owned memory 的 lifetime 污染進 rasterizer。
