THEORY TRACK · T-26
Regular Types / Equality / Hashing / Ordering
能 copy 不代表能當 value；能 compare 不代表能當 cache key
regular-type equality hashing ordering

T-23 已經提醒你：operator==、operator<=>、hash function 不是語法裝飾，它們會變成 cache、container、test diff 的語意 contract。現在要把這件事獨立講深：什麼樣的 type 可以像普通 value 一樣被 copy、compare、放進 container、當成 key？

Pixel-Renderer 很快會有 TextureId、PipelineState、DrawCommand、VertexLayout、RGBA8、DepthValue。如果 equality/hash/order 定義錯，錯誤不一定馬上 crash；它會變成 cache miss、cache hit 錯誤、golden test diff 不穩、resource registry 找不到東西。

本章範圍 — 建立 Regular / Semiregular / Movable / Copyable 的工作模型，拆 equality law、hash consistency、strict weak ordering、floating-point comparison、cache key design、command diff，並收束成 Pixel-Renderer 的 value type / resource id / pipeline state / command key policy。

1 naive model：只要有 operator== 就能當 key

naive 寫法通常是看到欄位就全部比較：

struct PipelineState {
  ShaderId shader;
  TextureId texture;
  float alphaThreshold;
};

bool operator==(const PipelineState& a, const PipelineState& b) {
  return a.shader == b.shader && a.texture == b.texture && a.alphaThreshold == b.alphaThreshold;
}

但這馬上會遇到問題：float exact equality 是你要的語意嗎？如果用 epsilon equality，hash 要怎麼跟它一致？如果 sorting 用另一套 comparison，container 的 invariant 還成立嗎？

核心修正 — equality / hash / ordering 是 law，不是 helper function。你不是在問「怎麼比較欄位」，而是在定義「哪些 object 在這個 domain 裡代表同一個 semantic value」。

2 Regular type：像 int 一樣可推理，不是像 int 一樣簡單

Regular type 的直覺是：它能像普通值一樣被建立、複製、指定、比較，而且 generic code 可以局部推理它。 

Regular-ish intuition:
  T a = make_value();
  T b = a;        // a remains same semantic value
  a == b;         // copied value compares equal
  b = c;          // assignment replaces b's value

This is why auto_ptr was a bad value type:
  T b = a;        // a changed, ownership was consumed
概念最低直覺Renderer 例子
Movable可以轉移 state，source 仍 valid。Framebuffer owner、backend wrapper。
Copyablecopy 不消耗 source。Vec3、RGBA8、TextureId。
Semiregular可 default construct / copy / assign / destroy。簡單 descriptor、optional state。
RegularSemiregular + equality 表示同一 semantic value。PipelineKey、VertexLayout。

3 equality law：不是每個近似都能叫相等

Equality 至少要讓 generic code 能假設： 

reflexive:
  a == a is true

symmetric:
  if a == b, then b == a

transitive:
  if a == b and b == c, then a == c

浮點 epsilon equality 常破壞 transitive： 

epsilon = 0.1
a = 0.00
b = 0.09
c = 0.18

a ~= b  true
b ~= c  true
a ~= c  false

這不代表 epsilon comparison 不能用。它適合 validation / tolerance check，例如 golden image diff 或 geometry test。但它通常不適合當作 unordered_map key equality。

Domain 區分 — 「測試容忍度」和「cache key equality」不是同一件事。Golden image 可以接受色差 tolerance；pipeline state cache key 必須有穩定、可 hash、可重現的 equality。

4 hash consistency：相等就必須同 hash

unordered container 的最低 contract： 

if a == b, then hash(a) == hash(b)

反過來不需要成立。不同 object 可以 hash collision，但 equality 相等的 object 必須落到同一個 hash value，否則 container 可能找不到明明存在的 key。

struct PipelineKey {
  ShaderId shader;
  TextureId texture;
  BlendMode blend;
  DepthMode depth;

  friend bool operator==(const PipelineKey&, const PipelineKey&) = default;
};

struct PipelineKeyHash {
  std::size_t operator()(const PipelineKey& k) const noexcept {
    return HashCombine(k.shader, k.texture, k.blend, k.depth);
  }
};

5 ordering：sorting 和 map 需要 strict weak ordering

std::sort、std::map、std::set 依賴 ordering law。比較函式不是「隨便回傳看起來大小」；它要形成 strict weak ordering。 

comp(a, a) must be false
if comp(a, b) is true, comp(b, a) must be false
equivalence groups must be transitive

Renderer 常見做法是不要對 domain object 發明不自然的 ordering。若只是為了 map key，建立明確排序 tuple：

auto Tie(const PipelineKey& k) {
  return std::tie(k.shader.value, k.texture.value, k.blend, k.depth);
}

bool operator<(const PipelineKey& a, const PipelineKey& b) {
  return Tie(a) < Tie(b);
}

6 float、NaN、-0.0：numeric state 不一定 Regular

Floating-point 有幾個會打破 naive equality 的點：

  • NaN == NaN 是 false，破壞 reflexive intuition。
  • +0.0 == -0.0 是 true，但 bit pattern 不同。
  • epsilon equality 可能不 transitive。
  • 不同平台 / compiler flags / SIMD path 可能產生微小差異。

所以對 renderer 來說，float 可以放在 transform、interpolation、depth calculation 裡，但不一定應該直接放進 long-lived cache key。若一定要放，先定義 bitwise exact semantics 或 quantized semantics。 

Good cache key candidates:
  shader id
  texture id
  blend mode
  depth mode
  vertex layout descriptor
  pixel format enum

Dangerous cache key candidates:
  raw float matrix
  epsilon-compared value
  pointer address
  transient frame-local object identity

7 command diff / golden tests：equality 要服務 debug

Command buffer replay 和 golden test 需要比較「兩次輸出是否同一件事」。但不同層級的 equality 不一樣：

層級Equality 目的Policy
Command exact diff找出 replay 是否一樣離散欄位 exact，比較 command payload。
Image golden diff驗證視覺輸出允許 tolerance / mask / histogram 視情境。
Pipeline cache key找同一 pipeline variant必須 deterministic、hash consistent。
Resource id指向 registry entryid + generation exact equality。

8 Pixel-Renderer type policy

Figure 1：Pixel-Renderer 裡不同 type 的 equality/hash/order policy。Regular value、resource id、cache key、debug validation 不應混成同一套比較。
Policy summary:

Vec2 / Vec3 / Rect:
  copyable value types
  equality if exact component equality is intended
  avoid unordered key use with floats unless bitwise/quantized

RGBA8 / PixelFormat:
  regular discrete values
  good equality/hash candidates

TextureId / ShaderId:
  regular id values
  equality/hash by id + generation if applicable

Framebuffer / DisplayBackend:
  owner types
  not Regular by default
  compare only through explicit debug identity or output image

PipelineKey / VertexLayout:
  explicit cache key types
  equality and hash must be implemented together

DrawCommand:
  value command for replay/diff
  avoid storing raw borrowed pointers as equality-relevant fields
本章學到的三件事：
  1. Regular type 是語意 law，不是語法清單；copy 不應消耗 source，equality 應表示同一 semantic value。
  2. Hash 和 equality 必須一起設計；若 a == b，就必須 hash(a) == hash(b)，否則 cache 和 unordered container 會失效。
  3. Pixel-Renderer 要分清 cache key equality、resource id identity、numeric tolerance、debug diff；float epsilon comparison 不應偷渡成 hash key equality。
