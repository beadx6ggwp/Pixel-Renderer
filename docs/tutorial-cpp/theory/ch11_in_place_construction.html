PART 4 · CHAPTER 11
In-place construction
不要先在別處建 T，再搬到 final storage
emplace placement-new storage

上一章用 return-by-value 建立了一個重要方向：如果 result object 的位置已經知道，C++ 有時候不需要先建立 source object，再 copy / move 過去。

這章把同一個思路推到 container、optional、arena、UI command buffer。問題不再是「move 夠不夠快」，而是：既然 final storage 已經知道，為什麼不讓 object 直接在那裡出生？

本章範圍 — 建立 destination-first construction 的模型。你會分清楚 push_back(T(...))、emplace_back(...)、placement new、std::construct_at 的關係，以及 storage existence 和 object lifetime 的差別。

1 naive model：先做一個 T，再塞進 container

假設未來 UI 系統每 frame 產生一串 draw commands：

struct DrawCommand {
  Rect bounds;
  Color color;
  TextureId texture;
};

std::vector<DrawCommand> commands;
commands.push_back(DrawCommand{bounds, color, texture});

這個寫法的直覺模型是：

create temporary DrawCommand
  | move/copy
  v
vector element slot

如果 DrawCommand 只是幾個 small value，這可能沒什麼問題。但如果 type owns buffer、handle、large payload，或乾脆是 move-only / non-movable，這個「先在外面建，再搬進去」的模型就開始有壓力。

2 emplace 的方向：把 constructor arguments 送到目的地

改成：

commands.emplace_back(bounds, color, texture);

語意方向變成：

vector reserves / chooses an element slot
  | forwards constructor arguments
  v
construct DrawCommand directly in that slot

這就是 in-place construction：不是先拿到一個已經 materialized 的 T，而是把建構 T 所需的參數送到 final storage，讓 T 的 lifetime 在那裡開始。

共同抽象

RVO、emplace_back、optional::emplace、placement new 都不是同一條語言規則，但都在往同一個方向走：final storage 已知時，讓 object 直接在目的地開始 lifetime。

3 emplace 不是 magic

一個常見誤解是：

emplace_back is always no move.

這句話太粗。先看這個寫法：

DrawCommand cmd{bounds, color, texture};
commands.emplace_back(std::move(cmd));

這裡 cmd 已經先被建立出來了。emplace_back 只能在 vector element slot 呼叫 DrawCommand(DrawCommand&&)。也就是：你只是把 move constructor 的呼叫地點放到 element slot，不是避免 materialization。

精準說法 — emplace_back(args...) 讓 container 有機會用 constructor arguments 在 element slot 建構新 element。如果你傳進去的已經是一個 T，那它仍然可能 move/copy 那個 T。

4 vector 的邊界：新 element vs 舊 elements

還有一個更重要的邊界：emplace_back 只是在講「新 element 如何建構」。它不保證整個 vector 操作都沒有 move/copy。

如果 vector capacity 不夠，vector 可能需要 reallocate：

old buffer: [A][B][C]
new buffer: [ ][ ][ ][ ][ ][ ]

move/copy A, B, C into new buffer
construct new D in final slot

所以 commands.emplace_back(...) 可以讓新的 DrawCommand 直接建在新 slot；但既有 commands 在 reallocation 時仍可能被 move/copy。這就是為什麼下一章會接 vector reallocation 與 noexcept move。

5 storage exists 不代表 object exists

要真的理解 in-place construction，必須拆開兩件事：

Storage

一塊符合 size / alignment 的 memory。它可以只是 raw bytes，還不一定有 T object。

Object Lifetime

T 的 constructor 已經開始建立 invariant；之後需要 destructor 結束 lifetime。

這就是 placement new / std::construct_at 要處理的核心問題：我已經有 raw storage，如何在指定地址開始 T object 的 lifetime？

6 placement new 與 construct_at

低階寫法是 placement new：

alignas(T) unsigned char storage[sizeof(T)];

T* p = new (storage) T(args...);

p->~T();

C++20 之後，generic code 更常看到 std::construct_at / std::destroy_at：

T* p = std::construct_at(reinterpret_cast<T*>(storage), args...);
std::destroy_at(p);

這類工具不是日常高階 code 的 default。它們是給 container、optional、variant、arena allocator、object pool 這種需要手動控制 storage / lifetime 的地方用。

低階 lifetime 警告 — raw storage 不會自動呼叫 constructor / destructor。你必須保證 alignment 正確、只 destroy 已建構的 object、destructor exactly once，並處理 constructor throw 時的 exception safety。

7 Pixel-Renderer 的落點

這章對 Pixel-Renderer 後續幾個方向很直接：

  • UI command buffer：每 frame 產生大量 draw commands，emplace_back 可以讓 command 直接在 vector slot 建立。
  • Resource cache：try_emplace 可以在 map node 裡直接建立 texture / shader metadata，避免先建立再搬入。
  • Optional backend resource：optional<SDLDisplayBackend> 或自訂 backend storage 可能需要 destroy old value 再 emplace new value。
  • Arena / frame allocator：未來如果 UI 或 renderer 有 frame-local allocation，placement new / construct_at 是理解 object lifetime 的基礎。

更深一層，這章是在幫你建立一個 renderer 架構問題：資料不是「放進某個容器」就結束。你要問的是 object 的 lifetime 從哪裡開始、最後會在哪裡被消耗、誰負責 destroy。

常見陷阱

1. 以為 emplace 永遠比較快 — 如果你傳入的已經是 T，還是可能 move/copy；如果 constructor arguments 很複雜，也不一定比較清楚。

2. 忘記 vector reallocation — emplace_back 可以直接建新 element，但既有 elements 在換 buffer 時仍可能 move/copy。

3. 把 raw storage 當成 object — 有 memory 不等於有 T。只有 constructor / placement new / construct_at 開始 lifetime 後，才有 object。
本章學到的三件事

1. In-place construction 的核心是 destination-first：final storage 已知時，讓 object 直接在目的地開始 lifetime。

2. emplace_back(args...) 的價值是把 constructor arguments 送到 element slot；但它不保證整個 container operation 零 move/copy。

3. storage 和 object lifetime 是兩件事；placement new / construct_at 是在 raw storage 上開始 object lifetime 的低階工具。
