Overload Resolution / Conversion / Operator Design
operator 不是語法糖而已，它會把 type 的語意、conversion policy、cache key contract 暴露到 call site
overload conversion operator strong-type

T-04 說 expression 有 value category，T-22 說 const 會限制 mutation capability。接著要問：當你寫 a + b、Draw(textureId)、if (handle)、state == other 時，compiler 到底選了哪個 function？哪些 conversion 被允許？operator 的回傳型別又在保證什麼？

這不是抽象語法問題。Pixel-Renderer 會很快有 Vec3、RGBA8、TextureId、ShaderId、PipelineState、Command。如果 conversion 和 operator 設計錯，type system 反而會幫你把錯誤變得更安靜：color 可以誤當 texture id、resource handle 可以誤當 int、cache key 的 equality/hash 不一致。

本章範圍 — 建立 overload / conversion / operator design 的工作模型：candidate set、viable function、ranking、standard conversion、user-defined conversion、explicit、explicit operator bool、hidden friend + ADL、math operator 回傳規則、equality/hash contract，以及 Pixel-Renderer strong type policy。

1 naive model：compiler 會選最合理的 overload

直覺上，你可能覺得 compiler 會選「人看起來最合理」的 overload：

void Draw(TextureId id);
void Draw(int debugMode);

TextureId id{3};
Draw(id);

如果 TextureId 可以隱式轉成 int，或可以從 int 隱式建構，overload set 會變得含糊。Compiler 不是理解你的 domain，它只照語言規則排序 conversion sequence。

核心修正 — overload resolution 解的是「哪個 function 在語言規則下最匹配」，不是「哪個 domain 意義最正確」。Domain 意義要靠 type design、explicit、operator policy 和 API boundary 先寫進 type system。

2 overload resolution 的粗略模型

完整 overload resolution 規則很細，但工程上可以先抓這個流程：

SVG：overload resolution 是語言規則，不是 domain reasoning。Strong type 的任務是把錯誤 conversion 從 candidate set 排除。
匹配類型直覺例子
exact match最好。Draw(TextureId) 接 TextureId。
standard conversion語言內建轉換。int 到 float、array-to-pointer、qualification conversion。
user-defined conversionconstructor 或 conversion operator。TextureId(int)、operator int()。
ellipsis / fallback最後手段。C-style variadic function。

3 implicit conversion：方便，但會偷穿過語意邊界

Strong type 的目的，是把相同 storage representation 的不同語意分開：

struct TextureId {
  explicit TextureId(uint32_t value) : value(value) {}
  uint32_t value;
};

struct ShaderId {
  explicit ShaderId(uint32_t value) : value(value) {}
  uint32_t value;
};

如果 constructor 不是 explicit，那 uint32_t 可能在 call site 被自動變成 id。這會讓原本該由 type system 擋下的錯誤通過。

void BindTexture(TextureId id);

BindTexture(7); // should this be allowed? usually no
renderer rule — domain id、pixel format、size、coordinate、handle 這類 type 預設應禁止 implicit conversion。你可以提供明確 factory 或 explicit constructor，但不要讓 raw integer 自動穿過語意邊界。

4 conversion operator：尤其小心 operator int

Conversion operator 也會進入 overload resolution：

struct TextureId {
  operator uint32_t() const { return value; }
  uint32_t value;
};

這看起來方便，實際上會讓 TextureId 幾乎到處可以當整數用。它可能參與 arithmetic、被傳進錯的 overload、或和 ShaderId 互相混用。

比較合理的是提供明確 accessor：

struct TextureId {
  explicit TextureId(uint32_t value) : value_(value) {}
  uint32_t Value() const { return value_; }
private:
  uint32_t value_ = 0;
};

5 explicit operator bool：handle 可以檢查有效，但不該變 int

Resource handle 常需要寫：

if (texture) { ... }

但你不想讓它變成任意整數。這時可以使用 explicit operator bool：

class TextureHandle {
public:
  explicit operator bool() const { return id != invalid; }
private:
  uint32_t id = invalid;
  static constexpr uint32_t invalid = 0xffffffff;
};

這允許 boolean context，但不會讓 handle 自動參與 arithmetic： 

if (texture)       OK
texture + 1        not OK
Draw(texture)      only OK if overload accepts TextureHandle

6 operator design：用語意模仿，不是用外觀模仿

Operator overload 的第一原則是：如果這個操作在 domain 裡沒有清楚語意，不要只是為了語法好看而 overload。

Type合理 operator可疑 operator
Vec2 / Vec3+、-、scalar *、+=* 若不清楚是 dot 還是 component-wise。
RGBA8==、explicit pack/unpack+ 若不定義 clamp/wrap/linear/sRGB。
TextureId==、maybe operator bool+、< 若沒有 domain 意義。
FramebufferSize==、area helper隱式轉成 int 或 uint32_t。

7 回傳型別：non-mutating by value，mutating by reference

Math type 的常見規則：

struct Vec3 {
  float x, y, z;

  Vec3& operator+=(const Vec3& rhs) {
    x += rhs.x; y += rhs.y; z += rhs.z;
    return *this;
  }
};

inline Vec3 operator+(Vec3 lhs, const Vec3& rhs) {
  lhs += rhs;
  return lhs;
}

operator+= 修改左邊，所以回傳 T& 讓 chaining 合理。operator+ 產生新值，所以回傳 value。不要為了避免 imaginary copy 回傳 local reference；T-04 已經說過那會 dangling。

8 hidden friend 與 ADL：math type 常見寫法

你常會看到 operator 寫在 class 裡當 friend：

struct Vec3 {
  float x, y, z;

  friend Vec3 operator+(Vec3 lhs, const Vec3& rhs) {
    lhs.x += rhs.x; lhs.y += rhs.y; lhs.z += rhs.z;
    return lhs;
  }
};

這種 hidden friend 不會變成一般 name lookup 裡到處可見的函式，但當參數有 Vec3 時，ADL 會找到它。對 math type 來說，這能讓 operator 靠近 type 定義，也避免污染 namespace 太多名字。

不要迷信 pattern — hidden friend 是工具，不是目標。若 operator 不需要 private access，也可以放在同 namespace。真正的判準是：這個 operator 是否是 type 的自然操作？是否會引入危險 conversion？

9 equality、ordering、hash：cache key 的語意 contract

Renderer 之後會有 cache： 

PipelineState -> compiled pipeline variant
TextureKey    -> loaded texture resource
ShaderKey     -> shader/material variant
CommandBuffer -> regression diff / replay

這些 key 需要 equality / hash / ordering。最重要的規則是： 

If a == b, then hash(a) == hash(b).
If a and b compare equal, they must represent the same semantic state.
Ordering must be strict and stable if used for sorting/map keys.

浮點尤其危險。若 PipelineState 裡放 float epsilon 或 transform matrix，直接把 float 做 hash/equality 可能不符合你想要的語意。這時要先問：它真的應該是 cache key 嗎？還是應該把 key 限制在 discrete state，例如 blend mode、depth test mode、shader id、texture id？

10 Pixel-Renderer operator / conversion policy

Strong ids:
  TextureId, ShaderId, BufferId
  explicit from integer
  no implicit operator int
  equality allowed
  ordering only if there is a real container need

Math value types:
  Vec2, Vec3, Mat4, Rect
  non-mutating operators return by value
  mutating compound operators return T&
  avoid ambiguous operator* if dot/component-wise/scalar meanings collide

Pixel formats:
  RGBA8, BGRA8, DepthValue
  explicit pack/unpack
  avoid implicit conversion to raw uint32_t unless crossing platform boundary

Cache keys:
  equality/hash must be defined together
  avoid floats in keys unless exact bitwise semantics are intended
  prefer discrete enums / ids / normalized descriptors
本章學到的三件事：
  1. Overload resolution 只依 C++ 語言規則選 function，不理解 renderer domain；錯誤 conversion 必須靠 strong type、explicit 和 operator policy 擋掉。
  2. Operator overload 應模仿語意，不是模仿外觀；math type 的 non-mutating operator 通常 by value，compound assignment 回傳 T&。
  3. Equality、ordering、hash 是 cache/test 的 contract；resource id、pipeline state、command key 若定義不一致，後面的 cache、diff、golden test 都會不可靠。
