地址、解引用、與別名
pointer reference address

打開你的渲染器程式碼，隨便看一眼就能找到這些東西：RenderDevice* device、ScreenManager* screen、unsigned char* frame_buffer、const Vertex& v1。星號和 ampersand 到處都是。你可能已經隱約知道它們跟「地址」有關，但如果有人問你：* 在宣告時和在使用時意思一不一樣？& 在宣告時和在使用時呢？你可能會猶豫。

這一章，我們要把 * 和 & 的四種含義一次搞清楚。不再含糊，不再混淆。

本章範圍 — 徹底搞懂 * 和 & 的四種含義（宣告指標、解引用、宣告參考、取地址），以及指標和參考各自的使用場景。你會看到你的渲染器裡每一個 * 和 & 背後的設計意圖。

1 地址：記憶體的門牌號碼

上一章你看到記憶體被分成 stack、heap、static、code 四個區域。但不管在哪個區域，每一個 byte 都有一個獨一無二的地址（address）。你可以把地址想成門牌號碼 — 每棟房子（每個 byte）都有自己的號碼，只要你知道號碼，就能找到那棟房子。

在 C++ 裡，& 放在變數前面（不是宣告裡）就是取地址運算子。它的意思是「把這個變數的門牌號碼告訴我」：

int x = 42;
printf("%p\n", &x); // 印出 x 的地址，例如 0x61fe1c

這裡的 &x 不是在宣告參考 — 它是在使用已經存在的變數 x，問它：「你住在哪裡？」。記住這個區別，因為 & 在宣告和使用時的意思完全不同。

2 指標：存放地址的變數

知道了地址，下一個問題是：我能不能把地址存起來？當然可以。存放地址的變數就叫做指標（pointer）。

int x = 42;
int* p = &x; // p 是指標，裡面存的是 x 的地址

這裡就是最容易混淆的地方了。* 出現在兩個完全不同的場景：

宣告時的 *

int* p — 星號出現在型別旁邊，意思是「p 是一個指標，它指向 int」。它在定義 p 的身分。

使用時的 *

*p — 星號出現在已存在的指標前面，意思是「去 p 存放的那個地址，把那裡的值拿出來」。這叫解引用（dereference）。

int x = 42;
int* p = &x; // 宣告：p 是指標，存放 x 的地址
printf("%d\n", *p); // 使用：去 p 指向的地方 → 印出 42
*p = 99; // 使用：去 p 指向的地方 → 把那裡改成 99
printf("%d\n", x); // x 現在是 99

看你的渲染器裡的 application.h：

class Application {
  RenderDevice* device = nullptr; // device 是指標，存放 RenderDevice 物件的地址
};

RenderDevice* 裡的 * 是宣告時的意思 — 告訴編譯器「device 不是一個 RenderDevice 物件，而是一個存放 RenderDevice 地址的變數」。當你在 application.cpp 裡寫 device = new RenderDevice(&screen) 時，new 在 heap 上建立物件並傳回它的地址，這個地址被存進 device。

3 指標運算實戰

箭頭運算子 ->

當你有一個指標 p 指向某個物件，你想存取物件的成員，你得先解引用再用點號：(*p).member。但這太囉唆了，所以 C++ 提供了箭頭運算子 -> 作為簡寫：

// 以下兩行完全等價
(*device).DrawTriangle(v1, v2, v3, color);
device->DrawTriangle(v1, v2, v3, color); // 更常用的寫法

你的 main.cpp 裡正是這樣呼叫的：device->DrawTriangle({400, 100}, {500, 300}, {300, 300}, 0xFF0000)。因為 device 是指標，所以用 ->，不能用 .。

nullptr：空指標

指標可以不指向任何東西，這時候它的值是 nullptr（C++11 以前用 NULL 或 0）。你的 Application 就是這樣初始化的：

RenderDevice* device = nullptr; // 還沒建立，先指向「沒有」

// 在建構子裡才真正建立：
if (running) device = new RenderDevice(&screen);

對 nullptr 做解引用（*device 或 device->...）是未定義行為，通常直接當掉。所以在使用指標之前，你有責任確認它不是 null。

this 指標與 SetWindowLongPtr

你的 ScreenManager 裡有一段很典型的 Windows API 用法：

// screen_manager.cpp — 建立視窗後
SetWindowLongPtr(window_handle, GWLP_USERDATA, (LONG_PTR)this);

// 在靜態的 WindowProc 裡取回：
ScreenManager* self = (ScreenManager*)GetWindowLongPtr(hwnd, GWLP_USERDATA);

this 是 C++ 在每個成員函式裡自動提供的指標，指向「呼叫這個函式的那個物件」。這裡的技巧是：把 this 指標藏進 Windows 的視窗資料裡，之後在靜態函式中用 GetWindowLongPtr 把它撈回來，轉型成 ScreenManager*，就能再次存取物件的成員。整個過程就是在搬運一個地址。

指標的指標（簡介）

既然指標是一個變數，它自己也有地址，所以可以有「指向指標的指標」：int** pp。在渲染器裡你不太會用到，但理解它的存在可以幫助你不被多層 * 嚇到。核心邏輯不變 — 每一層 * 都是「再往下找一層地址」。

4 參考：不能改指向的別名

C++ 除了指標以外，還有另一種跟地址有關的機制：參考（reference）。參考是一個變數的別名 — 它不是一個新的變數，它就是那個變數本身。

int x = 42;
int& r = x; // r 是 x 的別名。r 就是 x，x 就是 r。
r = 99; // 改 r 就是改 x
printf("%d\n", x); // 印出 99

注意這裡的 & 出現在宣告中（int& r），意思是「r 是 int 的參考」。這跟前面的 &x（取地址）意思完全不同。

總結 & 的兩種含義
int& r = x; — 宣告時：r 是 x 的別名（參考宣告）
&x — 使用時：給我 x 的地址（取地址運算子）

同樣地，* 也有兩種含義：
int* p = &x; — 宣告時：p 是指標
*p = 99; — 使用時：去 p 存放的地址（解引用）

參考和指標最大的區別：

  • 參考不能為 null。宣告時就必須綁定到一個變數。
  • 參考不能改指向。一旦綁定，終身不變。r = y 不是讓 r 改指向 y，而是把 y 的值拷貝給 x。
  • 參考語法更乾淨。不需要 * 解引用，也不需要 ->，直接用 .。

比喻一下：指標像一張便利貼，上面寫著一個地址，你可以擦掉重寫，也可以什麼都不寫（null）。參考像是在一個人身上掛了第二個名牌 — 名牌就是那個人的一部分，不能拆下來掛到別人身上。

你的渲染器裡兩種都有。為什麼 RenderDevice 的建構子用指標，而 DrawTriangle 用參考？

// 建構子 — 用指標
RenderDevice::RenderDevice(ScreenManager* s) : screen(s) {
  frame_buffer = screen->GetFrameBuffer();
}

// DrawTriangle — 用 const 參考
void RenderDevice::DrawTriangle(const Vertex& v1, const Vertex& v2,
                                const Vertex& v3, uint32_t color) {
  // ...
}

ScreenManager* s 用指標，因為 RenderDevice 需要把這個地址存起來（成員 screen），將來可能需要改指向（或一開始是 nullptr）。而 const Vertex& v1 用 const 參考，因為你只是想讀取頂點資料，不想拷貝整個 Vertex 結構，也不需要存起來。

5 函數參數：值、指標、參考

函數接收參數有三種方式。這是你在寫渲染器時最常面對的選擇：

傳遞方式 語法 特性 適用場景
傳值 void f(Vertex v) 拷貝整個物件；修改不影響外面 小型別（int, float）
傳指標 void f(Vertex* v) 傳地址；可以是 nullptr；可改指向 可能為 null、需要存起來
傳參考 void f(Vertex& v) 別名；不能是 null；可修改原物件 需要修改外面的變數
傳 const 參考 void f(const Vertex& v) 別名；不能修改；零拷貝 最常用：唯讀大物件

來看一個具體的比較：

傳值 — 拷貝整個 Vertex
void draw(Vertex v) {
  // v 是外面那個的複製品
  // 修改 v 不影響原本的
}

如果 Vertex 有 x, y, z, u, v, color... 每次呼叫都拷貝一整份，很浪費。

傳 const 參考 — 零拷貝
void draw(const Vertex& v) {
  // v 就是外面那個本身
  // const 保證不會改到它
}

不拷貝、不能改。你的 DrawTriangle 就是這樣做的。

多型派發用參考

在 myMiniRender 裡有一個進階用法：

void triangle2(Vec4f* pts, IShader& shader, TGAImage& image, float* zbuffer) {
  shader.fragment(bc, color); // 多型呼叫 — 走虛函式表
}

IShader& shader 用參考而不是指標，因為：(1) shader 不可能是 null — 畫三角形一定要有 shader；(2) 參考也支援多型，只要 IShader 有虛函式，傳進來的派生類別就會走自己的實作。參考和指標都能實現多型，但參考的語法更明確地表達「這個東西一定存在」的意圖。

6 你的渲染器裡的指標地圖

讓我們把你的 Pixel-Renderer 裡所有的指標和參考關係畫出來。每一根線都是一個你現在能看懂的 * 或 &。

三種指標，三種角色：

  • RenderDevice* device — 擁有指標（owning pointer）。Application 用 new 建立它，在解構子裡用 delete device 釋放它。忘記 delete 就是記憶體洩漏。
  • ScreenManager* screen（在 RenderDevice 裡）— 觀察指標（observing pointer）。RenderDevice 只是「看一眼」ScreenManager，不負責它的生命週期。screen 指向的物件是 Application 的成員變數，住在 stack 上，Application 結束時自動銷毀。
  • unsigned char* frame_buffer — 觀察指標。指向 Windows 的 DIB section 記憶體，由 OS 管理。ScreenManager 在 CreateDIBSection 時拿到這個地址，在 Shutdown 時由 DeleteObject 釋放。

至於 const Vertex& v1 和（myMiniRender 的）IShader& shader 都是參考 — 它們不涉及所有權，只是在函式呼叫期間「借用」物件，避免拷貝。

常見陷阱

1. 懸空指標（dangling pointer） — 物件已經被 delete 了，但指標還在。之後用這個指標就是未定義行為。解法：delete 之後把指標設為 nullptr。

2. 忘記檢查 nullptr — 你的 device 初始是 nullptr。如果視窗建立失敗（running == false），device 永遠是 nullptr。對它呼叫 device->DrawTriangle(...) 直接當掉。

3. 搞混 * 的兩種意思 — int* p = &x; 的 * 是宣告（p 是指標），*p = 99; 的 * 是解引用（去那個地址）。新手常把兩者混為一談。口訣：左邊是身分，右邊是動作。出現在型別旁邊就是宣告指標，出現在表達式裡就是解引用。
記住三件事

1. 指標是存放地址的變數。* 在宣告時是「我是指標」，在使用時是「去那個地址」（解引用）。& 在宣告時是「我是參考」，在使用時是「給我地址」。四個含義，不要混淆。

2. 參考是別名 — 綁定後不能改指向、不能為 null。指標可以改指向、可以是 null。

3. 函數參數最常用的模式：小型別傳值，大型別唯讀用 const T&，需要可能為 null 或需要存起來時用 T*。
