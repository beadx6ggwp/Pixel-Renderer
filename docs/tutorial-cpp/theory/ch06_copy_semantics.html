Copy 語意
copy is type operation, not byte movement
copy deep-copy ownership

上一章建立了 RAII：constructor 取得 resource，destructor 釋放 resource。這立刻逼出下一個問題：如果一個 object 擁有 resource，那複製它到底代表什麼？

很多 C++ 錯誤不是因為你不會寫 class，而是因為你把 copy 想成「複製 memory」。對 int、Point 這種 trivial type，這個直覺常常看起來沒問題；但只要 type 開始擁有 heap memory、file handle、window handle、GPU resource，byte-copy mental model 就會直接壞掉。

本章範圍 — 建立 copy semantics 的心智模型：copy 後的 destination 必須是一個可以正常使用、正常銷毀、符合 type 承諾的新 object。copy 是 type 定義的 operation，不是單純 memory operation。

1 容易誤導人的簡單 case

先看一個沒有 resource ownership 的 type：

struct Point {
  int x;
  int y;
};

Point a{1, 2};
Point b = a;

這裡你很自然會想：b.x = a.x、b.y = a.y。這個 mental model 在 Point 上通常沒問題，因為 Point 只包含兩個 int。copy 後 a 和 b 各自有自己的值，兩者都能正常使用，也都不需要釋放任何外部 resource。

陷阱 — Point 讓你以為 copy 就是複製 fields。這不是完全錯，而是只對簡單 value type 夠用。一旦 field 裡有 pointer，pointer 指向的東西是否也要複製，就不是 field copy 能回答的問題。

2 Copy 的真正要求

更有用的定義是：

Copy semantics — copy 後，destination 應該是一個新的 valid object。它可以正常使用、正常銷毀，並且符合這個 type 對 copy 的語意承諾。

這句話刻意沒有說「bytes 要一樣」。因為對 C++ object 來說，copy 的重點不是 representation 像不像，而是 operation 後語意是否成立。

Typecopy 後應該保證什麼可能成本
int兩個 int 值相同O(1)
Point兩個 Point 的欄位值相同O(1)
std::string兩個 string 都能獨立修改與銷毀可能 O(n)
std::unique_ptr不允許 copy，因為它代表 unique ownership不存在
std::shared_ptrcopy 共享 ownership，reference count 增加通常 O(1)

所以問題不是「copy 一定貴嗎？」而是：這個 type 的 copy 承諾是什麼？ 有些 copy 很便宜，有些 copy 正確但昂貴，有些 copy 根本不該存在。

3 C-style Buffer：representation copy 壞掉

現在看一個最小的 resource owner：

typedef struct {
  char* ptr;
  size_t size;
} Buffer;

假設 buffer_create 會配置 heap memory，buffer_destroy 會釋放：

Buffer a = buffer_create(1024);
Buffer b = a;

buffer_destroy(&a);
buffer_destroy(&b);

從 representation 看，Buffer b = a 只是複製兩個 fields：

b.ptr = a.ptr;
b.size = a.size;

但如果 ptr 是 ownership pointer，這件事就危險了。copy 後兩個 object 會指向同一塊 heap memory：

第一個 buffer_destroy(&a) 釋放 heap block 後，b.ptr 還指向同一個已經被釋放的位置。接著 buffer_destroy(&b) 就可能 double free。

錯誤不在 compiler — C struct assignment 正確地複製了 representation。壞掉的是：representation copy 沒有表達 ownership semantics。ptr 是 owner、borrower、view、shared handle，這些意義都藏在 programmer convention 裡。

4 正確 copy 有三種常見設計

如果 shallow copy 會錯，那正確答案不是固定的。你要先問：copy 後 a 和 b 應該是什麼關係？

Option 1：Shallow copy，當 type 是 view

Shallow copy 不是永遠錯。如果 type 的語意本來就是 view，那 copy pointer + size 完全合理。

struct BufferView {
  char* ptr;
  size_t size;
};

BufferView b = a; // copy view，不 copy owned memory

這像 std::span 或 std::string_view：copy view 很便宜，但 view 不負責 resource lifetime。呼叫端必須保證被指向的資料還活著。

Option 2：Deep copy，當 type 是可複製 owner

如果 type 的語意是「copy 後兩邊各自擁有完整資料」，那 copy constructor 就必須配置新的 memory 並複製內容：

class Buffer {
public:
  Buffer(const Buffer& other)
    : size(other.size), ptr(new char[other.size]) {
    std::memcpy(ptr, other.ptr, size);
  }

  ~Buffer() { delete[] ptr; }

private:
  size_t size = 0;
  char* ptr = nullptr;
};

這樣 a 和 b 指向不同 heap block。兩者都能正常 destroy，不會 double free。但代價是 allocate + copy bytes，所以可能很貴，也可能失敗。

Option 3：Deleted copy，當 type 是 unique owner

有些 type 的正確設計是：不允許 copy。像 window handle、file descriptor、GPU texture、unique framebuffer owner，可能就是 unique ownership。

class UniqueBuffer {
public:
  UniqueBuffer(size_t size);
  ~UniqueBuffer();

  UniqueBuffer(const UniqueBuffer&) = delete;
  UniqueBuffer& operator=(const UniqueBuffer&) = delete;
};

這不是少做功能，而是更精準地表達 type 的 operation set：這個 type 沒有 copy operation。你不能不小心寫出 UniqueBuffer b = a;，compiler 會直接擋下來。

5 Copy 是 C++ 的 semantic lifting

C 當然也能做對。你可以規定 Buffer 不能直接 assignment，只能用 buffer_clone() 做 deep copy，用 buffer_destroy() 釋放。問題是這些規則藏在命名、文件和團隊約定裡。

C++ 想做的是把這些 convention 提升成 type operations：

C conventionC++ type operation語意
buffer_createconstructor建立 valid object
buffer_destroydestructor結束 lifetime 並釋放 resource
buffer_clonecopy constructor建立另一個 valid object
禁止直接 assignment= delete把非法 operation 從 type 上移除
第一原則 — Type 不只是 layout。Type = valid states + operations + invariants + cost model。copy 是這組 operations 的一員，必須維持 invariant。
常見陷阱

1. 把 pointer field copy 當成 resource copy — pointer value 複製了，不代表 pointer 指向的 resource 也正確複製了。

2. 以為 shallow copy 一定錯 — 對 owner 可能錯，對 view 可能正確。關鍵是 copy behavior 要符合 type meaning。

3. 只寫 destructor，卻保留 compiler-generated copy — 這通常是 double free 的入口。下一章會正式處理這個壓力。
本章學到的三件事

1. copy 不是 byte operation。copy 後 destination 必須是 valid、usable、destructible 的 object。

2. resource owner 的 copy 必須明確設計：shallow view、deep copy、shared ownership，或 deleted copy。沒有預設正確答案。

3. C++ 把 C 裡靠 convention 維護的 create/destroy/clone/no-copy 規則，提升成 constructor/destructor/copy constructor/= delete 這些 type operations。
