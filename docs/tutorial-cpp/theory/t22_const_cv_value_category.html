Const / cv-qualification / Value Category
value category 說 expression 能不能被當成 movable source；const 說你有沒有 mutation capability
const cv-qualification value-category api-design

T-04 已經建立一個重要模型：expression 同時有 type 與 value category。可是實際寫 C++ 時，你很快會遇到另一個問題：如果 expression 是 xvalue，但它的 type 是 const T，到底能不能 move？如果一個 member function 後面有 const，它只是「我保證不改」，還是改變了 this 的型別？

這不是語法細節。Pixel-Renderer 之後會有 DisplayBackend::Present(const Framebuffer&)、RenderDevice(Framebuffer&)、resource cache lookup、UI command build、shader/material handle。這些 API 都在問同一件事：caller 給你的 expression 是可修改、只讀、可消耗，還是只能短暫觀察？

本章範圍 — 把 const / cv-qualification 和 value category 接起來：top-level const、low-level const、reference binding、std::move(const T&)、const T&&、member function const、mutable、ref-qualified member function、const_cast 的邊界，以及 Pixel-Renderer API 應如何表達 read-only view、mutable borrow、consume/move。

1 naive model：std::move 會 move，const 只是不要改

最常見的直覺是： 

std::move(x) means "move x"
const means "please don't modify this object"

這兩句都太粗。std::move(x) 只是把 expression cast 成 xvalue；真正能不能呼叫 move constructor，要看 overload resolution 和 type qualification。const 也不是禮貌用語，而是 type system 限制：你拿到的是不能透過這條 access path 修改 object 的 capability。

struct Image {
  Image(const Image&) { puts("copy"); }
  Image(Image&&) noexcept { puts("move"); }
};

const Image img;
Image a = std::move(img); // prints "copy", not "move"

原因是 std::move(img) 的結果不是 Image&&，而是 const Image&&。一般 move constructor 是 Image(Image&&)，它需要能修改 source，把 source 放進 moved-from state；但 const Image&& 不允許你修改 source。

重要規則 — xvalue 只表示「這個 expression 可以被視為 expiring value」。它不自動移除 const。如果 source 是 const，你通常不能真正 move from it。

3 top-level const 與 low-level const

const 放在哪裡很重要。尤其 pointer 類型最容易混淆：

寫法意思能不能改 pointer？能不能透過它改 pointee？
uint32_t* ppointer to mutable uint32_t可以可以
const uint32_t* ppointer to const uint32_t可以不可以
uint32_t* const pconst pointer to mutable uint32_t不可以可以
const uint32_t* const pconst pointer to const uint32_t不可以不可以

對 function parameter 來說，top-level const 常常不屬於 function type 的外部差異：

void Draw(Framebuffer* p);
void Draw(Framebuffer* const p); // not a different overload

但 low-level const 會改變你能否透過 pointer/reference 修改 object：

void Present(const Framebuffer& fb); // read-only view
void Clear(Framebuffer& fb); // mutable borrow

4 reference binding：const& 很方便，但不是 ownership

const T& 能綁到 lvalue，也能綁到 temporary。這讓它很適合 read-only parameter：

void DrawRect(const Rect& r);

Rect a{...};
DrawRect(a); // lvalue
DrawRect(Rect{...}); // temporary, OK during call

但這只保證 temporary 活到 full-expression 結束，或在某些 local reference binding 情境延長 lifetime。它不保證你可以把 reference 存進 command buffer 留到下一階段。

struct BadCommand {
  const Rect& bounds;
};

commands.push_back(BadCommand{Rect{...}}); // dangling after statement
renderer rule — Function call boundary 可以用 const T& 讀 temporary；command buffer / UI draw list / deferred render data 應該自己擁有跨 phase 需要的值。

5 move from const：為什麼通常會變 copy

Move constructor 通常長這樣：

Image(Image&& other) noexcept;

它接的是 mutable rvalue reference，因為 move 需要修改 source，把 source 放回可 destruct 的狀態。若你手上只有 const Image&：

void Store(const Image& img) {
  Image local = std::move(img); // const Image&&, cannot bind to Image&& move ctor
}

如果 copy constructor 存在，這通常會 copy；如果 copy 被刪掉，就會編譯失敗。這是好事，因為從 const source 偷走 resource 會違反 caller 對 const 的理解。 

want to observe:
  void Use(const Texture& texture)

want to mutate:
  void Clear(Framebuffer& framebuffer)

want to consume:
  void Submit(CommandBuffer&& commands)
  or void SetTexture(Texture texture)  // by value, then move into storage

6 member function const：改變的是 this 的型別

member function 後面的 const 不是裝飾，它表示在 function body 裡，this 類似於 const T*：

class Framebuffer {
public:
  uint32_t* Data() { return pixels.data(); }
  const uint32_t* Data() const { return pixels.data(); }

private:
  std::vector<uint32_t> pixels;
};

這兩個 overload 表達兩種 capability：mutable framebuffer 可以拿 mutable pixels；const framebuffer 只能拿 read-only pixels。這正好對應 renderer boundary： 

RenderDevice:
  holds Framebuffer&
  may call non-const Data()
  may write pixels

DisplayBackend::Present:
  takes const Framebuffer&
  may call const Data()
  may read/upload/present
  must not mutate renderer-owned pixels

7 mutable：logical const 的例外，不是偷改後門

mutable 允許某個 member 即使在 const member function 中也可修改。它適合用在不改變 logical value 的 cache / instrumentation：

class BoundsCache {
public:
  Rect Bounds() const {
    if (!valid) { cached = ComputeBounds(); valid = true; }
    return cached;
  }
private:
  mutable bool valid = false;
  mutable Rect cached{};
};

但對 renderer hot path 要小心：mutable 不是 thread-safety，也不是「const function 裡想改就改」。如果 const API 表面上 read-only，內部卻修改 shared cache，就要重新考慮 synchronization、lifetime、debug determinism。

8 ref-qualified member function：this 也有 value category

你可以限制 member function 只能在 lvalue 或 rvalue object 上呼叫：

class CommandBuilder {
public:
  CommandBuffer& Buffer() & { return buffer; }
  const CommandBuffer& Buffer() const& { return buffer; }
  CommandBuffer Finish() && { return std::move(buffer); }
private:
  CommandBuffer buffer;
};

這讓 API 更精準： 

builder.Buffer()
  lvalue builder: borrow internal buffer

std::move(builder).Finish()
  rvalue builder: consume builder and return owned CommandBuffer

這是 value category 和 API design 的交界。你不是只靠文件說「Finish 之後不要再用 builder」，而是用 function qualifier 把可呼叫情境寫進 type system。

9 const_cast：只改 access path，不改 object truth

const_cast 可以移除 type 上的 const qualification，但它不會改變 object 原本是否真的 const：

const int x = 3;
int& r = const_cast<int&>(x);
r = 4; // undefined behavior: original object is const

它可以在 legacy C API 邊界被隔離使用，例如 API 宣告不正確地少了 const，但實際不修改資料。它不應該出現在 renderer core 用來繞過 API 設計。

和 T-18 的關係 — const_cast 和 reinterpret_cast 都只是改變 expression/access path 的型別。它們不會創造 object lifetime、不會修正 alignment，也不會讓原本非法的 mutation 變合法。

10 Pixel-Renderer const policy

Read-only present:
  DisplayBackend::Present(const Framebuffer& fb)
  may read pixels, upload/copy to platform resource
  must not mutate renderer framebuffer

Mutable rendering:
  RenderDevice::RenderDevice(Framebuffer& fb)
  SetPixel / Clear mutate framebuffer through explicit mutable borrow

Borrowed view:
  FramebufferView        -> mutable pixels
  ConstFramebufferView   -> read-only pixels
  span         -> mutable contiguous range
  span   -> read-only contiguous range

Consume:
  Submit(CommandBuffer&& commands)
  SetMesh(MeshData mesh) // by value, then move into owned storage

Avoid:
  std::move(const Resource&)
  storing const& in deferred command data
  using const_cast inside renderer core
本章學到的三件事：
  1. Value category 和 const 是兩個軸：xvalue 表示 expression 可被視為 expiring，但不會移除 const；std::move(const T&) 通常不能真正 move。
  2. const 是 mutation capability boundary，不是 ownership、lifetime、thread-safety 或永遠不變的保證；member function const 會改變 this 的可用操作。
  3. Pixel-Renderer API 應明確分出 read-only present、mutable render borrow、borrowed view、consume/move；不要用 const_cast 或 long-lived const& 掩蓋資料生命週期問題。
