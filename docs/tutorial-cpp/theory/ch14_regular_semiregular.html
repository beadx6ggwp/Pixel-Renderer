PART 5 · CHAPTER 14
Regular / Semiregular types
generic algorithm 需要的是 operation laws，不是任意型別
regular concept generic

上一章說 type 是 valid states、operations、invariants、cost model。接下來要問：如果一個 algorithm 不知道你的 concrete type，它憑什麼能正確使用它？

這就是 generic programming 的核心。Template 不是「什麼 type 都可以丟進來」。真正可組合的 generic code 需要 type 滿足一組 operation requirements，也就是它能做哪些操作，這些操作要符合什麼語意。

本章範圍 — 用 Semiregular / Regular 建立 C++ generic programming 的第一個語意詞彙。重點不是背 C++20 concepts 定義，而是理解「像普通值一樣使用」到底要求哪些 operation laws，以及為什麼 resource owner 不一定應該 regular。

1 naive model：template 可以吃任何型別

看到 template 很容易有這個直覺：

template <class T>
void process(std::vector<T>& xs);

表面上 T 可以是任何型別。但 function body 只要做了一個操作，就已經開始要求 T 滿足某種語意：

T backup = xs[0]; // T 必須 copy constructible
xs[0] = xs[1]; // T 必須 assignable
if (xs[0] == backup) { } // T 必須 equality comparable

所以 template 的自由不是沒有規則，而是把規則從 concrete class 名稱改成 operation requirements。

2 從 ordinary value 開始

先看你直覺上最熟悉的 ordinary value：

int a = 3;
int b = a;
b = 4;
bool same = (a == b);

你會期待幾件事：

  • 可以 default construct / copy / assign / destroy。
  • copy 後兩邊都是獨立且有效的 value。
  • assignment 後 destination 代表新 value，source 不被破壞。
  • equality 有穩定語意：相等代表同一個 abstract value。

Regular / Semiregular 就是把「像普通值」這件事變成比較精準的 vocabulary。

3 Semiregular：像 value 一樣能被建立與複製

粗略地說，Semiregular type 像一般 value 一樣可以：

能力 直覺
default construct 可以建立一個預設有效狀態。
copy construct 可以從另一個 value 建立獨立 value。
copy assign 可以讓既有 object 變成另一個 value。
destruct 可以正常結束 lifetime，不需要外部儀式。

例如 renderer 裡的 Color、Vec3、Matrix4x4 通常就應該接近 semiregular：它們是 value type，不 owns unique external resource，copy 有清楚語意。

4 Regular：再加上 equality

Regular 可以先理解成：

regular = semiregular + equality comparable

也就是它不只可以像 value 一樣建立、複製、賦值，還可以問：

a == b

而這個 equality 必須有穩定語意。對 Vec3 來說，equality 可能表示三個分量都相同；但浮點數有精度問題，所以在 renderer 中你可能不想對所有 math type 都隨便提供 operator==，或要非常清楚它是 bit-exact 還是 epsilon-based。

設計提醒 — Regular 不是「越 regular 越高級」。它是一組 operation contract。如果 equality 沒有穩定語意，不要為了符合概念而硬加 operator==。

5 Resource owner 不一定該 regular

現在回到 resource owner：

std::unique_ptr<Texture> texture;

unique_ptr 明確不是 copyable，因為它的語意是 unique ownership。這不是缺陷，而是正確設計：copy unique owner 沒有合理語意。

同樣地，Pixel-Renderer 裡的 DisplayBackend、SDL texture wrapper、Win32 handle wrapper 可能也不該 regular。它們可能應該是 move-only：

copy: deleted
move: ownership transfer
destroy: release handle

這說明一件事：regular / semiregular 是 vocabulary，不是所有 type 的目標。你要先問 type 的語意，再問它應該滿足哪組 requirements。

6 Algorithm 需要 laws，不只是 syntax

Generic algorithm 不只需要語法上能編譯。它需要操作有合理 law。例如 sort 不是只要求你有 operator<；它要求 ordering 像一個 strict weak ordering。否則 sort 可能編譯過，但結果沒有意義。

std::sort(items.begin(), items.end(), compare);

compare(a, b) 必須表達穩定排序關係，而不是每次隨機回傳或依賴會突變的全域狀態。這和 Ch13 的 invariant 觀念是同一件事：operation 要有 laws，algorithm 才能推理。

7 Pixel-Renderer 的落點

這章可以直接幫 Pixel-Renderer 做 type 分類：

Type 可能的 category 設計理由
Color / Vec3 Semiregular，可能 Regular 像普通 value，copy 合理；equality 要看 float 語意。
Framebuffer 視 ownership policy 而定 如果 owns pixels，copy 可能 deep copy 或 delete；move 應清楚。
DisplayBackend Move-only owns platform handle / SDL resource，不應複製。
UICommand Semiregular 適合放進 vector、排序、批次處理；應避免藏 heavy ownership。
Shader interface 通常不是 Regular 可能靠 dynamic dispatch / state / resource；要用 pointer/reference/handle 表達。
架構判斷 — renderer 裡越靠近純資料流的 type，越適合 regular-like value semantics；越靠近 OS/GPU/backend resource 的 type，越可能是 move-only owner 或 non-owning view。

8 和下一章數學型別的關係

接下來會回到 renderer 最常見的 value type：Vec3、Matrix4x4、Color。這些型別看起來只是數學資料，但只要你替它們加上 operator、template、conversion，就開始在定義 operation laws。

  • Vec3 + Vec3 應該是 component-wise addition，且不改變 operands。
  • Matrix4x4 * Vec4 的乘法方向必須和你的座標慣例一致。
  • operator== 對 float math type 是否合理，要分清 bit-exact 和 epsilon-based equality。

也就是說，Ch15 會把 Ch13 / Ch14 的 type contract 拉回 renderer math types：讓 type 像數學物件一樣好用，但不能讓 operator 破壞語意。

常見陷阱

1. 以為 template 沒有限制 — function body 一旦 copy、assign、compare，就已經要求 type 滿足相應 operations。

2. 以為所有 type 都應該 regular — resource owner、backend handle、unique ownership type 不 regular 可能才是正確語意。

3. 只檢查能不能編譯 — generic algorithm 需要 operation laws；錯誤的 equality / ordering 會讓程式編譯過但語意壞掉。
本章學到的三件事

1. Generic programming 的核心是 operation requirements；template syntax 只是讓 requirements 延後到型別代入時檢查。

2. Semiregular / Regular 是描述 ordinary value behavior 的 vocabulary，不是所有 type 的目標。

3. Pixel-Renderer 應該區分 value-like data、move-only resource owner、non-owning view、runtime interface，不能全部用同一種 copy/move policy。
