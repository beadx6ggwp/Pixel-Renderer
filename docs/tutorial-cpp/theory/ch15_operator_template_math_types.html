Operator / Template 與數學型別
讓 Vec / Mat 好用，但不能讓語法破壞語意
operator template Vec

Pixel-Renderer 目前已經有很小的 Vec3f：

struct Vec3f {
  float x, y, z;
  Vec3f(float x = 0, float y = 0, float z = 0) : x(x), y(y), z(z) {}
};

下一步很自然會想：能不能讓它寫成 a + b、m * v、甚至做成 Vec<3, float>？可以。但這一章要先建立一個界線：operator / template 不是為了讓 code 看起來像數學，而是為了把正確的 operation laws 寫進 type。

本章範圍 — 用 renderer math types 連接 Ch13/Ch14：operator 是 type operation，template 是 operation family。你會看到何時該 overload、何時不該 overload、何時 template 能消除重複，何時它只是過早抽象。

1 naive model：operator 只是讓語法漂亮

最常見的想法是：

Vec3f c = Add(a, b);
// 改成
Vec3f c = a + b;

如果只看可讀性，這好像只是語法糖。但從 Ch13 的角度看，operator+ 是在宣告：

Vec3f 的 addition operation 存在，
它回傳一個新的 Vec3f，
它不修改 operands，
它的語意是 component-wise addition。

所以 operator overload 不是名字變短，而是把 operation 放進 type 的語意集合。

2 好的 operator 要符合直覺 law

對 Vec3f，這樣的 operator+ 很合理：

inline Vec3f operator+(const Vec3f& a, const Vec3f& b) {
  return Vec3f{a.x + b.x, a.y + b.y, a.z + b.z};
}

它大致符合幾個期待：

  • non-mutating：a + b 不改變 a 或 b。
  • closed：兩個 Vec3f 相加仍然得到 Vec3f。
  • component-wise：對每個分量做相同操作。
  • cheap enough：回傳小 value 很合理，不需要 heap allocation。

相對地，這種 operator 就很危險：

Vec3f operator+(Vec3f& a, const Vec3f& b) {
  a.x += b.x;
  a.y += b.y;
  a.z += b.z;
  return a;
}

它能編譯，但違反大多數人對 + 的 law：a + b 不應該偷改 a。如果你要 mutation，應該用 operator+=。

operator 的基本準則 — overload 只有在語意符合讀者對該符號的強烈期待時才值得做。語法看起來像數學，行為就必須真的接近數學。

3 Matrix multiplication 不是只差一個符號

對 matrix，operator 更容易藏規則：

Vec4f out = m * v;

這行牽涉的不只是乘法：

  • 你使用 row-major 還是 column-major memory layout？
  • vector 是 column vector 還是 row vector？
  • transform order 是 MVP * position 還是 position * MVP？
  • 座標系是 left-handed 還是 right-handed？

所以 operator* 的語意必須和 renderer pipeline 的慣例一致。不然你會得到最糟的 bug：程式碼看起來非常自然，但實際做了另一套數學。

4 operator 也有 value category 問題

前面幾章講過 prvalue、xvalue、return by value。Operator 也會碰到同一件事。看這行：

Vec3f c = a + b;

如果 operator+ 回傳 Vec3f by value，那 a + b 這個 expression 通常是 prvalue：它代表一個新產生的 Vec3f value，用來初始化 c。

inline Vec3f operator+(const Vec3f& a, const Vec3f& b) {
  return Vec3f{a.x + b.x, a.y + b.y, a.z + b.z};
}

這個設計對小型 math value type 通常是最穩定的 default：

  • operands 用 const&，避免複製，也表達不修改。
  • result 回傳 by value，表達 + 產生新 value。
  • return Vec3f{...} 可以走 prvalue direct initialization，不需要你手動 std::move。
  • Vec3f 是 inline value data，move 不會比 copy 有本質優勢。

你可能會看到更進階的 overload，試圖 reuse rvalue operand：

Vec operator+(Vec&& a, const Vec& b) {
  a += b;
  return std::move(a);
}

這個 overload 的想法是：如果左邊本來就是 temporary，例如 makeVec() + b，就直接修改那個 temporary 再回傳，避免多建一個 value。這對大型 dynamic vector 或 expression template library 可能有價值。

但對本教學目前的 Vec3f / Matrix4x4 階段，我不建議一開始就做這種 overload。它會讓 overload set 和 value category 行為變複雜，收益卻很小。先讓 operator 語意正確、簡單、可被 optimizer 處理，通常更好。

第一版建議 — 小型 renderer math type 先採用 const& operands + return by value。等你真的遇到大型 temporary chain 的效能問題，再考慮 rvalue overload、expression templates 或 SIMD-oriented design。

5 operator== 對 float math type 要小心

Regular type 需要 equality，但 renderer math type 裡的 equality 很微妙：

bool same = (a == b);

這裡的 == 可以有兩種語意：

Equality 語意 適用場景
bit-exact / component-exact 每個 float 分量完全相等。 cache key、deterministic tests、資料是否完全一致。
epsilon-based 誤差小於某個 threshold 就視為相等。 幾何計算、數值演算法容忍誤差。

問題是 epsilon equality 不一定符合 equality relation 的所有 law，尤其 transitivity 可能出問題。因此很多 engine 會提供 NearlyEqual(a, b, eps)，而不是讓 operator== 偷偷做 epsilon compare。

不要濫用 operator== — 如果 equality law 不穩定，就不要硬把它包成 operator==。把誤差容忍寫成明確函數，通常比讓 operator 偷藏 policy 更清楚。

6 Template：把 operation family 參數化

當你有 Vec2f、Vec3f、Vec4f，很容易重複寫很多相似 code。Template 可以把 dimension 和 scalar type 變成參數：

template <int N, class T>
struct Vec {
  T data[N];
};

這看起來很漂亮，但它也帶來新的責任：

  • 哪些 operation 對所有 N 都合理？
  • cross(a, b) 只對 3D 合理，要怎麼限制？
  • Vec<4, float> 是 homogeneous coordinate，還是單純 4D vector？
  • generic code 會不會讓錯誤在 template error 裡爆炸？

Template 不是把所有維度統一就自動變好。它是在建立一族 types，你要確認這族 types 共享同一組合理 operations。

7 何時不要 template

對 Pixel-Renderer 的早期階段，直接保留具名型別可能更清楚：

struct Vec3f { float x, y, z; };
struct Vec4f { float x, y, z, w; };
struct Matrix4x4 { float m[16]; };

原因不是 template 不好，而是目前 renderer pipeline 的語意還在形成。Vec3f 可能代表 position、normal、color RGB；它們 layout 一樣，但 operation 不一定完全一樣。過早用一個泛型 Vec<3, float> 吃掉所有差異，可能會讓 domain semantics 變模糊。

保守策略 — 先讓核心 math types 的語意穩定，再抽 template。當重複已經明確、operation laws 也一致時，template 才是在消除 real duplication，而不是提前建立不確定的抽象。

8 Pixel-Renderer 的落點

這章對後續 renderer 擴充可以整理成：

設計點 建議
Vec3f 可以先加 +、-、scalar multiply、dot；mutation 放在 +=。
Matrix4x4 先明確 row/column convention，再提供 operator*。
Color 不要和 geometric vector 混用；即使都是 3/4 floats，語意不同。
Template Vec 等 Vec2/Vec3/Vec4 的共同 operation 明確後再抽。
Shader input/output 不要只追求 generic；有些 vertex attributes 需要明確 domain names。

9 和下一章 smart pointer 的關係

到這裡，Part 5 完成了 value type 的主線：type contract、regular/value requirements、math operation laws。下一章會切到另一條線：ownership handle。

也就是說，Vec3f 這類 type 應該像 value；但 Texture、DisplayBackend、window handle、SDL texture wrapper 這些不是 value，它們需要用 smart pointer 或 move-only wrapper 表達 ownership policy。

常見陷阱

1. 為了漂亮語法濫用 operator — operator 必須符合讀者對符號的語意期待，否則會製造隱性 bug。

2. 過早寫 rvalue operator overload — 對小型 inline math value，reuse temporary 的收益通常不值得增加 overload 複雜度。

3. 對 float type 隨便定義 operator== — bit-exact 和 epsilon-based equality 是不同 policy，不要混在同一個符號裡。

4. 過早 template 化 Vec/Mat — layout 相同不代表語意相同。先確認 operation laws，再抽 generic type。
本章學到的三件事

1. Operator overload 是 type operation，不只是語法糖；它必須守住數學直覺與 type invariant。

2. Operator expression 也有 value category；小型 math value type 通常先用 const& operands + return by value。

3. Template 可以把一族 operations 參數化，但只有在各型別共享合理 laws 時才是好抽象。
