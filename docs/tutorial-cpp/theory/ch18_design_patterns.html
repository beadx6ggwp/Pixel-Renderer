設計模式速查
pattern 不是模板，而是替 architecture constraint 命名
design-pattern architecture RAII

前面幾章已經講了 ownership、move、smart pointer、virtual dispatch。這些不是分散的語法點，而是 C++ 用來描述大型程式架構的基本材料。

這章把它們收束成幾個對 Pixel-Renderer 真的有用的 design patterns：Strategy、Template Method、RAII、type erasure、Command Buffer。重點不是背名字，而是知道每個 pattern 解決哪一種約束。

本章範圍 — 用 Pixel-Renderer 的未來擴充需求重新理解 design patterns：DisplayBackend 要替換 Win32/SDL/macOS，Application 要固定 frame lifecycle，window/texture/framebuffer handle 要有明確 lifetime，shader/UI layer 需要可替換行為，render frontend/backend 之間可能需要 command stream。

1 naive model：design pattern = 套模板

很多人第一次學 design pattern 會得到一種很危險的模型：

我知道一個 pattern 的名字
所以我要在 code 裡塞進這個 pattern

這會讓架構變重，卻沒有讓語意變清楚。比較好的模型是：

repeated engineering constraint
  - who owns this object?
  - who decides the implementation?
  - who controls the lifecycle?
  - is this behavior or data?
  - must concrete type be hidden?
pattern name is the vocabulary for that shape

也就是說，pattern 不是「貼上某個 class diagram」。pattern 是你在 code review 或設計筆記裡，快速說明某段設計正在解決什麼問題的詞彙。

2 Strategy：替換行為

Strategy 的核心是：caller 只依賴一個 interface，具體 algorithm 或 backend 可以被替換。

class DisplayBackend {
public:
  virtual ~DisplayBackend() = default;
  virtual void PollEvents() = 0;
  virtual bool ShouldClose() const = 0;
  virtual void Present(const Framebuffer& fb) = 0;
};

Win32DisplayBackend 可以用 CreateDIBSection 和 BitBlt。SDLDisplayBackend 可以用 SDL_Texture 和 SDL_UpdateTexture。macOS 版本可以用 Cocoa/Metal/Quartz 的路徑。對上層 Application 來說，它只知道：

backend->PollEvents();
rasterizer.Draw(framebuffer);
backend->Present(framebuffer);

這就是 Strategy 的價值：替換 platform behavior，但不讓 rasterizer 被 platform API 污染。

不是所有地方都該用 Strategy

如果你在 SetPixel 或每個 triangle fragment 上都做 virtual dispatch，成本和 locality 都會變差。Strategy 適合放在 frame boundary、backend boundary、shader stage boundary；不適合無腦塞進 per-pixel hot loop。

3 Template Method：固定流程，開放步驟

Template Method 不是 C++ template。它是一種 object-oriented pattern：base class 固定演算法骨架，derived class override 某些步驟。

目前 Application 的 Run() 就接近這個形狀：

void Application::Run() {
  OnInit();
  while (!screen.ShouldClose()) {
    OnUpdate(delta_time);
    OnRender();
    screen.Present();
  }
  OnClose();
}

Base class 控制順序：init、loop、update、render、present、close。Derived app 只填「這個 demo 要怎麼 update/render」。這種 pattern 適合 engine/application lifecycle，因為 frame loop 順序通常不應該讓每個 demo 隨便改。

Template Method 和 Strategy 的差別

Template Method 是「base class 控制流程，derived class 填步驟」。Strategy 是「caller 持有一個可替換物件，把某個行為委派出去」。前者偏 lifecycle skeleton，後者偏 behavior replacement。

4 RAII：resource lifetime 的預設語言

RAII 常被放進 design pattern 清單，但在 C++ 裡它更像是 resource 管理的基本語法。它的規則很簡單：

constructor acquires resource
destructor releases resource
object lifetime == resource lifetime

對 Pixel-Renderer 來說，resource 不只 heap memory。這些都應該被 RAII 包起來：

Resource Acquire Release 可能的 wrapper
Win32 window / DIBSection CreateWindow, CreateDIBSection DestroyWindow, DeleteObject Win32DisplayBackend
SDL window / renderer / texture SDL_CreateWindow, SDL_CreateTexture SDL_Destroy* SDLDisplayBackend
CPU framebuffer std::vector<uint32_t> allocation vector destructor Framebuffer
future texture/shader resource backend-specific create backend-specific destroy TextureHandle, ShaderHandle

這也是為什麼 Ch16 的 unique_ptr 很重要。它是 heap object 的 RAII handle；而你自己寫的 backend class，則是 OS/SDL resource 的 RAII handle。

5 Type erasure：藏起 concrete type

type erasure 的問題意識是：caller 想操作「某種能做 X 的東西」，但不想知道 concrete type。

Virtual interface 其實就是 type erasure 的一種形式：

std::unique_ptr<DisplayBackend> backend;
// caller 不知道它實際是 Win32DisplayBackend 或 SDLDisplayBackend

std::function 也是 type-erased callable：

std::function<Vec4(const Vertex&)> vertex_shader;
// 可以裝 lambda、function pointer、function object

它和 template 的差別在於：

方法 concrete type 何時決定 常見代價 適合位置
template compile time code bloat、編譯期增加 math type、hot path、inline shader kernel
virtual interface runtime indirection、vtable dispatch backend、tool layer、coarse renderer stage
std::function runtime 可能 allocation、type-erased call overhead callback、editor tool、低頻事件

所以 type erasure 不是「比較高級」。它只是把 concrete type 從使用端拿掉，換來 ABI/API 穩定、runtime 替換能力，並付出 indirection 成本。

6 Command Buffer：把行為變成資料

如果你未來要做 immediate-mode UI、render frontend/backend 分離，或把 software renderer 當成 FPGA golden model，Command Buffer 會比單純 virtual call 更接近 GPU 的世界觀。

直覺上，frontend 不直接呼叫 backend 畫東西，而是產生一串 commands：

struct DrawTriangleCmd {
  Vertex v0, v1, v2;
  ShaderId shader;
};

struct ClearCmd {
  Color color;
};

std::vector<RenderCommand> commands;

然後 backend 或 renderer 在某個時間點 consume：

for (const RenderCommand& cmd : commands) {
  renderer.Execute(cmd, framebuffer);
}

這個設計的本質是：

  • UI layer 可以先產生 DrawRect、DrawText、ClipRect commands。
  • Render frontend 可以收集 Clear、SetShader、DrawIndexed commands。
  • Render backend 可以排序、batch、debug dump、或轉成另一種 backend-specific representation。
  • 未來 FPGA 方向也能把 command stream 對應到 command processor 的輸入。

7 Pixel-Renderer 的 pattern map

把 pattern 放回專案裡，可以得到這張決策表：

需求 適合 pattern C++ 工具 判斷重點
替換 Win32 / SDL / macOS display Strategy DisplayBackend + virtual + unique_ptr frame boundary 上的替換，成本可接受。
固定 init/update/render/present 順序 Template Method Application::Run + virtual hooks 流程由 engine 控制，demo 只填行為。
管理 window、texture、framebuffer resource RAII destructor、move-only wrapper、Rule of Zero/Five resource lifetime 必須跟 object lifetime 綁定。
讓 shader / callback / tool 行為可替換 Strategy 或 type erasure template、virtual、std::function 先判斷 hot path 還是 editor/tool path。
把 UI/render intent 傳給 backend Command Buffer std::vector<RenderCommand>、variant、handle id command 要盡量 self-contained，避免 dangling pointer。

這張表也回答一個常見問題：不是「PixelBuffer 是否就是前後端分離」。PixelBuffer 或 Framebuffer 是資料所有權與 memory layout 的問題；前後端分離還需要清楚的 boundary：誰產生 draw intent？誰持有 resource？誰把 framebuffer present 到 OS/window？

Application / UI / Scene
  - produces intent
  - maybe records commands

Rasterizer / RenderDevice
  - executes software rendering
  - writes owned CPU framebuffer

DisplayBackend
  - owns platform resources
  - presents CPU framebuffer to window

8 不要過早抽象

Pattern 最容易被濫用的地方，是你還沒有真實變化點，就先把所有東西切成 interface。這會讓 code 看起來很「架構化」，但每次修改都要穿過更多檔案。

比較務實的規則是：先找出真的會改變的軸。

  • Win32 / SDL / macOS 一定會變，所以 DisplayBackend 是合理 boundary。
  • per-pixel SetPixel 目前只需要穩定、可 inline、memory layout 清楚，不需要 virtual。
  • Shader 如果未來要學 GPU pipeline，初期可用簡單 interface；等 hot path 成本明確，再評估 template 或 function object。
  • UI 如果要 immediate-mode，command list 很自然，因為 UI 每 frame 產生的是 draw intent。
常見陷阱
  • 先選 pattern，再硬把問題塞進 pattern。
  • 用 shared_ptr 或 type erasure 掩蓋 ownership 邊界不清楚。
  • 在 per-pixel hot loop 使用 virtual dispatch，讓抽象成本進入最內層。
  • Command Buffer 裡塞裸 pointer，卻沒有定義 pointer 指向資料的 lifetime。
本章三個 takeaways
  1. Design pattern 是 recurring constraint 的名稱，不是要貼進 code 的模板。
  2. Strategy、Template Method、RAII、type erasure、Command Buffer 分別處理 dispatch、lifecycle、ownership、type hiding、behavior-as-data。
  3. Pixel-Renderer 應該把 pattern 放在真實邊界：DisplayBackend、Application loop、resource handles、shader stage、UI/render command stream。
