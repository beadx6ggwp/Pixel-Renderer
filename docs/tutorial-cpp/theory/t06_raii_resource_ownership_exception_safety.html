RAII / Resource Ownership / Exception Safety
resource lifetime should be a type invariant, not a cleanup convention
RAII unique_ptr exception-safety

T-05 說明 copy/move 決定 object 如何被交付。可是 resource-owning type 還有更基本的問題：resource 什麼時候取得？什麼時候釋放？如果中途 return、建構失敗、或發生 exception，誰保證 cleanup 還會發生？

這就是 RAII 的核心。它不是「C++ 會自動 delete」這麼淺，而是：把 resource lifetime 綁到 object lifetime，讓 constructor 建立 valid state，destructor 自動釋放 resource，並讓 ownership policy 寫進 type。

本章範圍 — 從 C-style acquire/release 的失敗模式推到 RAII；整理 constructor/destructor、destruction order、Rule of Zero/Five、unique_ptr、shared_ptr、observer pointer/reference、custom deleter、move-only handle、exception safety guarantee，最後回到 Pixel-Renderer 的 Framebuffer、Win32DisplayBackend、SDL_Texture wrapper、Application ownership map。

1 naive model：記得在最後 release 就好

C-style resource discipline 常長這樣：

Handle h = Acquire();
if (!h) return false;

DoWork(h);

Release(h);
return true;

問題是 control flow 很快會變複雜： 

Acquire A
Acquire B
if something fails -> release B? release A?
if early return -> release what?
if constructor halfway fails -> what has been acquired?
if exception happens -> which cleanup path runs?

你可以靠 goto cleanup、手動 flag、文件約定解決，但 C++ 給你的核心工具是 RAII。

第一個修正 — cleanup 不應該靠「記得」。如果 resource 有 owner，owner 的 destructor 就應該是 release path。

2 RAII：object lifetime owns resource lifetime

RAII 是 Resource Acquisition Is Initialization。名字看起來像歷史包袱，但工程意思很直接： 

constructor:
  acquire resource
  establish invariant

destructor:
  release resource
  must be safe during scope exit

object lifetime:
  resource is valid while object is valid

對 framebuffer，最簡單的 RAII member 是 std::vector：

SVG：RAII 的重點是把 cleanup 放進 destructor，所以 normal return、early return、exception unwinding、partial construction failure 都會走同一個 ownership 規則；設計時要同時看 constructor invariant、destructor safety 和 member destruction order。
class Framebuffer {
public:
  Framebuffer(int w, int h)
    : width_(w), height_(h), pixels_(static_cast<size_t>(w) * h) {}

private:
  int width_ = 0;
  int height_ = 0;
  std::vector<uint32_t> pixels_;
};

這個 class 沒有手寫 destructor，但它仍然是 RAII：std::vector 的 destructor 會釋放 owned memory。這就是 Rule of Zero 的精神。

3 destruction order 是 lifetime design

C++ member destruction order 是 member declaration order 的反向。這對 Pixel-Renderer 很重要，因為很多 object 是 borrow 關係。

class Application {
  Framebuffer framebuffer;
  RenderDevice device; // borrows framebuffer view
  Rasterizer rasterizer; // borrows device
};
// destroy order: rasterizer -> device -> framebuffer

如果 Rasterizer 借用 RenderDevice，那 Rasterizer 必須比 RenderDevice 早死。宣告順序不是排版細節，而是 lifetime contract。

第二個修正 — RAII 不只處理 heap memory。member declaration order、borrow relation、destructor order 都是 ownership design 的一部分。

4 Rule of Zero：優先讓 member 管 resource

Rule of Zero 的意思是：domain type 儘量不要自己寫 destructor/copy/move，而是把 resource 放進已經有正確 RAII 語意的 member。

Resource 優先 member 原因
dynamic array / pixels std::vector<T> 自帶 copy/move/destroy/capacity 管理。
unique heap object std::unique_ptr<T> move-only ownership，destroy 時 delete。
string data std::string 避免 raw char* ownership。
draw command list std::vector<DrawCommand> container owns per-frame commands。

例如 Framebuffer 用 std::vector<uint32_t>，你就不需要手寫 destructor。你只要決定 copy 是否公開：

Framebuffer(const Framebuffer&) = delete;
Framebuffer& operator=(const Framebuffer&) = delete;
Framebuffer(Framebuffer&&) noexcept = default;
Framebuffer& operator=(Framebuffer&&) noexcept = default;

5 Rule of Five：只有 raw resource owner 才手寫整套

當你真的 owns C API / OS API resource，標準 library member 不能直接幫你釋放，就需要自己寫 RAII wrapper。

class Win32Bitmap {
public:
  ~Win32Bitmap() { if (bitmap_) DeleteObject(bitmap_); }

  Win32Bitmap(const Win32Bitmap&) = delete;
  Win32Bitmap& operator=(const Win32Bitmap&) = delete;

  Win32Bitmap(Win32Bitmap&& other) noexcept
    : bitmap_(other.bitmap_) { other.bitmap_ = nullptr; }

private:
  HBITMAP bitmap_ = nullptr;
};

這個 wrapper 是 move-only handle。copy 被 delete，因為兩個 wrapper 不能同時 owns 同一個 HBITMAP。move 要把 source 清成 null，避免 double release。

Rule of Five 不是現代感 — 它是 raw resource owner 的負擔。能用 Rule of Zero 就用 Rule of Zero；只有包 C/OS/SDL handle 時才手寫。

6 unique_ptr、shared_ptr、observer：不要混淆 ownership

Smart pointer 的重點不是自動 delete，而是把 ownership policy 寫進 type。

Type 語意 Pixel-Renderer 用途
std::unique_ptr<T> unique ownership，move-only。 Application owns polymorphic DisplayBackend。
std::shared_ptr<T> shared ownership，copy increases ref count。 真的需要多個 owner 延長 resource lifetime 時才用。
T* / T& borrow / observer，不 owns。 Rasterizer borrows RenderDevice。
std::span<T> borrowed range view。 framebuffer pixels view、vertex buffer view。
不要用 shared_ptr 掩蓋 ownership 不清 — 如果你不知道誰 owns backend、texture、command data，先畫 ownership map。Reference count 會讓 lifetime 更分散，不會讓設計更清楚。

7 custom deleter：C API resource 也能放進 RAII handle

有些 resource 不是用 delete 釋放，例如 SDL texture 要 SDL_DestroyTexture，Win32 DC 要 DeleteDC 或 ReleaseDC。可以用 custom deleter 表達：

struct SdlTextureDeleter {
  void operator()(SDL_Texture* texture) const {
    if (texture) SDL_DestroyTexture(texture);
  }
};

using SdlTexturePtr = std::unique_ptr<SDL_Texture, SdlTextureDeleter>;

這樣 SDL texture 也變成 move-only RAII object。你不需要在每個 early return path 手動呼叫 destroy。

8 exception safety：先保持 invariant，再談功能

Exception safety 不是只有使用 exception 的專案才需要。它其實是在問：operation 失敗時，object 會不會留下破掉的 invariant？

Guarantee 意思 例子
basic guarantee 失敗後 object 仍 valid，沒有 leak。 backend init 失敗後 destructor 安全。
strong guarantee 失敗後狀態像沒發生過。 copy assignment 先配置新 buffer，成功後 swap。
no-throw guarantee operation 不會 throw。 destructor、move constructor for handle。

RAII 的力量是：只要每個 resource 都被 object owns，部分建構成功後若後面失敗，已經建好的 member 會自動依反向順序 destruct。 

construct Application
  construct backend
  construct framebuffer
  construct device
  construct rasterizer fails

C++ automatically destroys:
  device
  framebuffer
  backend
destructor 通常不應 throw — destructor 會在 scope exit、stack unwinding、partial construction failure 中被呼叫。如果 destructor throw，錯誤處理會變得不可控，甚至 terminate。

9 Pixel-Renderer ownership map

把 T-06 套回 I-08 的目標架構： 

Application
  owns DisplayBackend            unique_ptr
  owns Framebuffer               value member or unique owner
  owns RenderDevice              value member, borrows FramebufferView
  owns Rasterizer                value member, borrows RenderDevice

Framebuffer
  owns pixels                    vector

Win32DisplayBackend
  owns HWND/HDC/HBITMAP          RAII wrappers or custom destructor

SDLDisplayBackend
  owns SDL_Window/Renderer/Texture
                                  unique_ptr with custom deleter or wrapper classes

Rasterizer
  owns no platform resource       borrow RenderDevice

這張圖比「要不要用 smart pointer」重要。Smart pointer 只是其中一種表示法；真正的設計是誰 owns、誰 borrows、誰只 present。

10 verification：檢查 RAII 是否真的成立

  • 每個 new、Create*、SDL_Create* 是否有明確 owner？
  • owner destructor 是否釋放 resource？
  • copy 是否被正確 delete 或定義？
  • move 後 source 是否 safe-to-destruct？
  • constructor 中間失敗時，已取得的 resource 是否會被釋放？
  • borrowed pointer/reference 是否比 owner 活得短？
  • destructor 是否避免 throw？

本章學到的三件事：
  1. RAII 的本質是把 resource lifetime 綁到 object lifetime；constructor 建立 invariant，destructor 釋放 resource。
  2. 現代 C++ 優先 Rule of Zero；只有包 raw C/OS/SDL handle 時才承擔 Rule of Five，並讓 copy/move/noexcept policy 明確。
  3. Pixel-Renderer 要先畫 ownership map：Application owns backend/framebuffer，RenderDevice/Rasterizer borrow，backend resource 用 RAII wrapper 管，shared_ptr 不能拿來掩蓋不清楚的 ownership。
