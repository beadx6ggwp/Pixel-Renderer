Return by value 與 RVO
有時候不是 copy，也不是 move，而是直接形成 result object
RVO NRVO copy-elision

前面已經拆過 copy、move、std::move、value category。現在回到 C++ 最容易被低估的能力：return by value。如果你要寫 renderer factory、load texture、build mesh、產生 UI draw list，常常會想問：回傳一個大 object 會不會很貴？

naive 答案通常是「會 copy，所以要改成 pointer 或 out parameter」。但 modern C++ 的核心不是逃避 value return，而是理解 result object 到底怎麼被初始化。

本章範圍 — 建立 return-by-value 的 decision tree：return T{};、return x;、return std::move(x); 是三條不同 path。重點不是背最佳化名詞，而是分清楚 copy、move、direct construction、NRVO fallback。

1 naive model：return 一定先做完再搬回去

先看這段：

TextureData loadTexture() {
  return TextureData{};
}

TextureData data = loadTexture();

很直覺的模型會畫成：

callee temporary
  | move
  v
function return object
  | move
  v
caller variable data

這個模型的問題是：它預設一定先有一個獨立 source object，然後再 transfer 到 caller。但 return-by-value 不一定是這樣。

2 return T{}：有時候沒有東西可以 move

對這種形狀：

T make() {
  return T{};
}

T{} 是 prvalue。C++17 之後，這類 same-type prvalue return 的核心直覺不是「先建立 temporary，再 move」，而是：

return T{} 的好模型

function 需要產生一個 T result；T{} 可以直接初始化那個 result object。這裡沒有必要先建立一個獨立 source object，再從 source move。

所以這句話很重要：

Sometimes there is nothing to move.

Move 的前提是有一個 existing source object，destination 接手 source 的 resource。但 return T{}; 這條 path 更像是「直接在 result storage 形成 object」。

3 一個反直覺測試：delete copy/move 仍然可 return

如果 return T{}; 一定需要 move，那下面這段應該不能成立：

struct Token {
  Token() = default;
  Token(const Token&) = delete;
  Token(Token&&) = delete;
};

Token make() {
  return Token{};
}

在 C++17 的 same-type prvalue return 情境裡，這個例子可以成立，因為不需要 copy constructor，也不需要 move constructor 來完成這個 return。這不是「compiler 很努力省掉 move」，而是 source-level semantics 不要求那個 move 存在。

不要過度簡化 — 這個結論不是「所有 return 都不用 copy/move」。它只是在說：same-type prvalue return 是一條很重要的 no-transfer path。

4 return x：NRVO candidate

現在看另一個常見形狀：

T make() {
  T x;
  return x;
}

這裡真的有一個 named local object x。所以它和 return T{}; 不是同一條 path。可是它也不等於「一定 copy」。

return x; 這個形狀讓 x 成為 NRVO candidate。如果 compiler 採用 NRVO，概念上可以把 x 從一開始就建構在 function result object 的 storage 裡。

如果 NRVO 發生，x 不是先在 callee stack 上完整出生，再搬到 caller。更好的 mental model 是：x 的 construction 被安排到 result object 的位置。

5 如果 NRVO 沒發生呢？

NRVO 和 return T{}; 的 guaranteed prvalue path 不完全一樣。Named local return 通常是允許 compiler 做，但不是每個 shape 都保證。

所以要補上 fallback：

return x;

// mental model:
try NRVO first;
if not elided, move if possible;
otherwise copy if possible;

這不是完整標準 wording，但對第一輪理解很夠用。重點是：return x; 不是普通的 T y = x;。Return statement 對 local object 有自己的規則，compiler 不需要你用 std::move 告訴它「x 快死了」。

6 return std::move(x)：通常不是比較保險

現在比較：

T make() {
  T x;
  return x;
}

T makeMove() {
  T x;
  return std::move(x);
}

如果只用「copy 慢、move 快」思考，你會想寫 return std::move(x);。但上一章已經說過，std::move(x) 只是把 return expression 變成 xvalue。它不是幫 compiler 最佳化。

寫法 保留的 path 代價
return x; 保留 NRVO candidate shape；失敗時再 move/copy fallback。 需要理解 return statement 不是普通 lvalue copy。
return std::move(x); 明確產生 xvalue，可以選 move constructor。 通常破壞 plain local name 的 NRVO shape，可能把 no-transfer path 關掉。
核心 guideline — 回傳 local object by value 時，通常寫 return x;。不要把 return std::move(x); 當成保險習慣；它可能是 pessimizing move。

7 什麼時候 return std::move 是有語意的？

這章不是說 return std::move(...) 永遠不該出現。真正規則是：你要看自己是不是正在放棄一個更好的 no-transfer path。

例如這不是 local NRVO case：

TextureData takeTexture(TextureOwner& owner) {
  return std::move(owner.texture);
}

這裡你不是在 return local variable by name。你是在明確從 owner.texture 轉出 ownership。這時候 std::move 是語意訊號：callee 會消耗 owner 裡的 resource。

判斷問題 — 如果 return expression 是 local automatic object 的單純名稱，先保留 NRVO shape。只有在你真的要從某個 existing object 轉出 ownership，而且沒有更好的 direct-construction path 時，才使用 std::move。

8 Pixel-Renderer 的落點

這會直接影響 Pixel-Renderer 後續 API 設計：

  • factory function：Framebuffer makeFramebuffer(...)、TextureData loadTexture(...) 可以清楚地 return by value，不必先退回 out parameter。
  • resource loader：如果 function 內建立 local result，通常 return result;，保留 NRVO 與 fallback move。
  • resource transfer：如果要從 cache entry、owner member、container slot 轉出 resource，std::move 才是明確 ownership transfer。
  • UI draw data：一個 frame 的 command buffer 可以用 value return 表達「產生結果」；不要因為怕 copy 過早改成指標式 API。
常見陷阱

1. 把 RVO 當成 compiler trick — 對 C++17 prvalue，很多情況是語言語意上直接初始化 result object，不是先有 temporary 再省掉 move。

2. 以為 return x 一定 copy — local by-name return 是 NRVO candidate，fallback 也可能 move。

3. 以為 return std::move(x) 比較現代 — 它通常會破壞 NRVO shape，從 no-transfer path 退成 explicit move path。
本章學到的三件事

1. return T{}; 這種 same-type prvalue return 可以直接初始化 result object；有時候根本沒有 source object 可以 move。

2. return x; 保留 NRVO candidate shape；如果沒有 elide，return rules 仍可能走 move/copy fallback。

3. return std::move(x); 不是 return local 的安全預設；它表達 explicit xvalue path，常常放棄更好的 no-transfer path。
