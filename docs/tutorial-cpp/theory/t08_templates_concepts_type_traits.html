template concepts type-traits generic

T-07 講 callable 時已經看到一個很重要的分岔：shader hot loop 可能不想用 std::function，而是用 template 接 concrete functor，讓 compiler inline。這會立刻帶出下一個問題：如果 template 可以接「任何型別」，那怎麼保證傳進來的東西真的像 shader？

這就是泛型的核心。Template 不是文字替換魔法，也不是「我懶得寫型別」。它是把 function/class 的某些部分參數化，然後要求被代入的 type 滿足一組 operations、laws、layout 或 lifetime policy。

本章範圍 — 從 function/class template 的 instantiation model 開始，建立 type requirement、concept、type traits、if constexpr、policy-based design、CRTP 的基本語意。重點會回到 Pixel-Renderer：math type、shader functor、framebuffer view、command buffer、backend policy 什麼時候適合 template，什麼時候應該用 virtual 或 concrete type。

1 naive model：template 只是複製貼上

初學 template 很容易把它想成： 

template source code
  compiler copies it
  replaces T with Vec3
  done

這個模型有一點點直覺價值，但很快會失敗。因為 template 的重點不只是「產生很多版本」，而是 function body 裡的每個 operation 都在對 T 提要求。

template <class T>
T Add(T a, T b) {
  return a + b;
}

這段程式不是「任何 T 都能用」。它要求 T：

  • 可以被 copy 或 move 進參數。
  • 支援 a + b。
  • a + b 的結果可以回傳成 T。
  • 最好有合理的 addition law，否則語法能編譯也不代表泛型演算法有意義。
錯誤模型 — Template 不是無限制。限制只是從 function signature 裡的 concrete type，移到 function body 與 concepts 裡的 operation requirements。

2 template instantiation：compile-time 產生具體版本

當你寫：

auto a = Add(Vec3{}, Vec3{});
auto b = Add(Color{}, Color{});

compiler 會根據實際使用的 type 產生對應 specialization。可以先用這個結構理解： 

source template:
  template<class T>
  T Add(T a, T b) { return a + b; }

instantiated when used:
  Vec3  Add<Vec3>(Vec3 a, Vec3 b)
  Color Add<Color>(Color a, Color b)

所以 template 常常適合 hot path：compiler 看得到具體型別，就有機會 inline、constant fold、移除分支。但代價是 compile-time 變長、error message 變複雜、binary 可能因多個 instantiation 變大。

3 type requirement：泛型真正依賴什麼

比起問「這個 function 是不是 template」，更重要的是問： 

This algorithm requires T to support what?
Generic code 做了什麼 它要求 type 滿足什麼 Pixel-Renderer 例子
T x{}; default initializable 清空 temporary accumulator。
T y = x; copyable value-like Vec3、Color。
x + y addable，且語意合理 math vector / color accumulation。
shader.Vertex(v) shader callable contract software shader stage。
view.Data(), view.Pitch() framebuffer view contract present/upload helpers。

這就是 concepts 出現的理由：不要讓 requirement 藏在 template body 裡，等錯誤爆在很深的 instantiation stack；而是把 requirement 寫在 interface 附近。

4 concepts：把 requirement 寫成可讀的 contract

Concept 可以把「某型別必須支援哪些操作」命名：

template <class T>
concept Addable = requires(T a, T b) {
  { a + b } -> std::same_as<T>;
};

template <Addable T>
T Add(T a, T b) {
  return a + b;
}

這樣的好處不是讓 code 變短，而是讓錯誤類型更早、更接近問題本身。如果 TextureHandle 被丟進 Add，錯誤應該是「不滿足 Addable」，不是在函式深處看到一串 operator overload 失敗。

Concept 的語意 — 它是 compile-time predicate：某 type 是否滿足一組 syntactic requirements。真正的 semantic laws 仍然要靠你設計與測試，例如 addition 是否 associative、equality 是否穩定。

5 type traits：查詢型別性質

Concept 常常描述「能做什麼操作」。type traits 則常用來查詢型別本身的性質：

std::is_trivially_copyable_v<Pixel>
std::is_move_constructible_v<Framebuffer>
std::is_same_v<ColorFormat, ARGB8888>

對 renderer 來說，traits 的用途很實際：

  • 判斷某 pixel type 是否可以安全地 bulk copy。
  • 限制某 command type 必須 move constructible，才能放進 command buffer。
  • 根據 color format 選不同 upload path。
  • 用 static_assert 把不合法的 type 早點擋下來。
template <class Pixel>
void UploadPixels(const Pixel* pixels, int count) {
  static_assert(std::is_trivially_copyable_v<Pixel>);
  // safe to treat as bytes for upload boundary
}

6 if constexpr：compile-time branch

if constexpr 是 compile-time branch。沒被選到的 branch 不會被 instantiation 成需要合法的 runtime code。

template <class Pixel>
void ClearBuffer(Pixel* pixels, int count, Pixel value) {
  if constexpr (std::is_trivially_copyable_v<Pixel>) {
    std::fill(pixels, pixels + count, value);
  } else {
    for (int i = 0; i < count; ++i) pixels[i] = value;
  }
}

這不是為了把所有東西都塞進一個超大 template。它適合處理少數清楚的 compile-time policy。當分支太多、語意太模糊時，反而應該拆成明確 type 或 function。

7 shader functor：template 適合 hot path

如果 fragment shader 每個 pixel 都會呼叫，virtual 或 std::function 的彈性可能不是第一選擇。教學版可以先簡單，但你要知道 template shader 的語意：

template <class Shader>
void DrawTriangle(const Triangle& tri, Shader& shader) {
  // edge function / barycentric loop
  Color c = shader.Fragment(/* interpolated data */);
  device.SetPixel(x, y, c);
}

這會讓 Shader 的具體型別在 compile time 可見，compiler 有機會 inline Fragment。但如果 shader 要在 runtime 動態替換，或要從外部 script/material system 載入，就可能需要 virtual/type erasure/command representation。這不是誰比較高級，而是替換時間點不同。

替換時間 工具 適合場景
compile time template / concept / policy math、shader hot loop、pixel format policy。
runtime frame boundary virtual interface Win32 / SDL / macOS backend。
runtime low-frequency callback std::function UI button、debug action。
replay/sort/batch data command UI command buffer、render command stream。

8 policy-based design：把變化點變成 type parameter

Policy-based design 的想法是：如果某個選擇在 compile time 就能決定，而且會影響 hot path，可以把它放成 template parameter。

template <class DepthPolicy, class BlendPolicy>
class RasterPipeline {
public:
  void WritePixel(int x, int y, Color src) {
    if (depth_.Pass(x, y)) {
      Color out = blend_.Apply(src, target_.Get(x, y));
      target_.Set(x, y, out);
    }
  }
private:
  DepthPolicy depth_;
  BlendPolicy blend_;
};

這種設計適合 renderer core 的內部變化點，但不適合 platform backend selection。因為你不想為了在 runtime 選 Win32/SDL 重新編譯整個 app；那種邊界用 virtual interface 比較自然。

9 CRTP：static polymorphism，不是繼承萬靈丹

CRTP 常被介紹成「沒有 virtual cost 的 polymorphism」：

template <class Derived>
class ShaderBase {
public:
  Color Fragment(const FragmentInput& in) {
    return static_cast<Derived&>(*this).FragmentImpl(in);
  }
};

但它的本質不是魔法，而是 base template 在 compile time 知道 derived type。這適合建立 shared helper code 或 compile-time interface，但也會讓 error message 和 inheritance relation 變複雜。第一版 shader functor 通常不必急著用 CRTP；普通 concrete functor + concept 就夠清楚。

10 什麼時候不要 template

Template 是強工具，也很容易過早抽象。以下情況先不要 template：

  • 只有一個 concrete type，而且還不知道未來變化點。
  • layout 一樣但語意不同，例如 position、normal、color 都是三個 float，但 operation laws 不完全相同。
  • 錯誤訊息會把初學者帶離真正問題。
  • runtime 真的需要替換物件，例如 display backend。
  • 你只是想少打幾行 code，但還沒確認 abstraction 的 invariant。
過早泛型化的壞處 — 它會把 domain semantics 抹平。Vec3f、Normal3f、ColorRGB representation 可能相同，但 type meaning、valid operations、conversion policy 不一定相同。

11 Pixel-Renderer 的泛型地圖

系統位置 建議工具 原因
math helpers function/class template + concepts operation laws 清楚、hot path 可 inline。
pixel format conversion type traits / policy format 通常是 compile-time policy。
shader hot loop template callable / concrete functor 避免 per-pixel dynamic dispatch。
DisplayBackend virtual interface frame boundary，runtime selection 合理。
UI command buffer concrete value types / variant later 需要 sort/replay/batch，不適合藏任意 lambda。
FramebufferView concrete view first, template later 先穩定 pitch/format/lifetime contract，再抽 pixel type。

12 Summary

本章學到的三件事：
  1. Template 的核心不是「任意型別都可以」，而是把 algorithm 對 type 的 operation requirements 延到 compile time 檢查。
  2. Concepts 把 requirements 命名，type traits 查詢型別性質，if constexpr 讓少數 compile-time policy 分支保持可控。
  3. Pixel-Renderer 裡 hot path 和 math/pixel policy 適合 template；DisplayBackend 這種 runtime platform 替換點更適合 virtual interface。
