THEORY TRACK · T-05
Copy / Move / Object Transfer
copy 是 duplication，move 是 transfer，return by value 常常是 direct construction
copy move copy-elision

T-04 已經把 value category 說清楚：std::move(x) 只是把 expression 轉成 xvalue，不會自己搬東西。T-22 接著補上 const/cv-qualification：如果 source 是 const，xvalue 也不一定能被 move-from。T-23 又補了一層：call expression 會先經過 overload resolution、conversion、operator selection。T-24 再補上 deduction：auto / decltype(auto) 可能 copy、borrow 或保留 reference。那真正搬東西的是什麼？答案不是某個魔法語法，而是 type 自己定義的 object transfer operations：copy constructor、copy assignment、move constructor、move assignment，以及 return-by-value 的 result object formation。

這章要回答 Pixel-Renderer 會一直遇到的問題：當 framebuffer、texture data、command buffer、math value、backend handle 在 function、container、factory 之間流動時，object 的語意到底是複製、轉移、借用，還是直接在目標位置形成？

本章範圍 — 建立 object transfer 的 decision tree。你會分清 copy vs move、construction vs assignment、memberwise default operation vs custom operation、moved-from state、copy elision、NRVO、return std::move(x) 的風險、noexcept move 對 container 的影響，以及 Pixel-Renderer type 應該如何決定 copyable、move-only、borrowed view 或 Rule of Zero。

1 naive model：copy 慢，move 快

很多人學 move 後會把世界分成兩格： 

copy = slow
move = fast

這個模型有時會猜中結果，但它抓錯第一原則。Move 的本質不是「比較快的 copy」，而是 source object 的舊內容可以被放棄，所以 destination 可以接手它的 resource。

Operation 第一問題 語意 典型後果
copy source 之後還要保留原本語意嗎？ duplication 或 shared ownership。 source 和 destination 都 valid，通常都保留可用內容。
move source 可以被消耗嗎？ ownership/state transfer。 destination 接手；source 仍然活著，但內容通常 unspecified。
elision 是否根本不需要 source object？ direct result construction。 沒有 copy，也沒有 move；object 直接在目標 storage 形成。
第一個修正 — C++ object transfer 不是效能標籤，而是語意選擇。先問 type 的 ownership/invariant，再問成本。

2 Transfer 之前先問：object 擁有什麼？

T-02 說 type 是 valid states + operations + invariants；T-03 說 object 有 storage 與 lifetime。現在把它們合起來：copy/move 的設計取決於 object 擁有的是 value、resource、view，還是 platform handle。

Type kind 例子 Copy Move 直覺
small value Vec2, Color, Rect 通常保留。 通常和 copy 差不多。 資料就在 object 本體裡，沒有外部 resource 可偷。
owning buffer Framebuffer, Image, MeshData 可能昂貴，需明確 deep copy 或 delete。 通常有意義，可轉移內部 vector/heap block。 copy/move 都是 type policy。
borrowed view std::span<uint32_t>, pointer + pitch cheap shallow copy。 沒太大意義。 copy view 不 copy resource，但 lifetime 由別人保證。
unique handle Win32 HWND, SDL texture wrapper 通常 delete。 可能允許 move-only。 不能讓兩個 object 同時以為自己負責釋放同一個 handle。

因此 Framebuffer fb2 = fb1; 不是語法問題，而是設計問題：你希望它複製一整張畫面？共享同一塊 pixels？還是禁止，避免不小心複製幾 MB 的 buffer？

3 Copy construction vs copy assignment

copy constructor 和 copy assignment 常被混在一起，但它們處理的是兩種不同狀態： 

copy construction:
  destination does not exist yet
  source object -> construct destination

copy assignment:
  destination already owns some state
  replace destination's current state with source's state
Image a = LoadImage("a.png");
Image b = a; // copy constructor: b was not alive before

Image c = LoadImage("c.png");
c = a; // copy assignment: c already owns old pixels

這就是 assignment 比 construction 更容易寫錯的原因。左邊 object 已經活著，可能已經擁有 resource，所以 assignment 必須處理三件事：

  1. 不能在 self-assignment 時破壞自己。
  2. 不能先釋放舊 resource，再做可能失敗的新配置。
  3. 最後要讓 destination 回到 valid state。
Image& operator=(const Image& other) {
  if (this == &other) return *this;

  std::vector<uint32_t> new_pixels = other.pixels;
  width = other.width;
  height = other.height;
  pixels = std::move(new_pixels);
  return *this;
}

這裡刻意先建立 new_pixels。如果 allocation 失敗，原本的 object 還沒被拆掉。這已經把 T-06 會正式處理的 exception safety 帶進來了。

4 Move construction：轉移 ownership，不是瞬移 object

Move constructor 的 source 是 xvalue/prvalue。它表示 caller 給你一個可以被消耗的 source，但 source object 的 lifetime 仍然存在。

class CpuBuffer {
public:
  CpuBuffer(size_t n) : ptr(new uint32_t[n]), count(n) {}
  ~CpuBuffer() { delete[] ptr; }

  CpuBuffer(CpuBuffer&& other) noexcept
    : ptr(other.ptr), count(other.count) {
    other.ptr = nullptr;
    other.count = 0;
  }

  CpuBuffer(const CpuBuffer&) = delete;
  CpuBuffer& operator=(const CpuBuffer&) = delete;

private:
  uint32_t* ptr = nullptr;
  size_t count = 0;
};

move 後的 other 沒有消失。它只是被放進一個 valid but unspecified 的狀態。最少要求是：destructor 可以安全執行；通常也希望 assignment 可以讓它重新取得新內容。

move constructor 的最低責任 — 如果 destination 接手 source 的 resource，source 必須被改成不會再次釋放同一份 resource 的狀態。否則 move 只是 disguised shallow copy。

5 Compiler-generated operations：不是免費正確

C++ 會幫 class 生成一些 special member functions，但它只能根據 members 做 memberwise operation。Compiler 不知道某個 uint32_t* 是 owner、borrower、view，還是 C API 暫借出來的 pointer。

struct BadFramebuffer {
  uint32_t* pixels;
  int width;
  int height;
};

BadFramebuffer b = a; // memberwise copy: copies pointer value only

如果 pixels 是 view，這可能合理。如果 pixels 是 owner，這是 double free 的入口。如果 pixels 指到 Win32 DIBSection memory，copy 出去還可能讓你誤以為 renderer core 擁有 platform memory。

你寫的 member compiler 看到 compiler 看不到 設計動作
std::vector<uint32_t> pixels member 自己知道 copy/move/destroy。 semantic size 是否合理。 通常 Rule of Zero。
uint32_t* pixels 只是一個 address value。 ownership、capacity、pitch、allocator。 必須明確 delete/copy/move 或包成 RAII type。
SDL_Texture* 只是一個 pointer。 誰呼叫 destroy、是否可跨 renderer。 通常 move-only wrapper + custom destructor。
第二個修正 — compiler-generated copy/move 是 memberwise semantic composition。只有當每個 member 都已經有正確語意，整個 type 的 default operation 才可信。

6 Return by value：不是 out parameter 的敵人

Renderer 會有很多 factory-like function：

Image LoadImage(const Path& path);
MeshData BuildCubeMesh();
FrameCommands BuildUiCommands(const UiState& ui);

naive C-style 直覺會擔心：「回傳大 object 會 copy，所以應該用 output pointer/reference」。但 modern C++ 要先分清三種 return shape：

Return shape 主要 path 該怎麼想
return T{}; same-type prvalue direct result initialization。 通常沒有 source object 可以 move；result object 直接形成。
return local; NRVO candidate；失敗時通常 move fallback。 保留 compiler 把 local 建在 result storage 的機會。
return std::move(local); 明確 xvalue path。 通常破壞 plain local name 的 NRVO shape，可能變成 pessimizing move。
Image MakeImage(int w, int h) {
  Image img(w, h);
  Clear(img);
  return img; // keep NRVO candidate shape
}

Image MakeEmpty() {
  return Image{}; // direct result construction
}

這是為什麼「看到 return 大 object 就改 out parameter」不一定比較現代，也不一定比較清楚。Out parameter 的確有用途，例如要重用 caller 已有 capacity、回傳多個結果、或配合 C API；但它不是避免 copy 的第一反應。

Return-by-value guideline — factory、loader、builder 預設可以 return by value。先讓 type 的 move/copy/elision 語意正確，再用 profiler 判斷是否需要改成 reuse buffer / out parameter。

7 noexcept move：container 為什麼在意？

std::vector reallocation 是最容易把 copy/move 語意具體化的地方。假設 vector 需要從容量 4 長到 8，它要把舊元素放到新 storage： 

old storage: [ e0 ][ e1 ][ e2 ][ e3 ]
                 |
                 | relocate elements
                 v
new storage: [ e0 ][ e1 ][ e2 ][ e3 ][    ][    ][    ][    ]

如果 element type 的 move constructor 可能 throw，而 copy constructor 又可用，vector 可能偏好 copy，因為 copy 失敗時舊 storage 還完整，可以維持 strong exception guarantee。若 move 被標成 noexcept，vector 比較能安心 move。

Image(Image&& other) noexcept = default;
Image& operator=(Image&& other) noexcept = default;

對只移動 pointer、handle、std::vector、std::unique_ptr 的 type，move 通常應該能做到 noexcept。但不要為了讓 container 選 move 就亂標。noexcept 是 contract：如果真的 throw，程式會走向 std::terminate。

和 T-06 的連接 — noexcept move 不是孤立技巧。它牽涉 resource release、exception safety、container guarantee。下一章會把這些整理進 RAII / Resource Ownership。

8 historical detour：auto_ptr 不是也能 move 嗎？

你可能會想到一個歷史反例：C++11 之前的 std::auto_ptr 似乎也能把 ownership 從一個 object 交給另一個 object。

std::auto_ptr<Texture> a(new Texture{});
std::auto_ptr<Texture> b = a; // looks like copy, actually transfers ownership

這確實能做到 transfer，但問題是它把 transfer 藏在 copy syntax 裡。b = a 看起來像複製，實際上會讓 a 失去 ownership。這種 destructive copy 破壞了 generic code 對 Copyable type 的基本假設：copy 不應消耗 source。 

auto_ptr:
  copy-looking syntax -> destructive transfer
  source becomes empty
  container / algorithm assumptions become fragile

unique_ptr + move:
  copy syntax         -> compile error
  std::move(source)   -> explicit transfer intent
  source remains valid but moved-from

assignment note:
  b = a               -> RHS is lvalue, copy assignment channel
  b = std::move(a)    -> RHS is xvalue, move assignment channel
Deep dive — auto_ptr 的價值不在於現代程式要使用它，而是它能解釋 C++11 為什麼需要 rvalue reference、move constructor、deleted copy constructor 與 unique_ptr。完整歷史拆解放在 A-01 auto_ptr 與 Move Semantics 的誕生。

9 Rule of Zero：讓 member 自己帶語意

最穩的 C++ 設計常常不是手寫一整套 copy/move，而是讓 member 的 type 已經表達 ownership。

class Framebuffer {
public:
  Framebuffer(int w, int h)
    : width(w), height(h), pixels(static_cast<size_t>(w * h)) {}

  uint32_t* Data() { return pixels.data(); }
  const uint32_t* Data() const { return pixels.data(); }

private:
  int width = 0;
  int height = 0;
  std::vector<uint32_t> pixels;
};

這種版本下，Framebuffer 不需要自己寫 destructor，也不需要自己寫 move constructor。std::vector 已經知道如何 copy/move/destroy its storage。你要做的是決定 operation 是否應該公開：

Framebuffer(const Framebuffer&) = delete; // if accidental full image copy is unwanted
Framebuffer& operator=(const Framebuffer&) = delete;

Framebuffer(Framebuffer&&) noexcept = default;
Framebuffer& operator=(Framebuffer&&) noexcept = default;

注意這不是說 framebuffer 一定要禁止 copy。它是在說 copy 是否存在要由語意決定：教學或工具 code 可能希望 copy image；hot path renderer state 可能希望禁止 accidental copy。

10 Pixel-Renderer type policy

把這章收束到專案設計，可以得到一張初版 policy 表：

Type 建議語意 理由
Vec2, Vec3, Color copyable value type。 小型 inline data，copy 語意清楚，move 沒有特殊價值。
Framebuffer 第一版可 move-only；是否 copyable 要明確選。 owns CPU pixels；accidental copy 成本高，而且容易混淆 ownership。
FramebufferView copyable borrowed view。 copy pointer + size + pitch 合理，但不得超過 owner lifetime。
DisplayBackend 通常 non-copyable；可能 move-only 或固定 lifetime。 owns platform/window/texture handles。
DrawCommand prefer copyable/movable value command。 command buffer 應擁有跨 phase 所需資料，避免 dangling reference。
TextureData, MeshData return by value + movable owner。 loader/builder 產生資料後交給 renderer/cache，move/elision 都自然。 
Rasterizer core:
  Vec3 / Color / Rect
    -> small copyable values

Renderer state:
  Framebuffer
    -> owns pixels
    -> copy policy explicit
    -> move is useful

Platform layer:
  DisplayBackend
    -> owns OS/SDL/Win32 handles
    -> non-copyable resource wrapper

Frame data:
  DrawCommand / MeshData / TextureData
    -> value objects carried by vector
    -> rely on move/elision, avoid long-lived borrowed refs
不要把 transfer policy 藏在註解裡 — 如果 type 不該 copy，就 = delete。如果 type 應該 cheap view copy，就把名字叫成 View 或使用 std::span。如果 type owns resource，就讓 member type 或 special members 明確表達。

11 verification：怎麼真的看見 copy/move/elision？

你可以用 logging type 驗證自己的 mental model，但要注意 optimizer 與 C++17 guaranteed copy elision 會讓結果和舊教材不同。

struct Trace {
  Trace() { puts("default"); }
  Trace(const Trace&) { puts("copy"); }
  Trace(Trace&&) noexcept { puts("move"); }
  ~Trace() { puts("destroy"); }
};

Trace MakeA() { return Trace{}; }
Trace MakeB() { Trace t; return t; }
Trace MakeC() { Trace t; return std::move(t); }

可以用這些方向檢查：

  • MakeA 是否沒有 copy/move log？它是 same-type prvalue return。
  • MakeB 在不同 compiler/flags 下是否可能 NRVO？
  • MakeC 是否更容易看到 move？這不代表它比較好，可能只是關掉 NRVO path。
  • 對 container reallocation，移除 noexcept 後 std::vector<T> 是否改走 copy？
本章學到的三件事：
  1. copy/move 不是單純快慢差異；copy 是 duplication 或 shared semantics，move 是 ownership/state transfer，elision 則可能完全沒有 transfer。
  2. construction 和 assignment 的難度不同；assignment 要替已經活著、可能擁有舊 resource 的 object 安全換狀態。
  3. Pixel-Renderer 的 type 要明確標出 transfer policy：math value copyable、framebuffer owner 明確 copy/move、backend handle non-copyable、view cheap copy 但不 owns lifetime。

下一章要把 transfer policy 往更完整的 resource ownership 推進。Copy/move 解釋的是 object 如何被交付；RAII 解釋的是 object 如何在整個 lifetime 期間取得、維持、釋放 resource，並在 exception 或 early return 發生時仍然不漏資源。
