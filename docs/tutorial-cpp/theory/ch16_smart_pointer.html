PART 6 · CHAPTER 16
智慧指標
不是自動 delete，而是 ownership policy 的型別化
unique_ptr shared_ptr RAII

目前 Application 裡還看得到很典型的手動 ownership：

RenderDevice* device = nullptr;
Rasterizer* rasterizer = nullptr;

device = new RenderDevice(config);
rasterizer = new Rasterizer(device);

delete rasterizer;
delete device;

這段 code 的問題不是「不能用 raw pointer」。真正問題是：哪個 pointer owns object？哪個 pointer 只是觀察？這些語意現在主要靠人腦和註解維護。

本章範圍 — 智慧指標不是單純幫你 delete。它們把 ownership policy 寫進 type：unique_ptr 表示 unique ownership，shared_ptr 表示 shared ownership，weak_ptr 表示可檢查的 observer relation。你也會看到 raw pointer 仍然適合表達 non-owning borrow。

1 naive model：smart pointer = 自動 delete

很多人第一次學 smart pointer 會得到這個模型：

raw pointer 要手動 delete
smart pointer 會自動 delete

這句話不算錯，但太淺。更精準的模型是：

pointer-like type
  + ownership policy
  + lifetime behavior
  + copy/move semantics

也就是說，smart pointer 是 Ch13/Ch14 的延伸：同樣是 pointer-like representation，但不同 type 有不同 operation set 和 invariant。

2 unique_ptr：唯一擁有者

std::unique_ptr<T> 表示：這個 pointer owns 一個 T，而且同一時間只有一個 owner。

std::unique_ptr<RenderDevice> device;
device = std::make_unique<RenderDevice>(config);

它的 operation set 很清楚：

Operation unique_ptr 的語意
copy 不存在。copy unique ownership 沒有合理語意。
move ownership transfer。destination 接手，source 變 empty。
destroy 如果 owns object，就自動 delete。

所以 unique_ptr 不是比較方便的 raw pointer，而是一個 move-only ownership handle。

3 Application 裡的 owning pointer

目前 Application owns RenderDevice 和 Rasterizer。這種情境可以用 unique_ptr 表達：

class Application {
  std::unique_ptr<RenderDevice> device;
  std::unique_ptr<Rasterizer> rasterizer;
};

constructor 裡：

device = std::make_unique<RenderDevice>(config);
rasterizer = std::make_unique<Rasterizer>(device.get());

destructor 就不需要手動 delete 這兩個 member。當 Application 結束 lifetime，member destructor 會照反向宣告順序執行。

順序仍然重要 — 如果 Rasterizer 內部借用 RenderDevice，就要讓 rasterizer 在 device 之前被 destroy。C++ member destruction 是反向宣告順序，所以宣告順序本身就是 lifetime design。

4 raw pointer 仍然可以是 observer

但不是所有 raw pointer 都該改成 smart pointer。目前 Rasterizer 裡的這個 pointer：

class Rasterizer {
  RenderDevice* device;
};

它不是 owner。Rasterizer 不負責 delete RenderDevice，只是每次畫線、畫三角形時需要呼叫它。這種 pointer 是 observer 或 borrow。

你可以用 raw pointer 表達 nullable observer，也可以用 reference 表達 non-null borrow：

RenderDevice* device; // observer, may be null if design allows
RenderDevice& device; // observer, must be bound, non-null
判斷準則

如果這個 object 負責決定 resource 何時死亡，用 ownership handle。
如果它只是暫時使用別人的 object，用 pointer/reference/view，不要假裝 ownership。

5 shared_ptr：共享擁有權不是共享存取權

std::shared_ptr<T> 表示多個 owner 共同延長 T 的 lifetime。最後一個 shared_ptr 消失時，object 才被 destroy。

std::shared_ptr<Texture> texture = std::make_shared<Texture>(...);
std::shared_ptr<Texture> another = texture; // ref count +1

Copy shared_ptr 不是 deep copy Texture。它只是增加 reference count，讓多個 handle 指向同一個 object。

不要把 shared_ptr 當 default — shared ownership 會讓 lifetime 變成分散式決策。除非真的有多個 owner 都必須延長 resource lifetime，否則優先考慮 unique_ptr owner + raw pointer/reference observer。

6 weak_ptr：可檢查的 observer

std::weak_ptr<T> 配合 shared_ptr 使用。它不增加 reference count，所以不 owns object；但它可以檢查 object 是否還活著：

std::weak_ptr<Texture> weak = texture;
if (auto locked = weak.lock()) {
  locked->Bind();
}

它適合用在 cache、resource graph、UI references 這類「我想指向某個 resource，但不想因為我存在就讓它永遠活著」的情境。

7 Pixel-Renderer 的 ownership map

用目前專案和未來方向，可以先畫成這樣：

關係 建議表示 理由
Application owns RenderDevice unique_ptr<RenderDevice> 單一 owner，跟著 Application lifetime。
Rasterizer uses RenderDevice RenderDevice* 或 RenderDevice& observer，不負責 destroy。
Resource cache owns textures unique_ptr / value storage / handle table cache 是 owner，外部拿 handle 或 view。
UI widget references texture id / handle / weak reference UI 不應讓 resource 因為引用而永久活著。
Shared scene graph node 可能 shared_ptr，但要謹慎 只有在多個 owner 真的都擁有 lifetime 時才用。

8 和下一章 polymorphism 的關係

Smart pointer 常會和 polymorphism 一起出現：

std::unique_ptr<DisplayBackend> backend;
backend = std::make_unique<SDLDisplayBackend>(...);

這裡 unique_ptr 管的是 lifetime；virtual 管的是 runtime dispatch。兩者常常一起用，但解決的是不同問題。

下一章會接到 virtual / vtable：當你想讓 Win32 / SDL / macOS backend 在同一個 interface 後面替換時，C++ 怎麼做到 runtime polymorphism，以及這個 abstraction 的成本在哪裡。

常見陷阱

1. 把所有 raw pointer 都換成 shared_ptr — 這會把 borrow 關係誤寫成 shared ownership，讓 lifetime 更難推。

2. 忘記 unique_ptr 不能 copy — 這是語意，不是限制。轉移 ownership 要用 move。

3. 用 shared_ptr 解決架構不清 — 如果你不知道誰 owns resource，先畫 ownership map，不要先加 reference count。

4. 忽略 member destruction order — smart pointer 會自動 destroy，但順序仍由 member 宣告順序決定。
本章學到的三件事

1. Smart pointer 是 ownership policy 的 type，不只是自動 delete。

2. unique_ptr 表示唯一擁有權；raw pointer/reference 仍適合表達 non-owning observer。

3. shared_ptr 只應在真的有 shared ownership 時使用；否則會模糊 resource lifetime。
下一步 — Smart pointer 解決 lifetime ownership；但如果你要讓不同 backend / shader / renderer implementation 在同一個 interface 後面替換，就需要 runtime polymorphism。下一章進入 繼承、virtual 與 vtable。
