THEORY TRACK · T-18
Casts / Type Punning / Bit Cast
cast 不是叫 compiler 閉嘴，而是明確說你在轉 value、轉 type view，還是在複製 object representation
cast aliasing bit_cast object-representation

T-03 講過 object representation 與 strict aliasing，T-15 講過 packed pixel、endianness、numeric policy。現在要補中間缺的語意層：當你手上有一段 bytes、一個 uint32_t pixel、一個 void* callback context、一個 platform buffer pointer，你到底能不能把它 cast 成另一種 type？

C++ 的 cast 不是同一種操作。static_cast 常是 value conversion 或已知型別關係；const_cast 改 qualification；dynamic_cast 做 runtime checked polymorphic cast；reinterpret_cast 改變 pointer/integer interpretation，但不保證你能合法讀 object；std::bit_cast 則是把 object representation 複製成另一個 trivially copyable value。

本章範圍 — 建立 renderer 需要的 cast model：C-style cast 為什麼危險、static_cast / const_cast / dynamic_cast / reinterpret_cast 的語意差異、strict aliasing、alignment、object lifetime、std::memcpy / std::bit_cast、std::byte view、pixel buffer 和 file/network binary data 的處理規則。

1 naive model：cast 可以把 bytes 變成我要的 type

很常見的直覺是： 

I have bytes
I know what they mean
therefore:
  reinterpret_cast(bytes) should be fine

這個模型混淆了三件事：

  • 這段 storage 裡是否真的有一個 T object 活著。
  • 這個 address 是否滿足 T 的 alignment。
  • C++ aliasing rule 是否允許你用 T* 讀這個 object。
核心修正 — pointer cast 不會創造 object lifetime，也不會修正 alignment，更不會讓 strict aliasing 失效。它只改變你手上的 pointer type。

2 C-style cast：太多意思藏在一個語法裡

C-style cast 看起來短：

float f = (float)x;
Pixel* p = (Pixel*)bytes;

問題是它可能等價於 static_cast、const_cast、reinterpret_cast 的組合。讀 code 的人很難知道你是在做數值轉換、拿掉 const、還是硬改 pointer interpretation。

project policy — C++ code 裡避免 C-style cast。用具名 cast 不是為了形式，而是讓 code review 能看見你正在跨越哪一種語意邊界。

3 Cast family map

Cast主要語意Renderer 例子風險
static_cast 已知型別轉換、numeric conversion、up/downcast without runtime check。 int to size_t after bounds check；uint8_t to uint32_t before shift。 你仍要保證 range / precondition。
const_cast 改 const/volatile qualification。 極少需要；通常只在接不良 C API 時隔離使用。 若原 object 真的是 const，寫入是 UB。
dynamic_cast polymorphic runtime checked cast。 debug/tool layer 查某 backend concrete type。 需要 RTTI；不適合 hot path。
reinterpret_cast 低階 bit/pointer interpretation conversion。 C API user data、platform handle interop。 不保證 aliasing/lifetime/alignment 合法。
std::bit_cast copy object representation between same-size trivially copyable types。 inspect float bits、pack/unpack binary representation。 結果值是否有意義仍要由 format contract 決定。

4 reinterpret_cast 不等於 type punning 安全

假設你有：

std::array<std::byte, 4> bytes;
uint32_t* p = reinterpret_cast<uint32_t*>(bytes.data());
uint32_t v = *p;

這裡可能有多個問題：

  • bytes.data() 不一定滿足 uint32_t alignment。
  • 那段 storage 裡活著的是 std::byte array，不是 uint32_t object。
  • endianness 決定 bytes 變成 integer value 的方式。

比較穩的方式是明確 copy representation：

uint32_t v = 0;
std::memcpy(&v, bytes.data(), sizeof(v));

如果兩邊都是 trivially copyable、size 相同，而且你要的是 representation copy，C++20 可以用：

uint32_t bits = std::bit_cast<uint32_t>(someFloat);
和 T-15 的連接 — bit_cast 只保證 representation copy。它不替你定義 pixel format、endianness、channel order。那些仍然要由 renderer 的 PackRGBA8 / UnpackRGBA8 contract 決定。

5 Object representation vs object interpretation

SVG：同一組 bytes 可以被看成 raw representation、integer value 或 semantic pixel。這三層不能靠一個 pointer cast 混成同一件事。

6 char / unsigned char / std::byte view

C++ 允許用 character-like type 觀察 object representation。實務上，binary buffer 建議用 std::byte 表達「這不是數字，只是 raw bytes」。

std::span<const std::byte> bytes = ReadFile(path);
// parse bytes into explicit fields instead of pretending the file is a C++ struct

不要把 image file / mesh file / network packet 直接 cast 成 C++ struct。File format 有自己的 alignment、endianness、padding policy；C++ struct 有 compiler layout、padding、alignment。這兩套規則不是同一個東西。

7 static_cast：可見的 value conversion

static_cast 很適合把「我已經檢查過 precondition」寫清楚：

if (x < 0 || y < 0 || x >= width || y >= height) {
  return;
}
size_t index = static_cast<size_t>(y) * stride + static_cast<size_t>(x);

這裡 cast 本身不保證安全；安全來自前面的 bounds check。Cast 只是讓 signed coordinate 到 unsigned index 的轉換明確。

8 const_cast：隔離在 C API 邊界

const_cast 不是「我知道它其實不會改」的萬用工具。只有在你必須接某個 legacy C API，而且你確定 API 不會修改資料時，才應該把它隔離在很小的 adapter 裡。

// Adapter boundary only. Do not spread const_cast into renderer core.
legacy_c_api(const_cast<char*>(name.c_str()));

如果原 object 真的是 const，透過 const_cast 寫它就是 undefined behavior。這也是為什麼 core API 應該設計好 const-correctness，而不是靠 cast 補洞。

9 dynamic_cast：debug/tool boundary，不是 hot path default

dynamic_cast 對 polymorphic type 做 runtime check：

if (auto* sdl = dynamic_cast<SdlDisplayBackend*>(display.get())) {
  sdl->DumpBackendInfo();
}

這可以用在 debug/tool layer，但如果 renderer core 經常需要問「你到底是哪個 derived type」，那通常代表 interface 設計還沒有把需要的 capability 放在正確抽象上。

10 Pixel-Renderer cast policy

General:
  no C-style casts in C++ code
  use named casts to expose semantic boundary

Coordinates:
  static_cast only after bounds/range check

Binary data:
  parse from std::byte / unsigned char buffers
  do not reinterpret file bytes as C++ structs

Pixel formats:
  use Pack/Unpack functions for semantic channel order
  use memcpy/bit_cast only for representation copy

Platform/C API:
  isolate reinterpret_cast/const_cast in adapter layer
  do not let casts leak into renderer core

Polymorphism:
  dynamic_cast only for debug/tool paths
  use virtual interface or variant when dispatch is part of design
本章學到的三件事：
  1. Cast 不是單一工具：static_cast、const_cast、dynamic_cast、reinterpret_cast、bit_cast 分別跨越不同語意邊界。
  2. reinterpret_cast 不會創造 object、修正 alignment、解除 strict aliasing；要 copy representation 時優先考慮 std::memcpy 或 std::bit_cast。
  3. Pixel-Renderer 應把 C/API/platform cast 隔離在 adapter layer；renderer core 用明確的 Pack/Unpack、span/view、typed API 表達資料語意。
