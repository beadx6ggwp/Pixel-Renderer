THEORY TRACK · T-02
Type System / Initialization
type 是語意邊界；initialization 是 object 進入第一個合法狀態
type-system initialization explicit

T-01 說明 C++ 程式如何變成 binary。T-03 說明 object 如何在 storage 裡開始 lifetime。但中間還缺一層：一個 object 開始 lifetime 時，為什麼它是某個 type 的合法 object？

這就是 type system 和 initialization 的問題。對 Pixel-Renderer 來說，Width、Height、Color、TextureId、FramebufferDesc 不只是資料大小不同；它們代表不同語意。初始化也不是把 bytes 填進去而已，而是讓 object 從一開始就滿足它的 invariant。

本章範圍 — 建立 C++ type 與 initialization 的工作模型：type 作為 states / operations / invariants 的邊界，default/value/direct/copy/list initialization、aggregate、narrowing、explicit、implicit conversion、const/constexpr/consteval、enum class、member initialization order。這章補上 T-02，是為了讓 T-03 object lifetime 和 T-04 value category 的地基更完整。

1 naive model：type 只是記憶體大小和欄位排列

從 C 的角度看，type 很容易被理解成 layout： 

int      -> 4 bytes
float    -> 4 bytes
Vec3     -> 3 floats
Color    -> 4 bytes
TextureId -> maybe int

這個模型只看到 object representation，沒看到語意。兩個 type 可能底層都是 uint32_t，但不代表可以混用：

uint32_t color = 0xff3366aa;
uint32_t texture_id = 7;

DrawTexturedQuad(color); // layout fits, semantics wrong

真正有用的 type model 是： 

type =
  valid states
  + allowed operations
  + invariants
  + cost model
  + conversion rules

例如 FramebufferDesc 的 valid state 不是任意四個整數。width 和 height 應該大於 0，pitch 至少要能容納一列 pixels，pixels 指向的 storage 必須活得夠久。這些都是 type 設計要表達或保護的規則。

2 initialization 不是 assignment

最常見的混淆是把初始化想成「先有 object，再賦值」： 

wrong model:
  create empty object
  assign values into it

但 C++ 裡 initialization 是 object lifetime 開始時建立第一個值。Assignment 是 object 已經存在後，把它改成另一個值。

操作 object 是否已存在 呼叫什麼 例子
initialization 還沒有，正在開始 lifetime。 constructor / aggregate initialization。 Vec3 v{1,2,3};
assignment 已經存在。 operator=。 v = Vec3{4,5,6};

這跟 T-03 接起來就是： 

storage acquired
  |
  | initialization
  v
object lifetime begins with a valid value
  |
  | assignment may change existing value
  v
object remains alive
  |
  | destruction
  v
lifetime ends
第一個關鍵修正 — constructor 不是「先 default 再 assign」的語法糖。Constructor 是 object 進入第一個合法狀態的入口。

3 C++ 初始化語法為什麼這麼多

C++ 初始化語法多，是因為它要同時支援 C compatibility、class constructor、aggregate、initializer_list、narrowing 防護、copy/direct 語意。你不需要一開始背所有名字，但要知道它們不是完全等價。

形式 例子 直覺 注意
default initialization Vec3 v; 用 default constructor；primitive local 可能未初始化。 對 int x; 這類要小心 uninitialized read。
value initialization Vec3 v{}; 傾向產生 zero / default value。 常作為安全預設。
direct initialization Buffer b(1024); 直接呼叫 matching constructor。 允許 explicit constructor。
copy initialization Buffer b = 1024; 像是從右邊轉成左邊 type。 不允許 explicit constructor。
list initialization Vec3 v{1,2,3}; 用 braces 初始化。 可阻擋 narrowing，但有 initializer_list 優先規則。

對 renderer math/value type，通常偏好 braces，因為它清楚地表達「用這些分量建立一個值」：

Vec3 p{1.0f, 2.0f, 3.0f};
Color c{255, 128, 64, 255};

4 list initialization 與 narrowing

Braces 有一個重要工程價值：阻擋 narrowing conversion。

int a = 3.14; // allowed, loses information
int b{3.14}; // error: narrowing

這對 graphics 很重要，因為你常在 float screen-space、int pixel coordinate、normalized color、byte color 之間轉換。你不希望某個隱式轉換偷偷把資料截斷。

Renderer consequence — 從 clip space / NDC / screen space / pixel coordinate 的轉換應該是顯式設計，不應靠隱式 narrowing。座標轉換越靠近 rasterizer hot path，越需要清楚。

但 braces 不是完全沒有坑。若 type 有 std::initializer_list constructor，它可能優先被選到。這是 container 和 math type 設計時要注意的事，後面 STL/templates 章會再回來。

5 aggregate：資料協議可以公開，但不保護 invariant

Aggregate 很適合表達公開資料協議：

struct Vertex {
  Vec3 position;
  Color color;
};

Vertex v{Vec3{0, 1, 0}, Color{255, 0, 0, 255}};

這很適合 vertex、rect、clear color、config descriptor 這類「資料就是介面」的 type。但 aggregate 不會自動保護 invariant：

struct FramebufferDesc {
  int width;
  int height;
  int pitch;
};

FramebufferDesc bad{800, 600, 4}; // syntactically valid, semantically wrong

如果一個 type 需要保證 pitch >= width * 4，那它可能需要 constructor/factory validation，而不是純 aggregate。

6 explicit：阻止意外 conversion

假設你有一個 resource type：

class Buffer {
public:
  Buffer(size_t bytes);
};

如果 constructor 不是 explicit，這種 code 可能成立：

void Upload(Buffer buffer);
Upload(4096); // int/size_t implicitly converted to Buffer

這看起來方便，但語意很危險：一個數字突然變成 resource object。通常應該寫：

class Buffer {
public:
  explicit Buffer(size_t bytes);
};

Upload(Buffer{4096});

explicit 的本質是要求 caller 明確承認：「我正在從這個值建立一個 Buffer」。

準則 — 單參數 constructor 若會建立有語意成本或 resource/lifetime 的 type，預設考慮 explicit。讓隱式轉換只存在於真的像數學/值轉換的地方。

7 enum class：把整數常數變成不會混用的語意

C-style enum 和整數很容易混在一起。Renderer 會有很多 mode / format / backend kind：

enum class ColorFormat { RGBA8, BGRA8 };
enum class BackendKind { Win32, SDL, MacOS };
enum class DepthTest { Disabled, Less, LessEqual };

enum class 的好處：

  • 不會自動污染外層 scope。
  • 不會隨便跟 int 混用。
  • 不同 enum type 不會互相傳錯。

這不是語法潔癖，而是把 state space 收窄。ColorFormat::BGRA8 不應該能被拿去當 BackendKind。

8 const / constexpr / consteval：什麼時候值必須穩定

這三個詞常被混在一起：

Keyword 核心意思 renderer 例子
const 這個 object 不能透過這個 name 被修改。 const FramebufferDesc& 表示只讀借用 config。
constexpr 可在 compile time 求值，如果語境需要。 constexpr int BytesPerPixel = 4;
consteval 必須在 compile time 求值。 少用；適合強制 compile-time 產生 table 或檢查。

對目前 Pixel-Renderer，比較實用的是先把明確不變的編譯期常數寫清楚：

constexpr int BytesPerPixel = 4;
constexpr int DefaultWidth = 800;

但不要把 const 誤解成「compile-time」。const int width = window.GetWidth(); 是 runtime object 不能被改，不代表它是 compile-time constant。

9 member initialization order：不是照 initializer list 順序

Class members 的初始化順序是 declaration order，不是 constructor initializer list 的書寫順序。

class Renderer {
  Framebuffer framebuffer;
  RenderDevice device;
public:
  Renderer()
    : device(framebuffer), framebuffer(800, 600) {} // misleading
};

雖然 initializer list 寫 device 在前，實際仍然先初始化 framebuffer，因為它在 class 裡宣告比較早。這條規則是為了讓 destruction order 穩定：destructor 會用相反順序銷毀 members。

不要讓 member order 和 dependency order 相反 — 如果 RenderDevice 借用 Framebuffer，就讓 Framebuffer 在 class member declaration 中寫在 RenderDevice 前面。這是 type layout 層級的 lifetime 設計。

10 Pixel-Renderer type design 判準

接下來專案擴充時，可以用下面判準決定 type 應該長什麼樣子。

需求 適合形式 例子
公開資料協議，沒有強 invariant。 struct aggregate。 Vertex、Rect、ClearCommand。
必須保證 state 一致。 class + constructor/factory。 Framebuffer 保證 width/height/pitch/pixels 一致。
不能和 int 混用的 mode。 enum class。 ColorFormat、BackendKind。
單參數建立 resource 或語意 type。 explicit constructor。 TextureId、Buffer、Framebuffer。
編譯期固定常數。 constexpr。 BytesPerPixel、small lookup constants。

11 verification：如何檢查 type / initialization 問題

用 compiler 阻止 narrowing

int x{3.14}; // should fail

如果某段座標或顏色轉換會 narrowing，盡量讓它在 code 裡顯式出現，而不是默默發生。

用 explicit 阻止意外 conversion

explicit TextureId(uint32_t id);
BindTexture(3); // should fail
BindTexture(TextureId{3}); // explicit intent

用 warning 檢查 initialization order

-Wall -Wextra -Wreorder

當 initializer list 順序和 member declaration order 不一致，compiler 可以提醒你。這對 owner/borrower 成員順序很有價值。

本章學到的三件事：
  1. Type 不是 layout label，而是 valid states、operations、invariants、conversion rules 和 cost model 的邊界。
  2. Initialization 是 object lifetime 開始時建立第一個合法值；assignment 是既有 object 的狀態改變，兩者不是同一件事。
  3. Pixel-Renderer 的 type design 要主動阻止錯誤狀態：用 aggregate 表達資料協議，用 constructor/explicit/enum class/constexpr/member order 表達語意與 lifetime 約束。
