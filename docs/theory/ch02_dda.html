DDA 直線演算法

Bresenham 的前身 — 從微分方程到像素，以及它為什麼被取代

1
DDA 是什麼

DDA（Digital Differential Analyzer，數位差分分析器） 最初是 1940 年代的硬體機器，用數位電路模擬類比微分方程的求解器。 當時用來計算彈道、飛行軌跡等工程問題。

套用到畫直線，DDA 的思路非常直接：

直線就是微分方程 dy/dx = m（常數斜率） 的解。
每走一步 x，用 Euler 方法求下一步的 y：
y_next = y + m · Δx
令 Δx = 1，就是 y_next = y + m。
把每步的 y 四捨五入到最近的整數像素，就是直線。

這個想法幾乎不需要任何推導，直接從「直線是什麼」出發就能想到。

2
跟 Euler 方法的關係

數值分析的 Euler 方法：解常微分方程 y' = f(x, y)， 從初始值出發，用斜率估計下一步：

// Euler Method（一階 ODE 數值積分） y(x + h) ≈ y(x) + h · f(x, y(x)) // 直線：dy/dx = m（f 是常數） y(x + 1) = y(x) + 1 · m y_next = y + m

DDA 就是 Euler 方法的特例：步長 h = 1，斜率 f = m（常數）。 因為斜率是常數，Euler 方法在這裡沒有截斷誤差（對於直線，Euler 是精確的）。

直線特殊在斜率不變。如果是曲線（圓、拋物線），斜率每步都不同， Euler 方法就會累積截斷誤差——這是 DDA 推廣到曲線的困難所在。

3
基本演算法

從 (x₁, y₁) 到 (x₂, y₂)：

dx = x₂ − x₁ dy = y₂ − y₁ // steps：要走幾步？取較長的那個軸 steps = max(|dx|, |dy|) // 每步的增量（讓主軸每次恰好走 1 格） x_inc = dx / steps // 浮點數 y_inc = dy / steps // 浮點數 loop steps+1 次： 畫 (round(x), round(y)) x += x_inc y += y_inc

steps = max(|dx|, |dy|) 的原因：

  • 讓「主軸」每步精確走一格，保證不跳格、不重複
  • 緩坡（|dx| > |dy|）→ x 每步走 1，y 走 <1
  • 陡坡（|dy| > |dx|）→ y 每步走 1，x 走 <1
  • 斜率 = 1 → 兩軸都走 1（對角線）

具體例子：(0,0) → (7,3)

dx=7, dy=3，steps=7，x_inc=1.0，y_inc=3/7≈0.4286

4
四個 Case

steps = max(|dx|, |dy|) 讓 DDA 自動處理所有方向，不需要分 case。

DDA 的優雅之處：不需要分 case，不需要知道斜率大於還是小於 1。
steps = max(|dx|, |dy|) 讓主軸每步恰好走 1 格，保證不跳格。
x_inc 和 y_inc 自然地表達了「每步要走多少」。

5
DDA 的問題：浮點累積誤差

DDA 的問題不是「結果錯」，而是「表示方式帶來的代價」。

問題一：浮點運算昂貴（歷史問題）

1960 年代沒有浮點運算單元（FPU），加法和乘法的代價比現在高出幾個數量級。 每步做浮點加法 y += dy/dx 在當時是很重的操作。

現代 CPU 有硬體 FPU，這個問題幾乎消失了。

問題二：浮點精度的累積

浮點數不是精確的有理數。每次加法都有微小誤差，長距離的線會累積：

// 理想：每步加 3/7 = 0.42857142857... // 實際浮點（float32）： 3.0f / 7.0f = 0.42857142 857...（精確到約第 7 位） // 100 步後的累積誤差： 理想 y = 42.857142857... 浮點 y ≈ 42.857140... （誤差 ≈ 2.8e-6） // 1000 步後誤差可能大到影響 round() 的結果

對大多數螢幕解析度（1080p、4K），這個誤差不會造成視覺問題， 但在某些特殊情況下（非常長的線、精確像素輸出）會有問題。

問題三：round() 的捨入不一致

DDA 用 round(y) 決定要畫哪個像素， 但 round 的捨入方向在不同系統不一定一致（ties：0.5 要捨入到哪？）：

round(0.5) = ? C 標準：0.5 → 1（四捨五入） IEEE 754：0.5 → 0（取最近偶數，Banker's Rounding） Python：0.5 → 0，1.5 → 2（也是 Banker's Rounding）

同一條線，在不同環境跑 DDA，可能有個別像素不同。 Bresenham 沒有 round，判斷完全由整數比較決定，跨平台結果一致。

但在精確輸出（印刷、向量轉點陣、字型渲染）或嵌入式系統（無 FPU）的場景， Bresenham 的整數精確性仍然有優勢。

6
程式碼對比

▌ DDA
void DrawLineDDA(x1,y1,x2,y2,color) {
  int dx = x2-x1, dy = y2-y1;
  int steps = max(abs(dx),abs(dy));
  float xi = (float)dx/steps;
  float yi = (float)dy/steps;
  float x=x1, y=y1;
  for(int i=0; i<=steps; i++){
    SetPixel(round(x), round(y));
    x += xi;
    y += yi;
  }
}

// 優點：邏輯簡單，5分鐘能寫完
// 缺點：浮點、round()、跨平台不一致
▌ Bresenham
void DrawLine(x1,y1,x2,y2,color) {
  int dx=abs(x2-x1), dy=abs(y2-y1);
  int sx=(x1<x2)?1:-1;
  int sy=(y1<y2)?1:-1;
  int err=dx-dy;
  while(true){
    SetPixel(x1,y1);
    if(x1==x2 && y1==y2) break;
    int e2=2*err;
    if(e2>-dy){err-=dy; x1+=sx;}
    if(e2<dx) {err+=dx; y1+=sy;}
  }
}

// 優點：全整數、精確、無 round
// 缺點：需要理解推導才能修改

DDA 的現代用途

DDA 看似被 Bresenham 完全取代，但它的思路在現代圖形學仍然存在：

用途一：軟體光柵化的掃描線插值

三角形光柵化時，沿著邊緣從頂點走到底部，每步插值顏色、UV、深度。 這就是 DDA：每步 u += u_inc，v += v_inc。 這裡用浮點完全合理，因為插值結果本身就是浮點（寫入 framebuffer 前才量化）。

// 三角形邊緣的 DDA 插值（光柵化的基礎） 每列 y 掃描： x_left += x_inc_left x_right += x_inc_right u += u_inc depth += depth_inc // 正是 DDA 的思路：每步加常數增量

用途二：光線步進（Ray Marching / DDA Traversal）

光線在體素網格（Voxel Grid）或 SDF（Signed Distance Field）中前進， 用 DDA 決定光線下一步進入哪個格子——這比 Bresenham 更自然， 因為需要知道精確的進入點，不只是哪個格子。

// Voxel DDA（光線追蹤體素場景的標準算法） t_delta_x = 1.0 / |ray.x| // 穿越一格 x 需要的 t t_delta_y = 1.0 / |ray.y| // 穿越一格 y 需要的 t loop： 下一個碰到的格子邊界 = min(t_max_x, t_max_y) t_max_x += t_delta_x （DDA 增量）

用途三：抗鋸齒線（Anti-Aliased Line）

Wu's Line Algorithm（吳小林演算法，1991）是 DDA 的延伸： 不 round 到最近像素，而是把 y 的小數部分當作透明度， 在兩個相鄰像素上分配顏色，實現抗鋸齒：

y_float = 0.8 畫 (x, 0) 亮度 = 1 − 0.8 = 0.2 （離 y 的整數部分遠） 畫 (x, 1) 亮度 = 0.8 （離 y 的小數部分近）

Bresenham 只畫一個像素，Wu 用 DDA 的浮點 y 值算透明度，畫兩個。 這是 DDA 比 Bresenham 更自然的場景。

總結

DDA 的地位：

DDA 是「把連續數學直接翻譯成計算步驟」的最自然結果。
不需要任何特殊技巧，從微分方程出發，Euler 方法，加法步進。

它的問題不是「思路錯」，而是「浮點運算在當時太貴，精度也有隱患」。
Bresenham 解決的正是這個問題：把浮點決策替換成整數決策，不改變結果。

而 DDA 的思路本身（每步加常數增量）在現代光柵化、光線步進、抗鋸齒中繼續存在， 只是換了名字和應用場景。
