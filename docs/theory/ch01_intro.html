PART 1 · CHAPTER 01
什麼是光柵化？
Framebuffer、像素座標系、SetPixel 作為原子操作
framebuffer SetPixel 座標系 像素格

1
連續 vs 離散

數學裡的幾何對象是連續的：一條直線有無限多個點，一個三角形裡有無限多個面積元素。

螢幕不是連續的。它是一個有限的格子，每個格子叫做像素（pixel），每個像素只能是一個顏色。

光柵化（Rasterization）就是這個轉換的過程： 決定哪些像素要被點亮，以及各自的顏色。

2
Framebuffer：螢幕的記憶體表示

螢幕的每個像素在記憶體裡有一個對應位置，整塊記憶體叫做 Framebuffer（幀緩衝）。

在程式碼裡，Framebuffer 是一個 unsigned char*， 第 (x, y) 個像素的位置是：

// 第 (x, y) 個像素的記憶體位址 address = frame_buffer + y * pitch + x * 4 // pitch = 每一行的 byte 數（可能 > width * 4，因為對齊） pitch ≥ width * 4
為什麼不用 width * 4 而是 pitch？
記憶體對齊（Memory Alignment）：作業系統和顯示硬體有時要求每一行的起始位址對齊到 4、8 或 16 bytes 的倍數。 Pitch 把這個 padding 納入，讓計算保持正確。

3
座標系的約定

像素座標系和數學座標系有一個重要差異：y 軸方向。

▌ 數學座標系
▌ 螢幕座標系

螢幕的原點 (0,0) 在左上角，y 向下增加。 這是因為 CRT 顯示器的電子束從左上角開始掃描，沿用至今。

3D 場景通常用「y 向上」的座標系，但渲染到螢幕前需要一次 y 軸翻轉（通常在投影矩陣裡處理）。 本系列的光柵化部分統一用螢幕座標系（y 向下）。

4
SetPixel：原子操作

所有更複雜的繪製都建立在一個最基本的操作上： 在指定座標畫一個像素。

void RenderDevice::SetPixel(int x, int y, uint32_t color) {
  if (x >= 0 && x < width && y >= 0 && y < height) {
    // 找到這個像素在記憶體裡的位址，寫入顏色
    *(uint32_t*)(frame_buffer + y * pitch + x * 4) = color;
  }
}

它做了三件事：

  1. 邊界檢查：(x, y) 在螢幕範圍內才寫入，防止越界
  2. 地址計算：y * pitch + x * 4 找到對應的記憶體位置
  3. 寫入顏色：把 uint32_t（4 bytes = ARGB）寫進去

顏色格式：ARGB

// uint32_t color = 0xAARRGGBB A = Alpha（透明度） bits 31-24 R = Red bits 23-16 G = Green bits 15-8 B = Blue bits 7-0 // 常用顏色 0xFFFFFFFF // 白色 0xFF000000 // 黑色 0xFFFF0000 // 紅色 0xFF0000FF // 藍色

5
為什麼不能「直接計算」

既然有了 SetPixel，畫一條從 (0,0) 到 (7,3) 的線，最直覺的方法是：

// 笨方法：逐 x 計算對應的 y for (x = 0; x <= 7; x++) { y = round(x * 3.0 / 7.0); SetPixel(x, y, color); }

這個方法有什麼問題？

問題說明
浮點乘法 每步做一次 x * 3.0 / 7.0，比加法慢很多（在 1960 年代差距更大）
斜率 > 1 時跳格 若換成 (0,0) → (3,7)，每步 x 走 1，y 要跳 >1 格，產生斷線
round 不一致 0.5 要捨去到哪？不同語言實作不同，結果不可預期
根本問題：當斜率 > 1 時，x 每走一格，y 需要走超過一格。 如果只遍歷 x，中間的 y 位置就沒有像素。
解法：要麼遍歷「較長的那個軸」（DDA），要麼用誤差追蹤兩個軸（Bresenham）。

6
這個系列要解決的問題

從 SetPixel 出發，我們需要回答三個越來越複雜的問題：

問題難點解法章節
如何畫一條直線？ 斜率可能 >1，不能只遍歷 x DDA / Bresenham Ch02–07
如何填充一個三角形？ 需要知道哪些像素在三角形內 重心座標 + 包圍盒 Ch08–10
多個三角形重疊時誰在前？ 3D 場景需要深度排序 Z-buffer Ch11
軟體光柵化的核心就是這三個問題。GPU 做的事情在本質上相同，只是並行化了幾千倍。
