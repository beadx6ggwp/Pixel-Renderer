Bresenham 直線演算法 — 白話推導

從「每步要做什麼決定」出發，一個 case 一個 case 講清楚

1
問題到底是什麼？

想畫一條從 (0, 0) 到 (7, 3) 的直線。

螢幕是格子，每個格子是一個像素，只能點亮整數座標。

理想的直線是連續的，但我們只能選格子。所以問題是：

每走一格 x（x → x+1），y 要不要也往上加一格？

這條線斜率 = 3/7 ≈ 0.43，意思是「每走 7 格 x，y 才升 3 格」。 不是每步都升 y，所以要判斷。

2
最直覺的想法：追蹤「欠了多少 y」

理想直線的斜率 m = dy/dx = 3/7。

每走一步 x，理想的 y 應該增加 3/7。 但我們不能畫 0.43 個像素——只能等「欠的夠多了」再升一格。

定義一個 誤差值 err，表示「理想 y 比目前像素的 y 多出來多少」：

// 初始：還沒走，誤差為 0 err = 0 // 每走一步 x： err += m // 理想 y 增加了 m = dy/dx // 判斷：誤差超過 0.5，代表理想 y 已經超過下一個整數格的一半 if err ≥ 0.5 : y += 1; err -= 1

來跑一遍 (0,0) → (7,3)，m = 3/7：

步畫err 之前err += 3/7 後≥ 0.5？動作err 最後
0(0,0)03/7 ≈ 0.43否—0.43
1(1,0)0.436/7 ≈ 0.86是y++, err-=1-0.14
2(2,1)-0.143/7 ≈ 0.29否—0.29
3(3,1)0.296/7 ≈ 0.71是y++, err-=1-0.29
4(4,2)-0.291/7 ≈ 0.14否—0.14
5(5,2)0.144/7 ≈ 0.57是y++, err-=1-0.43
6(6,3)-0.430否—0
7(7,3)抵達終點—

結果：(0,0) (1,0) (2,1) (3,1) (4,2) (5,2) (6,3) (7,3) ← 完全正確！

但這裡有浮點數（3/7）。Bresenham 的目標是：只用整數，避免浮點運算。
3
消掉浮點數：兩步整數化

第一步：用 dy、dx 取代 m = dy/dx

把 err 乘以 dx，就不用除法了：

// 原本 err = 0 err += dy/dx → 判斷 err ≥ 0.5 → y++; err -= 1 // 兩側全部 × dx（dx > 0，不影響大小關係） err = 0 err += dy // 原來的 dy/dx × dx 判斷 err ≥ dx/2 // 原來的 0.5 × dx y++; err -= dx // 原來的 1 × dx

還剩一個問題：dx/2 如果 dx 是奇數就還是分數。

第二步：全部再乘以 2，徹底消掉 /2：

err = 0 err += 2·dy 判斷 err ≥ dx // 原來的 dx/2 × 2 = dx y++; err -= 2·dx
現在全是整數加減比較，沒有任何浮點數、沒有乘法（只有 ×2 可用位移）。

用整數版跑一遍同樣的例子（dx=7, dy=3）：

步畫err 之前err += 6≥ 7？動作err 最後
0(0,0)06否—6
1(1,0)612是y++, err-=14-2
2(2,1)-24否—4
3(3,1)410是y++, err-=14-4
4(4,2)-42否—2
5(5,2)28是y++, err-=14-6
6(6,3)-60否—0
7(7,3)終點

結果完全一樣，但現在全是整數運算。

4
斜率 > 1 的情況（陡線）

上面的版本只對 0 ≤ 斜率 ≤ 1 有效。如果斜率 > 1 呢？

舉例：(0,0) → (3,7)，斜率 = 7/3 ≈ 2.3。 每走一步 y，x 才加 3/7。主軸換成 y！

斜率 ≤ 1（x 是主軸）：

每步 x 必走 +1 判斷 y 要不要 +1

斜率 > 1（y 是主軸）：

每步 y 必走 +1 判斷 x 要不要 +1

解決方法很簡單：把兩個 case 合併成同一個邏輯。 用 dx 和 dy 的大小關係決定誰是主軸——不需要分開寫兩段。

5
負方向：sx 和 sy

上面都假設往右往上。如果 x2 < x1 或 y2 < y1 呢？

只需要改步進方向，誤差邏輯完全不變：

sx = (x1 < x2) ? +1 : -1 // x 走哪個方向 sy = (y1 < y2) ? +1 : -1 // y 走哪個方向 // 用絕對值 dx, dy 計算誤差（保持正數） dx = |x2 - x1| dy = |y2 - y1|

這樣 8 個方向全部統一成同一套邏輯，不用 8 個 if-else。

6
合併成通用公式（程式碼的寫法）

現在把「x 是主軸」和「y 是主軸」兩個 case 合併。 秘訣是把誤差的初始值改成 dx − dy， 讓同一個 err 能同時表達兩個方向的狀態：

換個角度理解 err：
err = dx − dy 相當於「x 還剩多少空間」減「y 還剩多少空間」。
正值 → x 方向空間多，主要走 x；負值 → y 方向空間多，主要走 y。

判斷條件變成：

err = dx − dy // 初始 每步： e2 = 2 × err // 先快照，避免 err 被改掉後影響第二個判斷 // Case A：要不要走 x？ if e2 > −dy : x += sx; err -= dy // Case B：要不要走 y？ if e2 < dx : y += sy; err += dx

為什麼是 > −dy 和 < dx？ 讓我們把兩個 case 分開看：

Case A：走 x 的條件

回到第 3 節的整數化結果（x 是主軸時），每步都走 x， 所以條件是「err 還沒超過 y 需要走的門檻」，也就是 err 不太負：

// 等價於：誤差還沒讓 y 走太多，x 可以繼續走 e2 = 2·err e2 > −dy ⟺ err > −dy/2 // 也就是說，誤差還沒負到讓 y 反過來壓制 x 的程度 // 走 x：err -= dy（因為 x 走了一步，y 的「欠債」減少）

Case B：走 y 的條件

// 等價於：誤差讓 y 已經欠得夠多，該升一格 e2 < dx ⟺ err < dx/2 // 走 y：err += dx（y 走了一步，還清了 dx 的債）

注意：兩個條件可以同時成立！

當斜率 ≈ 1（dx ≈ dy）時，某些步驟 x 和 y 都需要走一步（對角線移動）。 兩個 if 都觸發 → x+1, y+1，這是正確的！

用 e2（err 的快照） 而不是即時的 err 判斷兩個條件， 就是為了讓這兩個判斷基於同一個時間點的值。

7
跑一遍（通用版，跟程式碼一樣）

用同一個例子 (0,0) → (7,3)，dx=7, dy=3，sx=1, sy=1：

初始 err = dx − dy = 7 − 3 = 4

2(2,1)510 是 → x=3, err=2 否 2
3(3,1)24 是 → x=4, err=-1 是 → y=2, err=6 6

注意 高亮行 是兩個條件都成立的步驟（x 和 y 都走），對應斜率讓直線「跨過對角線」的時刻。

結果：(0,0) (1,0) (2,1) (3,1) (4,2) (5,2) (6,3) (7,3) — 和第 2 節的浮點版完全一致！

8
對應回程式碼
// dx=7, dy=3, sx=1, sy=1

int dx = abs(x2-x1); // = 7
int dy = abs(y2-y1); // = 3
int sx = (x1 < x2) ? 1 : -1; // 步進方向
int sy = (y1 < y2) ? 1 : -1;
int err = dx - dy; // 初始 = 7-3 = 4

while(true) {
    SetPixel(x1, y1, color); // 畫這格
    if (x1==x2 && y1==y2) break; // 到終點了

    int e2 = 2 * err; // 快照，避免兩個 if 互相干擾

    if (e2 > -dy) { // Case A：走 x
        err -= dy;
        x1 += sx;
    }
    if (e2 < dx) { // Case B：走 y（可以同時）
        err += dx;
        y1 += sy;
    }
}
