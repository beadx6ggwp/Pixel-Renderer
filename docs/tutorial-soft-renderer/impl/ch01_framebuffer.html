IMPL-01 · 像素與 Framebuffer
Framebuffer 與 SetPixel
逐行解析 RenderDevice 的初始化與最底層的像素操作
對應理論章節：Ch01 — 什麼是光柵化？ ——先讀理論再來看程式碼，效果最好。

1 資料結構：RenderDevice 知道什麼？

RenderDevice 只需要知道四件事，就能在任何位置畫像素： 哪塊記憶體（frame_buffer）、每行幾個位元組（pitch）、 畫面寬（width）、畫面高（height）。

render_device.h — private 成員
30private:
31 ScreenManager* screen; // 視窗系統橋接器
32 unsigned char* frame_buffer; // 記憶體起點
33 int pitch; // 每行的位元組數
34 int width; // 畫面寬（像素）
35 int height; // 畫面高（像素）
為什麼需要 pitch？
Framebuffer 在記憶體裡是一維陣列。理論上每行 = width × 4 bytes（ARGB 各 1 byte）， 但顯示驅動程式常因對齊需求在每行末尾加入 padding bytes。 pitch（又稱 stride）= 每行實際佔用的位元組數，包含 padding。 永遠用 pitch 計算位址，不要用 width × 4，否則跨行時位址會錯。

2 建構子：向 ScreenManager 問到所有資訊

render_device.cpp — RenderDevice::RenderDevice()
7RenderDevice::RenderDevice(ScreenManager* s) : screen(s) {
8 frame_buffer = screen->GetFrameBuffer(); // unsigned char* 起始位址
9 pitch = screen->GetPitch(); // 每行位元組數
10 width = screen->GetWidth(); // 畫面寬
11 height = screen->GetHeight(); // 畫面高
12}

ScreenManager 是平台抽象層（Windows GDI / SDL 等）， RenderDevice 只透過它取得 framebuffer 指標， 之後的所有像素操作都直接操作記憶體，不再呼叫任何平台 API。

3 Clear()：填滿整個畫面

render_device.cpp — RenderDevice::Clear()
16void RenderDevice::Clear(uint32_t color) {
17 for (int y = 0; y < height; y++) {
18 uint32_t* row = (uint32_t*)(frame_buffer + y * pitch);
19 std::fill(row, row + width, color);
20 }
21}
第 18 行：位址計算
frame_buffer + y * pitch 是第 y 行的起始位址（unsigned char* 型別）。
強制轉型為 uint32_t* 後，每次 +1 跳 4 bytes（一個 ARGB 像素）。

第 19 行：std::fill
用標準庫的填充函式，從 row[0] 填到 row[width-1]，每個寫入 4 bytes 的顏色值。 底層通常會編譯成 SIMD 指令（SSE/AVX），一次寫 16/32 bytes。

4 SetPixel()：在指定座標畫一個像素

render_device.cpp — RenderDevice::SetPixel()
23void RenderDevice::SetPixel(int x, int y, uint32_t color) {
24 if (x >= 0 && x < width && y >= 0 && y < height) {
25 *(uint32_t*)(frame_buffer + y * pitch + x * 4) = color;
26 }
27}
第 24 行：邊界檢查
DrawLine / DrawTriangle 傳入的座標可能超出畫面（例如 Bresenham 走到畫面外）。 這裡做一次統一的檢查，讓上層呼叫者不需要各自處理越界問題。

第 25 行：位址公式
frame_buffer + y * pitch → 跳到第 y 行
+ x * 4 → 在該行往右跳 x 個像素（每像素 4 bytes）
強制轉為 uint32_t* 再解引用，一次寫入 4 bytes 的顏色。
顏色格式：ARGB（0xAARRGGBB）
uint32_t 的 4 bytes 對應：A（透明度）· R（紅）· G（綠）· B（藍）。 例如不透明紅色 = 0xFF0000FF（視平台的位元組順序而定）。 目前 DrawTriangle 等函式直接傳入 uint32_t color 不做轉換， 顏色打包由呼叫端負責。

5 目前 render_device 的完整狀態

完成本章後，render_device 具備的能力：

✓ 向 ScreenManager 取得 framebuffer 指標與尺寸資訊
✓ Clear()：用指定顏色填滿整個畫面（逐行 std::fill）
✓ SetPixel()：在 (x,y) 位置安全寫入單一像素（含邊界檢查）
