PART 3 · CHAPTER 12
Z-buffer 與背面剔除
解決「哪個三角形在前面」的問題，以及「根本不用畫背面」的優化
Z-buffer depth test backface culling winding order 深度插值

到目前為止，DrawTriangle 已經能正確填滿一個三角形。 但當場景裡有多個三角形，而且它們在 3D 空間中前後交疊時，問題就出現了： 後畫的三角形會蓋掉先畫的，不管它實際上是在前面還是後面。

1
畫家算法（Painter's Algorithm）的問題

最直觀的解法是「畫家算法」：先把所有三角形按深度排序， 從最遠的畫到最近的，這樣近的自然蓋住遠的，跟真實世界一樣。

畫家算法有個致命缺陷：有些三角形根本排不出前後順序。

更根本的問題是：即使三角形不互相穿插，光是「對每個三角形選一個代表深度來排序」本身就不精確—— 兩個三角形可能在不同像素位置各有前後。

畫家算法的問題總結：
· 排序時間 O(n log n)，每幀都要重新排
· 循環遮擋無解（A 蓋 B，B 蓋 C，C 蓋 A）
· 三角形互相穿插（intersecting）時一定出錯
· 每個三角形代表深度難以定義

2
Z-buffer：把問題推到像素層級

Z-buffer（也叫 Depth Buffer）的思路完全不同。 它不試圖對三角形排序，而是把「哪個最近」的問題推到每個像素來解決：

對每個像素 (x, y)，額外維護一個值 zbuffer[y][x]，記錄「目前為止畫到這個像素的最近深度」。
當有新的三角形要畫這個像素時，只有它的深度比 zbuffer 裡記錄的更小（更近），才真正寫入 framebuffer 並更新 zbuffer。

這個做法的美妙之處在於：不需要排序。 三角形可以任意順序送進來，Z-buffer 自動保證每個像素顯示最近的那個。 複雜度從 O(n log n) 降到 O(1)（每個像素只做一次比較）。

3
Z-buffer 的資料結構

Z-buffer 是一個和 framebuffer 同樣大小的浮點陣列：

// render_device.h 中新增 float* zbuffer; // 大小 = width × height

初始化

每幀開始時，和 Clear() 一起重置：所有值設為「正無窮大」， 表示「還沒有任何三角形畫到這個像素」。

void RenderDevice::ClearDepth() { for (int i = 0; i < width * height; i++) zbuffer[i] = std::numeric_limits<float>::infinity(); }
慣例：z 值的範圍
在標準化裝置座標（NDC）中，z 的範圍是 [0, 1]（Near Plane = 0，Far Plane = 1）。 初始值設為 +∞ 或 1.0f 都可以——任何有效的三角形 z 值都會勝過它。

我們的 Vertex 的 z 目前還在螢幕座標（可以是任意值）， 後面引入投影矩陣後才會有正規化的 [0, 1] 範圍。

4
深度插值：重心座標的第一個實際應用

像素 (x, y) 的深度不是三個頂點深度的隨便一個——它應該根據像素的位置， 在三個頂點之間線性插值：

$z_P = w_1 z_1 + w_2 z_2 + w_3 z_3$
\(w_1, w_2, w_3\) — 重心座標（Ch10 求得）
\(z_1, z_2, z_3\) — 三個頂點的 z 值（Vertex.z）
\(z_P\) — 插值後像素的深度

這就是「重心座標可以插值任何頂點屬性」的第一個具體例子。 後面的顏色、UV、法向量，全部用同樣的公式。

注意：這裡暫時使用線性插值。
在 3D 投影後，z 值的線性插值其實是不精確的（透視失真）。 正確做法是對 1/z 做插值，叫做透視正確插值（Perspective Correct Interpolation）。 這個問題在引入 MVP 矩陣後（Part 4+）會詳細討論。

5
背面剔除（Backface Culling）

Z-buffer 解決了「畫錯像素」的問題，但它還是讓 GPU（或我們的 CPU）浪費時間 光柵化一個最後完全被擋住的三角形。 背面剔除（Backface Culling）是一個更早的優化： 在進入光柵化之前，直接丟棄「朝後」的三角形。

什麼是「背面」？

一個三角形有兩個面。哪一面是「正面」（朝向攝影機）由頂點繞向（Winding Order）定義：

用叉積判斷繞向

三角形的「有向面積」就是 Ch08 學的 2D 叉積——它的符號正好反映了頂點的繞向：

$\mathbf{e}_{01} = \mathbf{V}_1 - \mathbf{V}_0,\qquad \mathbf{e}_{02} = \mathbf{V}_2 - \mathbf{V}_0$ $E_{\text{total}} = e_{01}.x \cdot e_{02}.y - e_{01}.y \cdot e_{02}.x$ $\begin{array}{ll} \textcolor{#a6e3a1}{E_{\text{total}} > 0} & \Rightarrow\;\text{CCW，正面，繼續光柵化}\\[6pt] \textcolor{#f38ba8}{E_{\text{total}} < 0} & \Rightarrow\;\text{CW，背面，丟棄}\\[6pt] E_{\text{total}} = 0 & \Rightarrow\;\text{退化三角形（三點共線），丟棄} \end{array}$
CCW 還是 CW 是正面？這是約定，不是真理。
OpenGL 預設 CCW 為正面；Direct3D 預設 CW 為正面。 只要整個管線（建模工具、程式碼）統一即可。 我們的實作和 OpenGL 同一慣例：CCW = 正面（E_total > 0）。

為什麼這個優化有效？

一個封閉的 3D 物體（球、盒子、角色）有一半的三角形永遠朝後。 這些三角形在投影到螢幕後，它們的頂點繞向會反轉， 叉積符號就會跟著反轉——可以在光柵化前直接丟棄。 理想情況下省掉一半的工作量。

6
整合後的完整演算法

把背面剔除和 Z-buffer 加進 DrawTriangle，流程變成：

輸入
三頂點 V0, V1, V2（含 z 座標）
↓
Step 0 ← 新增
計算 E_total（有向面積）；E_total ≤ 0 → return（背面剔除）
↓
Step 1 ← Ch11
計算包圍盒，clamp 到畫面邊界
↓
Step 2 ← Ch09
對包圍盒內每個像素做 Edge Function 測試
↓
Step 3 ← Ch10
通過測試 → 計算重心座標 w1, w2, w3
↓
Step 4 ← 新增
插值深度 z_P = w1·z1 + w2·z2 + w3·z3；比較 zbuffer[y][x] → 失敗則 continue
↓
Step 5
更新 zbuffer[y][x] = z_P；SetPixel(x, y, color)

偽碼

1void DrawTriangle(Vertex v0, Vertex v1, Vertex v2) {
3 // ── Step 0：背面剔除 ─────────────────────────────────────────
4 float e_total = (v1.x-v0.x)*(v2.y-v0.y) - (v1.y-v0.y)*(v2.x-v0.x);
5 if (e_total <= 0.0f) return; // 背面或退化三角形，直接跳過
7 // ── Step 1：包圍盒 ───────────────────────────────────────────
8 int xmin = max(min(v0.x, v1.x, v2.x), 0);
9 int xmax = min(max(v0.x, v1.x, v2.x), width-1);
10 int ymin = max(min(v0.y, v1.y, v2.y), 0);
11 int ymax = min(max(v0.y, v1.y, v2.y), height-1);
13 for (int y = ymin; y <= ymax; y++) {
14 for (int x = xmin; x <= xmax; x++) {
16 // ── Step 2-3：Edge Function + 重心座標 ───────────────
17 float w0 = edge(v1, v2, x, y);
18 float w1 = edge(v2, v0, x, y);
19 float w2 = edge(v0, v1, x, y);
20 if (w0 < 0 || w1 < 0 || w2 < 0) continue;
21 w0 /= e_total; w1 /= e_total; w2 /= e_total; // 正規化
23 // ── Step 4：深度測試 ──────────────────────────────────
24 float z = w0*v0.z + w1*v1.z + w2*v2.z;
25 if (z >= zbuffer[y*width+x]) continue; // 更遠，丟棄
27 // ── Step 5：寫入 ──────────────────────────────────────
28 zbuffer[y*width+x] = z;
29 SetPixel(x, y, color);
31 } // end x
32 } // end y
33}

7
Z-buffer 的限制與現代做法

問題 說明 現代解法
Z-fighting 兩個三角形幾乎同深度，浮點誤差導致像素交替閃爍 Polygon Offset / Reverse-Z
精度不均勻 Near Plane 附近精度高，Far Plane 附近精度極差 Reverse-Z（把 Near 對應 1.0，Far 對應 0.0）
透明物件 透明三角形需要從後往前混合（blend），Z-buffer 會阻擋 OIT（Order Independent Transparency）/ 分層排序
記憶體頻寬 每個像素每幀都要讀寫 zbuffer，頻寬壓力大 HZB（Hierarchical Z-Buffer）/ Early-Z
Reverse-Z 是現代引擎的標準做法。 把 z 範圍反過來（Near = 1.0, Far = 0.0），搭配 32-bit float， 讓精度集中在遠處（浮點數在 0 附近精度更高）。 Unreal Engine 5 和 Unity HDRP 都預設使用 Reverse-Z。
