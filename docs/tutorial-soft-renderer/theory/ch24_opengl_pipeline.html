PART 7 · CHAPTER 24
OpenGL 管線剖析
把 Ch01–Ch21 建立的每一個環節，對應到 OpenGL 的渲染管線——看清楚哪些是固定機制、哪些是可程式化的鉤子
渲染管線 vertex shader fragment shader 固定管線 GPU 並行

從 Ch01 到 Ch21，我們從零建立了一條固定管線： Bresenham 畫線、邊緣函數填充三角形、重心座標插值、MVP 矩陣變換、Z-buffer 深度測試。 這條管線能正確渲染，但它是寫死的——顏色固定、光照模型固定、頂點處理固定， 想要任何不同效果都要改動渲染器核心。

OpenGL 2.0 在 2004 年用可程式化管線解決了這個問題， 它把整條管線只開放兩個「鉤子」讓你寫程式：Vertex Shader 和 Fragment Shader。 其他所有機制——裁剪、光柵化、深度測試——仍然是 GPU 固定執行的。 本章的任務是把我們手寫的每一個步驟，清楚地對應到 OpenGL 管線的每一個階段。

1
固定管線的根本限制

現在的 DrawTriangle 簽名是這樣的：

void DrawTriangle(const Vertex& v1, const Vertex& v2, const Vertex& v3, uint32_t color);

傳入三個頂點（已是螢幕座標）和一個固定顏色，整個三角形只能填一種色。 如果想做逐像素光照（每個 pixel 的亮度不同），就完全沒有辦法—— 一個 uint32_t 無法表達這件事。

想做任何效果都得在 DrawTriangle 外面預先算好：

// Flat Shading：整面同色——勉強可以 float intensity = dot(face_normal, light_dir); device->DrawTriangle(v1, v2, v3, apply_light(base_color, intensity)); // Gouraud Shading：每個頂點顏色不同，像素之間要插值——完全做不到 // Phong Shading：每個像素用插值法向量計算光照——更做不到 // 貼圖採樣：fragment 要查貼圖表——也做不到
固定管線的根本問題： 顏色計算邏輯「寫死」在 DrawTriangle 裡。 任何想客製化的效果，都要修改渲染器核心——這讓「引擎」和「遊戲/應用邏輯」完全耦合在一起， 每換一種光照模型就要重寫一次光柵化迴圈。

4
Vertex Shader 的輸入與輸出

Vertex Shader 是一個純函數，每個頂點獨立執行一次。它的輸入來自兩個地方：

Attributes — 每頂點獨有
  • 位置（Object Space）
  • 法向量
  • UV 座標
  • 頂點顏色
Uniforms — 整個 DrawCall 共用
  • MVP 矩陣
  • 光照方向
  • 貼圖（sampler）
  • 相機位置

Vertex Shader 必須輸出：

  • gl_Position：頂點在 Clip Space 的齊次座標 \((x, y, z, w)\)
  • Varyings：任何你想傳給 Fragment Shader 的值（UV、世界座標法向量…）——Rasterizer 會自動做插值
// GLSL Vertex Shader — 最典型的結構 layout(location = 0) in vec3 aPosition; // attribute：物件空間位置 layout(location = 1) in vec2 aUV; // attribute：UV 座標 layout(location = 2) in vec3 aNormal; // attribute：法向量 uniform mat4 uMVP; // uniform：MVP 矩陣 uniform mat4 uModelInvT; // uniform：法向量矩陣（M 的逆轉置） out vec2 vUV; // varying：傳給 fragment shader out vec3 vNormal; // varying out vec3 vWorldPos; // varying void main() { gl_Position = uMVP * vec4(aPosition, 1.0); // → Clip Space vUV = aUV; vNormal = (vec3)(uModelInvT * vec4(aNormal, 0.0)); // 法向量變換 vWorldPos = aPosition; // 暫時不做 Model 變換 }

5
Fragment Shader 的輸入與輸出

Fragment Shader 在光柵化後執行，針對每一個候選像素（fragment）計算顏色。 它接收 Rasterizer 插值好的 varyings，輸出最終顏色。

// GLSL Fragment Shader — Lambert 漫反射 + 貼圖 in vec2 vUV; in vec3 vNormal; uniform sampler2D uDiffuse; // uniform：diffuse 貼圖 uniform vec3 uLightDir; // uniform：光照方向（世界空間，已正規化） out vec4 fragColor; void main() { vec3 N = normalize(vNormal); float ndotl = max(dot(N, uLightDir), 0.0); // Lambert 漫反射 vec4 albedo = texture(uDiffuse, vUV); // 採樣貼圖 fragColor = vec4(albedo.rgb * ndotl, 1.0); // 最終輸出 }
Fragment ≠ Pixel（重要區別）： Fragment 是光柵化產生的「候選像素」，不保證最終寫入螢幕。 Fragment Shader 執行後，還要通過 Depth Test（比較 Z-buffer）和其他 Fragment Operations。 在我們的軟體光柵化中，if (depth >= zbuffer[…]) continue; 做的正是這件事—— 這一行發生在 fragment() 之前。

6
GPU 並行：為什麼 Shader 被設計成「純函數」

GPU 透過大量並行執行 Shader 呼叫來達到高效能。理解這個模型，你就理解了為什麼 Shader 要被設計成獨立執行、不能互相通訊。

Vertex Shader 並行

10 萬個三角形 → 30 萬次 vertex() 呼叫。 GPU 用數千個 core 同時執行，每個頂點完全獨立——沒有任何頂點需要等待另一個的結果。

Fragment Shader 並行

一個三角形可能覆蓋數千個像素 → 數千次 fragment() 呼叫。 GPU 把 32 個 fragment 打成一組（warp），同時執行相同指令流（SIMD）。

關鍵限制：Vertex Shader 執行時不能讀取「隔壁頂點」的資料； Fragment Shader 執行時不能讀取「隔壁像素」的顏色。 每次呼叫完全隔離——這是 GPU 能高效並行的根本原因。

軟體光柵化的對應： 我們的 DrawTriangle 雙層 for 迴圈在 CPU 上依序跑完所有像素，邏輯等價於 GPU 的光柵化。 GPU 做的事情在數學上完全相同——只是把「每個像素的 fragment() 呼叫」用數千個 core 並行展開。 理解了這一點，你就理解了為什麼 Shader 不能有「全域可變狀態」（global mutable state）： 沒有隔壁，就沒有共享狀態的必要。

Ch24 重點整理：
  • OpenGL 管線有六個階段，只有 ② Vertex Shader 和 ⑤ Fragment Shader 是可程式化的
  • 我們的 DrawTriangle 對應 ④ Rasterization——這部分邏輯保持不變
  • 目前「寫死的 MVP 變換」要搬進 vertex()，「寫死的 color」要搬進 fragment()
  • Varyings 是連接兩個 Shader 的橋樑：vertex 寫入，Rasterizer 插值，fragment 讀取
  • GPU 靠「每個 Shader 呼叫完全獨立」來做大規模並行——軟體版邏輯相同，只是單執行緒執行
