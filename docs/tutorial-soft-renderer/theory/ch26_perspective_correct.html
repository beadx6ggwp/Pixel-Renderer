DrawTriangle 重構 + 透視正確插值
把 DrawTriangle 改成接受 IShader 的可程式化版本； 從零推導透視正確插值（1/w trick）：螢幕重心座標和 3D 重心座標到底差在哪裡？
DrawTriangle 重構 1/w trick 透視修正插值 Z-buffer 完整管線

Ch24 告訴我們 OpenGL 管線的兩個可程式化鉤子（vertex、fragment）， Ch25 定義了 IShader 介面。 現在是最後一塊拼圖：把 DrawTriangle 的光柵化迴圈改造成呼叫 Shader 的版本。

但改造的過程中有一個隱藏的數學陷阱： 我們在 Ch10 用重心座標插值屬性，但那套公式假設頂點在同一個平面， 而透視投影會破壞這個假設——在螢幕空間算出的重心座標和 3D 空間的重心座標不一樣。 不修正就直接插值 UV，會產生明顯可見的扭曲。

本章的任務就是同時解決兩件事：API 重構和透視修正。

1
回顧：現有 DrawTriangle 的三個問題

Ch12 之後，我們的 DrawTriangle 已經能正確填充三角形。 但回頭看它的簽名，有三個根本限制阻止我們進入可程式化管線：

// render_device.h — 現有的簽名 void DrawTriangle(const Vertex& v1, const Vertex& v2, const Vertex& v3, uint32_t color); // 問題 ①：頂點已經是螢幕座標 // → MVP 變換必須在外部手動完成，vertex shader 無法參與 // 問題 ②：顏色是固定的 uint32_t // → fragment shader 完全沒有執行的機會 // 問題 ③：沒有 Z-buffer // → 深度排序完全依賴繪製順序（Painter's Algorithm 的困境）

新的 DrawTriangle 要一次解決這三個問題。 新簽名很簡單——但背後的光柵化迴圈需要精心改造：

// 新版簽名 void RenderDevice::DrawTriangle(IShader& shader, int face_idx);

2
新 DrawTriangle 的五個步驟

一次 DrawTriangle(shader, fi) 呼叫，光柵化器依序執行五個步驟：

步驟 ①~③ 是頂點階段（每個三角形執行一次）， 步驟 ④~⑤ 在像素層級的迴圈裡反覆執行（每個候選像素一次）。 透視修正發生在 ④ → ⑤ 之間——把螢幕空間重心座標轉換成透視修正版本， 再傳給 fragment()。

3
頂點階段：vertex → clip → NDC → 螢幕

新版 DrawTriangle 的前三步，對應 Ch18–Ch21 的 MVP 管線：

// ── ① 呼叫 vertex shader 三次 ──────────────────────────────── Vec4f clip[3]; for (int i = 0; i < 3; i++) clip[i] = shader.vertex(face_idx, i); // ── ② 透視除法 → NDC ──────────────────────────────────────── float w[3]; // ★ 保留每個頂點的 clip-space w 值 Vec3f ndc[3]; for (int i = 0; i < 3; i++) { w[i] = clip[i].w; ndc[i] = { clip[i].x / w[i], clip[i].y / w[i], clip[i].z / w[i] }; } // ── ③ Viewport 變換 → 螢幕座標（Ch20 的公式）─────────────── float sx[3], sy[3], sz[3]; for (int i = 0; i < 3; i++) { sx[i] = (ndc[i].x + 1.0f) * 0.5f * width; sy[i] = (1.0f - ndc[i].y) * 0.5f * height; // Y 翻轉 sz[i] = (ndc[i].z + 1.0f) * 0.5f; // 深度映射到 [0,1] }
為什麼要保留 w[i]？ 透視除法之後，NDC 座標裡的 w 資訊就「丟失了」。 但我們後面做透視修正插值必須用到每個頂點的 clip-space w 值。 如果只存 NDC 而不存 w，就無法修正。這是實作中最容易遺忘的細節。

4
透視投影的陷阱：一條邊的故事

接下來要做光柵化——用 Ch10 的重心座標插值屬性（UV、法向量…）。 但這裡隱藏了一個問題。我們先看一條線段的情況，把直覺建立起來。

側視圖：看清螢幕中點 ≠ 3D 中點

考慮三角形的一條邊 AB：A 在近處（depth 小），B 在遠處（depth 大）。 設 A 的 UV=0、B 的 UV=1，3D 中點 M 的 UV=0.5。

核心直覺：近處的頂點在螢幕上佔用的範圍比較大（因為離相機近，看起來比較大）， 所以螢幕中點在 3D 空間中其實偏向近處的頂點。 如果我們在螢幕中點用 UV=0.5（各佔一半），就是錯的—— 正確值應該更偏向 A 的 UV=0（近處頂點的權重應該更高）。

錯誤的根源： 我們在 Ch10 推導的重心座標公式，是用螢幕空間的 2D 座標計算的。 但「螢幕空間的重心座標」和「3D 空間的重心座標」在透視投影下不相等。 直接用螢幕重心座標插值 UV，就相當於在螢幕中點強制給 UV=0.5—— 而不是給物件在 3D 空間中真正對應的 UV 值。

5
從零推導：螢幕重心座標 vs 3D 重心座標

現在我們有了直覺，來做嚴格的數學推導。 這是整個教學系列最重要的一段數學——不跳步驟，完整走一遍。

Step 1：定義兩組重心座標

三角形三個頂點的 clip-space 座標為 \(\mathbf{v}_i = (x_i, y_i, z_i, w_i)\)，\(i = 0, 1, 2\)。

投影到螢幕後，2D 位置是 \(\mathbf{p}_i = (x_i/w_i,\; y_i/w_i)\)。 對螢幕上的任一點 P，我們用 2D 面積比算出螢幕空間重心座標 \(\mu = (\mu_0, \mu_1, \mu_2)\)，使得：

$P = \mu_0\,\mathbf{p}_0 + \mu_1\,\mathbf{p}_1 + \mu_2\,\mathbf{p}_2, \qquad \mu_0 + \mu_1 + \mu_2 = 1$

這就是 Ch10 的公式——完全正確，只是它算出的是螢幕空間的權重。

同時，在 3D 空間中，同一個點 P 對應的3D 重心座標 \(\lambda = (\lambda_0, \lambda_1, \lambda_2)\) 滿足：

$\text{正確的屬性插值} = \lambda_0\,f_0 + \lambda_1\,f_1 + \lambda_2\,f_2$

問題是：\(\mu_i \neq \lambda_i\)。 我們的光柵化器只能算出 \(\mu\)（因為光柵化發生在螢幕空間）， 但正確插值需要 \(\lambda\)。 我們需要找到一個從 \(\mu\) 轉換到 \(\lambda\) 的公式。

Step 2：建立 \(\mu\) 和 \(\lambda\) 的關係

在 clip space，三角形上的任一點可以用 3D 重心座標表示為齊次座標：

$\mathbf{v} = \lambda_0\,\mathbf{v}_0 + \lambda_1\,\mathbf{v}_1 + \lambda_2\,\mathbf{v}_2$

投影到螢幕後（除以 w 分量），螢幕位置是：

$P = \frac{(\lambda_0 x_0 + \lambda_1 x_1 + \lambda_2 x_2)} {\lambda_0 w_0 + \lambda_1 w_1 + \lambda_2 w_2} \;,\quad \frac{(\lambda_0 y_0 + \lambda_1 y_1 + \lambda_2 y_2)} {\lambda_0 w_0 + \lambda_1 w_1 + \lambda_2 w_2}$

但同時，P 也滿足螢幕重心座標的定義 \(P = \sum \mu_i\, \mathbf{p}_i = \sum \mu_i (x_i/w_i,\; y_i/w_i)\)。 合併兩個表達式，比較 x 分量：

$\sum_{i}\mu_i \frac{x_i}{w_i} = \frac{\sum_{i}\lambda_i x_i}{\sum_{i}\lambda_i w_i}$

令 \(D = \sum_i \lambda_i w_i\)（一個標量），上式變成：

$\mu_i \cdot \frac{x_i}{w_i} \;\leftrightarrow\; \frac{\lambda_i \cdot x_i}{D}$

要讓這個等式對任意 \(x_i\) 值都成立（不只是特定的三角形）， 唯一的可能是每一項都分別相等：

$\boxed{\frac{\mu_i}{w_i} = \frac{\lambda_i}{D}} \qquad \Longleftrightarrow \qquad \lambda_i = D \cdot \frac{\mu_i}{w_i}$

Step 3：求出 D

利用 \(\sum \lambda_i = 1\) 的約束：

$\sum_{i}\lambda_i = D \sum_{i}\frac{\mu_i}{w_i} = 1 \qquad \Longrightarrow \qquad D = \frac{1}{\displaystyle\sum_{i}\frac{\mu_i}{w_i}}$

Step 4：得到最終公式

把 D 代入，得到 3D 重心座標和螢幕重心座標的關係：

透視修正重心座標公式： $\boxed{\lambda_i = \frac{\mu_i \,/\, w_i}{\displaystyle\sum_{j=0}^{2}\;\mu_j \,/\, w_j}}$ 其中 \(\mu_i\) 是螢幕空間重心座標（Ch10 的公式算出的）， \(w_i\) 是第 \(i\) 個頂點的 clip-space w 值， \(\lambda_i\) 是正確的 3D 重心座標——用它來插值任何屬性都是正確的。

直覺驗證：w 值較小的頂點（離相機較近的）獲得更大的權重—— 符合我們在 Section 4 的側視圖直覺。

驗證：正交投影退化

正交投影時，所有頂點的 w 值相同（\(w_0 = w_1 = w_2 = w\)）：

$\lambda_i = \frac{\mu_i / w}{\sum_j \mu_j / w} = \frac{\mu_i}{\sum_j \mu_j} = \mu_i$

公式退化為 \(\lambda_i = \mu_i\)——螢幕重心座標等於 3D 重心座標。 正確！正交投影不存在透視扭曲，不需要修正。

6
透視修正重心座標的程式碼

把公式翻譯成 C++ 只需四行：

// μ = (l0, l1, l2)：螢幕空間重心座標（Ch10 的公式） // w[0..2]：三個頂點的 clip-space w 值 float S = l0/w[0] + l1/w[1] + l2/w[2]; // 分母（1/w 的加權和） Vec3f bar = { // λ = 透視修正重心座標 (l0 / w[0]) / S, (l1 / w[1]) / S, (l2 / w[2]) / S };

有了 \(\lambda\)（即 bar），fragment() 裡的所有屬性插值 都可以直接用簡單的線性組合——因為 \(\lambda\) 已經是正確的 3D 權重：

// fragment() 內：所有 varying 插值都用 bar（已修正） Vec2f uv = { bar.x * v_uv[0].x + bar.y * v_uv[1].x + bar.z * v_uv[2].x, bar.x * v_uv[0].y + bar.y * v_uv[1].y + bar.z * v_uv[2].y }; Vec3f normal = normalize({ bar.x * v_n[0].x + bar.y * v_n[1].x + bar.z * v_n[2].x, bar.x * v_n[0].y + bar.y * v_n[1].y + bar.z * v_n[2].y, bar.x * v_n[0].z + bar.y * v_n[1].z + bar.z * v_n[2].z });
這就是 Ch25 所說的： 「fragment() 收到的 bar 是透視修正後的重心座標」。 光柵化器計算修正版 \(\lambda\) 後傳給 fragment()， 所以 Shader 開發者完全不需要知道 1/w trick 的存在—— 他只要用 bar.x*f[0] + bar.y*f[1] + bar.z*f[2] 就能得到正確結果。

7
深度插值為何不需要修正？

你可能注意到一個奇怪的地方： Z-buffer 的深度值用的是未修正的螢幕重心座標 \(\mu\) 來插值， 而不是修正後的 \(\lambda\)。這不是錯了嗎？

// Z-buffer 深度插值——用 l0, l1, l2（螢幕重心座標 μ），不是 bar（修正版 λ） float depth = l0*sz[0] + l1*sz[1] + l2*sz[2];

沒有錯。原因在於我們插值的是 \(z_{\text{ndc}}\)，而不是原始的 3D 深度。 \(z_{\text{ndc}}\) 的定義是：

$z_{\text{ndc}} = \frac{z_{\text{clip}}}{w_{\text{clip}}}$

它已經除過 w 了。回想 Section 5 的推導： 任何 clip-space 量除以 w 的結果，都可以用螢幕重心座標 \(\mu\) 線性插值。 這正是我們推導過程的出發點——螢幕座標本身就是 \(x_{\text{clip}}/w\) 和 \(y_{\text{clip}}/w\)， 而 \(z_{\text{ndc}} = z_{\text{clip}}/w\) 具有完全相同的性質。

結論（業界標準實作）：
● 深度值 \(z_{\text{ndc}}\)：用螢幕重心座標 \((\mu_0, \mu_1, \mu_2)\) 插值——已經除過 w
● UV、法向量等屬性：用透視修正重心座標 \((\lambda_0, \lambda_1, \lambda_2)\) 插值——還沒除過 w，需要修正

8
新版 DrawTriangle 完整實作

把 Section 3（頂點階段）+ Section 6（透視修正）+ Section 7（深度）組合成完整函式：

void RenderDevice::DrawTriangle(IShader& shader, int face_idx) { // ── ① Vertex Stage ────────────────────────────────────────── Vec4f clip[3]; for (int i = 0; i < 3; i++) clip[i] = shader.vertex(face_idx, i); // ── ② 透視除法 ───────────────────────────────────────────── float w[3]; Vec3f ndc[3]; for (int i = 0; i < 3; i++) { w[i] = clip[i].w; ndc[i] = { clip[i].x/w[i], clip[i].y/w[i], clip[i].z/w[i] }; } // ── ③ Viewport ───────────────────────────────────────────── float sx[3], sy[3], sz[3]; for (int i = 0; i < 3; i++) { sx[i] = (ndc[i].x + 1.f) * .5f * width; sy[i] = (1.f - ndc[i].y) * .5f * height; sz[i] = (ndc[i].z + 1.f) * .5f; } // ── ④ 包圍盒（Ch11）────────────────────────────────────── int minX = std::max(0, (int)std::floor((std::min)({sx[0],sx[1],sx[2]}))); int maxX = std::min(width-1, (int)std::ceil ((std::max)({sx[0],sx[1],sx[2]}))); int minY = std::max(0, (int)std::floor((std::min)({sy[0],sy[1],sy[2]}))); int maxY = std::min(height-1,(int)std::ceil ((std::max)({sy[0],sy[1],sy[2]}))); // ── ⑤ 光柵化迴圈 ───────────────────────────────────────── float denom = (sy[1]-sy[2])*(sx[0]-sx[2]) + (sx[2]-sx[1])*(sy[0]-sy[2]); if (std::abs(denom) < 1e-6f) return; // 退化三角形 for (int y = minY; y <= maxY; y++) { for (int x = minX; x <= maxX; x++) { // 螢幕空間重心座標 μ（Ch10 的公式） float l0 = ((sy[1]-sy[2])*(x-sx[2]) + (sx[2]-sx[1])*(y-sy[2])) / denom; float l1 = ((sy[2]-sy[0])*(x-sx[2]) + (sx[0]-sx[2])*(y-sy[2])) / denom; float l2 = 1.f - l0 - l1; if (l0 < 0 || l1 < 0 || l2 < 0) continue; // 深度測試（用 μ 插值——Section 7 解釋了為什麼這是正確的） float depth = l0*sz[0] + l1*sz[1] + l2*sz[2]; int idx = y * width + x; if (depth >= zbuffer[idx]) continue; // ★ 透視修正重心座標 λ（Section 5 的公式） float S = l0/w[0] + l1/w[1] + l2/w[2]; Vec3f bar = { (l0/w[0])/S, (l1/w[1])/S, (l2/w[2])/S }; // Fragment Shader（bar 已修正，fragment 裡插值就是正確的） uint32_t color; if (!shader.fragment(bar, color)) continue; // 寫入 Framebuffer + 更新 Z-buffer zbuffer[idx] = depth; SetPixel(x, y, color); } } }
Z-buffer 的配置： zbuffer 是 float* 陣列，大小 width × height， 在 RenderDevice 建構時分配。 每幀開始時用 Clear() 重設為 1.0f（最遠深度，因為深度範圍是 [0,1]）。 深度測試 depth >= zbuffer[idx] 表示：新 fragment 比已有的更遠或等距 → 丟棄。

和舊版 DrawTriangle 的程式碼對比： 光柵化迴圈的核心結構（包圍盒 + 重心座標測試）和 Ch12 完全相同， 只增加了三處：vertex() 呼叫、1/w 修正、fragment() 呼叫。 我們沒有重寫光柵化——我們只是在它的外層加了 Shader 介面。

數值範例：手算透視修正

用一個具體的三角形，走一遍透視修正的計算。 這個範例能幫助你驗證自己的實作是否正確。

設定

三個頂點經過 MVP 變換後的 clip-space w 值（代表深度）和 UV 座標：

頂點clip wUV含義
\(V_0\)\(w_0 = 2\)(0, 0)最近（w 最小）
\(V_1\)\(w_1 = 4\)(1, 0)最遠
\(V_2\)\(w_2 = 3\)(0, 1)中間距離

假設螢幕上某個像素 P 計算出的螢幕空間重心座標為 \(\mu = (0.50,\; 0.30,\; 0.20)\)。

Naive 插值（錯誤）

如果直接用 \(\mu\) 插值 UV：

$\text{UV}_{\text{naive}} = 0.50\,(0,0) + 0.30\,(1,0) + 0.20\,(0,1) = \mathbf{(0.300,\; 0.200)}$

透視修正插值（正確）

Step 1：計算分母 \(S\) $S = \frac{\mu_0}{w_0} + \frac{\mu_1}{w_1} + \frac{\mu_2}{w_2} = \frac{0.50}{2} + \frac{0.30}{4} + \frac{0.20}{3} = 0.250 + 0.075 + 0.067 = 0.392$ Step 2：計算透視修正重心座標 \(\lambda\) $\lambda_0 = \frac{0.250}{0.392} = \mathbf{0.639}, \quad \lambda_1 = \frac{0.075}{0.392} = \mathbf{0.191}, \quad \lambda_2 = \frac{0.067}{0.392} = \mathbf{0.170}$ 驗證：\(0.639 + 0.191 + 0.170 = 1.000\) ✓ Step 3：用 \(\lambda\) 插值 UV $\text{UV}_{\text{correct}} = 0.639\,(0,0) + 0.191\,(1,0) + 0.170\,(0,1) = \mathbf{(0.191,\; 0.170)}$

比較結果

\(\mu_0 / \lambda_0\)\(\mu_1 / \lambda_1\)\(\mu_2 / \lambda_2\)UV.xUV.y
Naive（螢幕 \(\mu\)） 0.5000.3000.200 0.3000.200
修正（3D \(\lambda\)） 0.6390.1910.170 0.1910.170

修正後，\(V_0\)（最近的頂點，\(w=2\)）的權重從 0.500 跳到 0.639——增加了 28%。 \(V_1\)（最遠的頂點，\(w=4\)）的權重從 0.300 降到 0.191——減少了 36%。

結果符合直覺：近處的頂點在 3D 空間中佔據更大的「實際面積」， 所以它的屬性值在插值時應該佔更大的權重。 如果用 naive 插值，遠處頂點的 UV 被過度放大，就會產生「貼圖拉伸」的視覺錯誤。

舊版 vs 新版對比

舊版 DrawTriangle（Ch12）
  • 輸入：螢幕座標頂點 + 固定 uint32_t
  • MVP：在外部手動計算
  • 著色：整個三角形同一色
  • 深度：無 Z-buffer
  • 插值：螢幕空間線性（透視下不正確）
  • 擴展：換著色 = 改核心程式碼
新版 DrawTriangle（Ch26）
  • 輸入：IShader& + 三角形索引
  • MVP：vertex() 鉤子，Shader 自行決定
  • 著色：fragment() 鉤子，逐像素可程式化
  • 深度：Z-buffer 深度測試
  • 插值：透視修正重心座標（\(\lambda_i = \mu_i/w_i / S\)）
  • 擴展：換著色 = 換一個 Shader 物件

11
常見錯誤與除錯

錯誤症狀修正
w 值取錯 貼圖扭曲，越靠近相機越嚴重 必須用 clip[i].w（clip space w），不是 NDC z。
NDC 值已經除過 w，不能再用它做修正。
Z-buffer 沒清除 第二幀開始物件消失或全黑 每幀 Clear() 時必須同時重設 zbuffer 為 1.0f。
對深度也做透視修正 深度精度異常，Z-fighting 增加 深度值 \(z_{\text{ndc}}\) 已除過 w，用 \(\mu\) 插值。
只有 UV、法向量等「未除過 w」的屬性才需要用 \(\lambda\)。
fragment() 在 Z-test 前呼叫 效能極差（每個像素都執行 Shader） 先做深度測試，通過後才呼叫 fragment()。
被遮擋的像素直接跳過，不浪費計算。
denom 在迴圈內重算 效能浪費（不影響正確性） 重心座標的分母 denom 對同一個三角形是常數，
提到迴圈外面只算一次。
Ch26 重點整理：
