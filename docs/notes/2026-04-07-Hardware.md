## 你的想法哪裡有根本性的錯誤

**核心誤解：「軟體寫好，後端換成硬體就完成了」**

這個假設不成立。原因：

### 問題 1：軟體和硬體是不同的計算模型

你的 `Rasterizer::DrawTriangle()` 是這樣的：
```cpp
for (int y = min_y; y <= max_y; y++)       // 時間序列，一行一行
    for (int x = min_x; x <= max_x; x++)   // 一個一個像素
        if (barycentric_test)
            SetPixel(x, y, color);
```

硬體版本不是「跑同樣的迴圈，只是更快」。  
硬體是**同時存在的電路**，不是跑程式的機器：

```
Pixel (0,0) → [Bary Unit] → [Z-Test] → [Blend] → VRAM
Pixel (1,0) → [Bary Unit] → [Z-Test] → [Blend] → VRAM   ← 同時進行
Pixel (2,0) → [Bary Unit] → [Z-Test] → [Blend] → VRAM
```

你不是把 `for loop` 搬到 FPGA，你要**重新設計成流水線狀態機（Pipeline FSM）**，這是完全不同的設計思路。

---

### 問題 2：`CommandQueue` 的抽象在 CPU↔FPGA 邊界會洩漏

| | 軟體後端 | 硬體後端 |
|---|---|---|
| 呼叫延遲 | 奈秒（函式呼叫） | 微秒（AXI 匯流排傳輸） |
| 資料傳遞 | 直接記憶體存取 | DMA 搬運，需等待完成中斷 |
| 同步方式 | 函式回傳即完成 | 需輪詢狀態暫存器或等中斷 |
| 記憶體模型 | 共享 heap | FPGA 有自己的 BRAM/DDR，需明確 copy |

你的 `execute_all()` 換成硬體呼叫後，**裡面全部都得重寫**，不是單純替換。

---

### 問題 3：「自製 OpenGL」的難度被嚴重低估

OpenGL 不是「一個畫三角形的 API」，它是：
- **狀態機**（數百個全域狀態：混合模式、深度函數、面剔除方向...）
- **Shader 編譯器**（GLSL → 中間碼 → GPU 機器碼）
- **Context 管理**（多執行緒、多視窗、資源生命週期）
- **驅動層排程**（Command Buffer 重排、GPU 排程優化）

你現在做的 `CommandQueue` + `IShader` 比較接近的真實對應是：  
**Vulkan 的 VkCommandBuffer 錄製介面**，而不是 OpenGL。  
這其實是更現代的設計，但離「完整 GL 實作」還差很遠。

---

### 問題 4：Arty A7-100T 的資源不夠放「完整 GPU 管線」

| 模組 | 估計 LUT 消耗 |
|---|---|
| RISC-V 軟核（VexRiscv）| ~3,000 |
| DDR MIG 控制器 | ~8,000 |
| HDMI 輸出 | ~1,000 |
| 光柵化流水線（基本） | ~5,000 |
| **可程式化 Shader 核心** | ~20,000+ |
| **Texture Sampler** | ~5,000+ |
| **Total** | 接近或超過 100K LUT 上限 |

你大概只能放**固定管線光柵化**（Fixed-Function），不是可程式化 Shader。

// 你自訂的 Shader Assembly（類似 NV20 的 VS 1.1）
DP4  R0.x, v0, c0   // dot product：頂點位置 × MVP 矩陣 row0
DP4  R0.y, v0, c1
DP4  R0.z, v0, c2
DP4  R0.w, v0, c3

- **Shader Assembler**（把上面的 assembly 翻成你的硬體機器碼）

## 固定管線 vs 可程式化管線

### 固定管線（Fixed-Function Pipeline）
硬體電路**寫死**，只能做特定的計算，不能改變邏輯。

```
頂點進來 → [矩陣乘法電路] → [光照電路(只能算 Phong)] → [紋理電路] → 像素
               ↑                    ↑                      ↑
           你只能調參數          你只能調參數            你只能調參數
          （傳入矩陣數值）      （傳入光源位置）        （傳入貼圖）
           但電路邏輯不能改     但電路邏輯不能改        但電路邏輯不能改
```

你能做的只有**調整輸入參數**，不能改變「計算的方式」。  
早期 OpenGL 1.x 的 `glLightfv()`、`glMaterialfv()` 就是這個概念——你只是在填參數。

---

### 可程式化管線（Programmable Pipeline）
硬體裡放的是一顆**可以執行任意指令的小處理器**，你傳入的不是參數，而是**程式碼**。

```
頂點進來 → [Vertex Shader 處理器] → [光柵化（固定）] → [Fragment Shader 處理器] → 像素
                   ↑                                              ↑
            你傳入一段程式碼                                 你傳入一段程式碼
            （GLSL / 你的 ISA）                              （GLSL / 你的 ISA）
            愛怎麼算就怎麼算                                  愛怎麼算就怎麼算
```

---

### 對應到你的專案

| | 你的 C++ | 硬體對應 |
|---|---|---|
| **固定管線** | `Rasterizer.cpp`（掃包圍盒、重心座標，邏輯寫死） | 光柵化電路（固定電路，不可改） |
| **可程式化管線** | `IShader::vertex()` / `fragment()`（可繼承、可任意改） | Vertex/Fragment Shader Unit（小型處理器） |

---

### 為什麼光柵化永遠是固定電路？

因為「把三角形轉成像素」這件事**邏輯永遠一樣**——包圍盒掃描、重心座標判斷、Z-Test。沒有人需要改這個邏輯，做成固定電路反而速度最快，面積最小。

真正需要「可程式化」的只有兩個地方：
- **頂點怎麼變換**（你的 vertex shader）
- **像素顏色怎麼算**（你的 fragment shader）

這就是 NV20 的設計：**可程式 VS + 固定光柵化 + 可程式 FS**，也是你要做的目標。

### 一個可程式 Shader Unit 需要多少資源

最小可行的 Shader 處理器（類 NV20 VS 1.1）：

```
指令記憶體（128 條指令）  → ~4 個 BRAM block
暫存器組（16 個 vec4）    → ~2 個 BRAM block
MAD/DP4 ALU              → ~8 個 DSP block

```

A7-100T 總共有 135 個 DSP block，這個設計只用 8 個。  
**資源上完全可行**，問題在設計複雜度，不在資源。

### 第一層缺口：你需要理解 CPU 是怎麼執行一條指令的

Shader Unit 本質是一顆 CPU，所以你要先懂 CPU 才能設計它。

**最小必要知識：**
```
取指（Fetch）→ 解碼（Decode）→ 執行（Execute）→ 回寫（Writeback）

對應到 Shader Unit：
取指：從指令 BRAM 讀出一條 MAD 指令
解碼：判斷是 MAD，讀出 src1, src2, dst 暫存器編號
執行：DSP block 做乘加運算
回寫：結果存回暫存器組
```

**推薦路徑：**
- 你 notes 裡提到的成大 VLSI/ICLAB 課程第一個 Project 就是 RISC-V CPU
- Jserv 的 RISC-V 課程（他的開源課程連結你已經有了）

### 第二層缺口：GPU 跟 CPU 最關鍵的差異——SIMT

你設計的 Shader Unit 如果只有一條執行路徑，它就是一顆慢 CPU。  
NVIDIA 的突破在於 **SIMT（Single Instruction Multiple Threads）**：

```
【CPU 執行 fragment shader】          【GPU SIMT 執行 fragment shader】
for each pixel:                       32 個像素同時執行同一條指令
    R0 = tex_sample(uv)               [Warp of 32 threads]
    R1 = dot(normal, light)            每個 thread 有自己的暫存器
    color = R0 * R1                    同一時刻執行同一條指令
                                       但各自用自己的資料（不同 uv、不同 normal）
```

這個「32 個 thread 一起跑」的單位叫 **Warp**（NVIDIA 術語）/ **Wavefront**（AMD 術語）。  
你的 A7-100T 設計，最小可行版本可以做 **4-wide SIMD**（4 個像素同時計算）來模擬這個概念。

---

### 第三層缺口：Memory Hierarchy 是性能的真正瓶頸

很多人以為 GPU 快是因為計算單元多，實際上**記憶體頻寬才是真正的瓶頸**：

```
你的軟渲染器：
SetPixel(x, y, color) → 直接寫 host memory，延遲 ~1ns

FPGA 版本：
SetPixel → AXI write → DDR controller → DDR SDRAM → 延遲 ~100ns

如果光柵化每個像素都直接讀/寫 DDR，速度會比你的 CPU 軟渲還慢。
```

**解決方案就是 Tile-Based 的核心價值**：
- 把 64×32 的 tile 放在 on-chip BRAM（延遲 ~1ns）
- 整個 tile 算完再一次性寫回 DDR
- 這就是為什麼 Raster I 選 TBDR，而不是為了學 PowerVR

---

### 第四層：驅動程式是軟硬體的合約

你的 `CommandQueue::execute_all()` 換成硬體後，實際上你在寫的是**裸機 GPU 驅動**：

```c
// 你的驅動程式大概長這樣（裸機 C）
void gpu_draw_triangle(Vertex* verts) {
    // 1. 把頂點資料 DMA 傳到 FPGA BRAM
    axi_dma_transfer(VERTEX_BUFFER_BASE, verts, sizeof(Vertex) * 3);
    
    // 2. 寫控制暫存器，觸發 GPU 開始執行
    mmio_write(GPU_CMD_REG, CMD_DRAW_TRIANGLE);
    
    // 3. 等待完成（輪詢或中斷）
    while (mmio_read(GPU_STATUS_REG) != STATUS_DONE);
}
```

這正是 Linux `drm/i915`（Intel GPU 驅動）或 `nouveau`（開源 NVIDIA 驅動）在做的事，只是規模大上千倍。

### 一、ISA 設計決策：你的 Shader Unit 要做哪些選擇

**決策 1：純量（Scalar）還是向量（Vector）？**

```
【純量 ISA（RISC-V 風格）】       【向量 ISA（NV20 / GPU 風格）】
ADD  R0, R1, R2                  ADD  V0.xyzw, V1.xyzw, V2.xyzw
一次加一個 float                   一次加四個 float（vec4）

優點：電路簡單，好設計             優點：一條指令處理完整頂點座標
缺點：MVP 變換需要 16 條指令        缺點：需要 swizzle 電路（.xyzw → .wzyx 等排列）
```

NV20 選向量，所以它有 swizzle/mask 機制——`.x`、`.xy`、`.zw` 可以獨立讀寫。這讓電路多了一層複雜度，但指令數少很多。

---

**決策 2：指令格式長度**

```
固定長度（32-bit）：
[6-bit op][4-bit dst][4-bit src1][4-bit src2][4-bit swizzle][10-bit 立即數]
優點：取指電路極簡，一個 cycle 取一條
缺點：立即數空間有限，浮點常數要放常數暫存器

可變長度（類 x86）：
優點：編碼密集
缺點：取指需要先掃描長度，流水線 stall 問題複雜
```

你的 Shader Unit **一定選固定長度**，可變長在 FPGA 上幾乎不可行。

---

**決策 3：常數暫存器 vs 立即數**

NV20 的做法是分開兩組暫存器：
```
v0~v15  : Input registers（頂點屬性，只讀）
R0~R11  : Temp registers（計算用，讀寫）
c0~c95  : Constant registers（MVP 矩陣等，CPU 寫入，GPU 只讀）
oPos    : Output position（輸出到光柵化）
o0~o7   : Output registers（傳給 Fragment Shader 的 varying）
```

`c0~c95` 這 96 個常數暫存器就是 **Uniforms** 的硬體實現——你的 `IShader` 裡的 `uniforms` 就對應這裡。

### 二、編譯器工具鏈：Shader 如何從 GLSL 變成你的機器碼

現代路徑：
```
GLSL 原始碼
    │ glslang（前端編譯器）
    ▼
SPIR-V（標準中間碼，Vulkan 使用）
    │ 你的後端編譯器
    ▼
你的 ISA binary
    │ DMA
    ▼
FPGA Shader Unit 執行
```

你的最小可行路徑：

```
手寫 Shader Assembly（你自己的 .vsa 格式）
    │ Python Assembler（200 行）
    ▼
你的 ISA binary（.bin）
    │
    ▼
FPGA 執行
```

等軟體成熟後，可以考慮實作一個小型編譯器把 GLSL subset 翻成你的 Assembly。這本身就是一個完整的編譯器課題（詞法分析 → AST → 中間碼 → 指令選擇 → 暫存器分配）。

### 三、並行度的三個層次

你要從單核走向多核，有三條路可以走：

```
【Level 1：指令級並行（ILP）】
同一顆 Shader Unit 內，流水線化
取指  →  解碼  →  執行  →  回寫  （4 stage pipeline）
當 stage 1 在取第 2 條指令，stage 2 在解碼第 1 條指令
提速：~2x（受 data hazard 限制）

【Level 2：資料並行（SIMD）】
一條指令同時處理 N 個像素
MAD  [px0, px1, px2, px3],  [R1x4], [c0x4]
4-wide SIMD → 需要 4 組 DSP block
提速：~4x
NV20 就是這樣做的，4 個 float 同時計算

【Level 3：執行緒並行（SIMT）】
N 個「thread」同時執行，各自有獨立暫存器組
當 thread 0 在等 texture fetch（記憶體延遲）
GPU 切換去跑 thread 1、thread 2...
提速：隱藏記憶體延遲，讓計算單元不閒置
GeForce 3 開始引入，G80 完整實作成 Warp
```

**A7-100T 的務實目標**：Level 1（流水線）+ Level 2（4-wide SIMD），不做 Level 3。  
Level 3 需要大量 BRAM 存放多組暫存器，A7-100T 放不下足夠的 thread。

### 四、Texture Unit 是一個獨立的大問題

你以為 texture sampling 很簡單：UV → 查表 → 顏色。實際上硬體 Texture Unit 要處理：

```
問題 1：MIP Map
遠處物體用低解析度貼圖，避免摩爾紋
需要預先產生 N 個解析度層級
選哪一層：根據 ddx/ddy（屏幕空間導數）決定
→ 需要每 2×2 像素一起計算（quad dispatch）

問題 2：雙線性過濾
4 個相鄰 texel 加權平均
需要 4 次 BRAM 讀取 + 乘加
如果 BRAM 是單端口，4 次讀取需要 4 個 cycle
→ 流水線設計

問題 3：Cache
貼圖資料在 DDR，但每個像素都讀 DDR → 頻寬爆炸
需要 Texture Cache（通常是 4-way set associative）
→ 大量 BRAM + 命中率邏輯

問題 4：貼圖格式
RGB888 / RGBA8888 / 壓縮格式（DXT/BC1）
硬體解壓縮電路
```

這就是為什麼 Raster I 說「如果 BRAM 夠才加 Texture Unit」——光是 Texture Cache + 雙線性過濾就要吃掉很多 BRAM。

**你的 A7-100T 策略**：先用 BRAM 直接存小貼圖（64×64 @ RGBA8888 = 16KB = 2 個 BRAM block），不做 MIP Map，不做 Cache。等功能驗證完再優化。

### 五、驗證方法論：怎麼確認硬體是對的

這是硬體開發最關鍵、最容易被忽略的部分：

```
【軟體開發的 debug 方式】
執行程式 → 看輸出 → 不對就加 printf → 找到問題

【硬體開發的 debug 方式】
燒進 FPGA → 看輸出 → 不對 → 😱
  你不能加 printf
  你不能單步執行
  你不知道哪個 cycle 出問題
  你只能看最終結果錯了
```

**正確的硬體驗證流程（你會用到的）：**

```
Step 1：仿真優先（Simulation First）
   在燒板子之前，用 Vivado Simulator 跑測試
   用 C++ 軟渲染器產生「黃金向量」（Golden Vector）
   相同輸入 → 比對 HDL 仿真輸出 vs C++ 輸出
   這一步能找到 90% 的邏輯錯誤

Step 2：ILA（Integrated Logic Analyzer）
   Vivado 內建的硬體示波器
   在 RTL 中插入探針，燒進 FPGA 後抓波形
   可以看到每個 cycle 的訊號狀態
   缺點：會多佔 LUT，且只能抓有限深度的波形

Step 3：逐像素對拍
   FPGA 渲染完一幀 → 透過 UART/USB 把 Framebuffer 傳回 PC
   PC 上的軟渲染器渲染同一幀
   逐像素比對，找到第一個不一致的像素

Step 4：漸進式整合
   不要一次把所有模組都接起來
   先驗證光柵化電路（不接 Shader）
   再驗證 Shader Unit（不接光柵化）
   最後串接
```

這個驗證方法論，就是你軟體渲染器最重要的隱藏價值——它是你的硬體測試工具。

## 顯卡與渲染技術演進史

---

### 第零代：CPU 純軟體渲染（1980s）

**你正在做的事，當年是唯一的方法。**

```
代表作：
  id Software《Wolfenstein 3D》(1992)  — John Carmack 手刻軟渲
  id Software《DOOM》(1993)            — BSP Tree + 假 3D（實為 2.5D）
  id Software《Quake》(1996)           — 真正的 3D，純 CPU 軟渲

技術特徵：
  沒有 GPU，全部在 CPU 上算
  Quake 用的掃描線光柵化（scanline rasterizer）
  就是你 index.html 裡有的那些算法

Carmack 的貢獻：
  BSP Tree（空間分割，解決遮擋排序）
  Binary Space Partitioning → 你的 Frustum Culling 的前身
  Surface Caching（把光照預先算好貼在牆上，減少每幀計算量）
  → 這是 Lightmap 的起源
```

**效能瓶頸**：CPU 是序列的，每個像素一個接一個算，畫面越大越慢。

---

### 第一代：2D 加速卡 → 固定 3D 管線（1995~1998）

**硬體開始接手「搬像素」這件事。**

```
1995：
  S3 Virge、Matrox Mystique
  功能：硬體加速 2D（BitBlt、線段填充）
  3D 功能：只有貼圖填充，光照還是 CPU 算
  你的 ScreenManager::BitBlt() 就是這個年代的核心操作

1996：
  3dfx Voodoo 1  ← 第一張真正的 3D 加速卡
  功能：硬體三角形光柵化 + 貼圖採樣 + Z-Test
  API：Glide（3dfx 私有 API，不是 OpenGL）
  
  技術突破：
  把你的 Rasterizer::DrawTriangle() 做成硬體電路
  第一次讓《Quake》能流暢跑在硬體上

1997：
  OpenGL 1.0 普及、Direct3D 3.0
  第一次有「標準 API」，不用針對每張卡寫不同程式碼
  
  這等同於：
  你的 CommandQueue 第一次有了跨廠商的標準格式
```

**這個時代的管線：CPU 算頂點 → 顯卡光柵化像素**

---

### 第二代：T&L 硬體化，固定管線完成（1999~2000）

**NV10：GeForce 256，「世界第一張 GPU」（NVIDIA 自稱）**

```
1999：NVIDIA GeForce 256
  革命性改變：Transform & Lighting (T&L) 移到硬體
  
  之前（軟體 T&L）：
    CPU 要算 ModelMatrix × ViewMatrix × ProjectionMatrix × vertex
    場景有 10 萬個頂點，CPU 就要做 10 萬次矩陣乘法
    CPU 幾乎全程在做矩陣運算，沒有時間跑遊戲邏輯
  
  之後（硬體 T&L）：
    CPU 只負責「傳頂點資料 + 設定矩陣參數」
    顯卡內建固定電路做矩陣乘法
    → 你的 math_utils.h 裡的 mat4 * vec4 變成了晶片上的電路

  API：OpenGL 1.2、Direct3D 7
  設定方式：glMatrixMode(GL_MODELVIEW); glLoadMatrixf(...)
  → 你只能傳入矩陣數值，不能改變矩陣怎麼乘（固定管線）

2000：ATI Radeon DDR
  與 GeForce 256 級別相當
  開始了 NVIDIA vs ATI 的競爭時代
```

**這個時代對應你的軟體**：你在 `math_utils.h` 裡手算的那些 MVP 矩陣，GeForce 256 把它做成了電路。

---

### 第三代：可程式化 Vertex Shader 誕生（2001）

**NV20：GeForce 3，你的目標世代**

```
2001：NVIDIA GeForce 3
  革命性改變：Vertex Shader 可程式化
  
  之前（固定 T&L）：
    只能做 MVP 矩陣 + Phong 光照，邏輯寫死在電路裡
  
  之後（可程式 VS）：
    你可以傳一段「程式碼」給顯卡
    顯卡執行這段程式決定每個頂點的位置
    
  技術細節：
    VS 1.1：最多 128 條指令
    指令集：MAD、DP4、MOV、MUL、ADD、RCP...
    暫存器：12 個 temp、16 個 input、96 個 constant
    → 這就是你要為 FPGA 設計的 ISA
  
  Fragment 端：
    還是固定管線（8 個紋理混合 stage，register combiner）
    不能任意寫 Fragment Shader
  
  API：DirectX 8、OpenGL ARB_vertex_program 擴充
  
  對應你的軟體：
    你的 IShader::vertex() = GeForce 3 的 VS Unit
    你的 uniforms = GeForce 3 的 Constant Registers (c0~c95)
    你的 varyings = GeForce 3 的 varying output (o0~o7)

同年 ATI Radeon 8500：
  可程式 VS + 可程式 FS（Pixel Shader 1.4）
  Fragment 端先一步可程式化，但精度只有 8-bit integer
```

### 第四代：Fragment 也可程式化（2002~2003）

**NV30：GeForce FX，完整可程式化管線**

```
2002：NVIDIA GeForce FX (NV30)
  Fragment Shader 也可程式化了（PS 2.0）
  技術細節：
    浮點精度（16-bit / 32-bit float）← 重大突破
    最多 1024 條 Fragment 指令
    可以在 Fragment Shader 內做紋理採樣
  
  你的軟體對應：
    IShader::fragment() 裡寫的 Phong 光照計算
    = NV30 的 Fragment Shader 在每個像素執行的 PS 程式碼

2002：ATI Radeon 9700 (R300)
  DirectX 9 第一張完整支援的顯卡
  PS 2.0 + VS 2.0
  比 NV30 效能好非常多（NV30 設計失敗，發熱大、慢）
  這場戰役 ATI 贏了
  
  API：DirectX 9、GLSL（OpenGL Shading Language 第一版）
  第一次可以用高階語言（GLSL/HLSL）寫 Shader，不用寫 Assembly

這個時代的完整管線：
  CPU                    GPU
  app logic         →   Vertex Shader（可程式，你設計的 ISA）
  send vertices     →   Primitive Assembly（固定）
  send shader code  →   Rasterizer（固定）
                    →   Fragment Shader（可程式）
                    →   ROP / Z-Test（固定）
                    →   Framebuffer → Display
```

---

### 第五代：統一著色器架構（2006）

**G80：GeForce 8800，現代 NVIDIA 的起點**

```
2006：NVIDIA GeForce 8800 GTX (G80)
  革命性改變：Unified Shader Architecture
  
  之前（分離架構）：
    VS Unit：N 個頂點處理器
    FS Unit：M 個像素處理器
    N 和 M 是固定比例，但遊戲場景有時頂點多、有時像素多
    資源無法動態分配，經常有一邊閒置
  
  之後（統一架構）：
    128 個「Stream Processor」（全部一樣）
    複雜場景：128 個全跑頂點
    簡單場景：128 個全跑像素
    GPU 動態分配，資源不浪費
  
  同時引入：CUDA（2007）
    Stream Processor 不只跑圖形 Shader
    可以跑任意並行計算程式
    → GPU 從「圖形專用」變成「通用並行計算器」
  
  架構概念：
    SM（Streaming Multiprocessor）：32 個 SP 組成一個 SM
    Warp：32 個 thread 一起執行（SIMT 正式確立）
    一個 SM 同時管理多個 Warp（隱藏記憶體延遲）

  → 這是現代所有 NVIDIA GPU 的基礎架構
```
