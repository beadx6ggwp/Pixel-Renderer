UI 渲染架構與自動化測試
從渲染管線的視角，理解 Poco / UIAutomator / DevTools 為何長得不一樣
rendering-pipeline unity-ugui android-view browser-dom automation poco appium

你已經會寫 SetPixel，懂 framebuffer，知道 vertex → rasterize → fragment 的管線流程。 現在讓我們把這套思維搬到 UI 的世界：當你面對 Unity、Android、瀏覽器三套 UI 系統時，它們各自的「渲染管線」長什麼樣？

更重要的是 — 為什麼 Poco 需要嵌 SDK、UIAutomator 走 Accessibility Tree、而瀏覽器用 DevTools Protocol？ 答案藏在渲染架構的差異裡。

本篇的核心問題
不同 UI 系統的渲染架構差異，如何決定自動化測試工具的設計策略？

我們會：(1) 拆解三套渲染管線的前後端 (2) 標示自動化工具的 hook point (3) 連回你的 IMGUI 架構和 hybrid testing 思路。

1 所有 UI 渲染都是同一條管線

不管是 Unity、Android 還是瀏覽器，UI 渲染都遵循同一條邏輯管線。這跟你在軟體渲染器裡做的事完全一樣 — 只是抽象層級不同：

你在 Pixel-Renderer 裡的對照：UI Tree = 你的頂點資料、Layout = MVP 變換、Paint = Rasterizer 產生繪圖指令、Rasterize = SetPixel 寫入 framebuffer、Composite = BitBlt 到前台。

關鍵差異在三個地方：Tree 的結構、誰做光柵化、以及Tree 能不能被外部存取。

2 三套管線的前後端拆解

2a. Unity UGUI

  • Frontend：Canvas 管理一棵 GameObject 樹。每個 UI 元素（Button、Text、Image）都是 GameObject + RectTransform。Canvas 用 dirty flag 追蹤哪些元素需要重新計算布局或頂點。
  • Backend：Canvas Rebuild 階段遍歷整棵子樹，收集所有子元素的頂點、UV、顏色，做 mesh batching（合併同材質的 draw call），最後送 GPU 光柵化。
  • 關鍵特徵：UI 元素就是 GameObject — 它活在 C# runtime 裡，任何程式碼都可以透過 Reflection 或 API 直接遍歷整棵樹、讀取任何屬性。沒有 process boundary。

2b. Android View System

  • Frontend：View hierarchy 是一棵由 ViewGroup / View 構成的樹。每幀經歷三階段遍歷：measure()（計算尺寸）、layout()（計算位置）、draw()（產生繪圖指令）。
  • Backend：draw() 產出的指令被記錄為 DisplayList，交給 Skia（Google 的 2D 光柵化引擎）做光柵化。Skia 就像你的 Rasterizer — 它把向量指令轉成像素。最終由 SurfaceFlinger（系統合成器）把所有 App 的 Surface 合成到螢幕上。
  • 關鍵特徵：View Tree 活在 App 的 process 裡。Android 的安全模型不允許外部 process 直接存取另一個 App 的 View hierarchy。系統另外維護一棵Accessibility Tree — 它是 View Tree 的語意子集（過濾掉裝飾性元素，只保留有意義的節點），透過 Accessibility Service API 暴露給外部。

2c. 瀏覽器

  • Frontend：DOM + CSSOM → Style calculation → Layout。DOM 就是 UI Tree，CSS 定義外觀，Layout 計算每個元素的幾何資訊（位置、尺寸）。
  • Backend：Paint 產生繪圖指令，Compositor Thread 將頁面分層、獨立光柵化各層，最終 GPU 合成。
  • 關鍵特徵：DOM 從網路標準誕生之初就是一個公開的、可程式化的、有標準化遠端協議（CDP）的樹結構。Selenium/Puppeteer/Playwright 直接操作 DOM — 不需要嵌 SDK，不需要經過 accessibility 間接層。

3 自動化工具的 Hook Point

現在把三條管線並排，標示每個工具在哪裡 hook in。這張圖解釋了工具能力的上限 — 你 hook 的位置越靠近 frontend，拿到的資訊越完整；越靠近 backend 或語意抽象層，資訊越受限。

三個「為什麼」

Poco 為什麼要嵌 SDK？
Unity 的 GameObject hierarchy 沒有標準化的遠端存取協議（不像 DOM 有 CDP）。Poco 必須注入一個 C# agent 到遊戲 process 裡，用 Reflection 遍歷 hierarchy，再透過 socket 把結果回傳。

UIAutomator 為什麼資訊比較少？
Android 的安全模型不允許跨 process 直接存取另一個 App 的 View Tree。你只能透過 Accessibility Service — 一個系統級的語意抽象層 — 拿到過濾後的 tree。這就像你的渲染器只暴露「有哪些可見的三角形」，但不暴露頂點資料和 shader 參數。

瀏覽器為什麼最方便？
DOM 從 1998 年 W3C 標準化開始就是「公開的、可程式化的」。CDP 讓外部工具可以透過 WebSocket 完整存取 DOM、styles、layout — 相當於遠端存取整個 rendering frontend。這是設計哲學的差異，不是技術限制。

4 渲染前後端差異的具體影響

4a. 元素定位：為什麼同一顆按鈕要用不同方式找？

Unity + Poco

直接用 hierarchy path 定位：
poco("Canvas/Panel/BtnStart")

等同於 GameObject.Find() 遍歷樹。
拿到的是 RectTransform 的世界座標 — 解析度無關。

Android + UIAutomator

用 accessibility 屬性定位：
UiSelector().resourceId("btn_start")

只能找到 View 暴露給 Accessibility 的屬性。
如果開發者忘了設 contentDescription，UIAutomator 看不到這個元素。

Browser + Selenium

用 CSS selector / XPath 定位：
driver.find_element(By.CSS, "#btn-start")

直接查 DOM tree — 任何元素都能找到，可以讀 computed style、bounding rect、執行 JS。

4b. 事件注入：點擊一個按鈕，底層發生什麼？

Poco

透過 Reflection 取得元素的世界座標 → 轉換為螢幕座標 → 注入 touch event 到 Unity 的 EventSystem。

等同於模擬一個真實的觸控事件經過 Raycast → EventSystem → Button.onClick 的完整路徑。

UIAutomator

透過 Accessibility Service 對 node 執行 performAction(ACTION_CLICK)。

這是語意級的操作 — 系統告訴 View 「你被點了」，但不經過觸控事件的完整路徑。某些自定義 View 可能不回應。

Selenium

透過 CDP 發送 Input.dispatchMouseEvent，模擬完整的 mousedown → mouseup → click 事件序列。

或直接執行 JS：element.click()。兩種路徑都走完整的 DOM event bubbling。

4c. 為什麼 Unity Inspector 有時「抓不到」UI 元素？

Inspector 抓不到 ≠ 元素不存在

Unity UGUI 的元素可能因為以下原因對外部工具不可見：
  • Canvas.renderMode = WorldSpace：UI 元素被當作 3D 物件渲染，它的座標在世界空間而非螢幕空間 — Poco 的螢幕座標轉換可能失準。
  • CanvasGroup.alpha = 0：元素完全透明但仍存在於 hierarchy — 有些自動化工具會過濾 alpha=0 的元素。
  • 動態生成 / Object Pool：元素在 test 開始時不存在，需要等待 instantiate。
  • 第三方 UI 框架（如 FairyGUI、NGUI）：它們用自己的渲染管線繞過了 UGUI 的 Canvas 系統 — Poco 的 UGUI dumper 遍歷的是 Canvas hierarchy，所以看不到這些元素。
這就是你提到的「persona、world 有內嵌 Unity 的部分，Inspector 抓不到元件」的根本原因 — 那些元件可能不在標準的 UGUI Canvas tree 裡。

5 連回你的渲染器：Retained Mode vs. IMGUI

這裡有一個你正在學的概念，恰好是所有差異的根源：Retained Mode vs. Immediate Mode。

Retained Mode（UGUI / DOM / Android View）

系統維護一棵持久化的 UI Tree。你建立一個 Button，它就一直存在於 tree 裡，直到你銷毀它。

自動化友好：工具可以隨時遍歷 tree、定位元素、等待元素出現。Tree 在幀與幀之間是穩定的。

Immediate Mode（microui / Dear ImGui / 你的未來 UI）

沒有持久化的 UI Tree。每幀你呼叫 if(button("Start")) {}，UI 在那一幀被「畫出來」然後消失。

自動化困難：沒有穩定的 tree 可以遍歷。自動化工具必須在同一幀內抓取狀態，或者靠 screenshot + vision 做影像辨識。

你的 Pixel-Renderer 的 UI 階段（Part 4）

你計畫用 microui 風格的 IMGUI。如果將來想對這個 UI 做自動化測試，hook point 在哪？

  • 方案 A：在 IMGUI 的 render loop 中插入一個 snapshot 機制 — 每幀把所有 UI 元素的 ID、位置、狀態記錄下來。這等同於在 immediate mode 上模擬出一棵 retained tree。
  • 方案 B：直接用 framebuffer output 做影像辨識 — 就像 AirTest 的 template matching。但你已經知道這條路的問題：解析度、縮放、UI 變動。
這就是為什麼商業遊戲引擎（Unity、Unreal）最終都選擇了 Retained Mode UI（UGUI、UMG） — 它在工具鏈支持（Inspector、自動化測試、accessibility）上有結構性優勢。

6 Hybrid Testing 的渲染視角

你之前提出的 hybrid case（帳號註冊 → Agent 處理新手引導 → 串回腳本），用渲染架構的語言重新描述：

// 混合測試的三段式架構

// 1. 固定腳本段：操作 retained tree 的確定性路徑
playwright.fill("#email", "test@example.com") // DOM 是 retained → XPath 穩定
playwright.click("#btn-register")

// 2. Agent 段：UI tree 結構不可預測 → 動態 tree traversal
agent.goal = "navigate to main screen" // Agent 讀取當前 tree/screenshot
agent.run() // 自行決定如何遍歷未知的 UI tree

// 3. 驗證段：比對 render output vs 預期狀態
assert(agent.current_screen == "main_lobby") // 語意驗證 或 snapshot 比對
script.resume() // 串回確定性腳本

從渲染架構的角度看，這三段分別對應：

  • 固定腳本 = 依賴 UI Tree 結構穩定的前提。XPath / CSS selector 能精確命中，因為 tree 不變。
  • Agent = 面對未知或變動的 UI Tree。Agent 必須在 runtime 讀取 tree（DOM dump / screenshot），用 LLM 做即時的 tree traversal 決策。
  • 驗證 = 比對 render output（最終的 framebuffer / DOM snapshot / screenshot）vs 預期狀態。可以是 pixel-level 也可以是 semantic-level。
Agent 的本質：動態 UI Tree Traversal

傳統腳本 = 寫死的 tree 遍歷路徑（XPath）。
Agent = 讓 LLM 在 runtime 根據當前 tree 狀態動態決定遍歷路徑。

差異不在「智能」，而在遍歷策略是編譯期還是運行期決定的。就像你的渲染器：固定管線 vs. 可程式化管線。Agent 就是 UI 測試的「可程式化管線」。

7 本篇總結

三件事：

1. UI 渲染管線都是 Tree → Layout → Paint → Display，差異在於 tree 的存取方式和 process boundary。Unity 的 GameObject tree 在 process 內可見（Poco hooks here）；Android 的 View tree 被 process boundary 隔離，只暴露 Accessibility Tree（UIAutomator hooks here）；瀏覽器的 DOM 透過 CDP 標準化暴露（Selenium hooks here）。

2. 自動化工具的能力上限由 hook point 決定：hook 在 rendering frontend（Poco / DevTools）→ 資訊完整；hook 在 semantic abstraction（UIAutomator）→ 資訊受限。Hook 越深入渲染系統，能力越強但耦合越高。

3. Retained mode 天然支持自動化（持久化 tree 可遍歷），IMGUI 需要額外機制。這解釋了為什麼商業引擎選擇 retained mode，也暗示了你在 Pixel-Renderer 做 UI 系統時要考慮的 trade-off。
